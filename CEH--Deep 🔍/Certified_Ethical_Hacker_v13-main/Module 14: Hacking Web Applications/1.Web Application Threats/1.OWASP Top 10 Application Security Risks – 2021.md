OWASP is an international organization that maintains a list of the top 10 vulnerabilities and flaws of web applications. The latest OWASP top 10 application security risks are as follows. 
1. **A01 – Broken Access Control** →
   This happens when users can access things they shouldn’t, like someone else’s account or admin features. Example: changing a URL to view another user’s data. <br>
   Attacks that fall under this category include:
   - Directory Traversal
   - Hidden Field Manipulation

3. **A02 – Cryptographic Failures** →
   This is when sensitive data (like passwords or credit card info) isn’t properly encrypted. If it's sent in plain text or uses weak encryption, attackers can steal it. <br>
   Attacks that fall under this category include:
   - Cookie Snooping
   - RC4 NOMORE Attack
   - Same-Site Attack
   - Pass-the-Cookie Attack

5. **A03 – Injection** →
   When an application accepts untrusted input and runs it as code. Attackers can use this to run malicious commands. Example: SQL Injection lets them steal or change data. <br>
    Attacks that fall under this category include:
   - SQL Injection
   - Command Injection
   - LDAP Injection
   - Cross-Site Scripting (XXS)
   - Buffer Overflow

6. **A04 – Insecure Design** →
   Flaws in how an app is planned or designed, like missing security checks. It leads to attacks like CAPTCHA bypass or session hijacking. <br>
   Attacks that fall under this category include:
   - Business Logic Bypass Attack
   - Web-based Timing Attacks
   - CAPTCHA Attacks
   - Platform Exploits

8. **A05 – Security Misconfiguration** →
   This happens when systems are not properly set up. Using default passwords, showing error messages with too much info, or not updating software can make apps easy to hack. <br>
   Attacks that fall under this category include:
   - XML External Entity (XXE) Attack
   - Unvalidated Redirects and Forwards
   - Directory Traversal
   - Hidden Field Manipulation

10. **A06 – Vulnerable and Outdated Components** →
   Using old software parts (like plugins, libraries, etc.) with known bugs lets attackers exploit them. These weak parts often run with full access and can lead to data theft. <br>
   Attacks that fall under this category include:
      - Platform Exploits
      - Magecart Attack
      - Buffer Overflow

12. **A07 – Identification and Authentication Failures** →
   Weak login systems can be bypassed. Attackers may guess passwords or hijack sessions if tokens and credentials aren't protected properly. <br>
   Attacks that fall under this category include:
      - Cross-Site Request Forgery
      - Cookie/Session Poisoning
      - Cookie Snooping

14. **A08 – Software and Data Integrity Failures** →
   If software updates or data are not verified properly, attackers can inject malware. For example, insecure auto-update features or untrusted plugins. <br>
   Attacks that fall under this category include:
      - Insecure Deserialization
      - Unvalidated Redirects and Forwards
      - Watering Hole Attack
      - Denial-of-Service (DoS)
      - Buffer Overflow
      - Web Service Attacks
      - Platform Exploits
      - Magecart Attack

16. **A09 – Security Logging and Monitoring Failures** →
   If the app doesn’t log activities or alert about unusual actions (like failed logins), attacks can go unnoticed. This gives hackers more time to do damage. <br>
   Attacks that fall under this category include:
      - Web Service Attacks

18. **A10 – Server-Side Request Forgery (SSRF)** →
    When an app fetches a URL without checking if it's safe, attackers can use it to access internal systems behind firewalls or steal sensitive info. <br>
    Attacks that fall under this category include:
       - Injecting an SSRF Payload
       - Cross-Site Port Attack (XSPA)
       - DNS Rebinding Attack
       - H2C Smuggling Attack

## A01 – Broken Access Control
Broken Access Control happens when a web app doesn’t properly restrict what users can do. Attackers can exploit this flaw to bypass restrictions and gain access to functions or data meant only for admins or other users. This lets them view, change, or delete information they shouldn’t be able to. It usually occurs due to poor testing and missing security checks in the code.

According to OWASP 2021 R3 revision, the common vulnerabilities associated with access control are as follows: 
1. **Too Much Access by Default** → Users get more permissions than needed, instead of getting only what they require (least privilege not enforced).

2. **Bypassing Access Controls** → Attackers change URLs, parameters, or use tools to reach restricted pages or functions they shouldn't access.

3. **Account Takeover via ID** → Hackers guess or manipulate user IDs to access or change someone else’s account details.

4. **Unprotected API Methods** → Sensitive API actions like PUT, POST, or DELETE are exposed without proper access control checks.

5. **Privilege Escalation** → A normal user can become an admin or gain higher access by exploiting flaws in the system.

6. **Tampering with Metadata** → Attackers change cookies, hidden fields, or JWTs to bypass access restrictions or raise their privileges.

7. **CORS Misconfiguration** → APIs are accessed from unauthorized websites due to poor Cross-Origin Resource Sharing settings.

8. **Force Browsing** → Attackers directly visit admin or user-only URLs without logging in or after logging in as a low-privilege user.

## A02 – Cryptographic Failures/Sensitive Data Exposure 
Cryptographic failures, also known as sensitive data exposure, happen when web applications do not properly protect important data like passwords, credit card numbers, or personal information. This can occur due to weak encryption, insecure storage, or poor handling of cryptographic keys. Even if data is encrypted, attackers can still steal it if outdated or flawed algorithms like MD5 or SHA-1 are used, or if keys and certificates are stored in unsafe locations. Reusing weak initialization vectors or using bad encryption modes can also make data easier to break. To prevent these attacks, developers must use strong, up-to-date encryption, securely store keys, and ensure error messages don’t leak useful information.

## A03 – Injection Flaws
Injection flaws happen when a web application takes untrusted input and runs it as part of a command or query. Attackers use this to send harmful commands—like in SQL, LDAP, or XPath—that the application mistakenly executes. This can lead to data theft, data loss, or even full control over the system. These flaws are common in old or poorly written code and can be found using scanners or fuzzers. When exploited, attackers can read, change, or delete sensitive data by tricking the application into running their malicious input.

There are many types of injection flaws, some of which are discussed below: 
1. **SQL Injection** → Attackers inject harmful SQL code into input fields (like login forms) to access or steal data from the website's database without permission.

2. **Command Injection** → Hackers find weak spots in input handling and insert system commands that get executed on the server, which can lead to full system compromise.

3. **LDAP Injection** → Attackers insert malicious input into LDAP queries to gain unauthorized access or modify directory information (like login permissions or user data).

4. **Cross-Site Scripting (XSS)** → Hackers inject malicious scripts into web pages that run in users’ browsers, allowing them to steal sessions, deface websites, or redirect users to harmful sites.

## A04 – Insecure Design 
Insecure design happens when an application is built without proper security measures in place. This often results from developers not fully understanding security risks or skipping risk analysis during planning. Such flaws can lead to serious issues like SQL injection or exposed cloud storage. Attackers take advantage of this by analyzing how the app works to find and exploit these weaknesses in its design or structure.

The following are some exploitations that can be performed by software and data integrity failures:
- Request forgery
- Authentication hijacking
- Identity theft
- Data loss
- DoS attacks

**Example** <br>
Attackers often attempt to exploit poorly implemented APIs, which fail to filter requests properly. They search for weak APIs that are not integrated with security gateways to differentiate malicious inputs. Then, they attach malicious code to a vulnerable API. When a user accesses that API via an application, the malicious code loads along with the database content on the user’s browser.

## A05 – Security Misconfiguration 
Security misconfiguration happens when any part of a web application—like the server, platform, or code—is not set up properly. If developers or admins make mistakes in the setup, attackers can take advantage of issues like unvalidated inputs, weak encryption, or bad error handling. This can lead to serious security risks and make the entire application vulnerable to attacks.

### Unvalidated Inputs 
Unvalidated input is a security flaw that happens when a web application does not properly check the data it receives from users before processing it. If input validation is only done on the client side (like in the browser), attackers can easily bypass it by changing the data sent to the server—such as URLs, form fields, or headers. This weakness can lead to serious attacks like cross-site scripting, injection attacks, and buffer overflows, which can steal data or damage the system. Proper server-side validation is essential to prevent these risks.

### Parameter/Form Tampering
Parameter tampering is a web attack where attackers change values sent between the user and the server, such as prices, user roles, or quantities, to exploit application logic. These values are often stored in hidden form fields, cookies, or URLs. If the application does not properly validate them, attackers can manipulate these parameters to gain unauthorized access or perform restricted actions. This can lead to serious issues like XSS or SQL injection. Tools like WebScarab and WebSploit are commonly used to perform such attacks.

#### Detailed Description: 
Once a session is active between a user and a web application, the browser and server exchange parameters to keep track of the session. This helps avoid storing all session data on the server. These parameters are usually passed through URL queries, form fields, or cookies to maintain session information and ensure smooth interaction.

Parameter tampering happens when an attacker changes values in form fields, like hidden fields, combo boxes, or checkboxes, before they are sent to the server. These values are often used by web applications to process things like prices or user choices. For example, if a hidden field shows `<input type="hidden" name="price" value="99.90">`, an attacker could change the price value before submitting the form. In some cases, attackers may save the page, edit the HTML code, and reload it in the browser to manipulate values and trick the application.

For example, consider a form that includes the following combo box: 
```html
<FORM METHOD=POST ACTION="xferMoney.asp"> 
Source Account: <SELECT NAME="SrcAcc"> 
<OPTION VALUE="123456789">******789</OPTION> 
<OPTION VALUE="868686868">******868</OPTION></SELECT>
<BR>Amount: <INPUT NAME="Amount" SIZE=20> 
<BR>Destination Account: <INPUT NAME="DestAcc" SIZE=40> 
<BR><INPUT TYPE=SUBMIT><INPUT TYPE=RESET> 
</FORM> 
```
#### Bypassing:
This attack involves modifying the values in an HTML form or URL to perform unauthorized actions. HTML forms submit their results using one of two methods: `GET` or `POST`. When a user submits a form using the `GET` method, the data is visible in the URL. An attacker can view and change these values to manipulate the request. For example, if the original URL is:

`http://www.certifiedhackerbank.com/cust.asp?profile=21&debit=2500`

the attacker can change it to:

`http://www.certifiedhackerbank.com/cust.asp?profile=82&debit=1500`

This tricks the server into debiting a different account with a different amount. The attacker may also edit the page source to add a new account option in a combo box, which the browser then displays, allowing unauthorized selection.

The attacker can modify other URL parameters, including attribute parameters and internal modules. Attribute parameters are unique parameters that characterize the behavior of the uploading page. For example, in a content-sharing site, a regular user may only be allowed to view content using a URL like:

`http://www.certifiedhackerbank.com/stat.asp?pg=531&status=view`

An attacker can change the `status=view` to `status=delete`:

`http://www.certifiedhackerbank.com/stat.asp?pg=147&status=delete`

If the server doesn’t properly verify user permissions, this can allow the attacker to delete content they don’t own. Such attacks can lead to stolen services, unauthorized access, session hijacking, or exposure of sensitive system functions.

### Improper Error Handling 
Improper error handling happens when a web application or system reveals too much information during an error. This can help attackers find weak points like logic flaws, default accounts, or even system details. If error messages are not properly controlled, they may expose sensitive information such as server banners or internal code behavior. Attackers use this leaked data to plan further attacks like denial-of-service or web-based exploits. Proper handling of errors is important to prevent giving away useful information to potential attackers.

<img width="390" height="297" alt="image" src="https://github.com/user-attachments/assets/6a357c1d-1359-4458-ad86-1aaae262fdae" />

The attacker can gather the following information from improper error handling:
- Null pointer exceptions
- System call failure
- Database unavailable
- Network timeout
- Database information
- Web application logical flow
- Application environment

### Insufficient Transport Layer Protection
Insufficient transport layer protection happens when a website doesn't properly secure data as it travels over the network. This often involves using weak encryption methods, expired or invalid SSL/TLS certificates, or not using encryption at all. Without strong SSL/TLS protection, attackers can intercept or alter the data, leading to risks like account theft, phishing, or man-in-the-middle attacks. Improper SSL setup allows untrusted users to access sensitive information, putting both users and the application at risk.

**Example** <br>
Assume that a user logs into an online banking application that possesses insufficient transport layer protection (i.e., it is not SSL encrypted). The sensitive data in the communication (e.g., session ID) can be vulnerable to attack during transit in plaintext format. This allows an attacker to steal such data to perform various types of attacks on the application.

### Improper Restriction of XML External Entity (XXE)
Improper restriction of XML External Entity (XXE) happens when XML processors are not securely configured and allow external entity references. Attackers can exploit this to access sensitive internal files, perform internal network scans, execute malicious code, or cause denial-of-service (DoS) attacks like the “billion laughs” attack, which crashes the system by overloading it with nested XML data.

Some server configuration problems are as follows:
- Missing security hardening
- Server software flaws
- Enabling unnecessary services
- Improper authentication
- Unpatched security flaws
- Server configuration problems
- Default accounts with default credentials
- Legacy software

Automated scanners help to detect a few of these problems. Attackers can exploit default accounts, unused pages, unpatched vulnerabilities, and unprotected files if they are left active on a server. To prevent this, all unnecessary features should be removed or disabled, and proper authentication must be enforced. For example, if an admin console is installed with default settings and not secured, an attacker can find it, log in with default credentials, and take control of the server. Securing application files and access points is essential to stop unauthorized access and information leaks.

## A06 – Vulnerable and Outdated Components/Using Components with Known Vulnerabilities
Using outdated or vulnerable components like libraries or frameworks in a web application can lead to serious security risks. These components often run with full privileges, so if there's a flaw, attackers can exploit it to gain control or steal data. Attackers scan applications or check exploit databases like **Exploit Database** [https://www.exploit-db.com], **CXSecurity** [https://cxsecurity.com], and **Zero Day Initiative** [https://www.zerodayinitiative.com] to find known vulnerabilities. Once they find a weak component, they customize an exploit and launch an attack. Tools like **Nessus** or **GFI LanGuard** are commonly used to detect these weaknesses.

<img width="740" height="225" alt="image" src="https://github.com/user-attachments/assets/af6e46fa-d00f-4e57-8e30-d69dd025c994" />


The following are some of the conditions that make applications vulnerable: 
1. **Unknown Component Versions** - 
   If you don't know which versions of tools and libraries (on both server and client sides) are being used—including hidden dependencies—it's hard to spot risks.

2. **Outdated or Unsupported Software** - 
   Using old, unpatched, or no-longer-supported software (like OS, servers, or runtimes) leaves your application open to known attacks.

3. **Skipping Vulnerability Scans** - 
   If regular scanning is not done or security updates are ignored, you miss critical fixes that could protect your app.

4. **Delayed Updates to Platforms or Frameworks** - 
   When the core platform, framework, or libraries aren’t updated on time, your app stays exposed to old vulnerabilities.

5. **Not Testing Compatibility of Updates** - 
   If you apply patches or updates without checking if they work properly with your app, it can break functionality or introduce new risks.

6. **Poor Security in Config Files** - 
   Weak or misconfigured settings in configuration files can expose sensitive info or open backdoors for attackers.

## A07 – Identification and Authentication Failures/Broken Authentication
Broken authentication happens when web applications fail to protect user login and session functions properly. Even with strong login systems, weak features like "forgot password" or "remember me" can be exploited by attackers. If these areas aren't secured, attackers can steal session IDs or bypass authentication to impersonate users. This can lead to unauthorized access, data theft, or account takeovers. Using strong methods like biometrics or secure tokens can help prevent such attacks.

### Session ID in URLs
When a web application includes session IDs in the URL, it becomes vulnerable to session hijacking. For example, a URL like `http://certifiedhackershop.com/sale/saleitems=304;jsessionid=12OMTOIDPXM0OQSABGCKLHCJUN2JV?dest=NewMexico` exposes the session ID directly. If an attacker captures this URL—by sniffing network traffic or tricking the user—they can simply paste it into their browser to access the victim’s session without needing login credentials. This allows the attacker to impersonate the user and perform unauthorized actions. Storing session IDs in URLs increases the risk of session fixation and should be avoided.

### Password Exploitation
Password exploitation happens when attackers take advantage of weak or unencrypted passwords stored in a web application's database. If passwords are not properly protected using strong hashing methods, attackers can easily recover them and gain unauthorized access. Once inside, they can hijack user sessions, impersonate users, and carry out harmful actions within the system.

### Timeout Exploitation
Timeout exploitation happens when web applications keep user sessions active for too long without logging the user out. If the session ID doesn’t change after each login and the timeout is too long, attackers can hijack that session. For example, if a user forgets to log out on a public computer, an attacker can later reopen the browser and access the account with the same session ID, gaining the same permissions as the user.

## A08 – Software and Data Integrity Failures 
Software and data integrity failures happen when applications use outdated software or untrusted third-party components without proper checks. Many web apps depend on external libraries or plugins from public sources, which may be risky if not verified. If automatic updates or CI/CD pipelines install code without validation, attackers can inject malicious changes. This can lead to unexpected behavior or expose sensitive parts of the application. Issues like using code from untrusted sources, downloading files without verifying them, or handling unsafe data can make systems vulnerable.

The following are some exploits that can be performed using software and data integrity failures:
- Cache poisoning
- Code injection
- Command execution
- Denial of service
- Data theft

## A09 – Security Logging and Monitoring Failures/Insufficient Logging and Monitoring
Security logging and monitoring failures happen when a web application does not properly record or track important activities, like login attempts or suspicious actions. If logs are incomplete, missing, or contain sensitive data, it becomes hard to detect attacks. Attackers can take advantage of this by deleting or changing logs to hide their actions. Without proper logging and monitoring, attacks like brute-force password guessing can go unnoticed, putting sensitive information at risk.

<img width="716" height="206" alt="image" src="https://github.com/user-attachments/assets/751d8704-1b7e-4d58-8928-54eda4e4bd58" />

Given below are a few causes of security logging and monitoring failures:
- Logs that do not provide information about overall logins, failed login attempts, and important transactions.
- Warnings and error messages that provide vague and insufficient log information.
- Inappropriate monitoring of applications and API logs.
- Local storage of logs.
- Improper or no implementation of response escalation processes.
- Dynamic application security testing (DAST) tools such as OWASP ZAP that fail to generate alerts.
- Web applications that are not capable of detecting, escalating, or issuing alerts of suspicious activities in real time.

## A10 – Server-Side Request Forgery (SSRF) 
Server-Side Request Forgery (SSRF) is a vulnerability where attackers trick a web server into making unauthorized requests to internal systems. This happens when a web application allows users to provide a URL to fetch external content but doesn’t properly validate it. For example, if a site uses a URL like `https://xyz.com/feed.php?url=externalsite.com/feed/to` to load external data, an attacker can change the URL to point to internal resources, like `localhost`, allowing them to access sensitive internal systems behind firewalls or VPNs. Once exploited, SSRF can lead to data theft, scanning internal networks, or even remote code execution.

### Types of SSRF Attacks 

#### Injecting an SSRF payload 
This attack involves selecting a parameter and inserting an SSRF payload that can support a file or URL. It allows attackers to make certain modifications to the header field and change it to plaintext. The new payload is then inserted into the parameter in place of a file.

- **Gaining access to internal resources** <br>
   Attackers can gain access to internal resources through the `/admin` panel from the internal network. This allows them to access the files from the server via the `file://path/to/file` schema.

- **Gaining access to internal pages** <br>
   Attackers can use the following exploits to access internal pages:
   ```
   - https://www.certifiedhacker.com/page?url=http://127.0.0.1/a dmin
   - https://www.certifiedhacker.com/page?url=http://127.0.0.1/p gadmin
   - https://www.certifiedhacker.com/page?url=http://127.0.0.1/a ny_interesting_page
   - https://www.certifiedhacker.com/page?url=http://127.0.0.1/p hpmyadmin
   ```
- **Using a URL scheme to access internal files** <br>
   Attackers can access files by exploiting a URL scheme of a server. This helps them further attack its internal services. The following exploits can be used by attackers to access internal files:
   ```
   - https://www.certifiedhacker.com/page?url=file://etc/passwd
   - https://www.certifiedhacker.com/page?url=file://\/\/etc/pas swd
   - https://www.certifiedhacker.com/page?url=file:///etc/passwd
   - https://www.certifiedhacker.com/page?url=file://path/to/fil e
   ```
- **Using a URL scheme to access internal services** <br>
   Attackers can connect to different internal services by using a URL scheme. Some of the exploits that can be used for this purpose are as follows.
   ```
   For FTP
   - https://www.certifiedhacker.com/page?url=ftp://attacker.net :11211/
   - https://www.certifiedhacker.com/page?url=sftp://attacker.ne t:11111/
   - https://www.certifiedhacker.com/page?url=tftp://attacker.ne t:123456/TESTUDP

   Exploiting LDAP
   - https://www.certifiedhacker.com/page?url=ldap://127.0.0.1/% 0astats%0aquit
   - https://www.certifiedhacker.com/page?url=ldap://localhost:1 1211/%0astats%0aquit
   ```

#### Cross-Site Port Attack (XSPA)
This type of SSRF attack allows attackers to scan for the open ports of a server. Attackers use a loopback interface of that server such as `localhost` or `127.0.0.1`. Further, attackers can use scanned ports such as port 21, 22, and 25, along with the loopback interface. <br>
The following are some examples:
```
- https://www.certifiedhacker.com/page?url=http://localhost:22/
- https://www.certifiedhacker.com/page?url=http://127.0.0.1:25/
- https://www.certifiedhacker.com/page?url=http://127.0.0.1:3389 /
```
