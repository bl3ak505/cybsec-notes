# 📡 Attack Web Services
Web services are often integrated into web applications to provide specific functions, but if they are vulnerable, they can expose the entire application to attacks. Since these services interact closely with the application’s business logic, exploiting them can reveal or damage core functionality. Attackers may use different techniques depending on their goal, such as launching a DoS attack to overload the service and disrupt access for legitimate users.

## Web Services Probing Attacks
WSDL files are automated documents consisting of sensitive information about service ports, connections formed between two electronic machines, and so on. Attackers can use WSDL probing attacks to obtain information about the vulnerabilities in public and private web services, as well as to perform an SQL attack. 

A web service probing attack involves the following steps:
- In the first step, trap the WSDL document from web service traffic and analyze it to determine the purpose of the application, functional breakdown, entry points, and message types.
- Create a set of valid requests by selecting a set of operations and formulating the request messages according to the rules of the XML schema that can be submitted to the web service.
- Use these requests to include malicious contents in SOAP requests and analyze errors to gain a deeper understanding of potential security weaknesses.

## Web Service Attacks: SOAP Injection
SOAP injection is an attack on web services that use the SOAP protocol, where attackers insert malicious characters or code such as single quotes, double quotes, semicolons, and so on into user input fields. This can alter the SOAP message and bypass authentication, allowing unauthorized access to backend databases. It works in a similar way to SQL injection.

## Web Service Attacks: SOAPAction Spoofing
SOAPAction spoofing is an attack where the SOAPAction HTTP header in a web service request is manipulated to execute unauthorized operations. Since the header tells the service what action to perform without fully parsing the XML body, attackers can change it to trigger different or malicious operations than intended.

For example, consider a web service that includes two operations, createUser and deleteAllUsers, and is vulnerable to such an attack. Assume that this web service is protected by a gateway and only authorized users who have direct communication with the web service can perform the deleteAllUsers operation. An attacker in front of the gateway can perform a SOAPAction spoofing attack by manipulating the SOAPAction header as follows: 

An HTTP request message for creating a user: 
```http
POST /service HTTP/1.1
Host: certifiedHacker
SOAPAction: "createUser"

<Envelope>
    <Header />
    <Body>
        <createUser>
            <login>rinnimathews</login>
            <pwd>password</pwd>
        </createUser>
    </Body>
</Envelope> 
```
The attacker can modify the SOAPAction to “deleteAllUsers”, and the gateway passes this message because the SOAP body consists of the createUser operation. 
```http
POST /service HTTP/1.1
Host: certifiedHacker
SOAPAction: "deleteAllUsers"

<Envelope>
    <Header />
    <Body>
        <createUser>
            <login>rinnimathews</login>
            <pwd>password</pwd>
        </createUser>
    </Body>
</Envelope>
```
Attackers use tools such as WS-Attacker to perform SOAPAction spoofing: 

- **WS-Attacker** [https://github.com/RUB-NDS/WS-Attacker] <br>
  WS-Attacker is an open-source tool for automatically testing the security of web services. It supports multiple attack plugins and can load WSDL files to send SOAP messages to service endpoints. It helps identify vulnerabilities such as XML signature wrapping, SOAPAction spoofing, and denial-of-service attacks.

## Web Service Attacks: WS-Address Spoofing 
WS-Address spoofing is a web service attack that exploits insecure WS-Address headers in SOAP messages. Since the **ReplyTo** and **FaultTo** fields can be tampered with, an attacker can replace them with their own endpoint address. This diverts responses or error messages to the attacker’s chosen server, causing data redirection or generating excessive traffic that can lead to a denial-of-service attack. Tools like WS-Attacker are often used to detect and exploit this vulnerability.

## Web Service Attacks: XML Injection
XML injection is an attack where malicious XML data or tags are inserted into user input to manipulate how a web application processes XML. If the application uses XML and XPATH for storing or verifying credentials, attackers can exploit this to bypass authentication, gain unauthorized access, modify data, or cause denial-of-service attacks.

## Web Services Parsing Attacks 
Web services parsing attacks target weaknesses in how an XML parser processes data. By modifying a request and injecting malicious commands, attackers can cause errors, disrupt service, or run system commands like `.bat` or `.cmd` files. This can lead to denial of service or manipulation of how the web service handles requests.

- **Recursive Payloads** <br>
  The attacker queries for web services with a grammatically correct SOAP document that contains infinite processing loops, resulting in exhaustion of the XML parser and CPU resources.

- Oversize Payloads <br>
  Attackers send a payload that is excessively large to consume all the system resources, rendering web services inaccessible to other legitimate users.
