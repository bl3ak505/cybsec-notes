# 💉 Perform Injection/Input Validation Attacks
Injection attacks target weak input validation in web applications, allowing attackers to send specially crafted data that tricks the system into running unintended commands or queries. This can lead to unauthorized access, data theft, modification, or deletion, and in severe cases, full control over the application and its server. Common types include SQL injection, script injection, OS command injection, SMTP injection, LDAP injection, and XPath injection.

---
## Injection Attacks/Input Validation Attacks 
To perform injection attacks, supply crafted malicious input that is syntactically correct according to the interpreted language being used to break the application’s normal intended operation. <br> 
Some ways to perform injection attacks are described below: 
1. **Web Scripts Injection** → Insert malicious code into web scripts to make the server run unintended commands.
2. **OS Commands Injection** → Add harmful commands into input fields that execute directly on the operating system.
3. **SMTP Injection** → Insert fake SMTP commands to send spam or misuse email servers.
4. **SQL Injection** → Enter malicious SQL queries to steal or change database information.
5. **LDAP Injection** → Pass harmful data into LDAP filters to gain unauthorized database access.
6. **XPath Injection** → Alter XPath queries with malicious input to bypass logic or access hidden data.
7. **Buffer Overflow** → Overload input fields with excessive data to crash or control a system.
8. **File Injection** → Upload or include harmful files through insecure file handling in applications.
9. **Canonicalization** → Use path tricks like `../` to access restricted files or folders.

## Perform Local File Inclusion (LFI)
Local File Inclusion (LFI) is a vulnerability that lets attackers use a web browser to access or upload files on a server. It happens when an application includes files based on user input without proper validation, allowing attackers to use path traversal to view sensitive files or replace existing ones.

LFI vulnerability is often triggered in PHP-based websites. Simple PHP code susceptible to LFI is given below. Attackers can insert the URL parameter into `require ()` without proper validation. 
```
$file = $_GET[‘page’];
require($file);
```
In this case, an attacker can just insert this string and fetch the **/etc/passwd** file using the following URL: `http://xyz.com/page=../../../../../../etc/passwd`

### Evade added .php and other extensions of the file 
In general, file extensions are added using PHP code as follows: 
```
$file = $_GET[‘page’];
require($file.”.php”);
```
Now, php is appended to the file name, which means the user cannot find the required file because file `/etc/passwd.php` does not exist. If an attacker tries to insert null bytes (`%00`) at the end of the attack string, the .php can be easily evaded: `http://xyz.com/page=../../../../../../etc/passwd%00` <br>

Another method to evade the added php is to add a question mark (`?`) to the attack string: `http://xyz.com/page=../../../../../../etc/passwd?` 

### Bypassing .php execution
LFI vulnerability can read .txt files but not .php files because they are executed by the server and their file-ending comprises some code. This can be evaded using a built-in php filter as follows: `http://xyz.com/index.php?page=php://filter/convert.base64-encode/resource=index` <br>

Here, the php filter is used to convert everything into the Base64 format. Now, the entire page is Base64-encoded, which can be decoded and saved in a text file and executed: `base64 -d savefile.php`
