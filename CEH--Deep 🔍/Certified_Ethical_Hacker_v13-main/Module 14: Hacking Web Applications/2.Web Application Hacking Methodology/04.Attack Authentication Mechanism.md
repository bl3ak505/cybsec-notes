# 🔐 Attack Authentication Mechanism
Attackers target authentication mechanisms in web applications to exploit weaknesses like weak passwords, insecure credential transmission, or poor password strength checks. By using methods such as brute-force attacks, dictionary attacks, credential theft, or cookie replay, they can bypass login systems and gain unauthorized access. These vulnerabilities often stem from design flaws in how authentication is implemented.

- **Username Enumeration** <br>
  Attackers can enumerate usernames in two ways: verbose failure messages and predictable usernames.
  - **Verbose Failure Message** <br>
      In a typical login system, the user enters two fields, namely username and password. In some cases, an application will ask for additional information. If the user is trying to log in and fails, it implies that at least one field was incorrect. This provides grounds for an attacker to exploit the application. <br>
    **Examples:**
      - Account <username> not found
      - Incorrect password provided
      - Account <username> has been locked out
  - P**redictable Usernames** <br>
    Some applications automatically generate account usernames according to some predictable sequence. This makes it very easy for the attacker to discern the sequence for a potentially exhaustive list of all valid usernames.

- **Password Attacks** <br>
A password attack is a process of trying various password cracking techniques to discover a user account password by which the attacker can gain access to an application. Methods for cracking passwords include the following:
  - Password functionality exploits
  - Password guessing
  - Brute-force attack
  - Dictionary attack
  - Attack password reset mechanism

- **Session Attacks** <br>
  The following types of session attacks are employed by attackers against authentication mechanisms:
  - **Session Prediction** → The attacker tries to guess or predict a valid session ID by studying how session IDs are generated. If successful, they can access the application as a legitimate user without logging in.
  - **Session Brute-Forcing** → The attacker repeatedly tries different session ID values until they find a valid one. Once found, they can hijack a user’s session and gain unauthorized access.
  - **Session Poisoning** → The attacker modifies session data or injects malicious code into it. This can change the user’s experience or help the attacker steal sensitive information.

- **Cookie Exploitation** <br>
  Cookie exploitation attacks are of the following types:
  - **Cookie Poisoning** → The attacker changes the data stored inside a cookie to gain unauthorized access or steal user identity. It’s like editing the cookie to pretend to be someone else.
  - **Cookie Sniffing** → The attacker captures (sniffs) cookies from an unencrypted network. If the cookie contains a session ID, the attacker can use it to log in as the victim without needing a password.
  - **Cookie Replay** → The attacker reuses (replays) a valid session cookie to impersonate a user. It works only while the original user is still logged in; once they log out, the session becomes invalid.

- **Bypass Authentication** <br>
  - **Bypass SAML-based SSO** → Attackers exploit flaws like incorrect signature validation, expired sessions, replayed tokens, or misdirected SAML messages to skip the Single Sign-On (SSO) process and log in without valid credentials.
  - **Bypass Rate Limit** → Rate limiting is meant to block repeated login attempts. Attackers avoid this by using different IPs, modifying headers, or spreading attempts over time to brute-force passwords without getting blocked.
  - **Bypass Multi-Factor Authentication (MFA)** → Attackers find ways to bypass the second verification step (like OTP or authenticator apps) using phishing, session hijacking, or exploiting logic flaws in the MFA setup.

## Design Flaws in Authentication Mechanism
Authentication mechanisms are more vulnerable to attacks than other implementations involved in web application security. Applications usually validate a user via his/her login credentials; even a minor weakness in this authentication process can lead to serious consequences such as granting access to illegitimate users. 

- **Bad Passwords:** <br>
  Bad passwords are weak or easily guessable passwords, such as blank entries, very short values, common words, names, default settings, or using the username as the password. Such passwords make it easy for attackers to gain unauthorized access to an application.

- **Brute-Forcible Login:** <br>
  Brute-forcible login is an attack where an attacker repeatedly tries different username and password combinations to gain unauthorized access to an application. If there are no limits on failed login attempts, the attacker can keep guessing until the correct credentials are found, making the application vulnerable even to basic hacking attempts.

- **Verbose Failure Messages:** <br>
  Verbose failure messages occur when a login form reveals exactly which field—such as username or password—is incorrect during a failed login attempt. This detailed feedback can help attackers guess valid usernames or other required information, which can later be used for targeted attacks or social engineering.

- **Insecure Transmission of Credentials:** <br>
  Insecure transmission of credentials happens when sensitive information is sent over an unprotected HTTP connection or handled carelessly, even with HTTPS. This allows attackers to intercept data through MITM attacks. Risks include sending credentials in URL query strings or storing them in cookies, which can be easily stolen.

- **Password Reset Mechanism:** <br>
  Password reset mechanisms help users change passwords regularly or when credentials are compromised, but they can also be targeted by attackers. Weaknesses such as revealing if a username is valid, allowing unlimited guessing of the existing password, or verifying new password matches only after confirming the old one can be exploited to discover or compromise user credentials.

- **Forgotten Password Mechanism:** <br>
  The forgotten password mechanism can introduce security risks if not designed properly. Weaknesses like revealing valid usernames, using insecure secondary challenges, or allowing unlimited recovery attempts can be exploited through guessing or brute force. These flaws can compromise the application’s authentication system and give attackers unauthorized access.

- **“Remember Me” Functionality:** <br>
  The “Remember Me” feature lets users stay logged in without re-entering credentials by using persistent cookies. If these cookies are stolen or guessed, attackers can bypass login authentication and gain full access to the application, making this feature a potential security risk.

- **User Impersonation:** <br>
  User impersonation is when privileged users log in as other users to assist them, such as customer support accessing a user’s account to resolve an issue. If the impersonation feature has flaws, attackers can exploit it for vertical privilege escalation, gaining full control over the application.

- **Improper Validation of Credentials:** <br>
  Improper validation of credentials occurs when an application’s authentication system is poorly designed, ignoring strong password practices. Examples include only checking the first few characters of a password, ignoring case sensitivity, or stripping special characters before verification. Such flaws make it easier for attackers to run automated password guessing and compromise accounts with fewer attempts.

- **Predictable Usernames and Passwords:** <br>
  Predictable usernames and passwords occur when applications generate them in a fixed or guessable pattern. Attackers can use this to quickly obtain valid usernames and guess their initial passwords, especially if they are distributed through common sources. This weakness is often seen in intranet environments and can lead to further attacks.

- **Insecure Distribution of Credentials:** <br>
  Insecure distribution of credentials occurs when login details or activation links are sent through channels like SMS, email, or post without proper security. If attackers create multiple accounts, they can intercept or observe these messages to guess activation codes and gain access to other users’ accounts.

## Implementation Flaws in Authentication Mechanism
Implementation flaws in authentication happen when security features are designed well but mistakes occur during their setup or enforcement. These errors can allow attackers to bypass login, leak sensitive information, or weaken the entire security system. Such flaws are especially dangerous because they are often missed in regular testing. Some of the implementation flaws in authentication mechanisms are as follows: 

### Fail-Open Login Mechanism
A fail-open login mechanism is a flaw where errors in the authentication process, such as missing credentials causing exceptions, still allow a user to log in. This can let attackers bypass authentication and gain access to sensitive information or functions without valid login details.

**Example:**
```java
public Response verifyLogin(Session mySession) {
    try {
        String username = mySession.getParameter("username");
        String password = mySession.getParameter("password");
        User thisUser = db.getUser(username, password);

        if (thisUser == null) {
            // invalid credentials
            mySession.setMessage("Login Failed.");
            return doLogin(mySession);
        }
    } catch (Exception e) {
        // Handle exception silently
    }

    // valid user
    mySession.setMessage("Login successful!");
    return doMainMenu(mySession);
}
```
This code contains a logical flaw: if an exception occurs (e.g., DB error), the user is automatically logged in because the `catch` block skips the credential check. This could be exploited.

### Flaws in Multistage Login Functionality
Flaws in multistage login functionality occur when weaknesses exist in the logic of its multiple verification steps, such as username/password entry, partial character challenges, or token-based checks. Attackers can exploit these logic defects to bypass certain stages and gain unauthorized access.

### Insecure Storage of Credentials
Insecure storage of credentials happens when applications keep usernames and passwords in plain text or use weak encryption. This poor protection allows attackers to crack passwords through brute-force or other attacks if they gain access to the stored data.

## Password Attacks: Password Functionality Exploits 
1. **Password Changing** → Test the password change feature by entering random values in the "Old Password", "New Password", and "Confirm New Password" fields. Analyze the error messages to find weaknesses, like missing validation or lack of old password verification.

2. **Password Recovery** → If the "Forgot Password" option allows unlimited attempts, attackers can guess answers using social engineering. If successful, they may receive a password reset link or even the actual password via email.

3. **‘Remember Me’ Exploit** → This feature often stores usernames or session tokens in cookies. If these values are predictable (e.g., `RememberUser=jason`), attackers can guess them to gain unauthorized access without logging in.

## Password Attacks: Password Brute-forcing 
Password brute-forcing is a method attackers use to crack passwords by trying every possible combination of letters, numbers, and symbols until the correct one is found. Tools like Burp Suite and dictionaries of common passwords can speed up the process, but strong, complex passwords can make brute-forcing take years, making it less practical.

1. **Password List** → This list is created using personal information like birth dates, street names, nicknames, phone numbers, or pet names. Attackers gather this data through social engineering or footprinting, then try each possible password until they find the correct one.

2. **Password Dictionary** → A password dictionary is a collection of common or likely word-number combinations used as passwords. It's faster than brute force. Tools like Wordlist Generator (available on GitHub) help create these lists for dictionary-based password attacks.

## Password Attacks: Attack Password Reset Mechanism 
A password reset mechanism can be exploited if it does not properly validate and sanitize inputs, particularly the **Host** header in HTTP requests. This attack, known as **password reset poisoning**, takes advantage of the fact that many applications use the Host header value directly when generating password reset links.

In a normal process, when a user forgets their password, they request a reset link. The server creates a one-time token, embeds it in a URL, and sends it to the user via email. For example, a legitimate reset link might be:
```perl
https://certifiedhacker.com/reset-pwdd.php?token=87654321-8765-8765-8765-10987654321
```

However, if the application trusts the `$_SERVER['HTTP_HOST']` value without verification, an attacker can manipulate it. For instance, they can send a crafted HTTP request like:
```nginx
GET https://certifiedhacker.com/reset.php?email=foo@bar.com HTTP/1.1  
Host: badhost.com
```

If the server uses the `Host` header to build the reset link, it will generate:
```perl
https://badhost.com/reset-pwdd.php?token=87654321-8765-8765-8765-10987654321
```
This poisoned link will then be emailed to the victim. When the victim clicks it, they are directed to the attacker-controlled domain, where the attacker can capture the token and reset the victim’s password.

The vulnerability exists because developers assume the Host header will always come from the legitimate domain, without performing additional validation. Proper input checks, strict host validation, and avoiding reliance on client-supplied headers are essential to prevent such attacks.

The password reset poisoning attack involves the following steps:
- **Step 1:** The attacker obtains the target’s email address used on the website through techniques such as social engineering and OSINT.
- **Step 2:** The attacker sends a password reset request link to the victim using the altered Host header. For example,
  ```nginx
  POST https://certifiedhacker.com/reset.php HTTP/1.1 
  Accept: */* 
  Content-Type: application/json 
  Host: badhost.com 
  ```
  The resultant URL for resetting the password is
  ```nginx
  https://badhost.com/reset-password.php?token=87654321-8765-8765-8765-10987654321
  ```
- **Step 3:** Now, the attacker waits for the victim to receive the modified email. ▪ Step 4: Once the victim clicks on the malicious link embedded in the email, the attacker extracts the password reset token. Using this token, the attacker performs various malicious activities such as cloning web applications to steal the user’s credentials or acting as a proxy and mimicking the behavior and contents of the original website.

## Session Attacks: Session ID Prediction/Brute-forcing
Session ID prediction or brute-forcing is an attack where an attacker tries to guess a valid user’s session ID to hijack their session. For example, if a website uses session IDs with a fixed pattern or poor randomness, attackers can collect some valid IDs, predict the next one, and gain access without logging in. Strong randomness in session IDs is essential to prevent such attacks.

Session attacks are performed in the following steps: 
1. **Sniff Session IDs** → First, the attacker captures valid session ID values by sniffing network traffic from logged-in users.
2. **Analyze Session Structure** → The attacker studies the session ID to understand how it's created — whether it uses predictable data like username, IP address, or timestamps.
3. **Guessable Sessions** → If the session ID is generated using predictable patterns, the attacker can guess valid session IDs easily.
4. **Brute-force Attack** → Attackers can also use brute-force methods to generate and test multiple session IDs until they find one that gives access to a valid session.

## Cookie Exploitation: Cookie Poisoning
Cookie poisoning is an attack where an attacker changes the data stored in a website’s cookies to gain unauthorized access or manipulate a user’s session. Since cookies can hold sensitive details like usernames, passwords, account numbers, or session IDs, altering them can let an attacker impersonate another user or escalate privileges. For example, if a “Remember me” cookie stores login information, an attacker could edit it to log in as someone else. Even if cookies are encoded, methods like Base64 or ROT13 can easily decode them, making security measures ineffective if not properly encrypted.

Cookie poisoning is performed in the following steps:
- If the cookie contains passwords or session identifiers, steal the cookie using techniques such as script injection and eavesdropping
- Then, replay the cookie with the same or altered passwords or session identifiers to bypass web application authentication
- Trap cookies using tools such as OWASP Zed Attack Proxy, and Burp Suite.

## Bypass Authentication: Bypass SAML-based SSO
Single sign-on (SSO) allows users to log in once with a single set of credentials and access multiple applications without signing in again. For example, logging in with a Google account gives access to Gmail, Google Drive, and YouTube in the same session. This is achieved using the SAML protocol, where the identity provider (IdP) verifies the user and sends an authentication confirmation, called an SAML assertion, to the service provider (SP) to grant access.

Bypassing SAML-based SSO happens when attackers exploit weaknesses in the way SAML messages are handled between a service provider (SP) and an identity provider (IdP). These messages, encoded in Base64, contain a **signature** (to verify trust) and an **assertion** (to grant access). If signatures are misconfigured, session timeouts are weak, or messages are misdirected, attackers can intercept and modify them. For example, using tools like **SAML Raider**, an attacker could alter an assertion to make the SP treat them as a valid user, or replay an old session to gain unauthorized access.

4. **Capture SAML Data** → After some intercepted requests, a new “SAML Raider” tab appears once SAML data is detected.

5. **Forward Requests** → Keep clicking “Forward” in Burp to reach the IdP login page and enter credentials (e.g., [admin@xyz.org.fakedomain.com](mailto:admin@xyz.org.fakedomain.com)).

6. **Intercept SAML Response** → Once credentials are submitted, Burp captures more requests. Wait until the “SAML Raider” tab shows a response.

## Bypass Authentication: Bypass Rate Limit 
Bypassing rate limits is a technique attackers use to evade the restrictions placed on login attempts. Rate limiting helps stop brute-force attacks by limiting how many times someone can try to log in within a set time. Attackers bypass this by methods such as changing IP addresses, using multiple accounts, or spreading attempts across different sessions. For example, an attacker could use a botnet to try passwords from many different IPs, avoiding detection while continuing the attack.

### Explore Different Endpoints and Parameters 
Exploring different endpoints and parameters is a tactic where attackers try multiple variations of a web API or URL to find one that works. For example, if the target endpoint is `/api/v3/sign-up`, they may also try `/Sign-up`, `/SignUp`, `/signup`, `/api/v1/sign-up`, or `/api/sign-up`. This increases the chances of bypassing restrictions and successfully launching brute-force or other attacks.

### Including Blank Characters in Code or Parameters
Including blank characters in code or parameters is a technique where attackers insert special byte values like `%00`, `%0a`, `%0d`, `%09`, `%0C`, or `%20` to bypass authentication checks. For example, changing a parameter to `code=5678%0a` can add a newline character, potentially bypassing input restrictions. Similarly, adding a null byte (`%00`) at the end of an email address can sometimes evade rate limits or validation rules.

### Altering the IP Origin using Headers
Altering the IP origin using headers is a technique where attackers modify HTTP headers, such as **X-Originating-IP**, **X-Forwarded-For**, **X-Remote-IP**, **X-Remote-Addr**, **X-Client-IP**, **X-Host**, **X-Forwarded-Host**, and even adding multiple occurrences of **X-Forwarded-For**, to fake their IP address. For example, an attacker can send multiple requests with different fake IPs in the **X-Forwarded-For** header to bypass IP-based rate limits or detection systems. This makes it seem like the traffic is coming from different users, allowing attacks like brute force or vulnerability exploitation to continue undetected.

```
X-Originating-IP: 127.0.0.1 
X-Forwarded-For: 127.0.0.1 
X-Remote-IP: 127.0.0.1 
X-Remote-Addr: 127.0.0.1 
X-Client-IP: 127.0.0.1 
X-Host: 127.0.0.1 
X-Forwarded-Host: 127.0.0.1 
```
Or <br>
Attempt to use the X-Forwarded-For: IP header twice instead of once. <br>
Sometimes, there is a limit of 20 requests per account, and attackers can overcome this by switching to a different IP address after reaching 20 attempts.

## Bypass Authentication: Bypass Multi-Factor Authentication
Multi-factor authentication (MFA) bypass is when attackers find ways to avoid the extra verification steps meant to secure accounts. MFA requires multiple proofs of identity, such as a password and a one-time code. Attackers may bypass it using methods like stealing session cookies to skip verification or tricking users through phishing to capture both their password and code. This allows them to access systems or data without completing the intended security checks.

**Techniques to bypass multi-factor authentication**

### HTTP Response Body Manipulation 
Check the response of the MFA request. If the 'Success: false' parameter is noticed in the request, modify it to 'Success: true' and assess whether this adjustment aids in bypassing the 2FA implementation. 

Valid request with incorrect OTP: 
```
POST /otp-verify 
HOST: certifiedhacker.com 
<redacting_required_headers> 
{"otp":50563}
```
Valid response: 
```
200 OK
<redacted> 
__ {"success":false}__
```
Tampered response (with the same incorrect OTP): 
```
200 OK
<redacted>
{"success":true}"
```

### Status Code Manipulation 
If the response status code is 400, 401, 402, 403, etc. and showing “Invalid Token” message, then modify the response status code to '200 OK' with 'Success: true'. 

Valid request with incorrect OTP: 
```
POST /otp-verify 
HOST: certifiedhacker.com 
<redacting_required_headers> 
{"otp":50563}
```
Valid response: 
```
403 Forbidden
<redacted>
{"error":true, "message":"Invalid Token"}
```
Tampered response (with the same incorrect OTP): 
```
200 OK
<redacted>
{"success":true}
```
