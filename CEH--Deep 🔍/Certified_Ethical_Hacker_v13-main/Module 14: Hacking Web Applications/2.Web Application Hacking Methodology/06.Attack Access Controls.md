# 🚧 Attack Access Controls
Access controls are part of the application’s security mechanisms that are logically based on authentication and session management. An attacker walks through a website to identify the following access controls details of the application:
- Individual access to a particular subset of data
- Levels of grant access (employees, managers, supervisors, CEOs, etc.)
- Administrator functionality to configure and monitor
- Functionalities that allow escalating privileges

---
## Exploiting Insecure Access Controls 

- **Parameter-Based Access Control** <br>
  Parameter-based access control is when a web application decides user permissions based on request parameters like cookies or query strings. If attackers discover which parameters give admin rights, they can modify their own requests to include them and gain unauthorized administrative access.

- **Referer-Based Access Control** <br>
  In some web applications, the HTTP referer is the foundation for major access control decisions. The HTTP referer is considered unsafe; the attacker can use it and manipulate it to any value.

- **Location-Based Access Control** <br>
  Location-based access control restricts access to users based on their geographic location, often determined by IP address. Attackers can bypass this control using methods like proxies, VPNs, mobile data roaming, or altering client-side location data.

---
## Access Controls Attack Methods 
1. **Attack with Different User Accounts** → Try logging in with multiple accounts to see if one user can access another user’s data or functions due to weak access controls. Tools like Burp Suite help compare permissions between accounts.

2. **Attack Multistage Processes** → In multi-step workflows, test each request sent to the server for access control flaws. Use a proxy to replace session tokens with those from lower-privilege accounts to see if the process can be bypassed.

3. **Attack Static Resources** → Directly request protected files (like PDFs or images) via their URLs to check if unauthorized users can access them without logging in.

4. **Attack Direct Access to Methods** → Send requests directly to backend APIs and see if they respond without proper authorization checks.

5. **Attack Restrictions on HTTP Methods** → Test with different HTTP methods (GET, POST, PUT, DELETE, etc.) to see if changing the method bypasses restrictions and grants unauthorized access.
