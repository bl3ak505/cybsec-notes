# 🌐 Web Application Attack Countermeasures

## Broken Access Control 
1. **Check Permissions Before Giving Access** → Always verify user permissions **before** showing or sending them any sensitive resource.
2. **Use Secure, Hard-to-Guess IDs** → Avoid predictable IDs (like 123 or user1) that attackers can guess.
3. **Apply Least Privilege Rule** → Give users only the permissions they need — nothing extra.
4. **Remove Access Tokens on Logout** → Destroy session tokens on the server when a user logs out.
5. **Deny Access by Default** → Block everything unless it’s meant to be public.
6. **Enable Role-Based Access Control (RBAC)** → Assign permissions based on user roles for easier and safer management.
7. **Log and Monitor Unauthorized Attempts** → Keep detailed logs of failed access attempts and review them regularly.
8. **Restrict CORS Usage** → Allow cross-origin requests only from trusted domains.
9. **Apply Security Headers** → Use headers like **Content-Security-Policy (CSP)** and **X-Frame-Options** to block web attacks.
10. **Use Multi-Factor Authentication (MFA)** → Require an extra verification step for sensitive actions or data access.

## Cryptographic Failures/Sensitive Data Exposure
Many web applications do not properly protect sensitive data such as credit card numbers, SSNs, and authentication credentials with appropriate encryption or hashing. Attackers may steal or modify such weakly protected data to conduct identity theft, credit card fraud, or other crimes.

Some countermeasures against cryptographic failures/sensitive data exposure attacks are as follows:

1. **Use Strong Encryption Algorithms** → Always use modern encryption like **AES-256** for stored data and **TLS 1.3 with HSTS** for network traffic.
2. **Secure Key Management** → Store encryption keys in **Hardware Security Modules (HSM)** or **Key Management Services (KMS)**, never in code or config files.
3. **Rotate Keys Regularly** → Change keys periodically and have policies for **key generation, distribution, and destruction**.
4. **Encrypt Data in Transit & At Rest** → Protect data **while being sent** (TLS + Perfect Forward Secrecy) and **while stored** (database/file encryption).
5. **Hash Passwords Securely** → Use **bcrypt, scrypt, or Argon2** for password hashing. Avoid weak algorithms like MD5 or SHA-1.
6. **Classify & Limit Data** → Identify sensitive data, **store only what’s necessary**, and delete it once it’s no longer needed.
7. **Disable Caching of Sensitive Data** → Prevent browsers and servers from caching pages or requests containing personal/confidential data.
8. **Avoid Weak or Custom Crypto** → Never invent your own cryptographic methods; stick to well-tested libraries and keep them **up-to-date**.
9. **Use Authenticated Encryption** → Apply encryption modes like **AES-GCM** to ensure **both confidentiality and data integrity**.
10. **Protect APIs from Excessive Data Exposure** → Avoid sending unnecessary sensitive fields in API responses; use **data masking** when needed.

## Insecure Design

### Threat modelling:
- Implement a threat modeling system to detect potential threats before they are exploited. A threat modeling system scrutinizes the security and privacy requirements of an application during its development.
- Perform periodical assessments for every module and feature to be added in the application.
- Design tests for flow validation and verification to defend against the listed threats.
- Conduct threat modeling early in the design phase to identify potential threats and vulnerabilities.
- Use tools such as Microsoft Threat Modeling Tool or OWASP Threat Dragon.

### Security requirements definition:
- Define clear security requirements based on industry standards and best practices.
- Ensure these requirements are integrated into the design specifications.

### Secure design:
- Implement secure design that can help in maintaining proper security in the application through automated evaluation and testing for potential threats.
- Ensure that the application has been verified for errors. The estimations should be analyzed and recorded. Based on the recorded estimations, appropriate measures should be implemented.
- Follow security design principles such as:
  - **Least privilege:** Ensure users and processes operate with the minimum privileges necessary.
  - **Defense in depth:** Implement multiple layers of security controls.
  - **Fail securely:** Ensure the system remains secure even when errors occur.
  - **Secure by default:** Default settings should be secure out of the box.
  - **Minimize attack surface:** Reduce the number of entry points and potential attack vectors.

### Secure development life cycle:
- Implement a secure development life cycle, which helps in meeting the client requirements and developing applications according to security standards.
- Ensure that the development and security teams maintain healthy communication during the development of the application to implement security-and privacy-related controls for the application.
- Integrate security into every phase of the development lifecycle, from planning and design to testing and maintenance.
- Use frameworks such as Microsoft's Secure Development Lifecycle (SDL) or OWASP's Software Assurance Maturity Model (SAMM).

### Architectural risk analysis:
- Perform an architectural risk analysis to identify and address security risks associated with the design.
- Use techniques such as attack surface analysis and threat assessment.

Some additional countermeasures against insecure design issues are as follows: 
- Perform application reliability checks periodically at each stage of development.
- Employ limited privileged resource access depending on the application or service.
- Differentiate the list of user accessibility scenarios into two categories based on the use and misuse cases.
- Implement security controls layer-wise starting from the network to the system.
- Categorize the application users based on their authorization and access levels.
- Perform attack vector identification using all the security and access controls and business risk profiling.
- Utilize well-established security patterns and best practices for common design issues. Refer to resources such as the OWASP Security Knowledge Framework or the SEI CERT Coding Standards.

## Security Misconfiguration 
Security misconfiguration makes web applications potentially vulnerable and may provide attackers with access to them as well as to files and other application-controlling functions. Insufficient transport layer protection allows attackers to obtain unauthorized access to sensitive information as well as to perform attacks such as account theft, phishing, and compromising admin accounts. Encrypt all communications between the website and client to prevent attacks due to insufficient transport layer protection. 

Some countermeasures against security misconfiguration attacks are as follows: 

1. **Turn Off What You Don’t Need** – Disable unused services, ports, and features to reduce attack surfaces.
2. **Lock Down Accounts** – Remove or change default accounts/passwords and set proper roles and permissions.
3. **Stay Updated** – Regularly scan for vulnerabilities and apply the latest security patches.
4. **Force HTTPS Everywhere** – Redirect all HTTP traffic to HTTPS and ensure sensitive cookies have the `Secure` flag.
5. **Use Strong Encryption** – Configure SSL/TLS to allow only strong algorithms and ensure certificates are valid and match domains.
6. **Encrypt All Connections** – Use SSL/TLS or other encryption even for backend and internal communications.
7. **Separate Environments** – Use different credentials and configurations for development, testing, and production.
8. **Minimize Bloat** – Avoid installing unnecessary components, frameworks, or sample files.
9. **Restrict Management Access** – Hide admin panels from the public internet, protect them with MFA, strong passwords, and IP restrictions.
10. **Follow Hardening Standards** – Apply CIS Benchmarks, NIST guidelines, and use tools like Ansible, Puppet, or Chef to enforce secure configurations.

## XML External Entity

1. **Disable External Entities** → Configure your XML parser to **not process external entities** at all. This closes the main XXE attack path.
2. **Disable DTDs (Document Type Definitions)** → Turn off support for **any DTD declarations** inside XML documents unless absolutely needed.
3. **Use Secure XML Parsers/Libraries** → Pick parsers that have **XXE protection enabled by default** and keep them **up to date**.
4. **Enforce XSD Validation** → Validate XML against a **predefined XML Schema (XSD)** to block malicious or unexpected data.
5. **Implement Whitelisting & Input Validation** → Accept **only known safe XML structures** and filter out unwanted elements.
6. **Limit Document Size & Complexity** → Set **strict limits** for XML size, number of entities, and nesting depth to prevent DoS-style attacks.
7. **Use an XML Security Gateway** → Place an **XML firewall** to inspect and filter XML traffic before it reaches your app.
8. **Employ Security Tools** → Use **WAFs, API security gateways, and IAST tools** to detect and block XXE attempts in real time.
9. **Conduct Code Reviews & Security Audits** → Regularly review code and XML handling logic to **spot weaknesses early**.
10. **Log & Monitor Parsing Activities** → Keep detailed logs of XML parsing, and **watch for unusual patterns** that might signal attacks.

## Vulnerable and Outdated Components/Using Components with Known Vulnerabilities 

1. **Keep Everything Updated** → Always update both client-side and server-side components, including dependencies, to the latest stable versions.
2. **Monitor Vulnerability Databases** → Regularly check trusted sources like the **National Vulnerability Database (NVD)** to identify known risks in your components.
3. **Apply Security Patches Quickly** → Have a process to install patches as soon as they are released to fix known security holes.
4. **Use Dependency Management Tools** → Tools like **npm, pip, Maven, Composer** help track, update, and secure dependencies automatically.
5. **Scan Components Frequently** → Use security scanners or **Software Composition Analysis (SCA)** tools such as **OWASP Dependency-Check, Snyk, Black Duck, WhiteSource** to detect vulnerabilities early.
6. **Remove Unnecessary Components** → Delete unused libraries, dependencies, and files to reduce attack surface.
7. **Download from Trusted Sources Only** → Obtain components only from official and verified repositories; accept only signed packages.
8. **Limit Component Permissions** → Configure components to run with **minimum required privileges** to minimize damage if compromised.
9. **Isolate Third-Party Components** → Run untrusted or high-risk components inside **containers or sandboxes** (e.g., Docker) to limit their impact.
10. **Maintain an Inventory** → Keep a detailed, up-to-date list of all components with their **versions, sources, and purposes** for quick tracking and updates.

## Identification and Authentication Failures/Broken Authentication and Session Management 
Flaws in authentication and session management application functions allow attackers to gain passwords, keys, and session tokens or exploit other implementation vulnerabilities to gain other users’ credentials. <br> 
Session cookies are destined for client IPs by delivering a validation cookie, which includes a cryptographic token that verifies that the client IP is the one to which the session token was issued. Therefore, to perform the session attack, the attacker must steal the IP address of the target user. 

Some countermeasures against broken authentication and session management attacks are as follows: 

1. **Always Use HTTPS (SSL/TLS)** → Protect all login pages and authenticated areas with HTTPS to prevent credential theft during transmission.
2. **Strong Password Storage** → Store passwords only in a **hashed form** using secure algorithms like **bcrypt**, **scrypt**, or **Argon2**.
3. **Enforce Strong Passwords** → Use **secure password policies** (minimum length, complexity) or **passphrases** with at least 5 random words.
4. **Limit Login Attempts** → Set limits on failed login attempts and temporarily **lock the account** to stop brute-force attacks.
5. **Use Multi-Factor Authentication (MFA)** → Add an extra layer (e.g., OTP, authenticator app) to block attackers even if passwords are stolen.
6. **Secure Session Management** → Generate **long, random session IDs**, and **invalidate sessions on logout** to prevent session hijacking.
7. **Generic Error Messages** → Don’t reveal if the username or password is wrong — always say **"Invalid username or password"**.
8. **Implement Rate Limiting & CAPTCHA** → Restrict excessive login attempts from a single IP and use CAPTCHA to stop bots.
9. **Use Secure Password Reset Methods** → Send a **unique, time-limited reset link** to the registered email — never send plain-text passwords.
10. **Monitor & Alert** → Log **all authentication events** (login, logout, password changes) and set **alerts for suspicious activity**.

## Software and Data Integrity Failures

1. **Use Digital Signatures** → Sign software and data files so you can verify they haven’t been altered by attackers.
2. **Download from Trusted Sources** → Only get libraries, frameworks, and packages (like npm or Maven) from official repositories.
3. **Scan for Vulnerabilities** → Use tools like **OWASP Dependency-Check** or **CycloneDX** to detect security flaws in your software components.
4. **Verify with Checksums** → Compare file hashes (e.g., **SHA-256**) before installing or running software to confirm integrity.
5. **Secure the Build Environment** → Keep the CI/CD build system isolated and protected from both development and production systems.
6. **Deliver Updates Securely** → Send updates only over **HTTPS**, and ensure they are **signed and verified** before installing.
7. **Automate Integrity Checks** → Use scripts or tools to automatically verify file integrity and detect unauthorized changes in the build process.
8. **Control Access** → Apply strict permissions and isolation for sensitive data in the build, deploy, and runtime stages.
9. **Ensure Database Integrity** → Use built-in database checks and constraints to prevent corruption or tampering.
10. **Regular Backups with Verification** → Keep secure, restorable backups, and regularly test them to ensure data can be recovered without damage.

## Insecure Deserialization

1. **Validate Input Before Serialization** → Only allow safe, expected data to be serialized. Reject anything suspicious.
2. **Use Safer Formats (JSON/XML)** → Avoid native serialization if possible. Use formats like JSON or XML which are easier to validate.
3. **Whitelist Allowed Classes** → Restrict deserialization to only known, safe classes or object types.
4. **Integrity Checks or Encryption** → Protect serialized data with hashing or encryption to detect tampering.
5. **Run in a Low-Privilege Environment** → If deserialization is required, execute it in a sandbox or minimal-privilege environment.
6. **Avoid Dangerous Methods** → Ensure classes don’t have risky methods (`__wakeup`, `__destruct`, `__invoke` in PHP) that could be abused.
7. **No Code Execution on Object Creation** → Prevent automatic execution of code during object construction or deserialization.
8. **Use Secure Libraries** → Choose serialization frameworks with built-in security (e.g., Kryo for Java).
9. **Log & Monitor Deserialization** → Keep track of failed or unexpected deserialization attempts to detect attacks early.
10. **Avoid When Not Needed** → If serialization/deserialization isn’t essential, don’t use it at all.

## Security Logging and Monitoring Failures/Insufficient Logging and Monitoring

1. **Log Important Assets** → Always monitor and log activities from **critical systems** like databases, payment systems, and authentication services.
2. **Follow a Logging Standard** → Set a **minimum logging baseline** (what, when, and how to log) and make sure **all systems** follow it.
3. **Include User Details in Logs** → Record the **username, IP, and session info** in each log entry so you can trace actions back to specific users.
4. **Protect Logs from Tampering** → Use **integrity controls** (append-only databases, write-once storage) to stop attackers from deleting or changing logs.
5. **Store Logs Securely** → Keep logs in **secure backup servers** with encryption and **limited access** so attackers can’t steal or modify them.
6. **Sync All System Clocks** → Use **time synchronization** (NTP) so that all logs match the same exact time for accurate incident investigation.
7. **Detect Suspicious Activities in Real-Time** → Use **real-time monitoring tools** and **AI-based anomaly detection** to spot unusual behavior quickly.
8. **Use Centralized Logging** → Send logs from all systems to **one secure central location** for easier tracking and correlation.
9. **Control Access to Logs** → Allow only **authorized security team members** to view or modify logs — no one else.
10. **Use Log Management Tools** → Use advanced tools like **ELK Stack, Splunk, or Graylog** to store, analyze, and visualize logs for faster detection.

## Server-Side Request Forgery Attacks

1. **Use a Whitelist for URLs/IPs** → Only allow requests to safe, pre-approved domains or IP addresses. Block everything else.
2. **Validate and Sanitize User Input** → Carefully check all URLs provided by users and remove anything suspicious or unexpected.
3. **Block Internal Network Access** → Prevent the server from making requests to internal IP ranges (e.g., `127.0.0.1`, `192.168.x.x`).
4. **Restrict URL Schemes** → Allow only safe protocols like `https://` or `http://` and block others like `file://`, `ftp://`, etc.
5. **Disable Automatic Redirects** → Stop the server from following redirects in outbound requests to avoid bypass tricks.
6. **Limit Response Size and Type** → Set limits on what kind of data and how much data can be returned from external requests.
7. **Use a Web Application Firewall (WAF)** → Deploy a modern WAF or NGWAF to detect and block SSRF attempts in real time.
8. **Route Requests Through a Proxy** → Send all external requests via a controlled proxy server that can filter and log them.
9. **Segregate Network Access** → Isolate servers that handle external requests from internal systems to reduce attack impact.
10. **Strict Logging & Monitoring** → Record all outbound requests and monitor for unusual patterns to detect SSRF attempts early.

## Directory Traversal
Directory traversal enables attackers to exploit HTTP, gain access to restricted directories, and execute commands outside the web server’s root directory. Developers must configure web applications and their servers with appropriate file and directory permissions to avoid directory traversal vulnerabilities. 

Some countermeasures against directory traversal attacks are as follows: 

1. **Set Access Rights** → Give users only the necessary permissions to specific areas of the website.
2. **Update Web Server Regularly** → Install security patches quickly to fix known vulnerabilities.
3. **Validate User Input** → Allow only safe characters (letters/numbers) and block suspicious ones like `..`, `/`, `\`.
4. **Use a Whitelist** → Permit access only to approved files and directories.
5. **Normalize File Paths** → Convert file paths into a safe, standard format to remove dangerous sequences.
6. **Restrict File Names** → Accept only safe file names with known good characters.
7. **Avoid Direct File Access from User Input** → Never let user input directly decide which file to open.
8. **Set Proper File Permissions** → Make sure sensitive files are only accessible by authorized processes.
9. **Disable Directory Listing** → Hide directory contents so attackers cannot see file structures.
10. **Use a Chroot Jail (Unix)** → Limit the application's access to only a specific safe directory.

## Unvalidated Redirects and Forwards
In general, web applications redirect and forward users to other pages and websites. Therefore, if a web application does not validate the data, then attackers can redirect users to malicious websites or use forwarding to access unauthorized pages. Therefore, to prevent such attacks, it is best not to allow users to directly supply parameters to redirect and forward in web application logic.

Some countermeasures against unvalidated redirects and forwards attacks are as follows:

1. **Avoid Redirects and Forwards** → Don’t use them unless absolutely necessary to remove the attack surface.
2. **Use Static URLs** → Always use fixed, hardcoded redirect destinations instead of user-provided values.
3. **Whitelist Validation** → If redirects must be dynamic, only allow destinations from a safe, approved list.
4. **Map Inputs to Predefined Destinations** → Instead of directly using user input, map it to a safe destination via lookup tables or enums.
5. **Validate Destination Parameters** → Check if the redirect location is valid and the user is authorized to access it.
6. **Disallow Full URLs in Input** → Don’t allow users to provide a complete URL; restrict to safe paths within your site.
7. **URL Encoding & Sanitization** → Encode and sanitize all redirect parameters to block malicious code or injection.
8. **Token Verification** → Use a secure token or ID to confirm that the redirect request is legitimate.
9. **Internal Redirects Only** → Limit redirects to your own domain or subdomains to block external phishing pages.
10. **User Education & Warnings** → Inform users about phishing risks and show a warning popup before redirecting to unknown sites.

9. **Apply Web Filtering** → Use security tools that block access to infected or suspicious websites.
10. **Limit Website Permissions** → Restrict users from giving unnecessary access (e.g., camera, microphone, location) to sites.

## Cross-Site Request Forgery 
Using a CSRF attack, attackers lure a user’s browser into sending a fake HTTP request, including the user session cookie and other authentication information, to a legitimate (vulnerable) web application to perform malicious activities. 

Some countermeasures against cross-site request forgery attacks are as follows: 

1. **Use Unique CSRF Tokens** → Generate a random token for each user session and send it with every sensitive request to verify authenticity.
2. **Server-Side Token Validation** → Always check the CSRF token on the server before processing any state-changing action.
3. **SameSite Cookie Attribute** → Set cookies with `SameSite` to prevent them from being sent on cross-site requests.
4. **Custom Request Headers** → Require a custom HTTP header (e.g., `X-Requested-With`) for state-changing requests; attackers can’t set it cross-domain.
5. **Per-Request Tokens** → Use a new token for each request instead of reusing the same one for the entire session.
6. **Double-Submit Cookies** → Send the CSRF token both in a cookie and a request parameter, and check if they match on the server.
7. **Validate Origin & Referer Headers** → Accept state-changing requests only if they come from your domain.
8. **Re-Authentication for Critical Actions** → Ask users to re-enter their password for sensitive operations like changing email or payment details.
9. **Use Security Frameworks** → Leverage frameworks (e.g., Django, Rails, Spring) that have built-in CSRF protection.
10. **Correct HTTP Method Usage** → Use `GET` only for reading data, and protect `POST`, `PUT`, `DELETE`, and `PATCH` requests with CSRF tokens.

## Cookie/Session Poisoning 
Browsers use cookies to maintain a session state. They also contain sensitive, session-specific data (e.g., user IDs, passwords, account numbers, links to shopping cart contents, supplied private information, and session IDs). Attackers engage in cookie/session poisoning by modifying the data in the cookie to gain escalated access or maliciously affect a user session. Developers must hence follow secure coding practices to secure web applications against such poisoning attacks. They must use proper session-token generation mechanisms to issue random session IDs. 

Some countermeasures against cookie/session poisoning attacks are as follows: 

1. **Use HTTPS and Secure Attribute** → Always send cookies over encrypted HTTPS connections and set the `Secure` flag so they’re never sent in plain text.
2. **SameSite Attribute** → Set `SameSite` to prevent cookies from being sent with cross-site requests, reducing CSRF risks.
3. **Strong, Random Session IDs** → Generate long, unpredictable session identifiers so attackers can’t guess them.
4. **Regenerate Session IDs** → Change the session ID after login and periodically to stop session fixation attacks.
5. **Store Sensitive Data on Server** → Keep session data on the server and only store an ID in the cookie to avoid exposing private info.
6. **Cookie Timeout** → Set expiration times so stolen cookies quickly become useless.
7. **Sign Cookies** → Add a server-side signature to cookies and verify it before use to detect tampering.
8. **Validate Cookie Values** → Check that all cookie data is correctly formed and valid before processing.
9. **Restrict Cookie Scope** → Limit cookies to specific purposes and paths to minimize misuse.
10. **Logout Option** → Always provide a logout button that clears cookies and ends the session immediately.

## Web Service Attack 
Use multiple layer protection and standard HTTP authentication techniques to defend against web service attacks. Because most models incorporate business-to-business applications, it becomes easier to restrict access to only valid users. 

Some additional countermeasures against web service attacks are as follows: 

1. **Validate All Inputs** → Check all incoming XML/JSON data against strict rules to prevent malicious input.
2. **Strong Authentication** → Use OAuth, JWT, or API keys to verify users and systems.
3. **Role-Based Access Control (RBAC)** → Give each user only the access they actually need.
4. **Secure WSDL & SOAP** → Limit access to WSDL files and block risky SOAP actions or guessable names.
5. **Schema Validation** → Verify every request and response matches the approved XML/JSON structure.
6. **Firewall & IDS Protection** → Use web service–aware firewalls to block suspicious or malformed SOAP/XML traffic.
7. **Message Integrity & Signatures** → Sign messages with digital signatures to ensure they are not changed.
8. **Disable Unused Features** → Turn off unused SOAPAction, WS-Addressing, and auto-generated documentation.
9. **Encrypt Communication (TLS)** → Always secure data in transit between client and server with TLS.
10. **Hide Internal Details** → Use XML proxies and avoid exposing internal configurations in service responses.

## Clickjacking Attack

1. **Use X-Frame-Options Header** → Set `DENY` or `SAMEORIGIN` to stop your site from being loaded inside a frame on other domains.
2. **Apply Content-Security-Policy (CSP)** → Use `frame-ancestors` in CSP to control exactly which sites can embed your pages.
3. **Mask Content Until Verified** → Hide page elements and reveal them only after confirming the site is not inside a frame.
4. **SameSite Cookies** → Set cookies with `SameSite` to prevent them from being sent when your site is framed from another domain.
5. **Frequent Vulnerability Scanning** → Regularly use web vulnerability scanners to find and fix clickjacking risks.
6. **UI Frame Detection** → Add scripts to check if the page is the top frame, and block actions if it’s not.
7. **Click Confirmation Dialogs** → Use `window.confirm()` or similar prompts before important actions to ensure user intent.
8. **UI Manipulation Protection** → Implement checks to detect overlapping or hidden UI elements used in clickjacking.
9. **Auth0 or Similar Services** → Use authentication platforms that send both `X-Frame-Options` and `CSP` headers by default.
10. **Browser Extensions Recommendation** → Encourage users to install security extensions like NoScript or uBlock Origin to block malicious framing.

## JavaScript Hijacking

1. **Use `.innerText` instead of `.innerHTML`** → Prevents malicious scripts from executing by encoding text automatically.
2. **Avoid `eval()`** → `eval` is unsafe because it can run attacker-controlled code.
3. **Use Encoding Libraries** → Automatically escape unsafe characters to prevent injection attacks.
4. **Enable SSL/TLS** → Ensures all JavaScript and JSON data are encrypted in transit.
5. **Return JSON as Objects** → Always wrap JSON inside an object to avoid direct execution.
6. **CSRF Tokens** → Protects against unauthorized JSON requests from malicious sites.
7. **Use CORS Instead of JSONP** → JSONP can be abused; CORS safely controls domain access.
8. **Safe JSON Parsing** → Use `JSON.parse()` instead of `eval()` or unsafe parsing.
9. **Content-Type Validation** → Server should only process requests with correct `Content-Type` (e.g., `application/json`).
10. **API Keys or Tokens** → Ensures only authorized clients can request and access JSON data.

## Username Enumeration 

1. **Generic Error Messages** → Always show the same error for wrong username or password so attackers can’t guess if a username is valid.
2. **Uniform Response Time** → Make the login process take the same amount of time for both valid and invalid usernames to prevent timing attacks.
3. **CAPTCHA on Input Pages** → Use CAPTCHA on login, signup, and forgot password pages to block bots from testing usernames automatically.
4. **Rate Limiting** → Block or slow down requests from an IP after a few failed login attempts to stop brute-force username guessing.
5. **Two-Factor Authentication (2FA)** → Even if attackers guess a username and password, they can’t log in without the second verification step.
6. **Avoid Username Confirmation** → Never tell users if a username is already registered during signup or password reset.
7. **Random & Complex Usernames** → Use unique, hard-to-guess usernames (e.g., random strings) instead of simple patterns or sequential IDs.
8. **WAF (Web Application Firewall)** → Use a WAF to detect and block suspicious login patterns or multiple failed attempts from the same IP.
9. **Geo-Location Validation** → Check the location of login attempts and block suspicious or unusual geographic access patterns.
10. **Honeypot Accounts** → Create fake accounts that trigger alerts if someone tries to log in, helping detect attacks early.

## Attack on Password Reset Mechanism

1. **Validate Token & Email Together** → Always check that the reset token matches the correct email before allowing a password change.
2. **One-Time Use & Expiry** → Make reset links work only once and expire quickly (e.g., 10–15 minutes).
3. **Use CAPTCHA** → Stop bots by asking the user to solve a human verification challenge.
4. **Rate Limit Requests** → Restrict how many reset requests can come from the same IP or device in a short time.
5. **Enable MFA** → Require an extra verification step (code to phone/email) before completing the reset.
6. **Enforce HTTPS** → Send all reset links over secure HTTPS to prevent interception.
7. **Send Temporary Passwords** → Instead of changing passwords instantly, email a temporary password that the user must change after logging in.
8. **Use Strong Random Tokens** → Generate cryptographically secure, unique tokens for each reset request.
9. **Short Token Lifespan** → Keep token validity short to limit the attacker’s chance to use it.
10. **Add Confirmation Step** → Require users to enter a verification code sent to their email or phone before finalizing the reset.

## Same-Site Attacks

1. **CSRF Token per Session** → Give each user a unique secret code in forms and important requests, and check it on the server before doing anything.
2. **SameSite Cookies** → Set cookies so they aren’t sent when requests come from other sites, reducing cross-site attacks.
3. **Check Referer & Origin** → Make sure requests come from your own website by checking the browser’s “where it came from” headers.
4. **Dangling Domain Record Validation** → Check and confirm that no old or unused subdomains are still pointing to your server.
5. **DNS Misconfiguration Checks** → Regularly scan for and fix wrong or risky DNS settings.
6. **Update DNS Records** → Keep your DNS information correct and up-to-date to prevent hijacking.
7. **Educate on CNAME Risks** → Teach users/admins how CNAME entries work and the security problems if they point to wrong or unused domains.
8. **User Interaction Security** → Ask users to re-enter passwords or solve CAPTCHA before critical actions to confirm it’s really them.
