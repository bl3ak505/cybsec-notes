# 🔗 Web API and Webhooks
Web APIs allow applications to exchange data with multiple online sources and are widely used in platforms like social media, e-commerce, and search engines. Webhooks enable applications to send real-time data to other systems. Because APIs often handle sensitive personal or business data, any security weakness can lead to serious breaches, making strong API security measures essential.

---
## Web API 
A Web API is an interface that allows applications to communicate over the internet using protocols like HTTP. It provides tools and functions to retrieve or update data from multiple sources without complexity. By centralizing data and business logic, a Web API simplifies updates, improves consistency, and reduces the need to modify each client application separately.

## Web Service APIs 
1. **SOAP API** → A protocol that lets different systems (Windows, macOS, Linux, etc.) talk to each other using XML over HTTP, often used for creating, updating, or deleting data like profiles or credentials.
2. **REST API** → An architectural style for web services that uses simple, stateless communication between systems, making it easy to request and change web resources.
3. **RESTful API** → A web service following REST rules, using HTTP methods (GET, POST, PUT, DELETE) to improve speed, scalability, and reliability of applications. <br>
   APIs with the following features can be referred to as to RESTful APIs:

    - **Stateless** → The server doesn’t remember your previous requests; the client stores session data.
    - **Cacheable** → The client can store API responses to speed up future requests.
    - **Client-Server Environment** → The client (front end) and server (back end) work separately but communicate through requests.
    - **Uniform Interface** → Every resource has a unique URL, and you can use standard methods like GET, POST, PUT, DELETE to interact with it.
    - **Layered System** → Data can pass through multiple layers (like cache servers) without the client knowing exactly where it came from.
    - **Code on Demand** → The server can send temporary code to the client to add or change features.

5. **XML-RPC** → A simpler protocol than SOAP that sends data in XML format, using less bandwidth.
6. **JSON-RPC** → Similar to XML-RPC but sends data in lightweight JSON format for faster communication.

## Webhooks 
Webhooks are user-defined HTTP callbacks that send data to other applications in real time when specific events occur, such as posting a comment or updating code. They act like “reverse APIs,” automatically pushing information instead of waiting for requests. Common uses include sending alerts, messages, or notifications—like informing users when an out-of-stock item becomes available.

## Operation of Webhooks
Webhooks are mechanisms that send automated notifications to a specified URL when certain events occur. Once registered through a domain’s interface or API, they execute predefined code whenever the event is triggered. The receiving system can identify the event, its source, time, and details via an HTTP POST request, allowing it to react accordingly. Webhooks can be set as private or public.

### Webhooks vs. APIs 
- Webhooks are automated messages from websites to the server. APIs are used for server-to-website communication.
- Webhooks get reports or notifications via HTTP POST only when a new update is made. APIs make calls irrespective of the data updates.
- Webhooks update applications or services with real-time information. API needs additional implementations to perform this activity.
- Webhooks have less control over data flow. APIs have easy control over data flow.

## OWASP Top 10 API Security Risks 
Source: [https://owasp.org]

| API | Risk | Descriptions |
|---|---|---|
| API1 | **Broken Object-Level Authorization** | - APIs can have security issues when they expose endpoints that use object IDs without proper checks, making it easier for attackers to target them. <br> - Hackers can change the object ID in a request to access data they shouldn’t be allowed to see. <br> - This can lead to sensitive data being exposed, lost, or altered by unauthorized users. |
| API2 | **Broken Authentication** | - When authentication is not set up correctly, attackers can steal login tokens or take advantage of flaws in the system to pretend to be other users. <br> - Once they do this, they can take full control of the victim’s account, read private information, and perform sensitive actions as if they were the real user. |
| API3 | **Broken Object Property Level Authorization** | - Developers might send all data fields of an object to the user without checking which ones are sensitive, relying on the user to hide them. <br> - This can allow attackers to view, change, or delete private or important data. <br> - In some cases, it can let attackers gain higher access rights or even fully take over an account. |
| API4 | **Unrestricted Resource Consumption** | - Attackers can initiate multiple concurrent requests using automated tools designed to cause DoS via high loads of traffic, impacting APIs’ service rate. <br> - Attackers find APIs that do not limit client interactions or resource consumption and craft API requests including parameters that control the number of resources. |
| API5 | **Broken Function-Level Authorization** | - Complexity in access control policies through different hierarchies, groups, and roles between administrative and regular functions can cause authorization errors. <br> - Allow attackers to gain unauthorized access to administrative functions or users’ resources. |
| API6 | **Unrestricted Access to Sensitive Business Flows** | - Some APIs let users perform important actions like buying tickets or posting comments without limiting how often they can be done, which attackers can abuse. <br> - Hackers figure out which parts of the API handle these actions and how they connect to complete the process. <br> - They study how the business works through the API, find valuable functions, and then automate them to cause financial or operational damage. |
| API7 | **Server-Side Request Forgery** | - A server-side request forgery (SSRF) flaw enables an attacker to coerce the application to send a crafted request to an unexpected destination, even when protected by a firewall or a VPN. <br> - Attackers exploit this vulnerability by finding an API endpoint that accesses a URI provided by the client. <br> - Exploitation might lead to internal services enumeration (e.g., port scanning), information disclosure, bypassing firewalls, or other security mechanisms |
| API8 | **Security Misconfiguration** | - Attackers will often attempt to find unpatched flaws, common endpoints, services running with insecure default configurations, or unprotected files and directories to gain unauthorized access or knowledge of the system. <br> - Attackers use automated tools to detect and exploit misconfigurations such as unnecessary services or legacy options. <br> - Security misconfigurations not only expose sensitive user data but also system details that can lead to a full server compromise. |
| API9 | **Improper Inventory Management** | - Proper inventory management of hosts and deployed API versions is important to mitigate issues. <br> - Attackers usually gain unauthorized access through old API versions or endpoints left running unpatched and using weaker security requirements. <br> - Attackers can gain access to sensitive data or even take over the server when different API versions/deployments are connected to the same database with real data. |
| API10 | **Unsafe Consumption of APIs** | - Developers tend to trust data received from third-party APIs more than user input and thus tend to adopt weaker security standards. <br> - Attackers go after integrated third-party services instead of trying to compromise the target API directly. <br> - Successful exploitation may lead to exposure of sensitive information to unauthorized actors, different types of injections, or denial of service. |

## Webhooks Security Risks
Webhooks, while efficient for real-time data transfer between applications, can become a source of vulnerabilities if not properly secured. Here are some common risks associated with webhooks and strategies to mitigate them:
| Risk                              | Description |
|-----------------------------------|-------------|
| **Data Exposure in Transit**      | - Webhooks often transmit data in plain text over HTTP, making it susceptible to interception and tampering.<br>- Using HTTPS to encrypt all data in transit is crucial to protect the information from being exposed. |
| **Lack of Verification**          | - There is no proper mechanism to verify the authenticity and integrity of the data received through webhooks, making it susceptible to various online attacks.<br>- Implementing a hash-based message authentication code (HMAC) helps ensure that both the sender and the receiver are authenticated and the message has not been altered. |
| **Replay Attacks**                 | - Webhooks are vulnerable to replay attacks, where an attacker intercepts a legitimate webhook call and resends it, potentially causing unintended actions.<br>- To prevent this, include a timestamp in the webhook payload and verify it upon receiving to ensure it is within an acceptable time window. |
| **Unrestricted Sources**          | - By default, webhooks do not restrict where they can receive data from, opening up the possibility of accepting malicious data from unauthorized sources.<br>- Implementing IP whitelisting and mutual TLS can mitigate this by ensuring data is only accepted from trusted sources. |
| **Duplication and Forgery**       | - Webhooks can be duplicated or forged, which may lead to data integrity issues.<br>- Mutual TLS helps protect both the sender and receiver by ensuring the data is sent to and comes from authenticated parties. |
| **Endpoint Configuration Errors** | - Human errors in configuring webhook URLs can lead to data being sent to the wrong recipients, potentially causing data leaks or loss.<br>- Double-checking configurations and implementing additional checks to validate endpoint URLs can reduce this risk. |
| **Forged Requests**               | - Attackers might forge webhook requests to send malicious data to an application.<br>- To protect against this, use a secret token shared between the sender and the receiver to verify the integrity of the data. Only accept data that can be verified with the secret token. |
| **Unvalidated Redirects and Forwards** | - Webhooks can be used to redirect users to malicious sites if the URL redirection is not properly validated.<br>- Ensure that any URL redirection is validated against a whitelist of allowed URLs. |

## API Vulnerabilities
API vulnerabilities occur when issues like poor permission management, weak business logic, or exposure of sensitive data such as personally identifiable information (PII) are present. These weaknesses expand the attack surface, allowing attackers to exploit them for attacks like denial of service, code injection, and other malicious activities.

Some common API vulnerabilities are listed below: 
| # | Vulnerability | Description |
|---|---------------|-------------|
| 1 | **Enumerated Resources** | - Design flaws can cause serious vulnerability, disclosing information through unauthenticated public API.<br>- Allow attackers to guess user IDs easily, compromising the security of the user data. |
| 2 | **Sharing Resources via Unsigned URLs** | - API returns URLs to hypermedia resources such as image, audio, or video files that are vulnerable to hotlinking.<br>- This can cause issues such as poor analytics, strains on resources, and exploitation by attackers.<br>- Signed URLs can implement rate limiting, auto expiration, and scoped sharing. |
| 3 | **Vulnerabilities in Third-Party Libraries** | - Developers use third-party software libraries with open-source licenses.<br>- Avoiding regular updates and neglecting security fixes can result in many security flaws. |
| 4 | **Improper Use of CORS** | - Cross-origin resource sharing (CORS) allows web browsers to perform cross-domain requests; improper implementation can cause vulnerabilities.<br>- Allowing all origins on private APIs using the `Access-Control-Allow-Origin` header can lead to hotlinking. |
| 5 | **Code Injections** | - If input is not sanitized, attackers can use SQLi or XSS to inject malicious SQL statements or code into API fields.<br>- Can lead to theft of critical information such as session cookies and user credentials. |
| 6 | **RBAC Privilege Escalation** | - Role-Based Access Control (RBAC) flaws can allow privilege escalation when endpoint changes lack proper checks.<br>- Can allow attackers to gain access to sensitive user information. |
| 7 | **No ABAC Validation** | - Without Attribute-Based Access Control (ABAC), attackers can gain unauthorized access to API objects or perform unauthorized actions.<br>- Examples: view, update, or delete sensitive data. |
| 8 | **Business Logic Flaws** | - Many APIs have vulnerabilities in their business logic.<br>- Attackers can exploit legitimate workflows for malicious purposes. |

## Web API Hacking Methodology 
The rapid growth of web-based APIs for mobile and IoT devices has increased their exposure to attacks. To improve usability, developers sometimes compromise on security, leaving APIs vulnerable. Attackers analyze these APIs to identify their technologies, security standards, and attack surfaces, then exploit weaknesses to compromise the connected services.

Hacking a web API involves the following phases:
- Identify the target
- Detect security standards
- Identify the attack surface
- Launch attacks

---

## Detect Security 
Standards Although APIs claim to be secure as they incorporate security standards such as OAuth and SSL, they still include many vulnerabilities that can be exploited by attackers.

- APIs such as SOAP and REST implement different authentication/authorization standards such as OpenID Connect, SAML, OAuth 1.X and 2.X, and WS-Security.
- SSL provides transport-level security for API messages to ensure confidentiality through encryption and integrity through signature. Although SSL is used for security, in most API messages, only sensitive user data such as credit card details are encrypted, leaving other information in plaintext.
  
If these security standards are configured improperly, an attacker can identify vulnerabilities in these standards for further exploitation. For example, an attacker can capture and reuse a session token to retrieve a legitimate user’s account information that is not encrypted.

Identifying the attack surface of an API involves finding all possible points where it can be targeted. Unlike web apps with visible input fields, APIs only expose endpoints. Attackers study these endpoints, their messages, parameters, and behavior to discover where vulnerabilities may exist and plan potential attacks.

Use the following techniques to identify the attack surface of the target API:

### API Discovery
If an API does not have metadata, attackers monitor and record the communication between the API and an existing client to identify the initial attack surface. ## Launch Attacks 
After identifying the target API, analyzing the message formats and security standards, and
identifying the attack surface, attackers perform various attacks on the target API to steal sensitive information such as credit card details and credentials. Various attacks performed on APIs are discussed below: 

### Fuzzing
Attackers use the fuzzing technique to repeatedly send some random input to the target API to generate error messages that reveal critical information. To perform fuzzing, attackers use automated scripts that send numerous requests with varying combinations of input parameters. Attackers use tools such as Fuzzapi to perform fuzzing on the target API.

### Invalid Input Attacks
In some scenarios, fuzzing is difficult to perform due to its structure. In such cases, attackers will give invalid inputs to the API, such as sending text in place of numbers, sending numbers in place of text, sending a greater number of characters than expected, and sending null characters, etc., to extract sensitive information from unexpected system behavior and error messages. At the same time, attackers also manipulate the HTTP headers and values targeting both API logic and the HTTP protocol.

### Malicious Input Attacks
In the attack discussed above, attackers try to retrieve sensitive information from unexpected system behavior or error messages. A more dangerous attack is where the attackers inject malicious input directly to target both the API and its hosting infrastructure. ### Injection Attacks
Similar to traditional web applications, APIs are also vulnerable to various injection attacks. For example, consider the following normal URL:
```http
http://billpay.com/api/v1/cust/459
```
For the abovementioned URL, the API retrieves the customer details based on the customer ID 459 from the database using the following SQL query:
```sql
“SELECT * FROM Customers where custID=’” + custID + ”’”
```
Here, the custID is replaced with 459
```sql
“SELECT * FROM Customers where custID=’459’”
```
In the abovementioned URL, assume that an attacker injects the malicious input
```http
http://billpay.com/api/v1/cust/ '%20or%20'1'='1
```
The resultant malicious SQL query is
```sql
“SELECT * FROM Customers where custID=‘’ or ‘1’ = ‘1’”
```
The abovementioned query returns details of all the customers in the database. Using this information, an attacker may further delete or modify the data in the database or use the customers' information to perform other malicious activities on the database server. These API injection attacks are performed not only using SQL but also using JSON, JavaScript, XPath, XSLT, etc., which require parsers/processors for execution.

> **Note:** Similar to injection attacks, web APIs are also vulnerable to XSS and CSRF attacks 

### Exploiting Insecure Configurations
- **Insecure SSL Configuration** <br>
  Vulnerabilities in SSL configuration may allow attackers to perform MITM attacks. For example, using self-signed SSL certificates for secure API access may allow attackers to perform an MITM attack. An attacker may sniff the traffic between an API and a client, manipulate the client-side certificate, and start monitoring or manipulating the encrypted traffic between the client and the API.
  
- **Insecure Direct Object References (IDOR)** <br>
  In general, direct object references are used as arguments for API calls, and access rights are not imposed on the objects for which a user does not have access. These vulnerabilities can be identified through API metadata and exploited by attackers to identify the parameters and try all possible values for the parameters to access the data to which the user does not have access.

- **Insecure Session/Authentication Handling** <br>
  Vulnerabilities such as the reuse of session tokens, sequential session tokens, long session token timeout, unencrypted session token, and session token embedded into a URL, allow attackers to hijack and take over the client session and steal or manipulate the messages between the client and the API.

### Login/Credential Stuffing Attacks
Login or credential stuffing attacks exploit the common habit of users reusing the same password across multiple websites and services. Instead of guessing or brute-forcing passwords, attackers use stolen username–password pairs from previous data breaches and automate the login process against other platforms. Tools like Sentry MBA and PhantomJS allow them to quickly test large volumes of these stolen credentials across many accounts.

These attacks are often carried out with bots, making them difficult to detect with basic API security measures. The bots attempt logins repeatedly until a valid match is found, giving attackers full access to the compromised accounts. Once inside, they can perform unauthorized actions such as transferring funds, making purchases, stealing personal data, or running fraudulent campaigns.

In some cases, attackers may also use credential stuffing to overload login systems, blocking legitimate users from signing in and degrading API performance. This not only impacts security but also harms the service’s availability and user trust. Since the attack relies on valid stolen credentials rather than password cracking, it is harder to detect and requires stronger defenses like multi-factor authentication, bot detection, and monitoring for unusual login patterns.

API DDoS attacks overwhelm an API with traffic, often from botnets, to slow or disrupt its service for legitimate users. Attackers may stay within rate limits to evade detection and can bypass security systems like load balancers. These attacks are not always high-volume; they can also exploit API flaws to consume CPU and memory resources, causing delays and degrading performance.

### Authorization Attacks on API: OAuth Attacks 
According to [https://auth0.com], OAuth is an authorization protocol that allows a user to grant limited access to his/her resources on one site to another site without having to expose his/her credentials. <br>
OAuth grants authorization flows for many computing devices and applications, such as connecting users to different applications from one application to access the required information.

#### **Different actors involved in the OAuth process:**
- **Owner of the resource:** The resource owner is also known as a user who grants permission to an application to access his/her account. The access to the application is limited or conditional, such as providing only read and write permissions.
- **Authorization/Resource Server (API):** The resource server provides the secured user account, and the authorization server validates the user identity and then supplies the access token to the application.
- **Client or Application:** It is an application that seeks access to the user account. To access the account, the user must authorize the application; then, the API should validate the authorization.

#### **Steps involved in Authorization Code Grant**

There are four steps involved in the authorization code grant, through which attackers can perform various authorization attacks on the API. o The user passes the GET request to the client via the user agent to initiate the authorization process. This operation can be performed via the **“Login with or Connect”** button displayed on the client’s site.

- The user agent can be redirected to the authorization server by the client using the following parameters:
  - **response _type:** Code used for informing the server which permissions to execute
  - **client_id:** ID assigned to the client
  - **redirect_uri:** URI where the authorization server redirects the user agent when the authorization code is provided
  - **scope:** Defines the level of access to the application
  - **State:** Opaque value used for security implementations. The value is also used for maintaining the state between request and callback

- When the user is authenticated and authorized to access the resource, the user agent is redirected to redirect_uri by the authorization server. The server uses the following parameters to do this:
  - **Code:** Authorization code
  - **State:** Value supplied in the abovementioned request

- Using the authorization code, the client requests the access token by adding the following parameters in the body of a request:
  - **grant_type:** Authorization_code
  - **code:** Authorization code received in the previous message
  - **redirect_uri:** URI used in the first request

#### OAuth Attacks
Various attacks on OAuth performed by manipulating the requests stated above are described below: 

- **Attack on ‘Connect’ Request** <br>
  Most sites enable users to access other websites such as LinkedIn, Instagram, and Twitter, via OAuth. An attacker can exploit requests to connect one site to another, i.e., when the user hits the “Login with or Connect” button. Then, he or she can gain illegal access to the client-side user/victim’s account by connecting his/her account to the provider’s website.

  **Steps to perform an attack on “Connect” request:** <br>
  
  - The attacker first signs up for a normal (fake) account on the identity provider’s site—nothing special, just a legitimate account they control.
  - From that fake account, the attacker starts a “Connect” (e.g., OAuth/OpenID Connect link) to the client app, but blocks the redirect back—so the provider records that the client may access the attacker’s resources, while the client/victim never sees a consent screen.
  - The attacker hosts a malicious web page that silently forces the victim to log out of the provider (via CSRF), then immediately logs the victim back in—but using the attacker’s fake credentials, not the victim’s.
  - The same page then triggers the initial “Connect” request again (e.g., hidden in an `<iframe>`), so the client app now links “whoever is logged in at the provider” to the victim’s client session—which is actually the attacker’s provider account.
  - When the victim visits that page, their provider session gets swapped to the attacker’s account and the client becomes “connected” to the attacker’s provider identity without the victim noticing any consent step.
  - With that link in place, the attacker can sign in to the client using their provider account and gain access to the victim’s client-side account/data, because the client now trusts that provider identity as if it belonged to the victim.

- **Attack on ‘redirect_uri’** <br>
  An attack on `redirect_uri` happens when an attacker finds a flaw like XSS on an allowed page within the client’s domain. By exploiting it, they can intercept the authorization code during redirection, allowing them to gain unauthorized access to the application.

  **Steps to perform an attack on ‘redirect_uri’:**

  - The attacker finds a page on the client domain that leaks the full URL (including query and hash) and can run their JavaScript, e.g., `https://xyz.com/vuln`.
  - They place script on that page to read the current browser URL and quietly send it to their server.
  - They craft a phishing link to the OAuth authorize endpoint that uses the victim app’s `client_id` and sets `redirect_uri` to the leaking page (`https://xyz.com/vuln`).
  - When the victim clicks, the OAuth server redirects the browser to `https://xyz.com/vuln?code=CODE` after login/consent.
  - The malicious script on `/vuln` captures the `code` from the URL and exfiltrates it to the attacker.
  - The attacker redeems that stolen `code` against the real callback (`https://xyz.com/oauth/callback?code=CODE`) to get an access token and impersonate the user.

- **CSRF on Authorization Response** <br>
  The attacker performs a CSRF attack to connect a fake account on the provider with the victim’s account on the client side. This attack exploits a third request related to authorization code grant.

  **Steps to perform CSRF on authorization response:**

  - **Create fake provider account:** Attacker registers a new account on the OAuth provider.
  - **Obtain a valid authorization code:** Attacker starts a normal “Connect/Login with \<provider>” to the client using the fake account, approves consent, but **blocks the final redirect** back to the client. The attacker keeps the returned **authorization\_code**.
  - **Force the victim’s browser to send the attacker’s code:** Attacker lures the victim to load: `https://xyz.com/<provider>/login?code=Auth_Code` (e.g., via an `<img>` or `<script>` tag). The victim’s browser sends the attacker’s code to the client.
  - **Client links wrong accounts:** The client exchanges that code for tokens and, because the request came from the victim’s browser/session, **links the attacker’s provider account** to the victim’s client account.
  - **Account takeover via SSO:** Now the attacker can sign in to the client **as the victim** by logging in with the attacker’s fake provider account.

- **Access Token Reusage** <br>
  Access token reusage occurs when an OAuth access token issued for one client can also be used by another. If clients accept tokens without strict checks, attackers can use a token from “ClientA” to access resources on “ClientB,” bypassing proper authorization controls.

  **Steps to reuse access tokens:**

  - The attacker develops a legitimate client application “clientA” and enrolls it with some provider
  - Now, the attacker lures the victim into accessing “clientA” and gains illegal access to the victim’s access token on “clientA”
  - Let us consider that the victim accesses “client”, which uses the implicit grant. In such a case, the authorization server redirects the user agent to `https://clientB.com/callback#access_token=ACCESSTOKEN`. Now, the attacker can open this URL with client’s access_token.
  - The attacker is verified as a valid entity by “client”. Therefore, one access token is sufficient for use on many clients using the implicit grant.

- **SSRF Using Dynamic Client Registration Endpoint** <br>
  In black-box testing on an OAuth server, the analysis may not detect hidden URLs such as the Dynamic Client Registration endpoint. These URLs are used as special registration endpoints and are mapped to /register. The attacker can perform an SSRF attack using these URLs associated with the parameters, as demonstrated in the following POST request:

  ```http
  POST /connect/register HTTP/1.1
  Content-Type: application/json
  Host: server.certifiedhacker.com
  Authorization: Bearer eyJhbGciOiJSUzI1NiJ9.eyJ ...
  
  {
      "application_type": "web app",
      "redirect_uris": [
          "https://client.certifiedhacker.com/callback"
      ],
      "client_name": "Sample Test",
      "logo_uri": "https://client.certifiedhacker.com/logo.png",
      "subject_type": "pairwise",
      "sector_identifier_uri": "https://certifiedhacker.com/rdrct_uris.json",
      "token_endpoint_auth_method": "client_secret_basic",
      "jwks_uri": "https://client.certifiedhacker.com/public_keys.jwks",
      "contacts": [
          "hacker@certifiedhacker.com"
      ],
      "request_uris": [
          "https://client.certifiedhacker.com/rf.txt"
      ]
  }
  ```
  The above POST request has several URL references values that can trigger an SSRF attack. The vulnerable parameters are `logo_uri`, `jwks_uri`, and `request_uris`.

  - **logo\_uri** → When a new client registers, the server may fetch its logo from the URL given in `logo_uri`. If this URL points to a malicious server, it can trick the server into making hidden requests, leading to SSRF (Server-Side Request Forgery).
  - **jwks\_uri** → This points to where the server should get the client’s public keys to check tokens. If an attacker uses a malicious `jwks_uri`, the server could fetch keys from an attacker-controlled location, letting the attacker steal authorization codes from all users.
  - **request\_uris** → This is a list of URLs that store request details in JWT format. If one of these URLs is attacker-controlled, the server may fetch it and leak sensitive data, enabling an SSRF attack.

- **WebFinger User Enumeration** <br>
  WebFinger user enumeration is an attack that uses the WebFinger protocol to identify valid usernames on a server. By sending a GET request to “/.well-known/webfinger” with a chosen username, attackers can analyze the server’s response. If the username is valid, the server returns a URL like “[http://host/user,”](http://host/user,”) revealing that the account exists and enabling further targeted attacks.

- **Exploiting Flawed Scope Validation** <br>
  Exploiting flawed scope validation in OAuth occurs when attackers manipulate the scope parameter in an authorization request to gain more access than intended. If the service provider fails to validate this properly, the attacker can escalate the scope and retrieve additional sensitive data from the resource owner beyond what was originally authorized.

  **Steps to Exploit Flawed Scope Validation**

  Attackers use different grant types to exploit flawed scope validation. The following are the two grant types used by attackers during the attack.
  - **Authorization code grant type:** <br>
    - The attacker registers for the OAuth service that is used by the targeted resource owner for their malicious client application https://xyz.com.
    - When the victim attempts to open the attacker’s malicious client application, the attacker initiates a request to the OAuth service provider for access to the client’s mail address using the OpenID email scope.
    - When the user provides authorization for their request, the attacker attains an authorization code as a response.
    - Now, the attacker initiates the scope escalation process for the targeted client by controlling their malicious client application to add additional scope `client_id=12345&client_secret=SECRET&redirect_uri=https://xyz.com/callback&grant_type=authorization_code&code=a1b2 c3d4e5f6g7h8&scope=openid%20 email%20profile.`
    - After approval from the OAuth server, the attacker attains a new access token containing the newly added additional scope:
      ```
      "access_token":"z0y9x8w7v6u5",
      "token_type":"Bearer",
      "expires_in":3600,
      "scope": "openid email profile",
      ```
    - Now, the attacker attains a valid access token to access and pocket additional data by using the escalated scope to make usual API calls to the client.

  - **Implicit grant type:**
    - The attacker targets a vulnerable client application that utilizes an implicit grant-type process to attain access tokens from its clients through an open browser. `https://xyz.com/vuln`
    - When the targeted client application obtains approval from its client and the corresponding access token is generated, the attacker attempts to pocket it.
    - After attaining the access token from the targeted client application, the attacker initiates a new request to its corresponding OAuth service provider with an altered scope /userinfo.
    - As the client has already granted permission for data access to the targeted client application, the attacker is now able to access additional information from the user until the OAuth server verifies and validates the scope parameter.

### Other Techniques to Hack an API
Different ways to hack an API are discussed below: 

- **Reverse Engineering** <br>
  Reverse engineering in APIs involves attackers analyzing and calling the API’s functions in a different order than intended to find hidden flaws. For example, if an order process normally links an order to an account before acceptance, reversing this sequence can break the internal logic, potentially crashing the API and exposing sensitive user account details.

- **User Spoofing** <br>
  User spoofing is when an attacker hides their real identity and pretends to be a legitimate user, often with high privileges. Using stolen details from phishing or data leaks, they can gain access, escalate privileges, redirect functions, inject malicious code, or overload APIs, leading to unauthorized access and potential system damage.

- **Man-in-the-Middle Attacks** <br>
  A Man-in-the-Middle (MITM) attack happens when attackers intercept API communications or impersonate the server to trick users. They can use tactics like domain squatting, where a small change in the URL—such as `API.io` becoming `APO.io`—redirects the user to a malicious server. If unnoticed, the user may unknowingly send sensitive data to the attacker.

- **Session Replay Attacks** <br>
  Session replay attacks can be launched on websites and other sources that initiate and store sessions. These attacks are usually performed to obtain session IDs and replay them to the server. In this case, attackers rewind the session time and prompt the server to disclose the information as though a similar request is made once again.

- **Social Engineering** <br>
  Social engineering through APIs involves tricking users into giving away sensitive information, often using phishing or spear-phishing emails with fake links that appear legitimate. When users enter their credentials on these links, attackers steal the data to perform actions like changing account details or carrying out unauthorized transactions.

## Bypassing IDOR via Parameter Pollution 
Bypassing IDOR via parameter pollution is a technique attackers use to gain unauthorized access when a web application references internal objects like user IDs without proper validation. In a normal IDOR attack, an attacker changes a parameter value, such as `user_id=321` (victim) to `user_id=654` (attacker), to try accessing another account. If the application is secure, it rejects the request with an error like “401 Unauthorized.”

However, with parameter pollution, the attacker sends the same parameter name twice with different values, for example:
```
api.xyz.com/profile/user_id=654&user_id=321
```
If the application checks only the first `user_id` for authorization but later processes the second `user_id` when fetching data, it may mistakenly allow the request and return the victim’s data. This happens because some applications or APIs prioritize the first parameter for authentication but use the last one for actual data retrieval, creating a logical flaw.

Attackers commonly exploit this by intercepting and modifying traffic with tools like Burp Suite. They capture the legitimate request, inject both the victim’s and their own `user_id`, and send it to the server. If successful, they can view or modify another user’s data without permission.

This vulnerability is dangerous because it can be applied to any parameter in the application, not just `user_id`. It can allow attackers to steal sensitive information, manipulate application functions, or escalate privileges, making proper parameter validation and strict access control checks essential for security.

## Secure API Architecture 
Secure API architecture protects APIs from modern cyber-attacks by separating API development from API security. An API gateway, placed in the organization’s DMZ, controls traffic, detects threats, and acts as a secure proxy between internal systems and the internet. Developers focus on building and integrating APIs, while security professionals handle identity protection, threat detection, and data security using advanced tools and policies. This layered approach ensures proper access control, authentication, encryption, message validation, and rate limiting for all published APIs.

### Implementing Layered Security in an API
Considering the scenario of an API that fetches the transactions of a company, developers and security experts can implement the following layer-based security for the API: 

* **Layer 1:** The API first checks if the user is allowed to access the company’s data. If not, it rejects the request with an error like “Company Not Found” to stop unauthorized access.
* **Layer 2:** The API’s middleware ensures every database query is filtered by the correct company ID. If this filter is missing, it throws an “Unsafe Data Query” error to prevent fetching the wrong data.
* **Layer 3:** The system uses SQL joins to make sure the requested data belongs to the same user who made the API call, verifying the request matches the user’s own records.
* **Layer 4:** The database results are converted into safe, user-friendly models so that sensitive internal details are never shown to the public.
* **Layer 5:** A final response filter double-checks that the returned data belongs to the requesting user and removes anything that shouldn’t be visible, adding an extra safety check before sending the response.

## Best Practices for API Security 
Various best practices for securing APIs against cyberattacks are as follows: 

**1. Always use HTTPS (SSL/TLS)** → 
Make sure your API uses HTTPS so all data is encrypted during transfer, keeping it safe from eavesdropping.

**2. Validate and sanitize inputs** → 
Check all incoming data on the **server side**, and clean it to remove any harmful code or scripts before processing.

**3. Use authentication & authorization tokens** → 
Issue secure tokens (like OAuth2 or JWT) for each user or app, and verify them on every request.

**4. Implement rate limiting** → 
Set limits on how many API calls a user can make in a certain time to prevent abuse or DDoS attacks.

**5. Enable multi-factor authentication (MFA)** → 
Require more than just a password for access to sensitive API functions.

**6. Use API gateways** → 
Place an API gateway in front of your APIs to authenticate traffic, monitor usage, and block suspicious requests.

**7. Monitor & log activities** → 
Keep detailed access logs, monitor them for unusual patterns, and sanitize logs to avoid sensitive data exposure.

**8. Apply least privilege principle** → 
Give each user or system **only** the access they need — no extra permissions.

**9. Protect against injection attacks** → 
Always use parameterized queries and avoid dynamic SQL to block SQL injection attempts.

**10. Regularly test & audit APIs** → 
Run automated scans, penetration testing, and security reviews to find and fix vulnerabilities quickly.

## Best Practices for Securing Webhooks 
Various best practices for securing webhooks are as follows: 

**1. Always Use HTTPS** → 
Encrypt webhook traffic so no one can read or modify the data while it’s being sent.

**2. Authenticate Every Request** → 
Use API keys, shared secrets, or mutual TLS to ensure only trusted sources send data.

**3. Sign and Verify Payloads** → 
Use HMAC-based signatures to confirm that webhook data hasn’t been tampered with.

**4. Whitelist Source IPs** → 
Accept webhook requests only from known and trusted IP addresses.

**5. Prevent Replay Attacks** → 
Track unique event IDs and request timestamps so the same data can’t be processed twice.

**6. Limit Payload Size** → 
Set strict size limits to block attackers from sending huge payloads that could crash your server.

**7. Use Rate Limiting** → 
Restrict how often webhooks can be sent to avoid abuse or denial-of-service attacks.

**8. Keep Processing Idempotent** → 
Make sure running the same webhook twice doesn’t duplicate actions or cause errors.

**9. Avoid Sending Sensitive Data** → 
Never send passwords or other confidential info via webhooks—use secure APIs instead.

**10. Log and Monitor Activity** → 
Keep detailed logs of webhook requests for debugging, incident response, and threat detection.
