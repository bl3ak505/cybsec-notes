# ⚠️ Sniffing Technique: MAC Attacks 

## MAC Address
A **MAC address** is a unique identifier assigned to every device on a network. It’s linked to the physical network adapter (like a network card) and helps identify devices in communication, especially in technologies like Ethernet.

A MAC address is **48 bits long**, shown as **12 hexadecimal digits**, usually split into groups (e.g., D4-BE-D9-14-C8-29). The **first 6 digits** (D4BED9) identify the manufacturer (like Dell), and the **last 6 digits** (14C829) are the device's unique serial number. MAC addresses are essential for data transfer in network systems and help locate devices on the network accurately.

## CAM Table
A **CAM table** (Content Addressable Memory table) is used by network switches to store **MAC addresses** along with their corresponding **ports and VLAN info**. When data is sent across the network, the switch checks the destination MAC in the CAM table and forwards the data only to the correct port.

This targeted delivery is faster and more secure compared to a hub, which sends data to all devices. The CAM table helps make network communication efficient and reduces unnecessary traffic.

![image](https://github.com/user-attachments/assets/f4aae809-d656-43c3-927a-dbf3fb78959d)

### How CAM Works
**CAM (Content Addressable Memory) Table** is used by Ethernet switches to store MAC addresses and their corresponding ports, helping data reach the correct device. However, the CAM table has a size limit. If an attacker floods it with too many fake MAC addresses, the switch can’t store them all and starts acting like a hub—sending data to all connected devices instead of just one.

This makes it easy for attackers connected to the same network to **sniff** or capture sensitive data meant for others. This is a common method used in network attacks to steal information.

Refer to the diagrams of the working of the CAM table. Three machines are shown: Machine A, Machine B, and Machine C, each holding MAC addresses A, B, and C. Machine A, holding the MAC address A, wants to interact with Machine B.

Machine A broadcasts an ARP request to the switch. The request contains the IP address of the target machine (Machine B), along with the source machine’s (Machine A) MAC and IP addresses. The switch then broadcasts this ARP request to all the hosts in the network and waits for the reply.

![image](https://github.com/user-attachments/assets/36a1abe7-9944-410f-8b45-07090f480fff)

Machine B possesses the target/destination IP address, so it sends an ARP reply along with its MAC address. The CAM table stores this MAC address along with the port on which this machine is connected.

![image](https://github.com/user-attachments/assets/c7e82f5f-1f80-4660-bf42-0a51bf4996f6)

Now the connection is successfully established, and Machine A forwards the traffic to Machine B, while Machine C is unable to see the traffic flowing between them.

![image](https://github.com/user-attachments/assets/922e2f29-701e-4baf-b6d6-ad7f466e33f3)

### What Happens when a CAM Table is Full?
When a **CAM table** (which stores MAC addresses on a switch) becomes **full**, it can’t learn new addresses. This usually happens during a **MAC flooding attack**, where an attacker sends a large number of fake MAC addresses to overload the table.

Once full, the switch enters a **learning mode** and starts **broadcasting all traffic to all ports**, like a hub. This lets the attacker **capture and monitor network traffic** between other devices. The attack can also affect nearby switches by filling up their CAM tables too, making it easier to intercept sensitive data.

## MAC Flooding 
**MAC Flooding** is a network attack where the hacker overloads a switch with fake MAC addresses to break its normal functioning. Normally, a switch uses a **CAM table** to forward data only to the correct device, making it more secure than a hub.

But since the CAM table has limited space, the attacker floods it with fake entries. When the table gets full, the switch enters **fail-open mode** and starts sending data to all connected devices—just like a hub. The attacker then puts their system in **promiscuous mode** to capture all the traffic, allowing them to **sniff sensitive information** from the network.

### Mac Flooding Switches with macof 
🔗Source: [https://monkey.org]

**macof** is a Linux/Unix tool from the **dsniff** suite used to flood a network switch with fake MAC and IP addresses. This attack fills up the switch’s **CAM table** (which stores MAC-to-port mappings) by sending thousands of fake entries—up to **131,000 per minute**.

Once the table is full, the switch can no longer manage traffic properly and starts acting like a hub, broadcasting data to all ports. This lets an attacker **sniff or capture network traffic** that would normally be hidden. It’s a common technique to bypass switch-level isolation and steal data on local networks.

## Switch Port Stealing 
**Switch Port Stealing** is a sniffing technique where the attacker tricks a network switch into sending data meant for another device to their own port. This is done by flooding the switch with fake ARP packets using the victim’s MAC address. The switch gets confused and keeps switching the MAC address between the attacker’s and victim’s ports.

If the attacker is fast enough, they temporarily "steal" the victim’s switch port and can sniff the data. By sending ARP requests and waiting for replies, the attacker confirms the victim is active and uses this to continue intercepting packets meant for the target device.

Assume that there are three machines in a network: Host A, the target’s Host B, and the attacker’s Host C.
| Machine | MAC Address | IP Address | Ports |
|---|---|---|---|
| Host A | aa-bb-cc-dd-ee-ff | 10.0.0.1| Port A |
| Host B | bb-cc-dd-ee-ff-gg | 10.0.0.2 | Port B |
| Host C | cc-dd-ee-ff-gg-hh | 10.0.0.3 | Port C |

The switch's ARP cache and MAC table contain the following values:

**MAC Table**
| Vlan | MAC Address | Type | Learn | Age | Ports |
|---|---|---|---|---|---|
| 255 | Host A | aa-bb-cc-dd-ee-ff | 10.0.0.1 | 0 | Port A |
| 5 | Host B | bb-cc-dd-ee-ff-gg | 10.0.0.2 | 0 | Port B |
|5 | Host C | cc-dd-ee-ff-gg-hh | 10.0.0.3 | 0 | Port C |

**ARP Cache**
| IP | MAC |
|---|---|
| 10.0.0.1 | aa-bb-cc-dd-ee-ff |
| 10.0.0.2 | bb-cc-dd-ee-ff-gg |
| 10.0.0.3 | cc-dd-ee-ff-gg-hh |

- **Step 1:** - The attacker pretends to be the target (Host B) by **spoofing both its IP and MAC address**.

  | Machine | MAC Address | IP Address | Ports |
  |---|---|---|---|
  | Host A | aa-bb-cc-dd-ee-ff | 10.0.0.1 | Port A |
  | Host B | bb-cc-dd-ee-ff-gg | 10.0.0.2 | Port B |
  | Host C | *bb-cc-dd-ee-ff-gg* | 10.0.0.2 | Port C |

- **Step 2:** - The attacker turns on **promiscuous mode** on their network card, allowing it to **capture all traffic** on the network.

- **Step 3:** - Host A wants to talk to Host B, so it sends an **ARP request** asking,
  *"Who has IP 10.0.0.2?"*

- **Step 4:** - The **network switch broadcasts** this ARP request to all devices.

- **Step 5:** - Before the real Host B can reply, the **attacker quickly sends a fake ARP reply**, claiming,
  *"I am 10.0.0.2, and here’s my MAC address."*

  *(The attacker may slow down Host B using a DoS attack to respond first.)*

- **Step 6:** - The **switch believes** the attacker’s reply and **updates its ARP table** with the fake MAC and IP info.

  | IP | MAC |
  |---|---|
  | 10.0.0.1 | aa-bb-cc-dd-ee-ff |
  | 10.0.0.2 | bb-cc-dd-ee-ff-gg |
  | *10.0.0.2* | *bb-cc-dd-ee-ff-gg* |

- **Step 7:** - Now, the switch **thinks Host B is connected to the attacker’s port** and sends all traffic for Host B to the attacker.

  | Vlan | MAC Address | Type | Learn | Age | Ports |
  |---|---|---|---|---|---|
  | 255 | Host A | aa-bb-cc-dd-ee-ff | 10.0.0.1 | 0 | Port A |
  | 5 | Host B | bb-cc-dd-ee-ff-gg | 10.0.0.2 | 0 | Port B |
  |5 | Host C | *bb-cc-dd-ee-ff-gg* | *10.0.0.2* | 0 | Port C |

- **Step 8:** - The attacker **receives all the data** meant for Host B and can sniff sensitive information.

✅ **Result:**
The attacker **steals Host B's traffic** by fooling the switch, and can **spy on all data** sent to Host B.

## How to Defend against MAC Attacks
To **defend against MAC attacks**, network devices use a feature called **port security**. This limits which MAC addresses can access a specific port. Only devices with approved MAC addresses are allowed to send data through that port.

A **security violation** happens if:

* The number of allowed MAC addresses is already full.
* A device with an unknown MAC address tries to connect.

**Port Security** is a network switch feature that helps protect against MAC flooding attacks by limiting how many devices can connect to a single switch port.

You can add secure MAC addresses in three ways:
1. **Manually configure** all MAC addresses.
2. **Let the switch learn them dynamically** from connected devices.
3. **Combine both**—set some manually and let the rest be learned.

Once the limit is reached, the switch blocks other devices, locks the port, and sends an **SNMP alert** to notify administrators.


### Configuring Port Security on Cisco Switch 
🔗Source: [https://www.cisco.com]

```bash
Switch> configure terminal

# → Step 1: Select the port to configure
Switch(config)> interface gigabitethernet 3/1

# → Step 2: Set the port as an access port               
Switch(config-if)> switchport mode access

# → Step 3: Enable port security                   
Switch(config-if)> switchport port-security

# → Step 4: Sets the maximum number of secure MAC addresses for the interface. The range is 1 to 3072; the default is 1.                
Switch(config-if)> switchport port-security maximum <value>

# → Step 5: Sets the violation mode, the action to be taken when a security violation {restrict | shutdown} is detected.        
Switch(config-if)> switchport port-security violation {restrict | shutdown}

# → Step 6: Limit bad MAC rate 
Switch(config-if)> switchport port-security limit rate invalid-source-mac

# → Step 7: Set allowed MAC  
Switch(config-if)> switchport port-security mac-address <mac-address>

# → Step 8: Auto-learn sticky MACs     
Switch(config-if)> switchport port-security mac-address sticky

# → Step 9: Exit to EXEC mode             
Switch(config-if)> end                                        

# → Step 10: Verify configuration
Switch> show port-security address
```

#### Additional Optional Commands:
```bash
# → Set custom MAC limit
Switch(config-if)> switchport port-security maximum 3

# → Set MAC aging time in minutes        
Switch(config-if)> switchport port-security aging time 2

# → Age MACs only on inactivity      
Switch(config-if)> switchport port-security aging type inactivity

# → SNMP trap rate control  
Switch(config)> snmp-server enable traps port-security trap-rate 5  
```
