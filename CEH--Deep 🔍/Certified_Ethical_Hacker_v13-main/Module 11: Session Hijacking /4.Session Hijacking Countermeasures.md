# 🛡️ Session Hijacking Countermeasures
**Session Hijacking** is a serious threat where attackers take control of a user’s active session, risking identity theft, data loss, and fraud. Since all TCP/IP-based networks are vulnerable, it's important to use **best practices** to stay protected.

This section discusses session hijacking detection methods, session hijacking detection tools, various countermeasures to combat session hijacking attacks, and approaches causing vulnerability to session hijacking and their preventative solutions such as IP Security (IPsec).

---
## Session Hijacking Detection Methods 
Session hijacking attacks are exceptionally difficult to detect, and users often overlook them unless the attacker causes severe damage. The following are some symptoms of a session hijacking attack: 
- A burst of network activity for some time, which decreases the system performance
- Busy servers resulting from requests sent by both the client and hijacker 

### Methods to detect session hijacking

#### Manual Method
The manual method involves the use of packet sniffing software such as Wireshark and SteelCentral Packet Analyzer to monitor session hijacking attacks. The packet sniffer captures packets in transit across the network, which is then analyzed using various filtering tools. 

##### **Forced ARP Entry**

A forced ARP entry involves replacing the MAC address of a compromised machine in the ARP cache of the server with a different one in order to restrict network traffic to the compromised machine. A forced ARP entry should be performed in the case of the following: 
- Repeated ARP updates
- Frames sent between the client and server with different MAC addresses
- ACK storms

#### Automatic Method
The automatic method involves the use of an intrusion detection systems (IDS) and intrusion prevention systems (IPS) to monitor incoming network traffic. If the packet matches any of the attack signatures in the internal database, the IDS generates an alert, whereas the IPS blocks the traffic from entering the database.

## Protecting against Session Hijacking 
1. **Use HTTPS for All Sessions** - 
   Always send authentication cookies and session data over **encrypted HTTPS connections** to prevent sniffing.

2. **Generate Secure Session IDs** - 
   Only the server should generate **long, random session IDs** after login. Avoid accepting IDs from clients.

3. **Enable Session Timeouts** - 
   Automatically log users out after a period of **inactivity** to reduce the window for hijacking.

4. **Avoid Putting Session ID in URLs** - 
   Never include session IDs in the URL or query string — use cookies instead to keep them hidden.

5. **Use Secure, HttpOnly, and SameSite Cookies** - 
   Protect cookies with `Secure`, `HttpOnly`, and `SameSite` attributes to stop access by JavaScript and prevent cross-site attacks.

6. **Implement Multi-Factor Authentication (MFA)** - 
   Add an extra layer like **OTP or biometrics** so even if the session is stolen, it can't be misused.

7. **Bind Sessions to IP Address** - 
   Lock each session to the user's **IP address** to block access if it changes unexpectedly.

8. **Monitor Session Activity** - 
   Track unusual session behavior like **multiple logins from different locations** to catch hijacks early.

9. **Educate Users to Log Out** - 
   Teach users to always **log out**, especially on public or shared systems, to clear session data.

10. **Use VPNs and IPsec for Remote Access** - 
    Use **encrypted connections** like VPNs or IPsec to protect session data from being intercepted over networks.

## Web Development Guidelines to Prevent Session Hijacking 
An attacker usually hijacks a session by exploiting the vulnerabilities in mechanisms used for session establishment. Web developers often ignore security. During the development process, they should consider the following guidelines to minimize/eliminate the risk of session hijacking. 

1. **Use Strong, Random Session IDs** - 
   Generate session keys using long and unpredictable strings so attackers can’t easily guess them.

2. **Regenerate Session ID After Login** - 
   Always create a new session ID once the user logs in to prevent session fixation attacks.

3. **Enforce HTTPS Everywhere** - 
   Make sure every page of your site uses HTTPS to securely encrypt all user communication.

4. **Use Secure & HTTPOnly Cookies** - 
   Mark cookies as `Secure` (only sent over HTTPS) and `HTTPOnly` (inaccessible via JavaScript) to protect session data.

5. **Expire Sessions on Logout** - 
   Immediately destroy sessions once the user logs out to avoid misuse.

6. **Short Session Lifespan** - 
   Reduce how long a session or cookie stays active to limit exposure if hijacked.

7. **IP & User-Agent Check** - 
   Verify that each session request comes from the same IP address and browser type that started it.

8. **Use Risk-Based Authentication** - 
   For sensitive actions, ask users to re-authenticate or verify their identity again.

9. **Redirect HTTP to HTTPS** - 
   Automatically switch any HTTP requests to HTTPS to ensure secure communication.

10. **Avoid Sessions for Unauthenticated Users** - 
    Don't create sessions for users unless they've logged in or it's absolutely necessary.

## Web User Guidelines to Prevent Session Hijacking 
The following are some guidelines for web users to defend against session hijacking.

1. **Always Use HTTPS** - 
   Choose secure websites that use `HTTPS` to encrypt your data during transmission.

2. **Log Out Properly** - 
   Always click the logout button after using any sensitive service, especially on shared devices.

3. **Avoid Public Wi-Fi for Sensitive Work** - 
   Never access banking or private accounts over open or public Wi-Fi networks.

4. **Disable Auto-Connect to Wi-Fi** - 
   Turn off automatic connection to open Wi-Fi networks to avoid connecting to malicious hotspots.

5. **Clear Cookies and History** - 
   After completing sensitive transactions, clear browser history, cache, and cookies to remove session data.

6. **Don't Click Unknown Links** - 
   Avoid clicking links received through emails or messages unless you trust the source.

7. **Keep Browser and OS Updated** - 
   Regularly update your operating system, browser, and plugins to patch security holes.

8. **Use Anti-Virus and Keep It Updated** - 
   Install a trusted antivirus and keep it updated to block malware that may steal session info.

9. **Use Incognito Mode on Shared Devices** - 
   On public or shared computers, use private/incognito mode to avoid storing session data.

10. **Avoid Saving Passwords in Browsers** - 
    Don’t let browsers save your passwords; use a secure password manager instead.

## Approaches to Prevent Session Hijacking 

### HTTP Strict Transport Security (HSTS) 
**HTTP Strict Transport Security (HSTS)** is a security feature that forces websites to use **HTTPS** instead of **HTTP**. It protects users from **man-in-the-middle (MITM)** attacks by automatically upgrading all connections to secure, encrypted HTTPS. This ensures that all data sent between the browser and server is safe and comes from a trusted source.

<img width="481" height="160" alt="image" src="https://github.com/user-attachments/assets/06f39112-67db-4649-9765-97c99a69cc4f" />

### Token Binding 
**Token Binding** is a security method that protects users from **session hijacking**. Normally, when you log into a website, a session ID (token) is used to keep you connected. If an attacker steals this token, they can pretend to be you.

With token binding, the user's device creates a **unique public–private key pair** for each server connection. It signs the session using its **private key**, and sends the **signature** with its **public key** to the server. The server checks the signature to confirm it's from the real user. Since the private key never leaves the user's device, attackers can't reuse the session even if they capture the data. A new key pair is made for every connection, keeping sessions secure.

## Approaches to Prevent MITM Attacks 
Man-in-the-middle (MITM) attacks are the most common type of attack, wherein the attackers intercept the traffic between two endpoints. The victim may not realize the effect of this attack, because it is mostly passive in nature. Because the detection of MITM attacks is difficult, they can only be prevented using various measures. The following are some approaches to prevent MITM attacks: 

* **DNS over HTTPS** - 
**DNS over HTTPS (DoH)** is a secure version of the DNS protocol that hides your DNS queries by sending them through an encrypted **HTTPS connection on port 443**. This prevents attackers, ISPs, or third parties from spying on your web activity.

Unlike traditional DNS, which sends the full domain name openly, DoH only sends the needed part through a secure tunnel, keeping your browsing private. It also protects against **MITM attacks** and **session hijacking**. Modern browsers like **Chrome**, **Mozilla Firefox**, and **Edge** support DoH, with Mozilla using it by default in the U.S. since 2020.

<img width="791" height="242" alt="image" src="https://github.com/user-attachments/assets/b66483b0-063e-4aae-95c6-8c345cd5ba28" />

* **WPA3 Encryption** - 
  Protects wireless traffic using strong encryption, making it harder for attackers to break into Wi-Fi networks.

* **VPN (Virtual Private Network)** - 
  Creates a secure tunnel over the internet, encrypting all data between users and preventing attackers from spying.

* **Two-Factor Authentication (2FA)** - 
  Adds an extra security layer beyond passwords, stopping attackers from accessing accounts even if they steal credentials.

* **Password Manager** - 
  Helps generate and store strong, unique passwords securely, reducing the risk of password theft during MITM attacks.

* **Zero-Trust Principles** - 
  Verifies every user and device—inside or outside the network—before granting access, reducing the chance of unauthorized intrusion.

* **Public Key Infrastructure (PKI)** - 
  Uses trusted digital certificates to verify identities, ensuring communication happens only between verified parties.

* **Network Segmentation** - 
  Splits a large network into smaller, isolated sections so that attackers can’t easily intercept traffic or move across systems.

## IPsec 
**IPsec (Internet Protocol Security)** is a set of protocols used to securely send data over IP networks. It provides encryption, authentication, and data integrity to protect communications in IPv4 and IPv6. IPsec is commonly used in **VPNs** and **remote access** connections. It works in **transport** and **tunnel** modes and requires both devices to share a public key. IPsec settings can be managed through **Group Policy** in Active Directory for better control across networks.

The security services offered by IPsec include the following:
- Rejection of replayed packets (a form of partial sequence integrity)
- Data confidentiality (encryption)
- Access control
- Connectionless integrity
- Data origin authentication
- Data integrity
- Limited traffic-flow confidentiality
- Network-level peer authentication
- Replay protection

At the IP layer, IPsec provides all the above-mentioned services, offering the protection of IP and/or upper-layer protocols such as TCP, UDP, ICMP, and Border Gateway Protocol (BGP). 

### Components of IPsec
- **IPsec driver:** Software that performs protocol-level functions required to encrypt and decrypt packets.
- **Internet Key Exchange (IKE):** An protocol that produces security keys for IPsec and other protocols.
- **Internet Security Association and Key Management Protocol (ISAKMP):** Software that allows two computers to communicate by encrypting the data exchanged between them.
- **Oakley:** A protocol that uses the Diffie–Hellman algorithm to create a master key and a key that is specific to each session in IPsec data transfer.
- **IPsec Policy Agent:** A service included in Windows OS that enforces IPsec policies for all the network communications initiated from that system.

**The following are the steps involved in the IPsec process.**
- A consumer sends a message to a service provider.
- The consumer's IPsec driver attempts to match the outgoing packet's address or the packet type against the IP filter.
- The IPsec driver notifies ISAKMP to initiate security negotiations with the service provider.
- The service provider's ISAKMP receives the security negotiation request.
- Both principles initiate a key exchange, establishing an ISAKMP Security Association (SA) and a shared secret key.
- Both principles discuss the security level for the information exchange, establishing both IPsec SAs and keys.
- The consumer's IPsec driver transfers packets to the appropriate connection type for transmission to the service provider
- The provider receives the packets and transfers them to the IPsec driver.
- The provider's IPsec uses the inbound SA and key to check the digital signature and begin decryption.
- The provider's IPsec driver transfers decrypted packets to the OSI transport layer for further processing.

### Modes of IPsec
The configuration of IPsec involves two different modes: the **tunnel mode** and **transport mode**. These modes are associated with the functions of two core protocols: the **Encapsulation Security Payload (ESP)** and **Authentication Header (AH)**. The model selection depends on the requirements and implementation of IPsec. 

#### Transport Mode 
**Transport Mode** in IPsec encrypts only the **data part (payload)** of an IP packet, not the IP header. This keeps the routing information visible while protecting the actual data. It's mainly used for **end-to-end communication** between two devices. In this mode, encryption and authentication happen between the connected hosts, and it works well with **NAT**, making it useful for secure **VPN connections**.

<img width="481" height="280" alt="image" src="https://github.com/user-attachments/assets/696043da-0ddd-425e-9876-d4b6b391c806" />

#### Tunnel Mode
**Tunnel Mode** in IPsec is used to securely send data over the internet by encrypting the **entire IP packet**, including both the data and the original IP header. This encrypted packet is then placed inside a new IP packet with a new header.

Tunnel mode offers **stronger security** than transport mode and is commonly used for creating **VPNs** between networks, hosts, or remote users. It works well with **NAT** and supports NAT traversal.

* With **ESP**, tunnel mode encrypts and can authenticate the full inner IP packet.
* With **AH**, it authenticates the entire inner IP packet and parts of the outer header.

This mode is best for secure communication between **gateways** or **hosts and gateways**.

### IPsec Architecture
**IPsec Architecture** provides security at the **network layer**, allowing the use of different protocols and encryption algorithms to protect data. It offers key services like **access control**, **authentication**, **data integrity**, **anti-replay protection**, and **confidentiality**. IPsec uses two main protocols—**AH (Authentication Header)** and **ESP (Encapsulating Security Payload)**—along with key management methods to secure communication between systems.

<img width="604" height="439" alt="image" src="https://github.com/user-attachments/assets/8525417f-df65-4f89-b729-b6c0f0279dfc" />

* **Authentication Header (AH)** - 
  Provides data integrity and verifies that the data came from the claimed sender. It can also block replay attacks.

* **Encapsulating Security Payload (ESP)** - 
  Offers the same protections as AH but also **encrypts the data** to keep it confidential.

* **IPsec Domain of Interpretation (DOI)** - 
  Defines the formats and rules used to exchange security info (like encryption algorithms and policies) during IPsec negotiations.

* **ISAKMP (Internet Security Association and Key Management Protocol)** - 
  Handles key exchange and sets up secure connections by combining authentication, key management, and security rules (Security Associations).

* **Policy** - 
  IPsec policies decide **what data to protect, when to protect it, and how**. You can apply these at different levels (system, domain, organization).

### IPsec Authentication and Confidentiality 
IPsec uses two different security services for authentication and confidentiality.

* **Authentication Header (AH):**
  AH ensures that the data has not been changed and confirms where it came from. It **authenticates** parts of the IP packet and **protects against replay attacks**. However, it **does not encrypt** the data, so it **does not provide confidentiality**.

* **Encapsulation Security Payload (ESP):**
  ESP adds **encryption**, so it **protects data confidentiality**. It also offers **authentication, integrity, and anti-replay** services like AH. In **transport mode**, it protects only the **payload**. In **tunnel mode**, it secures both the **payload and IP header**.

<img width="674" height="478" alt="image" src="https://github.com/user-attachments/assets/80773a99-69ae-4c73-828c-45871c0212c2" />

> ✅ Use **AH** when you only need **authentication**.
> 
> ✅ Use **ESP** when you need **both encryption and authentication**.
