# 📱 Application-Level Session Hijacking
**Application-Level Session Hijacking** is a method where attackers take over a user’s active session on a website by stealing or guessing their **session ID**. This allows them to access the user’s account without logging in.

Attackers can **steal** session IDs using tools like Wireshark or by accessing system files or memory. They can **guess** session IDs if the server uses weak ID generation. Or, they can use **brute force** to try many possible session ID combinations quickly until one works. This attack often follows network-level hijacking and targets **HTTP sessions** to gain full control of a user’s session.

![image](https://github.com/user-attachments/assets/8a74a769-d26d-42bf-9e7f-bf2e692233b1)

As shown in the above figure, a legitimate user connects to a server with session ID VW30422101522507. Employing various combinations such as VW30422101518909 and VW30422101520803, an attacker attempts to brute force the session ID in the hope of eventually arriving at the correct session ID. Once the attacker obtains the correct session ID, they gain complete access to the user’s data and can perform operations on behalf of the legitimate user. 

> Note: A session ID brute-forcing attack is known as a session prediction attack if the predicted range of values for a session ID is very small.

A session token can be compromised in various ways:
```
- Session sniffing			- Session replay attack
- Predictable session token		- Session fixation attack
- Man-in-the-middle (MITM) attack	- CRIME attack
- Man-in-the-browser attack		- Forbidden attack
- Cross-site scripting (XSS) attack	- Session donation attack
- Cross-site request forgery attack	- PetitPotam hijacking
```

## Compromising Session IDs Using Sniffing 
**Compromising Session IDs Using Sniffing** is an attack where the hacker captures a user’s session ID to hijack their web session. After a user logs in, the server gives them a unique session ID, which can be sent through cookies, URLs, or HTTP body.

Using tools like **Wireshark** or **Riverbed Packet Analyzer Plus**, an attacker can sniff unencrypted traffic between the user and server. If they find the session ID in the captured data, they can send it to the server before the real user does. This tricks the server into thinking the attacker is the real user, giving them full access to the session.

## Compromising Session IDs by Predicting Session Token 
**Compromising Session IDs by Predicting Session Token** is a technique where attackers guess or predict session IDs to hijack a user's session and bypass login. If the web application uses weak algorithms to generate session IDs, attackers can analyze patterns in those IDs—either manually or using tools—to figure out how they’re created.

By collecting multiple valid session IDs within a short time, the attacker looks for similarities and patterns. Once the structure and algorithm are understood, the attacker can predict or brute-force a valid session ID to gain unauthorized access, impersonate users, and perform malicious actions on their behalf.

---
## How to Predict a Session Token 
Most web servers generate session IDs using custom algorithms or a predefined pattern that might simply increase static numbers, whereas others use more complex procedures, such as factoring in time and other computer-specific variables. Thus, attackers can identify session IDs generated in the following manner: 

### Analyzing Token Patterns

* **Sequential Tokens**
  Some session IDs increase in order like 1001, 1002, 1003. If attackers spot the pattern, they can guess the next token easily.

  ![image](https://github.com/user-attachments/assets/7b5f4ce2-95b7-484c-9a3a-dceae71f2c22)

* **Timestamp-Based Tokens**
  If tokens include a timestamp (e.g., `user123-20240709T1600`), attackers who know the format and current time can predict future tokens.

  ![image](https://github.com/user-attachments/assets/1cb8cba7-556a-4c11-8bbd-443624bce585)

### Brute-Force Attacks

* **Small Token Space**
  If there are limited token possibilities (e.g., 0000–9999), attackers can try every option using a script to find a valid session.

  ![image](https://github.com/user-attachments/assets/947e6f1d-6685-4f03-8934-57d821485537)

* **No Rate-Limiting**
  Without restrictions on how many guesses are allowed per minute, attackers can keep trying thousands of tokens quickly without being blocked.


### Weak Random Number Generators

* **Predictable PRNG**
  If a weak random generator is used (like one that starts with system time), attackers can recreate the same tokens if they figure out the seed or method.


These flaws make session prediction easier and can lead to **unauthorized access to user accounts or sessions**. Always use **strong randomness, long tokens, and rate-limiting** to protect session IDs.

Now, the attacker can mount an attack through the following steps.
- The attacker acquires the current session ID and connects to the web application.
- The attacker implements a brute-force technique or calculates the next session ID.
- The attacker modifies the current value in the cookie/URL/hidden form field and assumes the next user’s identity.

## Compromising Session IDs Using Man-in-the-Middle/Manipulator-in-the-Middle Attack
**Man-in-the-Middle (MITM) attacks** allow hackers to secretly intercept and control communication between a user and a server. In this attack, the attacker splits the connection into two parts—one between the **client and attacker**, and another between the **attacker and server**. This lets the attacker read, change, or inject fake data into the communication. When targeting **HTTP sessions**, the goal is to steal or manipulate session IDs to gain unauthorized access to user accounts.

## Compromising Session IDs Using Man-in-the-Browser/Manipulator-in-the-Browser Attack
**Man-in-the-Browser (MitB)** or **Manipulator-in-the-Browser** is a type of attack where a **Trojan** hides inside the web browser and secretly changes data between the browser and security tools. Unlike traditional MITM attacks, this one happens **inside the browser itself**, making it invisible to users and websites.

The attacker uses the Trojan to change web pages, fake transactions, or steal session IDs—mainly targeting **online banking**. Even if strong security like **SSL, PKI**, or **two-factor authentication** is used, the attack still works because everything looks normal to both the user and the system. The main goal is usually **financial theft**.

### Steps to Perform Man-in-the-Browser Attack: 
1. **Trojan Infection** - 
   The attacker installs a Trojan that infects the operating system or a software application.

2. **Malicious Browser Extension Installed** - 
   The Trojan adds malicious code (as an extension) to the browser’s configuration files.

3. **Extension Activated on Browser Restart** - 
   When the user restarts the browser, the malicious extension loads automatically.

4. **Extension Watches Page Visits** - 
   The extension monitors each webpage the user visits.

5. **Target Website Identified** - 
   If the visited website matches a known target (like a bank), the extension activates.

6. **User Logs in Normally** - 
   The user logs in to the legitimate website securely.

7. **Event Handler Triggers** - 
   The extension detects specific pages or buttons and prepares to capture actions.

8. **Form Data Captured and Modified** - 
   When the user submits a form (e.g., sending money), the extension changes the form data using the DOM (Document Object Model).

9. **Modified Data Sent to Server** - 
   The altered data is sent to the server without the user’s knowledge.

10. **Server Accepts the Data** - 
    The server accepts the modified form data as legitimate.

11. **Server Sends Back Receipt** - 
    The server completes the transaction and sends a receipt.

12. **Fake Receipt Shown to User** - 
    The browser displays a fake receipt with the user’s original input.

13. **User Gets Tricked** - 
    The user thinks the correct transaction was processed — but the attacker changed it silently.

## Compromising Session IDs Using Client-side Attacks 
**Compromising Session IDs Using Client-side Attacks** happens when attackers target weak spots in client applications like web browsers, Java, or unprotected websites. These attacks trick users—often through emails or malicious links—into connecting with harmful servers. When the client interacts with these servers, the attacker can steal session IDs and gain unauthorized access to user accounts. If there's no connection between the client and a server, the attack can’t happen. Apps like FTP clients or instant messengers become vulnerable when they auto-connect to remote servers, making them easy targets for such attacks. The following client-side attacks can be used to compromise session IDs. 

- **Cross-site scripting (XSS):** XSS enables attackers to inject malicious client-side scripts into web pages viewed by other users.
- **Malicious JavaScript codes:** An attacker can embed in a web page a malicious script that does not generate any warning but captures session tokens in the background and sends them to the attacker.
- **Trojans:** A Trojan horse can change the proxy settings in the user’s browser to send all sessions through an attacker’s machine.

## Compromising Session IDs Using Client-side Attacks: Cross-site Script Attack (XSS) 
**Cross-site Scripting (XSS)** is a client-side attack where hackers insert malicious scripts into vulnerable web pages. When a user visits such a page or clicks a crafted link, the malicious script runs in their browser without their knowledge.

This script can **steal session IDs**, cookies, or personal data, and even redirect users or run harmful code. For example, using code like `<SCRIPT>alert(document.cookie);</SCRIPT>`, an attacker can view a user’s session ID. They can then use that stolen ID to hijack the user’s session and access their account without needing a password.

## Compromising Session IDs Using Client-side Attacks: Cross-site Request Forgery Attack
**Cross-site Request Forgery (CSRF)** is a type of attack where hackers trick users into performing actions on a trusted website without their knowledge. If the user is already logged in, the attacker sends a fake form or link with malicious data. When the user unknowingly submits it, the website believes it's a legitimate request and processes it—like changing account details or making purchases. Unlike XSS, which tricks the user, **CSRF tricks the website into trusting the user's browser**.

A CSRF attack involves the following steps.
1. The attacker hosts a web page with a form that appears legitimate. This page already contains the attacker’s request.
2. A user, believing the form to be the original, enters a login and password.
3. Once the user completes the form, that page is submitted to the real site.
4. The real site’s server accepts the form, assuming that it was sent by the user based on the authentication credentials.

In this manner, the server accepts the attacker’s request.

## Compromising Session IDs Using Session Replay Attacks 
In a session replay attack, the attacker captures the authentication token of a user by listening to a conversation between the user and server. Once the authentication token is captured, the attacker replays the authentication request to the server with the captured authentication token to dodge the server; consequently, they gain unauthorized access to the server. A session replay attack involves the following steps.
- The user establishes a connection with the web server.
- The server asks the user for authentication information as identity proof.
- The user sends authentication tokens to the server. In this step, an attacker captures the authentication token of the user by eavesdropping on the conversation between the user and server.
- Once the authentication token is captured, the attacker replays the request to the server with the captured authentication token and gains unauthorized access to the server.

## Compromising Session IDs Using Session Fixation 
**Session Fixation** is a type of **session hijacking** where an attacker sets a session ID in the victim’s browser *before* they log in. Instead of stealing a session after login, the attacker tricks the user into using a session ID that the attacker already knows.

This happens because some web applications let users log in without creating a new session ID. If the victim logs in using the attacker’s fixed session ID, the attacker can take over the session and access the user’s account. It’s a serious flaw in how some websites manage session IDs.

An attacker uses various techniques to perform a session fixation attack:
- Session token in the URL argument
- Session token in a hidden form field
- Session ID in a cookie

The attacker must choose a technique based on how the target web application uses session tokens. The attacker exploits the vulnerability of a server that allows a user to use a fixed session ID. Then, the attacker provides a valid session ID to a victim and lures him to authenticate themselves using that session ID. A session fixation attack has the following three phases. 
- **Session set-up phase:** In this phase, the attacker first obtains a legitimate session ID by establishing a connection with the target web server. Few web servers support the idle session time-out feature. If the target web server supports this feature, the attacker needs to send requests repeatedly to keep the established trap session ID alive.
- **Fixation phase:** In this phase, the attacker introduces the session ID to the victim's browser, thereby fixing the session.
- **Entrance phase:** In this phase, the attacker waits for the victim to log in to the target web server using the trap session ID and then enters the victim’s session.

A session fixation attack is performed through the following steps. 
1. First, the attacker establishes a legitimate connection with the target web server.
2. The target web server (e.g., http://citibank.com/) issues a session ID, say `0D6441FEA4496C2`, to the attacker.
3. The attacker sends a link with the established session ID, say `http://citibank.com/? SID=0D6441FEA4496C2`, to the victim and lures the victim to click on it to access the website.
4. The victim clicks on the link, believing it to be a legitimate link sent by the bank. This opens the server’s login page in the victim’s browser for `SID=0D6441FEA4496C2`.
5. The web server checks that the session ID `0D6441FEA4496C2` is already established and is in an active state; hence, it does not create the new session.
6. Finally, the victim enters their login credentials in the login script, and the server grants them access to the bank account.
7. At this point, knowing the session ID, the attacker can also access the victim’s bank account via `http://citibank.com/? SID=0D6441FEA4496C2`.

Because the session ID is set by the attacker before the user logged in, the user can be said to have logged into the attacker’s session

## Session Hijacking Using Proxy Servers 
An attacker lures the victim to click on a fake link, which appears legitimate but redirects the user to the attacker’s server. The attacker then forwards the request to the legitimate server on behalf of the victim and serves as a proxy for the entire transaction. Acting as a proxy, the attacker captures the session information during the interaction between the legitimate server and user.

## Session Hijacking Using CRIME Attack 
**CRIME (Compression Ratio Info-Leak Made Easy)** is a session hijacking attack that targets the **compression feature** in HTTPS and other secure protocols like **TLS** and **SPDY**. It takes advantage of how cookies are **compressed and encrypted**, making it hard to steal with simple sniffing.

In this attack, the attacker tricks the victim (using social engineering) into clicking a **malicious link**. If the victim is already logged into a secure site, the attacker uses **sniffing methods** like ARP spoofing to capture the victim’s **HTTPS traffic**.

By sending crafted HTTPS requests and analyzing the **compressed cookie length**, the attacker can guess the actual cookie value. Once they get the authentication cookie, they can **impersonate the victim**, hijack the session, and steal sensitive data like passwords or credit card numbers. Tools like **CrimeCheck** help detect if a server is vulnerable to CRIME by checking if TLS or HTTP compression is enabled.

## Session Hijacking Using Forbidden Attack 
**Session Hijacking Using Forbidden Attack** is a type of **Man-in-the-Middle (MITM)** attack that happens when a **cryptographic nonce** (a number used once) is reused during an **HTTPS session**. According to TLS rules, this nonce should be unique every time.

If the server incorrectly reuses the same nonce while using **AES-GCM encryption**, attackers can take advantage of it to hijack the secure connection. They can then generate the session keys, intercept the data, and inject fake content like **malicious scripts** or fake web forms to steal sensitive information such as **passwords** or **social security numbers**.

A forbidden attack involves the following steps.
- The attacker monitors the connection between the victim and web server and sniffs the nonce from the TLS handshake messages.
- The attacker generates authentication keys using the nonce and hijacks the connection.
- All the traffic between the victim and web server flows through the attacker’s machine.
- The attacker injects JavaScript code or web fields into the transmission towards the victim.
- The victim reveals sensitive information such as bank account numbers, passwords, and social security numbers to the attacker.

## Session Hijacking Using Session Donation Attack 
**Session Donation Attack** is a trick where the attacker gives their own session ID to the victim. The attacker first logs into a service and gets a valid session ID, then sends it to the victim using methods like **cross-site cooking**, **MITM**, or **session fixation**.

When the victim clicks the link and enters their information (like username or payment details), it gets tied to the **attacker’s account**—not their own. This way, the attacker steals sensitive data without the victim realizing anything is wrong.

A session donation attack involves the following steps.
- First, the attacker logs into a service, establishes a legitimate connection with the target web server, and deletes the stored information.
- The target web server (e.g., http://citibank.com/) issues a session ID, say `0D6441FEA4496C2`, to the attacker.
- The attacker then donates their session ID, say `http://citibank.com/?SID=0D6441FEA4496C2`, to the victim and lures the victim to click on it to access the website.
- The victim clicks on the link, believing it to be a legitimate link sent by the bank. This opens the server’s page in the victim’s browser with `SID=0D6441FEA4496C2`. Finally, the victim enters their information in the page and saves it.
- The attacker can now login as themselves and acquire the victim’s information.
