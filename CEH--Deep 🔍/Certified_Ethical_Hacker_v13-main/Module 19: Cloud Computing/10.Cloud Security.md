# 🔐 Cloud Security
There are various risks and threats associated with cloud service adoption and migrating business-critical data to third-party systems. However, following security guidelines and countermeasures strengthens the business case for cloud adoption. <br>
This section deals with various cloud standards, countermeasures, and best practices to secure data hosted in the cloud environment.

---
## Cloud Security Control Layers 
The following layers show the mapping of the cloud model to the security control model:
1. **Application Layer** → Focuses on securing cloud-based applications by following standards like OWASP, ensuring they meet business and regulatory requirements. It includes controls such as secure development practices, code scanning, web firewalls, and transaction security.

2. **Information Layer** → Protects sensitive data from unauthorized access or alteration using controls like encryption, data loss prevention (DLP), content monitoring, and database activity tracking.

3. **Management Layer** → Handles administrative cloud security tasks to ensure continuous and reliable services. Controls include identity and access management (IAM), patch management, vulnerability assessment, compliance, and system monitoring.

4. **Network Layer** → Secures cloud networks against unauthorized access and attacks using firewalls, intrusion detection/prevention systems, DDoS protection, deep packet inspection, and secure protocols like OAuth and DNSSEC.

5. **Trusted Computing** → Builds a secure and verifiable computing environment using trusted hardware, secure APIs, and audit mechanisms to ensure data integrity and reliability.

6. **Computation and Storage** → Protects cloud data and processing systems by implementing encryption, host-based firewalls, intrusion detection, backups, and data integrity monitoring to ensure service availability and trust.

7. **Physical Layer** → Secures the physical infrastructure such as data centers with fences, guards, CCTV, access control systems, and patrols to prevent unauthorized physical access and ensure asset protection.

## Cloud Security is the Responsibility of both Cloud Provider and Consumer 
Cloud security follows a shared responsibility model where both the cloud provider and the consumer play key roles. While the provider secures the underlying infrastructure, the consumer is responsible for protecting data, configurations, and user access. The level of responsibility varies across service models—like IaaS, PaaS, and SaaS—but both sides must work together to ensure a secure and well-managed cloud environment.

**Example:** An IaaS platform provider usually performs account management controls for initial system privileged users, whereas a cloud consumer controls user account management for applications deployed in the IaaS.

## Cloud Computing Security Considerations 
1. **Tailor-Made Security** → Cloud providers should customize security settings based on each client’s specific needs and requirements.
2. **Secure Multi-Tenancy** → CSPs must ensure that multiple users can safely share cloud resources without risking data leaks.
3. **Disaster Recovery Plan** → Cloud services should have strong recovery systems to restore data quickly after unexpected failures.
4. **Continuous Monitoring** → Regularly track service quality to ensure both clients and providers meet agreed performance levels.
5. **Data Integrity** → Protect stored data from tampering or corruption to maintain accuracy and trust.
6. **Fast and Reliable Services** → Cloud systems must be quick, stable, and efficient to handle new requests smoothly.
7. **Strong Encryption** → Use both symmetric and asymmetric cryptography to keep data secure during storage and transmission.
8. **Secure Operations** → Cloud operations should follow secure engineering and integration practices aligned with company policies.
9. **Load Balancing** → Distribute workloads evenly to improve performance, speed, and system stability.
10. **Physical Security** → Ensure cloud data centers are well-protected from natural disasters and unauthorized physical access.
11. **Advanced Networking** → Public clouds should use secure networks like carrier-grade systems or dedicated VPNs for safe communication.
12. **Incident Response Plan** → Have clear procedures to detect, respond to, and recover from security incidents.
13. **Role-Based Protections** → Implement controls that allow only authorized users to perform specific actions based on their roles.
14. **Threat Intelligence** → Use global threat intelligence databases to detect and defend against new cyber threats.
15. **CASB Integration** → Include Cloud Access Security Brokers for secure gateways, monitoring, and data loss prevention.
16. **Zero-Trust Principle** → Always verify users and devices before granting access, no matter where they connect from.
17. **Identity and Access Management (IAM)** → Enforce strict control over who can access cloud systems and what they can do.
18. **Role-Based Access Control (RBAC)** → Assign permissions based on job roles to ensure the principle of least privilege.
19. **Regulatory Compliance** → Follow all industry standards and legal regulations such as GDPR, ISO, or HIPAA.
20. **Backup and Recovery Testing** → Regularly test backup systems to confirm they can successfully restore data when needed.

## Placement of Security Controls in the Cloud 
Placing security controls correctly in the cloud is vital to protect against threats and reduce risks. Organizations should assess potential vulnerabilities and impacts to decide which controls are needed and where to apply them. When implemented properly within the cloud architecture, these controls strengthen defense, close security gaps, and minimize the damage caused by attacks, ensuring a more secure and resilient cloud environment.

Categories of security controls:
- **Deterrent controls** – These controls reduce attacks on the cloud system. <br>
  **Example:** A warning sign on the fence or property to inform potential attackers of adverse consequences if they proceed to attack.

- **Preventive controls** – These controls strengthen the system against incidents by minimizing or eliminating vulnerabilities. <br>
  **Example:** A strong authentication mechanism to prevent unauthorized use of cloud systems.

- **Detective controls** – These controls detect and react appropriately to occurring incidents. <br>
  **Example:** Employing IDSs, IPSs, etc. helps detect attacks on cloud systems.

- **Corrective controls** – These controls minimize the consequences of an incident by limiting the damage. <br>
  **Example:** Restoring system backups.

## Assessing Cloud Security using Scout Suite 
Attack surface discovery involves identifying all potential points of entry that an attacker can exploit to gain unauthorized access to a cloud network or virtual environment. This process includes asset identification, vulnerability assessment, access control review, network mapping, penetration testing, compliance checks, and threat analysis. Security professionals can use tools such as Scout Suite to identify potential attack surfaces in their cloud environments. 

- **Scout Suite** [] <br>
  Scout Suite is a multi-cloud security-auditing tool that helps security professionals assess the security posture of cloud environments. It leverages APIs exposed by cloud providers to gather configuration data for manual inspection and automatically highlights areas of risk. Instead of navigating through numerous pages on web consoles, Scout Suite offers a clear and concise view of the attack surface.

## Best Practices for Securing the Cloud 
Discussed below are various best practices for securing a cloud environment: 
1. **Data Protection and Backup** → Regularly back up critical data, encrypt it during storage and transfer, and ensure recovery plans are in place to prevent data loss or breaches.
2. **Strong Authentication and Access Control** → Use multi-factor authentication (MFA) and role-based access control (RBAC) to ensure only authorized users can access cloud resources.
3. **Regular Patching and Vulnerability Management** → Keep cloud systems updated with the latest patches and fix vulnerabilities quickly to block potential attacks.
4. **Network Security Monitoring** → Continuously monitor network traffic using firewalls, IDS, and IPS to detect and stop unauthorized or suspicious activities.
5. **Secure API Management** → Protect cloud APIs through authentication, encryption, and regular testing to prevent misuse or data leaks.
6. **Compliance and Auditing** → Ensure cloud vendors undergo regular security audits (like SSAE 18 Type II) and comply with security standards and regulations.
7. **Encryption and Key Management** → Use strong encryption for data and securely manage cryptographic keys throughout their lifecycle to prevent unauthorized decryption.
8. **Physical and Infrastructure Security** → Ensure the cloud provider maintains 24/7 physical security, proper isolation of storage and network resources, and strong server management.
9. **Incident Response and Notification** → Have a defined process for detecting, reporting, and responding to security breaches, including timely customer notifications.
10. **Employee Awareness and Policy Enforcement** → Enforce strict security policies, prevent credential sharing, and regularly train employees to recognize and avoid security risks.

## Best Practices for Securing AWS Cloud 
The following are some best practices for securing AWS cloud environments: 

### Basic AWS Security Practices 
1. **Categorize User Identity** → Manage access by assigning users to proper accounts, roles, and groups so permissions stay organized and secure.
2. **Use Temporary Credentials** → Avoid long-term access keys and use short-lived temporary credentials to reduce the risk of misuse.
3. **Rotate Credentials Regularly** → Change access keys, passwords, and other credentials often to prevent unauthorized access from old or leaked keys.
4. **Apply Least Privilege Principle** → Give users and services only the minimum permissions they need to perform their tasks, reducing potential damage if compromised.
5. **Enable Multi-Factor Authentication (MFA)** → Add an extra security layer that requires users to verify their identity with a code in addition to a password.
6. **Use IAM Access Analyzer** → Audit permissions and policies to find and fix overly permissive or risky access in your AWS environment.
7. **Leverage AWS Trusted Advisor** → Use AWS’s built-in tool to detect and fix security misconfigurations or noncompliance in your setup.
8. **Keep Systems Updated** → Regularly update AWS services, systems, and applications with the latest security patches to block known vulnerabilities.
9. **Segregate AWS Assets** → Separate user data, applications, and resources to contain breaches and improve overall security management.
10. **Securely Delete Unused Data and Groups** → Remove unnecessary users, roles, and data to minimize exposure and prevent unauthorized use of outdated resources.

### AWS Infrastructure Security Practices
1. **Use Information Security Management Systems (ISMS)** → Regularly review and update AWS security policies and controls to ensure continuous protection and compliance.
2. **Perform Network Segmentation** → Divide your AWS network into separate zones to control access and limit the spread of attacks within the infrastructure.
3. **Use Load Balancers, CDNs, and WAFs** → Deploy these tools to defend against DoS, DDoS, XSS, and SQL injection attacks, ensuring stable and secure application performance.
4. **Customize AWS Security Hub Insights** → Monitor and manage all AWS security alerts in one place by tailoring insights to your organization’s specific needs.
5. **Implement Data Loss Prevention Policies** → Apply consistent rules across AWS services to prevent unauthorized access, sharing, or leakage of sensitive data.
6. **Perform Automated Security Assessments** → Use Amazon Inspector to automatically scan AWS resources, detect vulnerabilities, and suggest fixes for better security posture.

### AWS Financial Services Security Practices
1. **End-to-End and Transport Data Encryption (TDE)** → Use encryption to protect data while it’s being sent and stored, ensuring secure communication between AWS and third-party services.
2. **Penetration Testing on AWS Services** → Regularly test AWS components like EC2, RDS, and load balancers to find and fix security weaknesses before attackers can exploit them.
3. **Use CloudTrail and CloudWatch** → Enable these tools to continuously monitor, log, and audit AWS activities, helping detect suspicious behavior or policy violations.
4. **Centralized Account Management with AWS Organizations** → Manage multiple AWS accounts from one place and enforce security rules using Service Control Policies (SCPs) for better governance and compliance.

### AWS Security Hub Practices 
1. **Leverage AWS Labs Script** → Use AWS-provided automation scripts to quickly enable Security Hub across all AWS accounts for centralized security management.
2. **Establish Threat Detection Systems** → Set up services like Amazon GuardDuty and Inspector to continuously detect, monitor, and respond to potential threats and vulnerabilities.
3. **Enable AWS Config & CIS Standards** → Turn on AWS Config and apply CIS Foundations benchmarks across all regions to ensure compliance and consistent security posture.
4. **Assign Tags to Resources** → Use tags to organize Security Hub resources and manage access controls more effectively across teams and accounts.
5. **Control IAM Policies & Use CIEM** → Create specific IAM policies for each user type and use Cloud Infrastructure Entitlement Management (CIEM) tools to centrally manage roles, permissions, and account access.
6. **Build Custom Actions** → Set up custom workflows in Security Hub to automatically copy findings for internal review or external remediation efforts.
7. **Use IAM Roles Instead of IAM Users** → Assign IAM roles to applications and services instead of creating user credentials, reducing the risk of credential leaks.
8. **Use IAM Access Analyzer** → Regularly scan for resources shared with external accounts and ensure they are properly secured and not exposing sensitive data.

### AWS Security Groups Practices
1. **Allocate Policies to Groups** → Assign permissions to groups instead of individual users to simplify management and ensure consistent security across AWS resources.
2. **Implement AWS VPC Flow Logs** → Use VPC Flow Logs to monitor and analyze network traffic, helping detect suspicious IP activity or attack patterns within your AWS environment.
3. **Isolate Resources with VPCs** → Keep resources separated using Amazon VPCs and control traffic flow with security groups and network ACLs to prevent unauthorized access.
4. **Enforce Strict Security Rules** → Apply tight inbound and outbound traffic rules in security groups and ACLs to minimize exposure and limit access only to what’s necessary.
5. **Automate Email Alerts** → Set up automated alerts to instantly notify administrators about critical security events or unusual activities for faster response.

### AWS Backup Data Practices 
1. **Ensure and Automate Frequent Backups** → Regularly back up your AWS data automatically to avoid data loss during system failures or attacks.
2. **Safeguard Backups Using Immutable Storage** → Store backups in immutable storage so that no one, even an attacker, can modify or delete them once created.
3. **Incorporate Backup in Recovery Plans** → Include backup processes in disaster recovery and business continuity plans to quickly restore data after any incident.
4. **Implement Configuration Audit and Monitoring** → Continuously audit and monitor AWS configurations to detect unauthorized changes or misconfigurations early.
5. **Enable and Monitor AWS Logs** → Activate logging services like VPC Flow Logs, S3 Access Logs, and CloudFront Logs to track activities and identify suspicious actions.
6. **Examine Data Recovery Abilities** → Regularly test and verify that backup data can be successfully restored to ensure reliability during real incidents.

## Security Assertion Markup Language (SAML) 
SAML is a popular open-standard protocol used for authentication and authorization between two communicating entities. It provides a single sign-on (SSO) facility for users to interact with multiple applications or services with one set of common credentials. SAML can be offered as software-as-a-service, which can be installed at the service provider (SP) and identity provider (IdP) to simplify the federated authorization and authentication mechanisms for users. The SAML protocol consists of the following three entities. 
- **Client or user:** It is an entity with a valid account that requests a service or resource through a web browser.
- **Service provider (SP):** It is a server hosting applications or services for the users.
- **Identity provider (IdP):** It is an entity within a system that stores user directories and validating mechanisms.

When SAML federation software is installed or configured, it builds a trust relationship between SP and IdP, enabling secure communication. When a user wishes to access any service or resource, he/she must be authenticated by the IdP. Soon after a service request is initiated from the user, the SP sends an SAML request to the IdP to validate the user.

The IdP then creates an XML-based SAML authentication assertion that describes what type of login attempt has been initiated (password, two-factor, etc.); the SAML attribute assertion, which contains specific details about the user; and the authorization assertion, which describes whether the user can be allowed or denied access to the service. XML-based assertions are then forwarded to the SP. Once the authentication process is completed successfully, the user is free to access the protected resources or services.

## Cloud Network Security 
A cloud network is virtual IT infrastructure managed by cloud service providers (CSPs), where network resources are supplied on demand in the form of private and public clouds. By creating a virtual environment within the cloud through an existing physical network, CSPs can perform network operations on the public cloud using individual client accounts. Cloud network security can be achieved in the following ways. 

### Virtual private cloud (VPC)
A Virtual Private Cloud (VPC) is a private network you get inside a public cloud provider where you can run apps, store data, and control networking just like a standalone datacenter. It isolates your traffic from other customers, lets you create subnets and IPv6 ranges, and combines the scalability and on-demand resources of the public cloud with strong data separation and configuration control so you can expand and secure your environment as needed.

### Public and private subnets
In a VPC, public and private subnets define how virtual machines connect to the internet. Instances in a public subnet can directly send and receive traffic through an Internet Gateway (IGW), allowing full web access if security rules permit it. In contrast, instances in a private subnet cannot be reached directly from the internet; they connect outward only through a NAT gateway, which hides their private IPs. This setup keeps private subnets isolated from external access while still allowing them to reach the internet securely for updates or communications.

### Transit gateways
A transit gateway is a central hub that connects on-premises networks with multiple cloud VPCs, making communication easier and reducing the need for complex peering setups. It allows network administrators to manage traffic efficiently and apply access controls based on IPs or ports using cloud-specific ACLs. This centralized setup provides better visibility and control over the entire network, even when devices connect through SD-WAN, ensuring smoother and more secure data flow across environments.

### VPC endpoint
A VPC endpoint creates a private, internal link between your VPC and a cloud service so traffic never traverses the public Internet or needs gateways, NAT, VPNs, or public IPs. Endpoints act like virtual networking devices that are redundant, scalable, and highly available, letting virtual machines in the VPC talk to cloud services directly with lower risk of exposure and without added bandwidth or availability bottlenecks. The following are the two types of VPC endpoints.
- **Interface endpoint:** It is an elastic network interface (ENI) that has a private IP address within the limit of a defined subnet. It operates as an initial point of source for traffic towards a VPC or supported cloud services.
- **Gateway-load-balancer endpoint:** It is also an ENI and operates as an initial point of source to impede traffic flow and divert it to a service that has been configured through a gateway load balancer, which is then used for security inspection.

## Cloud Security Controls 
Cloud security controls protect a cloud environment from any type of vulnerability and minimize the impacts of cyberattacks. These controls may include practices, procedures, guidelines, and policies that are enforced to secure the cloud infrastructure. A few examples of cloud security controls are discussed below. 

### Cloud Application Security 
Cloud application security focuses on protecting data and user activity within cloud-based apps like Google Workspace, Slack, or Office 365. It involves using security policies, controls, and monitoring tools to manage how data is shared and stored between users and platforms. Acting as a “safety net” in a zero-trust setup, it ensures that even trusted cloud apps are continuously verified and protected. This layer of security specifically targets the application components of SaaS, IaaS, and PaaS environments to prevent data leaks and unauthorized access.

Implementing cloud application security prevents exploits such as cross-site scripting (XSS), cross-site request forgery (CSRF), session hijacking, SQL injection, and weak authentication.

### High Availability Across Zones
High availability across zones ensures that cloud applications remain operational even during network outages or hardware failures. It divides resources into multiple availability zones, where each zone runs independent nodes that replicate data between them. If one node or zone fails, the other instantly takes over without data loss or service interruption. This setup provides continuous uptime, centralized monitoring, and reliable performance, allowing maintenance or upgrades to occur on one node while the other keeps the application running smoothly.

### Cloud Integration and Auditing
Cloud integration connects multiple cloud platforms and on-premises systems into a unified environment, allowing administrators to manage data, services, and applications efficiently. It simplifies operations, improves visibility, and ensures all risk indicators from APIs and logs can be monitored together. Following defined security policies and regular audits helps maintain compliance and reduce errors. <br>
Cloud auditing, on the other hand, verifies that cloud services meet security, privacy, and performance standards. It ensures consistent service availability, automates data collection for evaluation, and provides a cost-effective way for organizations to maintain a secure and compliant cloud environment.

### Security Groups 
A security group is a basic security measure implemented in cloud infrastructure to provide security to virtual instances. It serves as a security solution for virtual machines. The security group is located between the Internet and virtual instances, and it controls the inbound and outbound traffic. A properly configured security group prevents denial-of-service (DoS) attacks and unauthorized access to IT resources.

### Instance Awareness
Instance awareness is about recognizing individual cloud application instances to prevent attackers from exploiting fake or malicious instances for command-and-control or data theft. Traditional security tools often cannot distinguish between real and fake instances, making cloud environments vulnerable to threats like data exfiltration and SaaS phishing. Using instance-aware tools for services like Google Drive or OneDrive helps detect these threats and better protect cloud resources.

The rising popularity and acceptance of Kubernetes technologies across various organizations, including the top public cloud providers, have eventually amplified critical vulnerabilities. It has become obligatory for security professionals to detect vulnerabilities and implement enhanced security solutions.

Discussed in the table are some of the Kubernetes vulnerabilities and solutions associated with each vulnerability.
| No. | Vulnerabilities | Solutions |
| :--- | :--- | :--- |
| 1. | **No Certificate Revocation** | - Ensure that nodes maintain the certificate revocation list (CRL) and check each time they are presented with a certificate. <br>- Insist that administrators use online certificate status protocol (OCSP) stapling for revoking certificates in the cluster via an OCSP server. |
| 2. | **Unauthenticated HTTPS Connections** | - By default, authenticate all HTTPS connections within the system. <br>- Ensure that all the components use CA maintained by the kube-apiserver. <br>- Implement **two-way TLS** for all connections. |
| 3. | **Exposed Bearer Tokens in Logs** | - Remove the bearer token from the system logs and avoid logging any authentication credentials. <br>- Perform code reviews to ensure sensitive data is not logged. <br>- Implement logging filters to remove sensitive data before storing in logs. |
| 4. | **Exposure of Sensitive Data via Environment Variables** | - Avoid collecting sensitive data directly from environment variables. <br>- Use **Kubernetes secrets** in all system components. |
| 5. | **Secrets at Rest not Encrypted by Default** | - Define and document configurations required for different levels of security. |
| 6. | **Non-constant Time Password Comparison** | - Use a safe **constant-time comparison function**, such as `crypto.subtle.ConstantTimeCompare`. <br>- Disapprove of basic authentication mechanisms for secure options. |
| 7. | **Hardcoded Credential Paths** | - Define a configuration method for credential paths, and **avoid hardcoding credential paths**. <br>- Allow cross-platform configuration through path generalization. |
| 8. | **Log Rotation is not Atomic** | - Implement a copy-then-rename technique to ensure logs are not lost during log rotation. <br>- Avoid using log rotation and implement persistent logs that add log data linearly and create a new log whenever log rotation is required. |
| 9. | **No Back-off Process for Scheduling** | - Implement a **back-off process for kube-scheduler** to prevent tight-loops. |
| 10. | **No Non-repudiation** | - Use **secondary logging mechanisms** for processes that require strict non-repudiation and auditing. <br>- All authentication events should be logged and retrievable from a **central location** within the cluster. |

## Serverless Security Risks and Solutions 
Serverless computing has become popular owing to its remarkable features such as zero administration, pay-per-use service, and self-scalable ability. Although serverless technology has several advantages, it also introduced new risks that need to be mitigated. Discussed in the table are the top 10 serverless security risks and solutions, according to OWASP.

| Risks | Solutions |
| :--- | :--- |
| **A1 – Injection** | - Do not trust or make any assumptions regarding the input and its validity from any source. <br>- Implement safe API, and employ **parameterized interfaces** or object relational mapping tools. <br>- Implement **application whitelisting** wherever necessary. <br>- Avoid special characters using a specific escape syntax in dynamic SQL queries. <br>- Evaluate all entry points and event types of the system. <br>- Run the functions with the **least privileges** necessary to execute the task. <br>- Protect functions in execution state by using runtime defense solutions. |
| **A2 – Broken Authentication** | - Employ identity and access control solutions provided by the cloud service provider, such as **AWS Cognito**, AWS Single Sign-On, Azure Active Directory B2C, Azure App Service, and Google Firebase Authentication. <br>- Implement **strong authentication and access control** on external-facing resources. <br>- Employ **secure service authentication methods**, such as Federated Identity (SAML, OAuth2, Security Tokens, etc.), for authenticating internal resources. |
| **A3 – Sensitive Data Exposure** | - Identify and classify sensitive data. <br>- **Minimize the storage of sensitive data**; only store necessary data. <br>- **Encrypt data both in transit and at rest**. <br>- Implement **HTTPS endpoints** for APIs. <br>- Employ CSP services for key management and encryption of stored data, secrets, and environment variables to the functions in runtime and data in transit. |
| **A4 – XML External Entities (XXE)** | - Use only the CSP's software development kits, whenever possible. <br>- Perform **vulnerability scanning on supply chain libraries**. <br>- Test API calls for XXE vulnerabilities. <br>- Always **disable entity resolution**. |
| **A5 – Broken Access Control** | - Follow the **least-privilege principle** while granting permissions to functions. <br>- **Review each function** to detect excess privileges. <br> - Follow the cloud service provider's best practices, such as **AWS IAM and Azure Identity Management** best practices.|
| **A6 – Security Misconfiguration** | - Use the cloud provider's built-in services, such as **AWS Trusted Advisor**, to identify public resources. <br>- **Enforce strong access control on cloud resources**. <br>- Identify functions with unlinked triggers. <br>- Set the functions with a minimum timeout required. <br>- Employ **automated tools** to detect security misconfigurations in serverless applications. |
| **A7 – Cross-Site Scripting (XSS)** | - **Encode all untrusted data** before transmitting to the client. <br>- Use only well-known frameworks and headers. |
| **A8 – Insecure Deserialization** | - Ensure **validation of serialized objects** originating from untrusted data. <br>- Scan **third-party libraries** for deserialization vulnerabilities. <br>- **Monitor deserialization usage** and exception to identify probable attacks. |
| **A9 – Using Components with Known Vulnerabilities** | - Perform **continuous monitoring of third-party libraries and dependencies**. <br>- Deploy only **signed packages and components** from official sources. <br>- **Continuously check the vulnerability databases**, such as the CVE and national vulnerability database. <br>- Perform vulnerability scanning of third-party dependencies for known vulnerabilities using tools such as **OWASP Dependency Check and Dependency-Track**. |
| **A10 – Insufficient Logging and Monitoring** | - Employ **CSP monitoring tools**, such as **Azure Monitor, or AWS CloudTrail** to detect anomalous behavior. <br>- Employ **auditing and monitoring mechanisms** for data not originating from the CSP. |

## Best Practices for Container Security 
Discussed below are various best practices for securing the container environment. 
1. **Monitor CVEs** → Regularly check container software for known vulnerabilities and fix them promptly.
2. **Use Minimal Base Images** → Keep images lightweight to reduce attack surfaces and potential risks.
3. **Run as Non-Root User** → Avoid privilege escalation by running applications with normal user permissions.
4. **Authenticate Registry Access** → Protect sensitive images by ensuring only authorized access to registries.
5. **Regular Image Scanning** → Scan container images for vulnerabilities or misconfigurations before deployment.
6. **Enable Logging & Auditing** → Track changes, access, and activities for accountability and threat detection.
7. **Immutable Containers** → Prevent changes to containers after deployment to reduce tampering risks.
8. **Use Mandatory Access Controls (SELinux/AppArmor)** → Enforce strict access rules to protect apps and services.
9. **Harden Host & Stack** → Remove unnecessary services and secure the host operating system.
10. **Store Sensitive Data Externally** → Keep secrets outside containers and allow controlled runtime access.

## Best Practices for Docker Security 
Discussed below are various best practices for securing Docker environment. 
1. **Avoid exposing Docker daemon** → Don’t let the Docker API be accessible publicly; it’s a main entry point for attackers.
2. **Use trusted images only** → Pull images from verified sources to avoid backdoors and malware.
3. **Run containers as non-root** → Always configure containers to run with unprivileged users to prevent privilege escalation.
4. **Limit container capabilities** → Drop unnecessary capabilities using `--cap-drop` and allow only what’s required.
5. **Enable security options** → Use `--security-opt=no-new-privileges` and Linux security modules (AppArmor, SELinux, seccomp) for process control.
6. **Restrict inter-container communication** → Disable default communication (`--icc=false`) and link containers explicitly if needed.
7. **Limit resources** → Set CPU, memory, and process limits to prevent Denial-of-Service attacks.
8. **Use read-only filesystems and secrets management** → Protect container data with `--read-only` and store sensitive info securely via Docker secrets.
9. **Verify image integrity** → Enable Docker Content Trust (DCT) to ensure images are authentic and untampered.
10. **Secure API and communications** → Use HTTPS/TLS and basic authentication for Docker daemon and API access.

## Best Practices for Kubernetes Security 
Discussed below are various best practices for securing the Kubernetes environment. 
1. **Validate files and paths** → Always check file contents and their locations before use to prevent malicious access or execution.
2. **Avoid hardcoded credentials** → Use configurable paths for secrets and credentials instead of embedding them in code.
3. **Use secure logging methods** → Persist logs safely and use copy-then-rename during rotation to avoid losing important information.
4. **Handle JSON safely** → Use tested libraries and proper data structures when working with Kubernetes APIs to avoid data corruption or injection attacks.
5. **Authenticate HTTPS by default** → Ensure all connections use verified certificates to prevent Man-in-the-Middle (MITM) attacks.
6. **Encrypt secrets properly** → Use key management systems and strong encryption methods; avoid weak modes like AES-GCM/ECB.
7. **Use network policies** → Restrict pod-to-pod communication to only what is necessary to reduce attack surfaces.
8. **Monitor and log effectively** → Implement comprehensive logging and monitoring with tools like Prometheus, Grafana, or ELK Stack.
9. **Enforce security policies** → Use tools like Open Policy Agent (OPA) or Kyverno to automatically enforce security rules in the cluster.
10. **Keep Kubernetes updated** → Regularly patch Kubernetes components and dependencies to fix vulnerabilities and improve security.

## Zero Trust Networks
Zero Trust is a security approach that assumes no user or device is automatically trustworthy. Every access request is verified before granting entry, ensuring users can only reach the resources they need for their role. This model protects networks from compromised accounts or untrusted connections by continuously validating identities and enforcing strict access controls.

### Representation of Zero Trust Network 
The figure illustrates a Zero Trust Network, where a cloud-based control plane manages and verifies all access requests before allowing network traffic. Only authenticated and authorized users or devices can reach resources, and access is enforced based on roles, device type, and time. Once approved, the data plane permits traffic solely from that client. Zero Trust uses micro-segmentation to divide the network into smaller zones, limiting the impact of any breach, and incorporates techniques like encryption, multifactor authentication, and privileged access management to maintain secure, tightly controlled, and continuously monitored access to network resources.

## Cloud Access Security Broker (CASB) 
Cloud access security brokers (CASBs) are on-premise or cloud-hosted solutions. They are responsible for enforcing security, compliance, and governance policies in cloud applications. A CASB is located between the on-premise infrastructure of an organization and the infrastructure of a cloud provider. It acts as a gatekeeper that enables organizations to extend their security policies beyond their own infrastructure.

**Features of CASB**
- **Visibility into cloud usage** <br>
  It finds shadow IT cloud services and provides visibility into the user activities with the allowed cloud applications.

- **Data security** <br>
  It enforces data-centric security encryption, tokenization, access control, and information rights management.

- **Threat protection** <br>
  It detects and responds to malicious insider threats, privileged user threats, and compromised accounts.

- **Compliance** <br>
  It discovers critical data in the cloud and enforces DLP policies to satisfy the data residency and compliance requirements.

**CASBs offer the following:**
- Firewalls to identify malware, thereby preventing the malware from entering the enterprise network.
- Authentication for user credentials, ensuring that only the allowed users can access the required organizational resources.
- WAFs to prevent malware from breaching security at the application level instead of the network level.
- DLP prevents users from transferring critical information outside the organization. 

**How CASBs work:**
A CASB works by
- Ensuring network traffic between on-premise devices and the cloud provider complies with the organizational security policies.
- Provides insights into the use of cloud applications across cloud platforms and identifies unsanctioned use.
- Uses auto-discovery to identify
  - Cloud applications in use
  - High-risk applications
  - High-risk users
- Enforcing different security access controls such as encryption and device profiling.
- Providing services such as credential mapping when SSO is not available.
