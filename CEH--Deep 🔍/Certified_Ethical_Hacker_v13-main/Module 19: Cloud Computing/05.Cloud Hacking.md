# 🕵️ Cloud Hacking 
Though most organizations adopt cloud technologies for the variety of cost-effective services, security remains a significant concern because it depends on sharing. Security gaps and vulnerabilities of the underlying technologies can allow attackers to launch various types of cloud attacks, affecting confidentiality, integrity, and availability of resources and services in cloud systems. This section discusses the various techniques and tools attackers use for hacking the cloud environment.

Cloud hacking encompasses a broad range of activities aimed at compromising the cloud infrastructure and services. This can include both cloud-hosted Web applications and system hacking, but extends to compromising the overall cloud security. The three elements that attackers often target to compromise cloud security are as follows: 

### Web Applications Hacking 
Attackers often target web application programming interfaces (APIs), which are vital sources for relaying cloud services and resources. APIs are the gateways cloud apps use to talk to each other, and attackers focus on them because they’re often exposed on the Internet. They first map and scan APIs to find weak spots—poor authentication, broken authorization, or injection flaws—and then exploit those weaknesses to access cloud resources or data. Because cloud services are shared, compromising one API can let an attacker reach large amounts of data or move laterally across tenants, making API attacks especially dangerous.

### System Hacking
System hacking in cloud environments means finding and abusing weaknesses in virtual systems like VMs, containers, and serverless functions that run on cloud platforms. Attackers start with reconnaissance to discover exposed interfaces, weak protocols, or misconfigurations, then exploit those flaws to move deeper into cloud resources and set up ways to stay there long-term. Because cloud services share infrastructure and use many automated interfaces, successful attacks can spread quickly and be hard to detect, so strong configuration, access controls, and continuous monitoring are essential.

### Cloud Hacking
Cloud hacking is when attackers target weaknesses in cloud systems—like open ports, weak passwords, unpatched software, or misconfigured settings—to gain unauthorized access to data or disrupt services. They start with reconnaissance (scanning services and mapping the cloud network), identify exploitable gaps, and then move in to steal data, deploy malware, or lock users out, often trying to cover their tracks. The result is data loss, service disruption, and damaged trust in cloud providers and customers.

> **Note:** Each cloud provider, such as AWS, Azure, or GCP, imposes specific rules, policies, or permissions regarding ethical hacking. As an ethical hacker, the provider must be notified before performing hacking operations. In addition, checking which activities are permitted for specific operations (e.g., some organizations or cloud service providers may restrict certain types of attacks that could cause service disruptions).

## Cloud Hacking Methodology 
The following are the different phases of cloud hacking: 
- **Information Gathering** <br>
  Information gathering is the first step in a cloud attack where the attacker collects everything they can about the target’s cloud environment—network layout, IPs, domains and subdomains, user accounts, and any public data. This reconnaissance uses techniques like network scans, DNS checks, web scraping, and social engineering, often automated with tools such as Nmap, Shodan, and Recon-ng. The goal is to build a clear map of the environment so attackers can find weak points and plan more effective follow-up exploits.

- **Vulnerability Assessment** <br>
  Vulnerability assessment is the process of finding and evaluating security gaps in a cloud environment—like misconfigured settings, missing patches, or weak network and application controls—that attackers could exploit to gain access, escalate privileges, or disrupt services. Security teams (and attackers) use both automated scanners and manual checks to map these weaknesses; common tools include Nessus, OpenVAS, and Qualys, which produce detailed reports that show where the cloud is exposed. The goal is to discover and prioritize these issues so they can be fixed before they become entry points for further compromise.

- **Exploitation** <br>
  In the exploitation phase, attackers use the weaknesses they found to break into cloud systems and take control. They often run custom scripts or tools like Metasploit, sqlmap, or thc-hydra to bypass authentication, inject malicious code, or exploit application flaws. Once inside, attackers aim to access sensitive data, hijack cloud resources, or disrupt services—actions that can cause data breaches, downtime, financial loss, and lasting damage if not detected and fixed quickly.

- **Post-Exploitation** <br>
  Post-exploitation is the stage after an attacker successfully breaks into a cloud resource, where they focus on staying inside, hiding their tracks, and moving deeper into the environment. Attackers create persistence (backdoors, privilege escalation, C2 channels) and use tools like Cobalt Strike or Metasploit to control systems, steal data, and prepare further attacks. Their aim is long-term access and covert data exfiltration, which makes detection and cleanup harder and much more expensive, and risks ongoing loss of sensitive information and disruption to cloud services.

> **Note:** Ethical hacking in a cloud environment is typically feasible only through internal means, ensuring compliance with security policies and avoiding unauthorized access.

## Identifying Target Cloud Environment 
Identifying a target cloud environment means discovering which cloud provider and services an organization uses—like AWS, Azure, or GCP—and profiling their specific configurations. This lets attackers tailor techniques to cloud-specific weaknesses and misconfigurations; they commonly use internet-scanning services such as Shodan and Censys to map exposed assets and gather details needed for targeted exploitation.

Shodan search filters that can be used by attackers to gather information on a target cloud infrastructure are as follows:
- **Search for HTTPS services:** `port:443` <br>
  Obtains devices and services accessible through HTTPS, which are commonly used for web services in cloud environments.

- **Search for AWS services:** `ssl.cert.issuer.cn:Amazon` <br>
  Obtains information on SSL certificates issued by Amazon that can help identify services hosted by AWS.

- **Search for Kubernetes:** `product:Kubernetes` <br>
  Obtains information on instances of Kubernetes.

- **Search for AWS services:** `org:Amazon` <br>
  Obtains information on services hosted specifically on Amazon’s infrastructure, which also reveals AWS-hosted services.

- **Search for Azure-hosted services:** `ssl.cert.subject.cn:azure` <br>
  Obtains information on SSL certificates that mention Azure in the subject’s common name, indicating Azure-hosted services.

- **Search for any cloud asset:** `tag:cloud` <br>
  Obtains information on any assets tagged as being part of a cloud infrastructure. <br>
  Note: The "tag" filter is only available to enterprise users.

- **Search within a specific IP range:** `net:52.0.0.0/8` <br>
  Searches within a specific IP range, such as those assigned to AWS, where 52.0.0.0/8 is a common range for AWS.

- **Search within the instances:** `http.html:"s3.amazonaws.com"` <br>
  Searches for instances where the HTML content mentions "s3.amazonaws.com", which could indicate the AWS S3 buckets used.

- **Search for AWS-hosted services and infrastructure used by Facebook:** `Amazon web services Facebook` <br>
  Searches for various AWS-hosted services and infrastructure used by Facebook such as IP addresses, open ports, and service banners. This includes details on publicly accessible S3 buckets, EC2 instances, and other AWS resources associated with Facebook.

## Discovering Open Ports and Services Using Masscan 
Masscan is a network port scanner designed to scan large networks and the entire Internet within minutes. It is particularly useful for identifying open ports and services running on a cloud infrastructure. By leveraging their scanning capabilities, attackers can quickly discover services that may be vulnerable to exploitation. Masscan can be configured to scan specific IP addresses or ranges, allowing attackers to target cloud service providers, such as AWS, Azure, or Google Cloud.

### Identifying Open Ports using Masscan 
- Run the following Masscan command to scan the target IP address for open ports:
  ```bash
  sudo masscan -p0-65535 <target_IP_address> --rate=<rate>
  
  o -p0-65535: # -> Scans all ports from 0 to 65535.
  o <target_IP_address>: # -> Replace with the target IP address.
  o --rate=<rate>: # -> Sets the rate of packets per second.
  ```

## Vulnerability Scanning Using Prowler 
🔗Source: [https://github.com/prowler-cloud/prowler] <br>
Prowler contains more than 240 controls covering the CIS, NIST 800, NIST CSF, CISA, RBI, FedRAMP, PCI-DSS, GDPR, HIPAA, FFIEC, SOC2, GXP, AWS Well-Architected Framework Security Pillar, AWS Foundational Technical Review (FTR), ENS (Spanish National Security Scheme) and custom security frameworks. If attackers gather the necessary credentials in the enumeration process above, they can use Prowler to check for security loopholes, including unsecured data transmission channels, overly permissive policies, and other potential vulnerabilities. 

### Some of the Prowler commands to Perform Vulnerability Scanning
- Run the following command and specify the provider to start scanning: `prowler <provider>`

  ```bash
  prowler gcp --project-ids <Project_ID_1> <Project_ID_2> ... <Project_ID_N>
  ```

## Identifying Misconfigurations in Cloud Resources Using CloudSploit 
Cloud environments like AWS, Azure, and Google Cloud are large and change frequently, which makes them prone to mistakes called misconfigurations. Tools such as CloudSploit automate the scanning of cloud accounts to find these errors—things like overly permissive IAM roles, public storage buckets, open databases, and loose network rules. Attackers use the same scans to find weak spots, then exploit them to access data, escalate privileges, and move through the environment. Left unchecked, these misconfigurations can give attackers a persistent foothold and lead to major data breaches.

For this purpose, attackers can use tools such as CloudSploit, as discussed below: 

### CloudSploit 
🔗Source: [https://github.com/aquasecurity/cloudsploit] <br>
CloudSploit is a tool designed to identify misconfigurations and security risks across a wide range of cloud resources and can assist in gaining access to cloud resources. CloudSploit also supports mapping its plugins to particular compliance policies such as the HIPAA, and CIS Benchmarks. CloudSploit generates detailed reports outlining potential security issues. An attacker analyzes these reports to identify specific vulnerabilities that can be exploited to gain access and perform lateral movements within the target cloud environment.

- Set up the cloud provider's credentials and configuration file containing the credentials of the targeted cloud service:
  - Configuration file format for AWS:
    ```
    {
        "accessKeyId": "YOURACCESSKEY",
        "secretAccessKey": "YOURSECRETKEY"
    }
    ```
    
  - Configuration file format for Azure:

## Cleanup and Maintaining Stealth 
After breaching a cloud environment, attackers remove traces and hide changes so defenders don’t notice the intrusion. They erase or alter logs, undo obvious modifications, and mask malicious tools to stay under detection thresholds. This low-profile behavior lets them keep access long term, quietly monitor systems, and extract data over time without triggering alerts. Maintaining stealth and persistence maximizes the attacker’s chance to reuse the environment for further exploitation while making the breach harder to investigate and remediate.

To achieve cleanup and maintain stealth, attackers can use the following methods:
- **Log Manipulation:** Once attackers compromise the target cloud environment, they can delete logs or modify them to remove or hide entries that record malicious actions.
- **Removing Credentials and Access Management:** This method involves removing temporary credentials such as temporary access tokens or keys. Attackers can create hidden backdoors by establishing hidden accounts or access methods that blend into legitimate activities.
- **Manipulating System and Service Configurations:** This is the process of eliminating visible changes during an attack. Additionally, attackers can disable or modify alerts that could reveal their presence.
- **Implementing Persistence Mechanisms:** Using this method, attackers can hide malicious code through legitimate processes or services. Alternatively, legitimate built-in cloud tools and scripts can be used to avoid suspicion.
