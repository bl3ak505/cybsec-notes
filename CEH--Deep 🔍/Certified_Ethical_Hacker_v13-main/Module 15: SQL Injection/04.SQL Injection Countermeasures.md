# 🛡️ SQL Injection Countermeasures
Previous sections discussed the severity of SQL injection attacks, their various techniques, tools used to perform SQL injection, techniques used to bypass IDS/firewall signatures, and so on. These discussions were about offensive techniques that an attacker can adopt for SQL injection attacks. This section discusses defensive techniques against SQL injection attacks and presents countermeasures to protect web applications.

### Why are Web Applications Vulnerable to SQL Injection Attacks? 
1. **Database Server Runs OS Commands** → If attackers inject SQL, they can make the database run system commands and take control of the server.
2. **Privileged Accounts** → If the database connection uses an admin-level account, attackers can gain full control over the database and even the OS.
3. **Error Messages Leak Info** → Detailed error messages can reveal database details (like table names, structure, or queries) that attackers can use to hack deeper.
4. **No Input Validation** → Without proper filtering of user input, attackers can insert harmful SQL code directly into queries.
5. **Complex Software Stacks** → Modern apps use many layers of tech, and weak security in one layer can open doors for SQL injection.
6. **Legacy Code** → Old code often lacks modern security measures, making it easier for attackers to exploit SQL injection vulnerabilities.
7. **Concatenated Queries** → Building queries by joining strings allows attackers to inject their own SQL commands into the query flow.

Implementing consistent coding standards, minimizing privileges, and firewalling the server can all help to defend against SQL injection attacks. 

- **Minimizing Privileges** <br>
  Minimizing privileges means giving applications and accounts only the access they need to function, instead of full administrative rights. By starting with low-privilege accounts and adding permissions only when required, developers can reduce security risks early in the development cycle. This approach not only makes it easier to identify and fix issues as features are built but also results in a more secure product that avoids last-minute security fixes and works safely within organizational policies.

- **Implementing Consistent Coding Standards** <br>
  Implementing consistent coding standards ensures that all developers follow the same secure practices when building databases and web applications. Clear policies for tasks like data access, input validation, and error handling reduce security risks and make systems easier to maintain. For example, enforcing input validation on both client and server prevents malicious data from bypassing checks, while using custom error messages avoids exposing sensitive system details.

- **Firewalling the SQL Server** <br>
  Firewalling the SQL Server ensures that only trusted systems, such as the web server and backup server, can connect to it. By restricting access to required ports like TCP 1433 and UDP 1434 while blocking unnecessary ones, the risk of attacks is reduced. This helps protect against threats caused by insecure scripts, unpatched systems, or administrative mistakes, strengthening overall database security.

### Countermeasures Against SQL Injection
To defend against SQL injection, the developer needs to take proper care in configuring and developing an application to create one that is robust and secure. The developer should use the best practices and countermeasures to prevent applications from becoming vulnerable to SQL injection attacks. 

Some countermeasures to defend against SQL injection attacks are listed below: 

1. **Use Parameterized Queries / Prepared Statements** → Always separate user input from SQL commands so attackers can’t inject harmful queries.
2. **Input Validation (Whitelist)** → Accept only expected and safe values (like numbers, emails) and reject everything else.
3. **Avoid Dynamic SQL** → Never build SQL queries by joining user input with strings; this is the main cause of SQL injection.
4. **Least Privilege Principle** → Give the application only the minimum database permissions it needs (e.g., read-only for reading data).
5. **Stored Procedures** → Use stored procedures with strict input validation instead of raw SQL queries.
6. **Error Handling** → Don’t show database error messages to users, as they can leak valuable information to attackers.
7. **Regular Security Testing** → Perform penetration testing, code review, and automated scans to detect SQLi vulnerabilities early.
8. **Keep Software Updated** → Regularly patch databases, frameworks, and applications to close known security holes.
9. **Web Application Firewall (WAF)** → Use a WAF to filter and block malicious SQL injection attempts before they reach your app.
10. **Secure Password Storage** → Store user passwords with strong hashing (like SHA-256 with salt) instead of plaintext.

### Use Type-Safe SQL Parameters
Using type-safe SQL parameters ensures user input is handled as data, not executable code, which prevents SQL injection. By enforcing type and length checks, input is validated before reaching the database.

**Vulnerable code:**
```csharp
SqlDataAdapter myCommand =
new SqlDataAdapter("LoginStoredProcedure '" + Login.Text + "'", conn);
```
This directly concatenates user input into the query, making it vulnerable.

**Secure code:**
```csharp
SqlDataAdapter myCommand = new SqlDataAdapter(
  "SELECT aut_lname, aut_fname FROM Authors WHERE aut_id = @aut_id", conn);
SqlParameter parm = myCommand.SelectCommand.Parameters.Add("@aut_id", SqlDbType.VarChar, 11);
parm.Value = Login.Text;
```
Here, `@aut_id` is treated strictly as a value with defined type and length, blocking malicious input from being executed.

To defend against SQL injection, systems should combine secure coding and protective measures. Applications must sanitize and filter user input, use stored procedures or parameterized queries, and avoid verbose error messages by showing custom error pages. Developers should regularly patch software, minimize third-party use, and review source code for vulnerabilities. On the server side, WAF/IDS can filter malicious traffic. For databases, always connect with non-privileged accounts, apply least privilege access to tables and columns, and disable risky commands like `xp_cmdshell`.

## Defenses in the Application 

### Input Validation 
Input validation is the process of checking and sanitizing user-supplied data before it is processed by the application or database. By using methods like whitelisting (allowing only safe inputs) and blacklisting (blocking harmful inputs), developers can ensure that malicious data does not alter the logic of the code, helping prevent attacks such as SQL injection or XSS.

- **Whitelist Validation** <br>
  Whitelist validation, also called positive validation, is a secure practice where only pre-approved input types, values, ranges, or characters are allowed. It helps block malicious input by accepting only what is explicitly defined as safe. This is often done with regular expressions.
  
  For example, a whitelist may allow only specific characters such as:
    ```
  ^ \ { } ( ) @ | ? $
  ```
  This ensures that inputs outside the approved set are rejected, reducing the risk of attacks like SQL injection or cross-site scripting.

- **Blacklist Validation** <br>
  Blacklist validation, also called negative validation, is a method where applications block inputs that contain known malicious patterns or characters. It is usually implemented with regular expressions to reject dangerous inputs before they reach the database. However, since attackers constantly find new ways to bypass filters, blacklist validation alone is not enough and is often combined with whitelisting and output encoding for stronger protection.

  **Example of characters commonly blacklisted:**
  `' | % | -- | ; | /* | \* | _ | [ | @ | xp_`
  
  These characters are often used in SQL injection attempts, so blocking them helps reduce risk.

### Output Encoding
Output encoding is a technique used to sanitize user input before sending it to the database, helping prevent SQL injection. Unlike simple whitelisting, it ensures special characters are safely handled. For example, names like **O’Henry** include a single quote (`'`), which could break a SQL query if not encoded.

If the query is written as:
```java
String myQuery = "INSERT INTO UserDetails VALUES ('" + first_name + "','" + last_name + "');"
```
An attacker could inject malicious input such as:
```sql
', ''); DROP TABLE UserDetails--
```
This would generate:
```sql
INSERT INTO UserDetails VALUES ('', ''); DROP TABLE UserDetails--
```
To stop this, output encoding replaces the single quote with a safe format. In MySQL, this can be done by doubling the quote (`''`) or escaping it with a backslash (`\'`). For example, in Java:
```java
myQuery = myQuery.replace("'", "\\'");
```
This ensures the single quote is treated as part of the string, blocking SQL injection attempts. However, the drawback is that encoding must be applied every time input is passed to the query.

### Enforcing Least Privileges 
Enforcing least privileges means giving users or applications only the minimum access rights needed to perform their tasks. For example, if an application only needs to read data, it should be granted **read-only access** instead of full control. Similarly, the database should never run with **root or administrator rights**, as this increases risk. By restricting privileges, the chances of unauthorized access or successful SQL injection attacks are greatly reduced.

### LIKE Clauses
LIKE clauses in SQL use wildcards such as `%`, `_`, and `[` to match patterns, but if not handled properly, they can be abused for SQL injection. To prevent this, developers should escape these characters before passing user input into a query. This can be done using the `Replace()` method, which ensures wildcards are treated as normal characters instead of special operators.

Example code:
```csharp
s = s.Replace("[", "[[]");  
s = s.Replace("%", "[%]");  
s = s.Replace("_", "[_]");  
```
This ensures that inputs containing `%`, `_`, or `[` are safely processed, protecting the query from injection attacks.

### Wrapping Parameters with QUOTENAME() and REPLACE()
Wrapping parameters with **QUOTENAME()** and **REPLACE()** is a secure coding practice to prevent SQL injection in dynamic SQL queries. These functions ensure that input values are properly handled before being included in the query. If the input string is 128 characters or fewer, `QUOTENAME()` is used; if longer, `REPLACE()` is applied to escape single quotes.

**Example:**

**Before (unsafe):**
```sql
SET @temp = N'SELECT * FROM employees WHERE emp_lname =''' + @emp_lname + N'''';
```
**After (safe):**
```sql
SET @temp = N'SELECT * FROM employees WHERE emp_lname = ''' + REPLACE(@emp_lname,'''','''''') + N'''';
```
This ensures user input is sanitized and cannot break or inject malicious SQL into the query.

## Detecting SQL Injection Attacks 
Security professionals must develop and deploy rules in the IDS to detect regular expressions used in SQL injection attacks on a web server. For this purpose, they must use regular expressions to detect the SQL injection meta-characters such as single-quote (‘) and double-dash (--). The regular expressions for detecting SQL injection-specific characters and their meanings are listed below:

| Characters | Explanation |
|------------|-------------|
| ’          | Single-quote character |
| \|         | or |
| %27        | Hex equivalent of single-quote character |
| --         | Double-dash |
| %2D        | Hex equivalent of double-dash |
| #          | Hash or pound character |
| %23        | Hex equivalent of hash character |
| i          | Case-insensitive |
| x          | Ignore white spaces in pattern |
| %3D        | Hex equivalent of = (equal) character |
| %3B        | Hex equivalent of ; (semi-colon) character |
| %6F        | Hex equivalent of o character |
| %4F        | Hex equivalent of O character |
| %72        | Hex equivalent of r character |
| %52        | Hex equivalent of R character |
| %3C        | Hex equivalent of < (opening angle bracket) character |
| %3E        | Hex equivalent of > (closing angle bracket) character |
| %2F        | Hex equivalent of / (forward slash for a closing tag) character |
| \s         | Whitespaces equivalents |
| ^\n        | Hex equivalent of a non-newline character |

Security professionals can use regex search to detect SQL meta-characters. 

### Regular expression for detection of SQL meta-characters 
The regular expression `/(\')|(\%27)|(\-\-)|(#)|(\%23)/ix` is used to detect SQL meta-characters commonly abused in SQL injection attacks. It looks for inputs such as the single quote `'`, its hex value `%27`, the double-dash `--`, and the hash `#` with its hex `%23`. These characters are often used by attackers to break queries or comment out parts of SQL statements.

Security professionals apply this regex in logs from devices like IDS or WAF to spot malicious requests. For example, in the Snort log:
```
alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS $HTTP_PORTS (msg:"SQL Injection – Paranoid"; 
flow:to_server, established; uricontent:".pl"; 
pcre:"/(\')|(\%27)|(\-\-)|(#)|(\%23)/ix"; 
classtype:Web-application-attack; sid:9099; rev:5;)
```
Here, the alert shows an SQL injection attempt detected in HTTP traffic. It uses TCP, targets HTTP servers on ports like 80 or 8080, and checks for suspicious characters in requests, especially for `.pl` (Perl-based) web applications.

### Modified regular expression for detection of SQL meta-characters 
The given regular expression
```
/((\%3D)|(=))[^\n]*((\%27)|(\')|(\-\-)|(\%3B)|(;))/ix
```
is used to detect SQL meta-characters in user inputs. It first looks for an equals sign (`=`) or its hex value `%3D`, followed by any non-newline characters. Then it checks for symbols commonly used in SQL injection, such as a single quote (`'` or `%27`), double dash (`--`), or semicolon (`;` or `%3B`).

**Example matches:**
* `user=admin'` → detects `=` and `'`
* `id=10--` → detects `=` and `--`
* `name%3Dvalue%27` → detects `%3D` and `%27`
* `product=1;DROP TABLE Users` → detects `=` and `;`
This helps security professionals block malicious inputs that may indicate SQL injection attempts.

### Regular expression for typical SQL injection attack 
The given regular expression:
```
/\w*((\%27)|(\'))((\%6F)|o|(\%4F))((\%72)|r|(\%52))/ix
```
is used to detect patterns of SQL injection attempts. It looks for alphanumeric or underscore characters followed by a single quote (`'` or `%27`), and then the keyword “OR” in any form (`or`, `Or`, `oR`, `OR`) or its hex values. This helps identify attackers trying to inject conditions like `‘ OR ‘1’=‘1` into SQL queries.

**Example detections:**
* `admin' OR '1'='1`
* `user%27oR%271%27%3D%271`
* `test' oR 'a'='a`
These match the expression because they use a single quote with the `OR` condition, which is a common SQL injection pattern.

### Regular expression for detecting SQL injection with the UNION keyword
A regular expression can be used to detect SQL injection attempts that include the `UNION` keyword. Attackers often combine `UNION` with malicious queries to extract data. The following regex is an example:
```
/((\%27)|(\'))union/ix
```
This pattern looks for a single quote (`'`) or its encoded form (`%27`), followed by the word `union`. It helps detect suspicious HTTP requests that may be part of a SQL injection.

Similar expressions can be created for other SQL keywords, such as:
* `insert`
* `update`
* `select`
* `delete`
* `drop`
These regex patterns allow security professionals to monitor and block common SQL injection attempts in web applications.

### Regular expression for detecting SQL injection attacks on a MS SQL Server 
A regular expression can be used to detect SQL injection attempts that target MS SQL Server’s stored or extended procedures. Attackers often abuse commands like `xp_cmdshell`, `xp_regread`, or `xp_regwrite` to execute system commands or alter registry settings. The regex
```
/exec(\s|\+)+(s|x)p\w+/ix
```
looks for the keyword `exec`, followed by spaces or their encoded equivalents, and then checks for `sp` (stored procedures) or `xp` (extended procedures), ending with alphanumeric or underscore characters.

**Example detections:**
* `exec xp_cmdshell`
* `exec sp_helpdb`
* `exec xp_regread`
This helps identify suspicious queries that attackers may inject to misuse SQL Server functionality.

## SQL Injection Detection Tools
SQL injection detection tools help in the detection of SQL injection attacks by monitoring HTTP traffic and SQL injection attack vectors, and they determine if the web application or database code suffers from SQL injection vulnerabilities. 

- **OWASP ZAP** [https://www.zaproxy.org] <br>
  OWASP Zed Attack Proxy (ZAP) is a free penetration testing tool used to find security weaknesses in web applications. It provides automated scanners along with manual testing features, making it useful for both beginners and experienced professionals. Developers and testers can use ZAP to detect vulnerabilities early, fix them effectively, and reduce the chances of successful attacks.

- **Snort** [https://www.snort.org] <br>
  Snort is an open-source intrusion detection and prevention system that can detect attacks by matching malicious code patterns in network traffic. Security professionals write custom Snort rules to identify threats like SQL injection attempts. For example, attackers often use functions or characters such as `sleep()`, quotes (`'` or `"`), semicolons (`;`), comments (`--`, `/*`), or encoded symbols to bypass filters. Snort rules can block these patterns when detected in HTTP requests or headers.

**Examples of Snort expressions:**

* `/User-Agent\x3A\x20[^\r\n]*sleep\x28/i` → Detects `sleep()` used in HTTP headers.
* `/[?&]selInfoKey1=[^&]*?([\x27\x22\x3b\x23]|\x2f\x2a|\x2d\ -\d)/i` → Detects injection characters in parameters.
* `/^\s*?MAIL\s+?FROM\x3a[^\r\n]*?\x28\x29\s\x7b/i` → Detects malicious mail commands.
* **Rule example:**

  ```
  alert tcp any any -> any $HTTP_PORTS 
  (msg:"SQL use of sleep function in HTTP header - likely SQL injection attempt"; 
  flow:to_server,established; http_header; content:"User-Agent|3A| "; 
  content:"sleep(",fast_pattern,nocase; 
  pcre:"/User-Agent\x3A\x20[^\r\n]*sleep\x28/i"; 
  classtype:web-application-attack; sid:38993; rev:9;)
  ```

This way, Snort helps detect and block SQL injection patterns before they compromise a web application.
