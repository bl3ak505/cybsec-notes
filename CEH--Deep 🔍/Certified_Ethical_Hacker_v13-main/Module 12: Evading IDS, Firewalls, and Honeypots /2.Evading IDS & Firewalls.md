# 🧨Evading IDS/Firewalls
**Evading IDS/Firewalls** means attackers use smart techniques to avoid being detected by security systems while trying to break into a network. They do this by hiding their attack code (obfuscation), using encrypted communication, or taking advantage of unknown (zero-day) flaws. For ethical hackers and security teams, knowing these tricks helps build stronger defenses and improve how IDS and firewalls detect advanced threats.

---
## IDS/Firewall Evasion Techniques
**Bypassing an IDS/Firewall** means an attacker changes the way an attack is carried out to avoid being detected by security systems. Since IDS/firewalls follow fixed rules, skilled attackers can use special techniques to trick them into ignoring harmful traffic, allowing the attack to succeed without being blocked.

Some IDS/firewall bypassing techniques are as follows:
```
- Port Scanning					- ACK Tunneling
- Firewalking					- HTTP Tunneling
- Banner Grabbing				- SSH Tunneling
- IP Address Spoofing				- DNS Tunneling
- Source Routing				- Through External Systems
- Tiny Fragments				- Through MITM Attack
- Using an IP Address in Place of a URL		- Through Content
- Using Anonymous Website Surfing Sites		- Through XSS Attack
- Using a Proxy Server				- Through HTML Smuggling
- ICMP Tunneling				- Through Windows BITS
```

## IDS/Firewall Identification 

### Port Scanning 
**Port Scanning** is a technique used to find open ports on a computer or server. These ports are like doors that allow data to go in and out. Attackers scan them to see which services are running and look for weaknesses they can exploit. Often, they use automated tools to quickly scan and gather this information as the first step in a cyberattack.

#### How Attackers Scan Ports 
**Port scanning** is a technique attackers use to find open ports on a system by sending messages to each port and analyzing the response. Open ports can reveal running services and potential weaknesses. Attackers also use port scanning to detect firewalls and intrusion detection systems (IDS).

For example, tools like **ManageEngine Firewall Analyzer** and **Forcepoint NGFW** can be identified by the specific ports they use (like UDP 514, TCP 443). IDS tools like **Snort** and **Suricata** monitor traffic on specific ports (e.g., TCP 80 or UDP 53) to detect malicious activity. By scanning ports, attackers learn about these defenses and plan how to bypass them.

### Firewalking
**Firewalking** is a technique used to gather information about networks behind firewalls. It works by sending packets with carefully set **TTL (Time-To-Live)** values to see how far they travel before being blocked. If a packet reaches just past the firewall, it triggers an **ICMP "TTL exceeded"** message, helping identify which ports and services are allowed through the firewall.

Tools like **Firewalk** and **Nmap firewalk scripts** perform this by first discovering the network, then scanning it. This method helps reveal firewall rules and potential weaknesses, which attackers might use to bypass security systems like **IDS** and plan more targeted, stealthy attacks.

### Banner Grabbing
**Banner Grabbing** is a technique used to collect information from a system by connecting to its open services like **FTP**, **Telnet**, or **web servers**. When a connection is made, these services often send back a **banner** that reveals details like the software name, version, and sometimes even configuration info.

Attackers use this information to identify vulnerabilities and create targeted exploits that avoid detection by **IDS (Intrusion Detection Systems)**. Since the connection looks normal, firewalls usually don’t block it. For example, a simple SMTP banner grab might look like:

```
telnet mail.targetcompany.org 25
```
The syntax is `“<service name > <service running > <port number>”`

For example, when the user opens a telnet connection to a known port on the target server and presses Enter a few times, if required, it displays the following result:

## IP Address Spoofing 
**IP Address Spoofing** is a trick used by attackers to hide their identity and bypass firewalls or intrusion detection systems (IDS). They change the source IP address in a data packet to make it look like it's coming from a trusted source. This helps them sneak past security filters that rely on IP addresses to allow or block traffic.

By pretending to be a trusted device, attackers can access networks without being noticed, making it hard to trace the real source. They often use spoofing for spamming, phishing, or launching **DoS attacks**. Tools like **Hping** help create fake packets with forged IP addresses to carry out these attacks.

For example, Host A sends malicious packets to Host B but modifies the packet’s source IP address to match that of Host C. Host B recognizes Host C as a trusted source and accepts the packets, allowing Host A to bypass the firewall/IDS.

## Source Routing 
**Source Routing** is a technique where the sender decides the path a data packet takes through the network, instead of letting routers choose the route. Attackers use this to bypass secure or monitored areas, reducing the chance of detection by firewalls or IDS.

There are two types:

* **Loose Source Routing**: The sender picks some key points the packet must pass through.
* **Strict Source Routing**: The sender controls the exact full path of the packet.

This method helps attackers avoid security systems by taking less-protected routes.

## Tiny Fragments 
**Tiny Fragments** is a technique used by attackers to break a malicious packet into very small pieces. This splits up the TCP header, making it hard for **Intrusion Detection Systems (IDS)** to detect the full attack. If the firewall or IDS only checks the first fragment, the rest—carrying harmful data—can slip through undetected. This method helps attackers bypass security filters that rely on analyzing complete packet headers.

## Bypass an IDS/Firewall Using a Proxy Server 
Steps to be followed to bypass a IDS/firewall using a proxy server: 
1. Find an appropriate proxy server.
2. In a Windows system, Go to **Control Panel**, select **Network and Internet** → **Internet Options** and in the **Internet Options** dialog box under **Connections** tab, click **“LAN settings”**
3. Under **LAN Settings**, click on the **“Use a proxy server for your LAN”** checkbox
4. In the **Address** box, type the **IP address** of the proxy server
5. In the **Port** box, type the **port number** that is used by the proxy server for client connections (by default, 8080)
6. Click to select the **“Bypass proxy server for local addresses”** checkbox if you do not want the proxy server computer to be used when connected to a computer on the local network
7. Click **OK** to close the **LAN Settings** dialog box
8. Click **OK** again to close the **Internet Options** dialog box

## Bypassing an IDS/Firewall through the ICMP Tunneling Method 
**ICMP Tunneling** is a method attackers use to bypass firewalls and IDS by misusing the ICMP protocol, which is normally used for tools like **ping**. Since ICMP is often left open and its **data section isn’t inspected**, attackers hide malicious data inside ICMP Echo packets.

By using tools like **ICMPTX** [https://codeberg.org], they can send commands and even create a backdoor shell through the ICMP payload. Because most security systems don’t check this part of the packet, it becomes a hidden channel for attackers to access and control target networks without being detected.

Steps Involved in ICMP Tunneling to Bypass Firewall/IDS
- The attackers install the ICMPTX tool (https://codeberg.org) on their system.
- Attackers insert malicious client commands or payloads into the data portion of the ICMP Echo packets. ICMP packets with embedded payloads are sent through the network.
- The firewall/IDS, assuming that ICMP packets are part of legitimate traffic, allows them to be sent to the client without thorough inspection.

## Bypassing an IDS/Firewall through the ACK Tunneling method 
**ACK Tunneling** is a technique attackers use to sneak past firewalls and IDS by hiding malicious data in **ACK packets**. Normally, firewalls focus on **SYN packets** that start a connection and trust ACK packets, which are used to confirm data transfer in an already established session. Since firewalls and IDS often don't check ACK packets closely, attackers exploit this by placing harmful code inside them. This lets them run backdoor programs and avoid detection while passing through network security filters.

Steps Involved in ACK Tunneling 
- An attacker establishes a legitimate TCP connection with a target server.
- Using tools such as Hping or Nping, an attacker crafts ACK packets containing malicious payloads. These packets are sent to the target by leveraging an existing session to avoid detection.
- Assuming that the ACK packets are part of legitimate traffic, the firewall allows them without thorough inspection. Similarly, an IDS may overlook these packets if it focuses primarily on SYN packets or other types of traffic.

## Bypassing an IDS/Firewall through the HTTP Tunneling Method 
**HTTP Tunneling** is a technique attackers use to bypass firewalls and IDS by hiding malicious traffic inside normal-looking HTTP traffic on **port 80**, which is usually allowed by most networks. Since many firewalls don’t check the contents of HTTP packets, attackers can tunnel other data through it without detection. Tools like **Chisel**, **HTTPort**, **HTTHost**, and **Tunna** help attackers do this, allowing them to secretly access blocked services or control compromised systems.

### Why do I Need HTTP Tunneling?
**HTTP Tunneling** is used when firewalls or proxies block certain applications or ports. It allows restricted apps—like FTP, chat tools, or games—to work by hiding their data inside normal web traffic (HTTP on ports 80 or 443). This way, the blocked traffic passes through firewalls as if it were regular web traffic. The tunnel creates a two-way virtual connection, letting users access blocked services, browse restricted sites, or securely share data—even when direct access is denied by network rules.

## Bypassing Firewalls through the SSH Tunneling Method 
**SSH Tunneling** is a method attackers use to bypass firewalls and IDS by hiding blocked or unencrypted traffic inside a secure SSH connection. For example, if **FTP** is blocked, the attacker can send FTP data through an **SSH tunnel**, making it look like allowed traffic. Tools like **OpenSSH** help encrypt and tunnel this traffic from the local machine to a remote server, allowing attackers to avoid detection and firewall restrictions by routing it through allowed ports like **port 80**.

**Example:**
```bash
# Command to forward local port 5000 to remote server's port 25 (SMTP) over SSH
ssh -f user@certifiedhacker.com -L 5000:certifiedhacker.com:25 -N

# Breakdown:
# ssh                        => Starts the SSH client
# -f                         => Sends the session to the background after authentication
# user@certifiedhacker.com   => Logs into the remote server as 'user'
# -L 5000:certifiedhacker.com:25 => Forwards local port 5000 to certifiedhacker.com's port 25 (SMTP)
# -N                         => No remote command will be executed (used for tunneling only)
```

### Performing SSH Tunneling for Bypassing Firewalls Using Bitvise 
🔗Source: [https://www.bitvise.com]

**Bitvise SSH Server** allows secure remote access to Windows systems by encrypting data during transmission. It's commonly used for remote administration, secure file transfers (SFTP), and setting up VPN-like connections using SSH tunneling. Attackers can misuse this tool to **bypass firewalls** or access **restricted network services** by creating **SSH tunnels** through **port forwarding**, making it a powerful tool for both legitimate and malicious purposes.

The following are the various port-forwarding methods used to perform SSH Tunneling using the Bitvise tool: 

#### Local Port Forwarding
**Local Port Forwarding** is a technique where attackers use an **SSH tunnel** to redirect a local port to a remote port, making remote services appear as if they are running on the local machine. This allows them to secretly access internal systems that are normally not reachable from outside the network.

Steps to perform local port forwarding: 
1. **Connect to Target Server**
   Open Bitvise SSH Client and enter the target server’s IP, SSH port (usually 22), and login credentials.

2. **Configure Port Forwarding**
   Go to the *C2S* tab → click **Add** → set:

   * **Type**: Local
   * **Listen Interface**: `127.0.0.1`
   * **Listen Port**: (any unused local port, e.g., `8080`)
   * **Destination Host**: Internal server IP
   * **Destination Port**: Service port to access (e.g., `80`, `443`, etc.)

3. **Establish Connection**
   Click **Login** to start the SSH session and enable port forwarding.

4. **Access the Internal Service**
   On your browser, visit `http://localhost:8080`. Traffic is securely tunneled to the internal service via SSH.

#### Remote Port Forwarding 
**Remote Port Forwarding** is a technique that lets attackers make a local service on a victim's machine accessible from their remote system. This allows them to reach and interact with internal apps or development tools on the target network from a distance, even if those services aren’t normally exposed to the internet.

Steps to perform remote port forwarding: 
1. **Open Bitvise SSH Client** and enter the SSH server’s IP, port, and your login credentials.
2. Under **C2S (Client-to-Server)**, click **"Add"**, choose:

   * **Type**: Remote
   * **Listen Interface**: `0.0.0.0` (listen on all interfaces)
   * **Listen Port**: any unused remote port
   * **Destination Host**: `localhost`
   * **Destination Port**: local service port
     
3. Click **Login** to start the SSH session.
4. Now from another remote machine, you can visit
   `http://<remote-server-ip>:<listen-port>`
   to access the local service via the remote server.

#### Dynamic Port Forwarding
**Dynamic Port Forwarding** lets attackers create a **SOCKS proxy** through an **SSH tunnel**, allowing them to secretly route any network traffic. This method is very flexible and helps bypass **firewalls**, **IDS**, and other security controls, making it easier to access unauthorized services without being detected.

Steps to perform dynamic port forwarding:
1. **Open Bitvise SSH Client** and enter the **target server’s IP and port**. Fill in your **SSH login credentials**.
2. Under the **C2S tab**, click **“Add”**, choose type **Dynamic**, set `127.0.0.1` as the **Listen Interface**, and assign a **local port** (e.g., `1080`).
3. Click **“Login”** to establish the SSH connection.
4. Finally, **set your browser or app** to use a **SOCKS proxy** at `localhost:1080`.

## Bypassing Firewalls through the DNS Tunneling Method 
**DNS Tunneling** is a method attackers use to bypass firewalls and IDS by hiding data inside normal-looking DNS traffic. Since DNS is usually trusted and uses **UDP** with a 255-byte limit, attackers split and encode malicious data into DNS queries and responses. This hidden data travels through the firewall unnoticed. Even **DNSSEC** can't detect it because the traffic appears legitimate. Tools like **iodine** and **dnscat2** help attackers create and manage these tunnels, making it easier to secretly steal data from a network.

### DNS Tunneling using iodine 
Attackers use tools such as iodine and dnscat2 to tunnel traffic across DNS port 53. 

#### iodine 
🔗Source: [https://code.kryo.se]

**Iodine** is a tool that lets users bypass firewalls by sending internet data through **DNS queries**. It's useful when normal internet access is blocked, but DNS is still allowed.

It has two parts:

* **Iodined (server)** runs on a server with a public IP. It decodes data from the client, accesses the internet for them, and sends responses back as DNS replies.
* **Iodine (client)** runs on the user’s device, converting internet traffic into DNS queries and sending them to the iodined server.

Attackers can use this to secretly tunnel data through DNS when other traffic is blocked.

## Bypassing an IDS/Firewall through MITM Attacks 
**Bypassing IDS/Firewalls through MITM Attacks** involves using **Man-in-the-Middle (MITM)** techniques to sneak past security systems. Instead of attacking the firewall or IDS directly, attackers target **DNS servers**—by hijacking them or sending fake DNS responses. This lets them redirect traffic and bypass security checks without being noticed, making it a stealthy and effective method to evade network defenses.

Steps to be followed to bypass a firewall/IDS through MITM attacks: 
1. Attacker performs DNS server poisoning, corrupting the DNS cache with false information
2. User A requests for www.certifiedhacker.com from the corporate DNS server
3. Corporate DNS server sends the IP address (127.22.16.64) of the attacker
4. User A accesses the attacker’s malicious server
5. The attacker connects to the real host and tunnels the user’s HTTP traffic, making it appear as legitimate traffic to the firewall/IDS
6. The malicious code embedded in the attacker’s webpage is downloaded and executed on the user’s machine, bypassing firewall/IDS detection

## Bypassing an IDS/WAF using an XSS Attack 
**Bypassing an IDS/WAF using an XSS attack** involves injecting malicious HTML or JavaScript code into a vulnerable web application. Attackers exploit how the app handles user input and server responses to sneak past security systems like **IDS (Intrusion Detection Systems)** or **WAF (Web Application Firewalls)**, allowing them to run harmful scripts on the site without being detected.

### Using ASCII values 
**Using ASCII Values** is a technique attackers use to bypass security filters like IDS or WAF. Instead of writing a script directly (like `<script>alert("XSS")</script>`), they convert the payload into **ASCII values** so it’s harder for filters to detect.

For example, the payload `alert("XSS")` becomes:
`String.fromCharCode(97, 108, 101, 114, 116, 40, 34, 88, 83, 83, 34, 41)`

Then it’s used like this:
`<script>String.fromCharCode(97, 108, 101, 114, 116, 40, 34, 88, 83, 83, 34, 41)</script>`

When this runs, JavaScript converts it back into `alert("XSS")`, bypassing filters that would normally block the direct script.

### Using Hex Encoding
**Using Hex Encoding** is a trick attackers use to hide harmful XSS code by converting it into hex values. This helps them **bypass WAF or IDS filters** that normally block such scripts. Attackers use online websites such as [http://www.convertstring.com/EncodeDecode/HexEncode] to convert the XSS payload into equivalent Hex values. For example, a simple XSS payload like `<script>alert("XSS")</script>` can be hex-encoded as:

```
%3C%73%63%69%72%70%74%3E%61%6C%65%72%74%28%22%58%53%53%22%29%3C%2F%73%63%72%69%70%74%3E
```

This encoded version looks harmless to filters but still works the same in the browser, making it a useful evasion technique for attackers.

### Using Obfuscation
Attackers use the obfuscation technique to bypass the IDS/WAF. In this technique, attackers use a combination of upper-and lower-case letters in the XSS payload. For example, consider the following XSS payload: 

`<script>alert("XSS")</script>` 

Using obfuscation, the above payload is replaced with 

`<sCRiPt>aLeRT("XSS")</sCriPT>` 

The above payload bypasses the IDS/WAF successfully

## Other Techniques for Bypassing WAF 

### Using HTTP Header Spoofing
**HTTP Header Spoofing** is a technique where attackers fake HTTP headers to make it look like their request is coming from inside the network. Some web application firewalls (WAFs) trust internal traffic and allow certain requests. Attackers take advantage of this by spoofing headers to bypass security checks and trick the server into accepting their malicious requests.

The following extension headers are automatically appended to the requests, which can be exploited by the attackers to bypass WAF: 
```
X-Originating-IP: 127.0.0.1
X-Forwarded-For: 127.0.0.1
X-Remote-IP: 127.0.0.1
X-Remote-Addr: 127.0.0.1
```
Attackers use tools such as Burp Suite to exploit HTTP headers and bypass WAF.

### Using Blacklist Detection 
**Using Blacklist Detection** is a method attackers use to bypass Web Application Firewalls (WAFs). They first analyze the WAF to find which keywords are blocked (blacklisted). Then, they craft new payloads using different words or patterns that are not on the blacklist, allowing them to slip past the WAF undetected.

```mysql
# Blacklist Filtering Example 1:
# Blocked keywords: and, or, union
# Blocked query:
union select username, pwd from employees
# => Detected and blocked due to use of "union"

# Bypassed using alternative logic:
1 || (select username, pwd from employees where userID = 1001) = 'admin'
# => This bypasses the blacklist by avoiding "union" and using a logical OR with a subquery

# Blacklist Filtering Example 2:
# Blocked keywords: and, or, union, where, limit
# Blocked query:
1 || (select username from employees limit 1) = 'admin'
# => Detected and blocked due to use of "limit"

# Bypassed using GROUP BY and HAVING:
1 || (select username from employees group by userID having userID = 1001) = 'admin'
# => Avoids the banned "limit" and "where" by using "group by" and "having" instead
```

### Using Fuzzing/Brute-forcing
**Fuzzing/Brute-forcing** is a technique where attackers test a **Web Application Firewall (WAF)** by sending many known payloads to find out which ones get blocked and which ones pass through. First, they test on a WAF in their own setup to identify bypassing payloads. Then, they use those working payloads against the target WAF.

To do this, attackers use wordlists like **Assetnote Wordlists** [https://wordlists.assetnote.io] and **SecLists** [https://github.com/danielmiessler/SecLists], load them into a fuzzer, and start brute-forcing. They track the responses for each payload, then hide their identity using **random user-agents** and **proxy chains** to avoid detection.

### Abusing SSL/TLS ciphers 
**Abusing SSL/TLS Ciphers** is a method attackers use to bypass Web Application Firewalls (WAFs). Sometimes, a web server supports more encryption ciphers than the WAF. Attackers first check which ciphers the WAF and server support using tools like **sslscan2** [https://github.com/rbsec/sslscan]. If they find a cipher that the server supports but the WAF doesn't, they use that cipher to sneak past the WAF and connect directly to the server. Tools like **abuse-ssl-bypass-waf.py** [https://github.com/LandGrey/abuse-ssl-bypass-waf] and **curl** [https://curl.se] help perform this attack.

## Bypassing an IDS/Firewall through HTML Smuggling 
**HTML Smuggling** is a sneaky web attack where hackers hide malicious code inside HTML or JavaScript files to bypass firewalls and security tools. The attacker sends a phishing email with a **malicious HTML link or attachment**. When the victim opens it, the browser runs hidden JavaScript that builds and downloads the malware directly on the system—without raising alarms. Tools like **HTML Smuggler** help attackers create these payloads. Once executed, the malware gives the attacker remote access for further attacks.

### Working of HTML Smuggling 
```bash
# Step 1: Attacker prepares a malicious HTML or web page with a disguised file
# This is a simple download link that looks innocent to the user
<a href="malicious.doc" download="Myfile.doc">Click Here</a>
# This triggers the download of 'malicious.doc' as 'Myfile.doc'

# Step 2: JavaScript version – attacker hides payload generation in script
var myAnchor = document.createElement("a");
myAnchor.download = "Myfile.doc";  # File name shown to the user

# Step 3: Attacker generates a fake file using Blob API
var myfakeFile = "malicious code here";  # Hidden payload
var fakeBlob = new Blob([myfakeFile], { type: 'application/octet-stream' });

# Step 4: Create a hidden download URL from the blob and auto-click it
var myfileUrl = window.URL.createObjectURL(fakeBlob);
myAnchor.href = myfileUrl;
myAnchor.click();  # Simulates user click — starts download automatically

# Result: 
# - No external request is sent for the file (evades firewall/IDS)
# - File is generated in the browser and saved locally
# - Antivirus may not scan blob-generated files in memory

# The JavaScript hides the malicious content in the blob,
# evading detection by network-based security tools.
```
### Signs of HTML Smuggling 
Following are the signs of malware-smuggling HTML attachments:
- A ZIP file with JavaScript
- An encrypted attachment
- Existence of suspicious a script code within a HTML-based file
- Decoding of HTML-based file using Base64

### Countermeasures
- Block auto-execution of .js and .jse files.
- Ensure Office 365 email security filtering is actively barring auto-download of malware-loaded mails.
- Verify the perimeter operation of security devices such as firewall and proxy restricting arbitrary server connections to the Internet.
- Recommend user to access web browser activated with Microsoft Defender SmartScreen and network protection to avert and block malicious access.
- Enable cloud delivery-based protection using AI and ML technology for identification and aversion of threats.
- Implement CSP headers on the web server.
- Define a whitelist of allowed HTML tags, attributes, and protocols.
- Encode all dynamic content before rendering it in the browser.

## Evading an IDS/Firewall through Windows BITS 
**Evading IDS/Firewall through Windows BITS** involves abusing a trusted Windows service called **Background Intelligent Transfer Service (BITS)**, which is used for updates. Since BITS traffic is often ignored by security tools, attackers use it to secretly download or upload malicious files. Even when browsers like Chrome or Firefox are closed, BITS can still transfer data. This makes it easy for attackers to create hidden tasks that bypass firewalls and IDS, allowing malware to run or backdoors to open without being detected.

**Background Intelligent Transfer Service (BITS)** allows programs to transfer files in the background, even when they’re not actively running. Attackers can abuse this by creating BITS jobs that trigger malicious commands or programs once the transfer is complete. These jobs can be created using **bitsadmin** commands or API calls, helping malware stay hidden and persistent.

- **Downloading the malicious binary:** Run the following `bitsadmin` command to create a job, transfer the malicious file to the remote system, and store it at the specified location.


- Avoid downloading suspected programs or files from the Internet or email.

- Integrate with a reputed SIEM solution to automate the monitoring and alerting process of suspicious BITS activities.
- Use Group Policy Objects (GPO) to manage BITS settings.
- Configure the settings for BITS to limit the maximum job age and prevent jobs from running immediately.
- Conduct regular audits of BITS jobs.
- Limit the ability of user accounts to create BITS jobs unless necessary.

## Other Techniques for IDS Evasion 

### Insertion Attack 
**Insertion Attack** is a technique where attackers confuse an IDS by sending invalid or altered packets that the real system ignores but the IDS accepts. This tricks the IDS into seeing fake data, making it miss the actual attack.

For example, the attacker sends a packet that appears harmful to the IDS but is rejected by the target system due to errors like a bad **checksum** or **TTL** value. The IDS may detect something like “phoneyf” instead of the real attack string “phf,” causing it to miss the threat. As a result, the IDS thinks the traffic is safe, while the attacker successfully hides the attack from detection.

<img width="751" height="292" alt="image" src="https://github.com/user-attachments/assets/a4d07117-52f4-43f6-a189-3fe9220a07d8" />

### Evasion 
**Evasion** is a technique where attackers send malicious data in a way that the **host accepts**, but the **IDS ignores or misses**. This causes the IDS to see an incomplete version of the attack, making it fail to detect it. For example, if an attacker sends harmful data **byte by byte** and the IDS drops just one byte, it might not recognize the full attack. Another trick is putting data inside **TCP handshake packets**, which some IDS may ignore—allowing attackers to bypass detection and compromise the target system silently.

<img width="714" height="305" alt="image" src="https://github.com/user-attachments/assets/a860d201-d14c-41a0-b24f-811476d83b8a" />

### Denial-of-Service Attack (DoS) 
**Denial-of-Service (DoS) attacks** against IDS are designed to overwhelm and disable the system so it can't detect threats. Attackers target resources like **CPU**, **memory**, **disk space**, and **network bandwidth**. They overload the IDS with unnecessary tasks, forcing it to waste processing power and memory on useless data. Attackers can also fill up disk space by triggering too many fake alerts or flood the network to make the IDS miss real threats. If a **central log server** is used, attackers may crash it with a DoS attack to stop alerts from being recorded, allowing further attacks to go undetected.

Using this evasion technique, an attacker
- Causes the device to lock up
- Causes personnel to be unable to investigate all the alarms
- Causes more alarms than can be handled by management systems (such as databases, etc.)
- Fills up disk space, preventing attacks from being logged
- Consumes the device’s processing power and allows attacks to sneak by

### Obfuscating 
**Obfuscating** is a technique used to hide the true purpose of code or data, making it hard to read or detect. Attackers use it to bypass **IDS** by encoding attack traffic in a way that the **target system can understand**, but the IDS cannot. They may use **Unicode characters**, **polymorphic code**, or **steganography** to hide malicious payloads. This helps them sneak past security systems, especially over encrypted channels like **HTTPS**, and deliver malware without raising alerts.

### False Positive Generation 
**False Positive Generation** is a trick where attackers flood an IDS with fake alerts by sending harmless packets designed to look suspicious. This doesn’t harm the system directly but overwhelms the IDS with noise, making it hard to spot real attacks. By hiding true malicious activity among the false alarms, attackers can slip past security unnoticed, especially if the IDS is already known to them.

### Session Splicing 
**Session Splicing** is an IDS evasion technique where attackers split malicious data into many tiny packets, making it hard for the IDS to detect the attack. Since some IDS don’t fully rebuild these packets before checking them, the attack slips through unnoticed. Attackers can also add time delays between packets, causing the IDS to stop tracking the session. This allows them to send malicious data without being detected. Tools like **Nessus** can be used to carry out such attacks.

### Unicode Evasion Technique 
**Unicode Evasion** is a technique where attackers use special character encodings to hide malicious input and bypass security filters. Unicode allows text from all languages to be displayed consistently, and formats like **UTF-8** and **UTF-16** are widely supported. Attackers take advantage of this by encoding characters in different ways.

For example, instead of using a normal **/**, they might use **%u2215** in UTF-16, or **%c2%a9** to represent **©** in UTF-8. This tricks security systems into missing the real meaning of the input, allowing harmful data to slip through undetected.

**Problems with Unicode:**
**Problems with Unicode** arise because a single character can have multiple valid Unicode representations. This makes it hard for IDS to correctly detect threats, as pattern-matching becomes unreliable. Attackers take advantage of this by converting attack strings or URLs into Unicode, which can confuse IDS systems and let malicious traffic pass through undetected.

### Fragmentation Attack 
**Fragmentation Attack** is a technique where attackers break malicious IP packets into smaller fragments to bypass security checks. Since networks split large packets into pieces, systems and IDS tools reassemble them to read the full message. Attackers take advantage of differences in how long systems wait to reassemble fragments. By sending the fragments slowly or in tricky ways, they can sneak harmful data past the IDS without triggering an alert, making it harder to detect the attack.

#### Attack Scenario - 1
In this **attack scenario**, the attacker takes advantage of a timing mismatch between the IDS and the target system. If the IDS drops packet fragments after **10 seconds**, but the target waits **20 seconds**, the attacker sends fragments **15 seconds apart**. The IDS ignores them, thinking they’re incomplete, but the target reassembles them correctly. This way, the attacker delivers the attack payload without being detected by the IDS.

<img width="656" height="498" alt="image" src="https://github.com/user-attachments/assets/0ec31f5d-c933-4b4a-a52a-e168c5d79273" />

#### Attack Scenario - 2
In this **fragmentation attack**, the attacker takes advantage of different timeout settings between the **IDS** and the **victim system**. The IDS waits **60 seconds** to reassemble packet fragments, while the victim only waits **30 seconds**.

The attacker first sends **fake fragments (frag-2' and frag-4')** to both the IDS and the victim but doesn't send **frag-1**, so the victim drops them after 30 seconds. The IDS, however, still holds onto them. Later, the attacker sends **frag-1 and frag-3** with valid data. Now, the IDS has a mix of fake and real fragments and discards them due to a checksum mismatch.

Finally, the attacker sends **frag-2 and frag-4 again**, this time with valid data. The IDS ignores them, thinking the attack is already handled, but the **victim receives all valid fragments** and reassembles them into a complete, malicious packet—successfully bypassing the IDS.

<img width="732" height="516" alt="image" src="https://github.com/user-attachments/assets/8ea3e274-d415-4f59-81e3-3e624949230a" />

### Time-To-Live Attacks
**Time-To-Live (TTL) Attacks** exploit how routers handle TTL values in IP packets. Each time a packet passes through a router, its TTL decreases by 1. When it hits 0, the packet is dropped. Attackers use this to guess how many routers exist between them and a target, and even identify the target's operating system.

In one method, attackers who know the network layout send a malicious packet split into fragments:

* **Frag-1** and **Frag-3** have high TTLs and reach both IDS and victim.
* **Frag-2′** (a fake piece) has a TTL of 1 and only reaches the IDS.
  The IDS wrongly reassembles fragments with the fake **Frag-2′** and thinks it's safe. Later, the real **Frag-2** with the attack code is sent to the victim, which reassembles the full malicious packet. This tricks the IDS and allows the attack to succeed unnoticed.

<img width="707" height="447" alt="image" src="https://github.com/user-attachments/assets/80f407cc-20e9-4b5b-8b7b-dbaf868955cc" />

### Urgency Flag 
**Urgency Flag** in TCP marks certain data as urgent so it can be processed quickly. It uses an **Urgent Pointer** to show where the urgent data ends, and the system skips anything before it. Some **IDS tools ignore this flag** and read all the data, while the actual system processes only the urgent part.

Attackers use this mismatch to trick IDS. They hide garbage data before the urgent part, which IDS sees but the system ignores. This causes the IDS and the target system to process different data, helping attackers bypass detection.

**Example:**
```
Packet 1: XYZ  
Packet 2: LMN  (Urgent Pointer: 3)  
Packet 3: PQR  
```
**Final Output:** `XYZLMNQR` (but **1 byte after urgent data is lost** as per RFC 1122).

### Invalid RST Packets
**Invalid RST Packets** are used by attackers to trick an IDS into thinking a network connection has ended. They do this by sending a **fake RST (reset) packet** with a **wrong checksum**. The IDS sees this and may stop analyzing the session, thinking it's over. But the actual receiving system checks the checksum, sees it's invalid, and ignores the packet—**keeping the connection alive**. This lets the attacker keep sending data without being detected by the IDS.

### Polymorphic Shellcode 
**Polymorphic Shellcode** is a technique attackers use to hide malicious code from security systems like IDS. Instead of sending the same code every time, they **encrypt the payload** and add a **decoder** in front of it. Each time the shellcode is sent, it looks different, making it hard for signature-based IDS to detect it. The attack often uses **buffer overflow**, setting the return address to run the decoder, which then unlocks the actual malicious code. Since the shellcode changes constantly and avoids known patterns, IDS systems fail to recognize and block it.

### ASCII Shellcode
**ASCII Shellcode** is a special type of shellcode made using only ASCII characters. Attackers use it to bypass filters that block non-ASCII input and to avoid detection by IDS, as these systems often fail to catch ASCII-based patterns. While using only ASCII is limiting—since not all instructions can be directly written in ASCII—attackers work around this by using alternate or combined instructions that still perform the intended actions.

An ASCII shellcode example is given below: 
```ASCII
char shellcode[] =
"LLLLYhb0pLX5b0pLHSSPPWQPPaPWSUTBRDJfh5tDS"

"6rfYf1Lki0tkkh95h8Y1LkmjpY0Lkq0tkrh2wnuX1"
"Dks0tkwjfX0Dkx0tkx0tkyCjnY0LkzC0TkzCCjtX0"
"DkzC0tkzCj3X0Dkz0TkzC0tkzChjG3IY1LkzCCCC0"
```

### Application-Layer Attacks
**Application-Layer Attacks** using media files work by hiding malicious code inside **compressed images, audio, or video**. Since compression changes how data looks, many **IDS tools can’t scan** it properly for threats. Attackers take advantage of flaws in how apps handle this compressed data, making it easy to sneak in attacks unnoticed. They can also use tricks like **integer overflow** in different ways, making it even harder for IDS to detect. This makes application-layer attacks through compressed files a serious challenge for traditional security systems.

### Desynchronization 

#### Pre-Connection SYN 
**Pre-Connection SYN Attack** is a trick where attackers send a fake SYN packet with a bad TCP checksum before starting a real connection. This can confuse the **IDS**, making it miss or misread the actual traffic. If the IDS syncs with the fake sequence number, it becomes desynchronized from the real session, allowing attackers to bypass detection. Smart IDS systems may ignore bad checksums to avoid this trick.

#### Post-Connection SYN
**Post-Connection SYN** is an IDS evasion trick where attackers send a fake SYN packet after a connection is already established. The real host ignores it because the connection is active, but the IDS may get confused and think a new connection has started. This causes the IDS to lose track of the real data flow. After that, the attacker sends a fake RST packet to shut down the IDS’s view of the connection, while the real connection continues unnoticed.

### Domain Generation Algorithms (DGA) 
**Domain Generation Algorithms (DGA)** are tools used by attackers to create large numbers of random domain names. These domains help malware secretly connect to **command-and-control (C2) servers** without using fixed IPs or known domain names, making detection harder.

The domains often look like gibberish (e.g., `isdfcbdjdnfuylt.ru`) and are created using a shared **seed** known to both the malware and the attacker’s server. If one domain is blocked, the algorithm simply generates a new one, helping the attacker stay connected and continue the attack without being easily detected by security systems.

<img width="707" height="178" alt="image" src="https://github.com/user-attachments/assets/c90b2b27-17a5-4b17-9b6d-e1641c95ebbe" />

#### Types of DGAs
Attackers can develop various types of DGAs to enhance their capabilities for creating malicious domains and evading detection mechanisms. Some of the major types of DGAs are as follows:
- **Character-based DGAs:** This method uses random seeds to create domain names containing letters or numbers. For example, “ab5cde8.com” or “xy12z34.net”.
- **Pseudorandom number generator (PRNG) DGAs:** This method uses a random seed that often includes the system date and time to create sequences of domains. In addition, both attackers and malware can predict these sequences.
  
  For example, if the random seed generates the sequence 837291056, and each number is mapped to a corresponding letter (e.g., 1 = ‘a,’ 2 = ‘b,’ etc.), the generated domain could be something like “hbeajidfg.com”.

- **Dictionary-based DGAs:** It generally randomly combines words to create domains that appear readable. These domains are more difficult for security systems to detect because they resemble real systems. For example, “applebanana.com”, “orangekiwi.net”, etc.
- **High-collision DGAs:** These mimic real domain names and use popular top-level domains (TLDs), such as .com, .org, and .net. As a result, there is a high probability that the generated domain has already been taken, which can confuse detection systems. For example, “test.com”, “demo.org”, etc.

### Encryption 
**Encryption** can be used by attackers to hide their malicious traffic from network-based IDS. When they create secure connections like **SSH**, **SSL**, or **VPN tunnels**, the data is encrypted, so the IDS can't see what's inside. This allows attackers to send harmful commands or data without being detected.

### Flooding 
**Flooding** is an IDS evasion technique where attackers overload the IDS with fake or noisy traffic. This drains its memory and processing power, making it too busy to detect real threats. Once the IDS is overwhelmed, attackers send the actual malicious traffic, which often goes unnoticed and reaches the target system.
