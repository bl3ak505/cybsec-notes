## Intrusion Detection using YARA Rules 
🔗Source: [https://yara.readthedocs.io] 

**YARA** is a security tool used to detect and identify malware by creating custom rules based on patterns found in files. It works on **Windows, macOS, and Linux**, and lets analysts define **text, hex, or binary patterns** to spot threats. These rules use **Boolean logic** to decide if a file is harmful. YARA is useful for scanning files, malware databases, or entire systems to detect intrusions and classify malware quickly and accurately.

YARA rules adhere to a syntax where each rule starts with the word “rule” before specifying its name. A rule is comprised of three parts, which are as follows: 
* **Metadata**: Gives general info about the rule—like author, description, or purpose—for easier identification.
* **Strings**: Lists the text, hex, or regex patterns to search for in files.
* **Condition**: Uses Boolean logic to decide **when** the rule should trigger a match based on the defined strings.

> Think of YARA as a malware detection "search filter"—metadata labels it, strings define what to look for, and the condition tells when to flag a file.

An example of a YARA rule as per YARA’s official documentation is as follows: 
```yara
rule silent_banker : banker
{
    meta:
        description = "This is just an example"
        threat_level = 3
        in_the_wild = true

    strings:
        $a = { 6A 40 68 00 30 00 00 6A 14 8D 91 }
        $b = { 8D 4D B0 2B C1 83 C0 27 99 6A 4E 59 F7 F9 }
        $c = "UVODFRYSIHLNWPEJXQZAKCBGMT"

    condition:
        $a or $b or $c
}
```
From the above syntax, security experts can identify that any of those three strings defined must be considered as the “silent_banker” Trojan. In addition, the strings in the above example are comprised of both hex and text patterns.

## Intrusion Detection Tools
**Intrusion Detection Tools** are security programs that watch network traffic for suspicious activity. They read all data packets, rebuild user sessions, and look for signs of attacks or unusual behavior. These tools help detect threats like malware, spyware, port scans, viruses, and even DoS or DDoS attacks. Running in real-time, they also offer protection against new (zero-day) threats, helping prevent systems from being compromised.

### Snort
🔗Source: [https://www.snort.org]

**Snort** is a free, open-source tool used to monitor network traffic in real time and detect suspicious activity. It analyzes data packets, checks for known attack patterns like **buffer overflows**, **port scans**, and **CGI attacks**, and logs them for review. Snort uses custom rules to decide what traffic to watch or ignore and has a flexible plug-in system to extend its detection abilities. It’s widely used for identifying threats on IP networks quickly and effectively.

Uses of Snort:
- Straight packet sniffer such as tcpdump
- Packet logger (useful for network traffic debugging, etc.)
- Network intrusion prevention system

#### Snort Rules 
**Snort Rules** are used to detect and alert on suspicious network activity by defining what is considered malicious. Snort uses libraries like **libpcap** or **Winpcap** to capture network traffic, similar to **tcpdump**. When Snort is in **promiscuous mode**, it watches all packets on the network. Based on the rules set in its config file, it analyzes each packet and triggers alerts if something matches known attack patterns. This helps separate normal traffic from potential threats.

Snort allows users to write their own rules. However, each of these Snort rules must describe the following: 

* Any activity that breaks the company’s security policy or threatens the network.
* Common or known attacks trying to exploit weaknesses in the system.
* Any packet that looks suspicious or doesn’t seem genuine.

> These rules help Snort spot and alert on possible intrusions in real-time.

**Snort rules** are used to detect and respond to suspicious network activity. These rules are written in a simple, lightweight language that makes them easy to create and understand. For strong security, Snort rules should be **robust**—able to closely monitor traffic and alert admins about threats—and **flexible**—able to respond quickly based on the type of intrusion. This balance helps protect networks effectively and take action when needed.

Both flexibility and robustness can be achieved using an easy-to-understand and lightweight rule-description language that aids in writing simple Snort rules. Consider the following two primary principles while writing Snort rules: 

* **Keep rules short and on one line** — this makes them easy to read and manage.
* **Split each rule into two clear parts**:

  * **Rule Header**: Defines what traffic to watch (like IPs, ports, and protocol).
  * **Rule Options**: Describes what to look for in the traffic (like keywords, alerts, etc.).

> This structure makes Snort rules lightweight, easy to write, and simple to understand.

The rule header contains the rule’s action, the protocol, the source and destination IP addresses, the source and destination port information, and the Classless Inter-Domain Routing (CIDR) block. The rule option section includes alert messages in addition to information about the inspected part of the packet to determine whether to take any rule action.

#### Snort Rules: Rule Actions and IP Protocols
**Snort Rules** help detect and respond to suspicious network traffic. Each rule starts with a **rule action** that tells Snort what to do when a packet matches certain conditions. Actions include **alert**, **log**, **pass**, **drop**, **reject**, and **sdrop**. The **rule header** defines who sent the packet, where it’s going, what protocol is used, and what to do if it matches.

**IP Protocol** is used to send data across the internet. It breaks data into packets, each containing the source, destination, and message, allowing systems to communicate with unique addresses.

Snort supports three available IP protocols to tackle suspicious behavior:
- **TCP:** The Transmission Control Protocol (TCP) is a part of the IP. It is used to connect two different hosts and exchange data between them.
- **UDP:** The User Datagram Protocol (UDP) is used for broadcasting messages over a network.
- **ICMP:** The Internet Control Message Protocol (ICMP) is a part of the IP. The OS uses ICMP in a network to send error messages.

#### Snort Rules: Direction Operator and IP Addresses 

##### Direction Operator
**Direction Operator** in Snort defines the direction of network traffic to monitor—whether it's one-way or both ways. The bidirectional operator `<>` means the rule will check traffic going in **both directions** between the source and destination.

**Example:**
```
log tcp !192.168.1.0/24 any <> 192.168.1.0/24 23
```
This rule logs any TCP traffic between **non-local IPs** and **local IPs** using **port 23**, in **both directions**.

##### IP Addresses 
**IP Addresses in Rules** help define where network traffic is coming from or going to. You can specify exact IPs, use **"any"** for all addresses, or use **CIDR notation** to cover a range. You can also **exclude** certain IPs using negation.

**Example:**
```snort
alert tcp !192.168.1.0/24 any -> 192.168.1.0/24 111 (content: "|00 01 86 a5|"; msg: "external mountd access";)
```
This rule alerts when traffic comes from **outside** the 192.168.1.0/24 network and targets port **111** inside that network, helping detect suspicious external access.

#### Snort Rules: Port Numbers
**Snort Rules: Port Numbers** help define which network traffic should be detected or logged. You can use **specific ports**, **port ranges** (with `:`), **"any"** to match all ports, or **"!"** for negation to exclude certain ports.

The **direction operator `->`** shows traffic going from a **source** (IP/port on the left) to a **destination** (IP/port on the right). Use the **`<>` operator** for **bidirectional traffic**, which is useful for tracking both sides of a conversation, like in Telnet or POP3.

You can specify one IP or a **list of IPs** using square brackets with no spaces:
`[192.168.1.1,192.168.1.45,10.1.1.24]`

You can also use **CIDR notation** and the **"!"** operator to exclude certain IPs or ranges.

**Example of port negation** in a Snort rule:

```
log tcp any -> 192.168.1.0/24 !6000:6010
```

This rule logs all TCP traffic to the local network **except** traffic to ports **6000 to 6010**.

| Protocols             | IP Address                 | Action                                                                 |
|-----------------------|----------------------------|------------------------------------------------------------------------|
| Log UDP any ->        | 192.168.1.0/24 1:1024      | Log UDP traffic coming from any port and destination ports ranging from 1 to 1024 |
| Log TCP any ->        | 192.168.1.0/24 :5000       | Log TCP traffic from any port going to ports less than or equal to 5000 |
| Log TCP any :1024 ->  | 192.168.1.0/24 400:        | Log TCP traffic from the well-known ports and going to ports greater than or equal to 400 |
