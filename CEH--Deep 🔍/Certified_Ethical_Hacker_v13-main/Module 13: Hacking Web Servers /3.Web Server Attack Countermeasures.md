# 🛡️ Web Server Attack Countermeasures
This section focuses on how to **secure web servers** using various tools and techniques. It explains how to:

* **Detect attacks** on web servers
* **Apply countermeasures** and defense techniques
* **Use security tools** to find vulnerabilities, malware, or hacking attempts

These tools help scan the server and web applications, alert when there’s a threat, and keep the server safe from attacks.

---
## Place Web Servers in Separate Secure Server Security Segment on Network 
To protect web servers, they should be placed in a **separate network segment** called the **DMZ (Demilitarized Zone)**. This DMZ sits between the public Internet and the internal network. It acts as a buffer zone, adding extra security.

By placing web servers in the DMZ:

* You keep them **isolated** from sensitive internal systems.
* **Firewalls** can be used to control traffic between the Internet, DMZ, and internal network.
* Even if an attacker hacks the web server, they **can’t easily reach** the internal network.

This setup reduces the risk of attacks from both outside and inside the organization.

## Countermeasures: Patches and Updates 
The following are various countermeasures for secure update and patch management of web servers: 
1. **Scan & Patch Regularly** → Continuously check for vulnerabilities and apply patches to keep the server secure.

2. **Test Before Deployment** → Always test patches and updates in a non-production environment before applying them live.

3. **Review Before Applying** → Read and peer review all patch documentation to avoid unexpected issues.

4. **Have a Backup Plan** → Always back up systems and keep a recovery plan ready in case the update fails.

5. **Avoid Default Configurations** → Do not rely on default web server settings; customize for better security.

6. **Use Virtual Patching** → Apply virtual patches to detect and log threats even before official patches are released.

7. **Automate Patch Management** → Use tools like SolarWinds to automate and streamline patching processes.

8. **Set Patch Alerts** → Enable notifications for new patches to stay updated in real time.

9. **Inventory All Systems** → Maintain a list of all systems, services, and dependencies to manage updates efficiently.

10. **Integrate into SDLC** → Make patching a regular part of your development and deployment lifecycle for long-term security.

✅ **Bonus Tip:** Add patching into your **SDLC** (Software Development Life Cycle) to ensure long-term security.

## Countermeasures: Protocols and Accounts 

### Countermeasures: Protocols
The following are various countermeasures for using secure protocols on web servers: 

1. **Block Unused Ports & Protocols** → Disable unnecessary ports, ICMP traffic, and insecure protocols to reduce attack surface.

2. **Apply Patches Regularly** → Keep your OS and software updated to fix known vulnerabilities in protocols and services.

3. **Secure Remote Access** → Use tunneling and encryption (like SSH or VPN) to protect remote connections from eavesdropping.

4. **Use TLS/SSL Protocols** → Always use HTTPS (not HTTP) for encrypted and secure communication between users and servers.

5. **Redirect HTTP to HTTPS** → Force secure connections by automatically redirecting all HTTP traffic to HTTPS.

6. **Enable HSTS Headers** → Use HTTP Strict Transport Security (HSTS) to prevent protocol downgrade attacks.

7. **Use FTPS for File Transfers** → Encrypt all file transfers over the web using FTPS instead of plain FTP.

8. **Isolate Critical Services** → Keep servers like LDAP on a separate subnet and filter access with firewalls.

9. **Hide Server Info** → Configure HTTP service banners to hide OS version and server details from attackers.

10. **Automate SSL/TLS Certificate Renewal** → Prevent expired certificate issues by setting up auto-renewal.

### Countermeasures: Accounts 
The following countermeasures can be adopted to secure user accounts on a web server: 

1. **Use Least Privilege** → Give users and services only the minimum access they need. This limits damage if an account is compromised.

2. **Disable Unused Accounts** → Turn off default or inactive accounts to prevent attackers from misusing them.

3. **Strong Password Policy** → Enforce strong, complex passwords and set rules to slow down brute-force attacks.

4. **Enable 2FA or MFA** → Add another layer of security beyond passwords to protect user accounts.

5. **Account Lockout Policy** → Lock accounts after a few failed login attempts to stop password guessing.

6. **Use Password Hashing** → Store passwords using secure algorithms like bcrypt or Argon2 to prevent theft in case of a breach.

7. **Audit Login Attempts** → Monitor and log all login activities to detect unusual or unauthorized access.

8. **Remove Unused Modules** → Uninstall unnecessary software or extensions to reduce potential attack surfaces.

9. **Secure Account Recovery** → Design recovery options (like security questions) carefully to avoid account hijacking.

10. **Use CAPTCHA on Logins** → Prevent bots from trying to break into accounts by using CAPTCHA challenges.

## Countermeasures: Files and Directories
The following countermeasures can be adopted for securing files and directories on a web server: 

1. **Run Web Server with Least Privileges** → Only give the web server access to the directories and files it truly needs to minimize risk.

2. **Disable Directory Listings** → Prevent attackers from viewing the contents of folders on the web server.

3. **Use File Integrity Checkers** → Regularly scan and verify web files to detect unauthorized changes or intrusions.

4. **Store Web Files Separately from OS** → Keep website files in a different drive/partition than the OS and logs to reduce damage if hacked.

5. **Filter User Input** → Block meta characters (like `../`, `<`, `>`, etc.) to prevent directory traversal and command injection.

6. **Secure Configuration Files** → Make sure important files like `.htaccess` and `web.config` are not publicly accessible.

7. **Scan Uploaded Files** → Check user-uploaded files for malware and store them outside the web root to avoid direct access.

8. **Use WAF (Web Application Firewall)** → Defend against common attacks like SQL Injection that can access or alter files.

9. **Avoid Serving Unnecessary Files** → Remove or restrict access to backup files, text files, and archives that may reveal sensitive data.

10. **Use SFTP Instead of FTP** → Always encrypt file transfers using SFTP to protect sensitive data from being intercepted.

## Detecting Web Server Hacking Attempts 
**Detecting Web Server Hacking Attempts** involves spotting early signs of compromise, like backdoors secretly added by attackers. These backdoors let attackers control the server, send spam, or redirect visitors without the owner knowing—until it’s too late.

To catch these attacks early, a **Website Change Detection System (WDS)** can be used. It monitors the server’s files (like HTML, PHP, JS, etc.) by comparing their current hash values with original ones. If any file is added or changed, it alerts the user immediately.

**Example:** Tools like **DirectoryMonitor** scan web folders, detect code changes, and send email alerts, helping identify hacking attempts before major damage occurs.

## How to Defend against Web Server Attacks 
Defenses against web server attacks include the following. 

### Ports 
Regularly monitoring all ports on a web server helps detect and block unwanted or harmful traffic. Public access to common ports like **80 (HTTP)** and **443 (HTTPS)** should be restricted to prevent attacks like **DoS**, which can overload the server. It's important to **encrypt or limit intranet traffic** for extra security. Attackers may **spoof IP addresses** to hide their identity, but you can block such threats by analyzing logs and using **firewall rules** or **blackhole routing** to deny access to suspicious IPs.

### Server Certificates
**Server certificates** help keep online communications secure by proving a server's identity, usually through a trusted certificate authority (CA). However, attackers can use **fake or forged certificates** to launch **Man-in-the-Middle (MITM)** attacks and intercept secure data.

To avoid this, follow these key techniques: <br>
* **Directly validate certificates** without relying on third parties.
* Use **secure protocols** that don't depend on external validation.
* Let websites **check their own certificates** using known user credentials.
* Apply **strong cryptographic methods** to confirm server identity.
* Make sure certificate dates are valid and they’re used only for their correct purpose.
* Confirm that the certificate is **not revoked** and the **public key is trusted** up to the root certificate authority.

### Machine.config
The **machine.config** file is a system-level configuration file in the .NET framework that controls settings for all applications on a machine. It helps improve **security** by:

* Blocking access to sensitive resources
* Removing unused modules
* Disabling debugging and tracing
* Hiding detailed error messages from users
* Securing session settings

Changing this file applies the settings to every app running on that machine.

### Code Access Security 
**Code Access Security** helps protect web servers from running harmful or unauthorized code. To ensure this:

* Use **secure coding** to stop attacks like input tampering or source-code leaks.
* Set strict **permissions**, so code from the internet or internal network can’t run without approval.
* Block dangerous URL patterns like `"../"` to prevent **path traversal** attacks.
* Lock down system tools using **ACLs (Access Control Lists)** and keep software **updated** with the latest patches.

If these steps aren’t followed, attackers may be able to run **malicious code** on the server.

**The following are some other measures to defend against web server attacks:**

1. **Apply Restricted ACLs** → Limit who can access or manage the server using Access Control Lists.

2. **Use NTFS Permissions** → Secure files, folders, and logs by setting strict NTFS file permissions.

3. **Remove Unnecessary Features** → Delete unused ISAPI filters, script mappings, and file shares to reduce attack surfaces.

4. **Use a Dedicated Web Server** → Host your website on a separate, non-domain controller system to isolate risk.

5. **Segment the Network** → Use VPNs and secure protocols (like HTTPS) to protect communication and limit unauthorized access.

6. **Implement Role-Based Access Control (RBAC)** → Give users only the access they need and nothing more.

7. **Enable Centralized Log Monitoring** → Track server and app logs in one place to detect and respond to threats quickly.

8. **Deploy IDPS** → Use Intrusion Detection and Prevention Systems to watch and react to malicious traffic.

9. **Physically Secure the Server** → Keep the web server in a locked, secure room to prevent tampering.

10. **Turn Off Unused Features** → Disable things like directory browsing and unused scripting to reduce entry points.

## How to Defend against HTTP Response-Splitting and Web Cache Poisoning 
To defend against HTTP Response-Splitting and Web Cache Poisoning, it's important to remove any carriage return (CR) and linefeed (LF) characters from user input before adding it to HTTP response headers, especially when setting cookies. Using security testing tools can help identify and fix CRLF injection flaws. Keeping application engines updated also strengthens defense. Additionally, using UDP source port randomization helps prevent forged responses, while limiting recursive DNS queries and increasing TTL values helps reduce the impact of cache poisoning attacks.

The following are some methods to defend against HTTP response-splitting attacks and web cache poisoning:
- **Server Admin**
    - Use the latest web server software
    - Regularly update/patch the OS and web server
    - Run a web vulnerability scanner

- **Application Developers**
    - Restrict the web application’s access to unique IPs
    - Disallow CR (%0d or \r) and LF (%0a or \n) characters
    - Comply with RFC 2616 specifications for HTTP/1.1
    - Parse all user inputs or other forms of encoding before using them in HTTP headers

- **Proxy Servers**
    - Avoid sharing incoming TCP connections among different clients
    - Use different TCP connections with the proxy for different virtual hosts
    - Implement “maintain request host header” correctly

## How to Defend against DNS Hijacking 
The following techniques can be used to defend against DNS hijacking: 

1. **Use DNSSEC** → DNS Security Extensions add cryptographic protection to DNS, preventing tampering and spoofing of DNS records.

2. **Choose a Trusted Registrar** → Register your domain with an ICANN-accredited registrar and enable **REGISTRAR-LOCK** to avoid unauthorized changes.

3. **Strong Passwords & MFA** → Use strong, unique passwords and enable **Multi-Factor Authentication (MFA)** to secure DNS accounts and server access.

4. **Constant DNS Monitoring** → Regularly monitor your DNS servers to ensure the domain is pointing to the correct IP and detect any suspicious changes.

5. **Secure Your Router** → Change the default router username and password, and keep firmware updated to prevent router-based hijacks.

6. **Use DNS Filtering or Secure DNS Providers** → Use trusted DNS services like **Cloudflare or Google DNS** that provide built-in protection against malicious domains.

7. **Enable DNS over HTTPS (DoH) or DNS over TLS (DoT)** → These protocols encrypt your DNS queries, stopping attackers from spying or redirecting them.

8. **Restrict Access with ACLs** → Use **Access Control Lists (ACLs)** to limit who can query or manage your DNS servers.

9. **Avoid Suspicious Downloads** → Never install codecs or downloaders from shady websites, as they may modify DNS settings silently.

10. **Use a VPN (Reputed Only)** → Secure your internet traffic with a **reliable VPN** to avoid DNS leaks and protect against interception.
