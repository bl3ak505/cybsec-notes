# 💥 Web Server Attacks
An attacker can use many techniques to compromise a web server, such as DoS/DDoS, Domain Name System (DNS) server hijacking, DNS amplification, directory traversal, man in the middle (MITM)/sniffing, phishing, website defacement, web server misconfiguration, HTTP response splitting, web cache poisoning, Secure Shell (SSH) brute force, and web server password cracking. This section describes these attack techniques in detail. 

---
## DNS Server Hijacking 
The Domain Name System (DNS) resolves a domain name to its corresponding IP address. A user queries the DNS server with a domain name, and the DNS server responds with the corresponding IP address. <br>
**DNS Server Hijacking** is when an attacker takes control of a DNS server and changes its settings. So, when a user types a real website address, they're secretly redirected to a fake, attacker-controlled site. This trick is used to steal information or spread malware.

## DNS Amplification Attack 
Recursive DNS query is a method of requesting DNS mapping. The query goes through DNS servers recursively until it fails to find the specified domain name to IP address mapping. <br> 
The following are the steps involved in processing recursive DNS requests; these steps are illustrated in the below figure.
- **Step 1:** Users who desire to resolve a domain name to its corresponding IP address send a DNS query to the primary DNS server specified in its Transmission Control Protocol (TCP)/IP properties.
- **Steps 2 to 7:** If the requested DNS mapping does not exist on the user’s primary DNS server, the server forwards the request to the root server. The root server forwards the request to the .com namespace, where the user can find DNS mappings. This process repeats recursively until the DNS mapping is resolved.
- **Step 8:** Ultimately, when the system finds the primary DNS server for the requested DNS mapping, it generates a cache for the IP address in the user’s primary DNS server.

<img width="721" height="323" alt="image" src="https://github.com/user-attachments/assets/ed50f069-80a5-4464-ba8b-dffe0e78a46f" />

Attackers exploit recursive DNS queries to perform a DNS amplification attack that results in DDoS attacks on the victim’s DNS server. <br>
The following are the steps involved in a DNS amplification attack; these steps are illustrated in the below figure.
- **Step 1:** The attacker instructs compromised hosts (bots) to make DNS queries in the network.
- **Step 2:** All the compromised hosts spoof the victim’s IP address and send DNS query requests to the primary DNS server configured in the victim’s TCP/IP settings.
- **Steps 3 to 8:** If the requested DNS mapping does not exist on the victim’s primary DNS server, the server forwards the requests to the root server. The root server forwards the request to the .com or respective top-level domain (TLD) namespaces. This process repeats recursively until the victim’s primary DNS server resolves the DNS mapping request.
- **Step 9:** After the primary DNS server finds the DNS mapping for the victim’s request, it sends a DNS mapping response to the victim’s IP address. This response goes to the victim because bots use the victim’s IP address. The replies to copious DNS mapping requests from the bots result in DDoS on the victim’s DNS server.

## Directory Traversal Attacks 
**Directory Traversal Attack** is a web attack where hackers access restricted folders on a server by using tricks like `../` in the URL. This happens when a web app or server doesn’t properly validate user input. Attackers can then reach sensitive files outside the web root, potentially stealing or executing them. Poor server configuration or unpatched code often causes this vulnerability.

## Website Defacement
**Website defacement** is when a hacker breaks into a web server and changes how a website looks. They may add unwanted images, text, or popups, or even replace the whole website. This is done by injecting malicious code to alter the site’s content without permission.

<img width="519" height="392" alt="image" src="https://github.com/user-attachments/assets/85499140-9dbd-42ad-9ca0-0e58caa6c92b" />

They often use methods like SQL injection to break in. Defaced sites may show false messages or propaganda, and can also infect visitors’ computers with viruses. It damages the organization’s image and puts users at risk.

## Web Server Misconfiguration 
Web server misconfiguration refers to the configuration weaknesses in web infrastructure that can be exploited to launch various attacks on web servers, such as directory traversal, server intrusion, and data theft. The following are some web server misconfigurations:
- Verbose debug/error messages
- Anonymous or default users/passwords
- Sample configuration and script files
- Remote administration functions
- Unnecessary services enabled
- Misconfigured/default SSL certificates

**Web Server Misconfigurations** happen when admins set up a server incorrectly, leaving security gaps. ## HTTP Response-Splitting Attack 
**HTTP Response Splitting** is a web attack where the attacker injects **new line characters** into the **response headers** of a vulnerable web server, like Apache. This makes the server split one HTTP response into **two separate responses**.

The attacker crafts a request with malicious header data. If the server **doesn’t validate input properly**, it includes the attacker’s input in the response header. This tricks the browser into processing **two HTTP responses**—the first one controlled by the attacker.

This can lead to:

* **Redirecting users** to fake websites
* **Launching other attacks** like XSS, CSRF, or SQL injection

The core issue is **poor input validation**, which allows the attacker to break the response format and insert harmful content.

## Web Cache Poisoning Attack 
**Web Cache Poisoning** is an attack where the attacker tricks a web cache into storing **malicious content** instead of the correct one. When other users request the same URL, they receive the **infected (poisoned)** version from the cache.

The attacker sends a **specially crafted request** that gets saved in the cache. If the web server has vulnerabilities like **HTTP response splitting**, the malicious version gets served to everyone using that cache—until it is cleared. <br>
This attack misleads users and can be used to spread fake or harmful content.

## SSH Brute Force Attack
An **SSH Brute Force Attack** is when an attacker tries to break into a system by guessing the correct **username and password** for an SSH server (usually running on **TCP port 22**). The attacker uses bots to **scan for systems with open SSH ports** and then uses tools like **Nmap** and **Ncrack** to try many password combinations quickly. Once successful, the attacker gains **unauthorized access** and can use the **encrypted SSH tunnel** to **secretly send malware** or perform other malicious activities without being detected.

## HTTP/2 Continuation Flood Attack
**HTTP/2 Continuation Flood Attack** is a type of **DoS (Denial-of-Service)** attack that targets how Apache handles **HTTP/2 CONTINUATION frames**.

In normal cases, when headers are too big, HTTP/2 splits them into one **HEADERS frame** followed by multiple **CONTINUATION frames**. Attackers misuse this by sending **a large number of CONTINUATION frames without ending the headers**.

This causes the Apache server to keep waiting and using memory, which **drains system resources** and can crash the server or make it slow—resulting in a **DoS**.

### How HTTP/2 Continuation Flood Attack Works 
1. The attacker connects to the Apache server using TCP.

2. A normal-looking HEADERS frame is sent by the attacker to start an HTTP/2 request.

3. The server allocates memory and prepares to process the incoming headers.

4. Instead of ending the headers END_HEADERS, the attacker sends many CONTINUATION frames.

5. These frames tell the server more header data is coming, keeping it waiting.

6. Each CONTINUATION frame causes the server to use more memory and CPU.

7. The server stays in this incomplete state, waiting for the final headers.

8. With enough frames, the server's resources get overloaded.

9. This can lead to the server slowing down, freezing, or crashing.

10. The result is a **Denial of Service (DoS)**—the server becomes unresponsive.

## Frontjacking Attack 
**Frontjacking** is a web attack where an attacker changes or injects code into a website’s front-end (like scripts or HTML) to hijack how users see or interact with the site. It mainly targets **poorly configured Nginx reverse proxy servers** in shared hosting.

By abusing weak input checks—like unsafe use of `$uri` or `$document_uri`—the attacker can inject a **fake Host header**. This tricks the proxy server into connecting to a **malicious server** controlled by the attacker.

As a result, attackers can:

* Show fake content
* Redirect users to phishing sites
* Run XSS or other harmful scripts

This attack combines methods like **CRLF injection, header injection, and XSS** to take control of the web interface.

### How a Frontjacking Attack Works 
- The attacker creates an HTTP request containing CRLF characters in the URI to inject a malicious host header and sends the request to the vulnerable reverse proxy server.
- The vulnerable reverse proxy accepts a request containing the malicious host header injected via the CRLF injection.
- Once the reverse proxy processes the injected host header, it routes the request to the attacker-controlled server instead of the legitimate backend server.
- The attacker-controlled server responds with malicious content such as phishing pages, malware, or other harmful scripts.

## Other Web Server Attacks 

### Web Server Password Cracking 
**Web Server Password Cracking** is when an attacker tries to guess or steal login passwords to gain access to web servers. They often target services like **SMTP, FTP, SSH, web shares, or login forms** using weak or common passwords like *admin*, *123456*, or *qwerty*.

Attackers use tricks like **phishing**, **Trojan viruses**, **social engineering**, **spoofing**, or **keylogging** to steal passwords. Cracking passwords is usually the first step to pretending to be a valid user and breaking into the system.

#### Web Server Password Cracking Techniques 
**Web Server Password Cracking** is a method attackers use to break weak login systems and gain unauthorized access. Once the password is cracked, they can launch further attacks. Here are the common techniques:

* **Guessing**: Attackers try common or simple passwords like names, dates, or "admin" using tools or manual attempts.
* **Dictionary Attack**: Uses a list of known words to try as passwords. It's quick but fails if the password is complex.
* **Brute-Force Attack**: Tries all possible combinations of characters. It's accurate but slow, especially for long passwords.
* **Hybrid Attack**: Combines dictionary and brute-force methods, making it faster and more effective for complex passwords.

Tools like **THC Hydra** and **Ncrack** automate these methods to crack passwords on web, FTP, or SMTP servers.

### DoS/DDoS Attacks
**DoS/DDoS attacks** overload a web server with a massive number of fake requests, making it crash or slow down. This causes the server to become **unavailable to real users**. Attackers often target important websites like **banks, payment gateways, or DNS servers** to disrupt their services.

<img width="715" height="225" alt="image" src="https://github.com/user-attachments/assets/729434ef-752b-4451-8d2a-4a0e04a7a264" />

To crash a web server running an application, the attacker targets the following services to consume the web server’s resources with fake requests:
- Network bandwidth
- Server memory
- Application exception handling mechanism
- CPU usage
- Hard-disk space
- Database space

### Man-in-the-Middle Attack
A **Man-in-the-Middle (MITM) attack** happens when an attacker secretly intercepts and possibly alters the communication between a user and a web server. The attacker pretends to be a trusted proxy, so the user unknowingly sends sensitive data—like login credentials or banking info—through the attacker. This lets the attacker **eavesdrop or change the data** before passing it to the real server, effectively stealing private information.

### Phishing Attacks 
**Phishing attacks** happen when an attacker sends a fake email with a **malicious link**, tricking the user into clicking it. The link leads to a **fake website** that looks like a real one. When the victim enters sensitive details like **usernames, passwords, or bank info**, the data is sent to the attacker. The attacker can then use this stolen information to **log in to real websites** and perform **fraudulent actions**.

### Web Application Attacks
Web application attacks happen when a website’s code is not written securely. Even if the web server is protected by firewalls or other tools, weak coding in the web app can give attackers a way in. If developers don’t follow secure coding practices, attackers can exploit these weaknesses to hack the web application and even the server behind it.

1. **SSRF (Server-Side Request Forgery)** - 
   Attackers trick the server into sending requests to internal systems, exposing sensitive data.

2. **Parameter/Form Tampering** - 
   Hackers change form values (like price or permissions) to manipulate application behavior.

3. **Cookie Tampering** - 
   Cookies are modified to gain unauthorized access or change user data.

4. **Unvalidated Input/File Injection** - 
   Hackers send harmful input or upload files to exploit weak input validation.

5. **Session Hijacking** - 
   Attackers steal or guess session IDs to access a user’s active web session.

6. **SQL Injection** - 
   Malicious SQL code is injected to control or steal data from a database.

7. **Directory Traversal** - 
   Attackers access restricted files or directories by altering the URL path.

8. **Denial-of-Service (DoS)** - 
   The server is flooded with traffic to crash it and make it unavailable.

9. **Cross-Site Scripting (XSS)** - 
   Hackers inject malicious scripts into websites to target other users.

10. **Buffer Overflow** - 
    Attackers send too much data to crash the app or take control of it.

11. **Cross-Site Request Forgery (CSRF)** - 
    A logged-in user is tricked into sending unwanted requests to a web app.

12. **Command Injection** - 
    Hackers run system-level commands by exploiting unvalidated form fields.

13. **Source Code Disclosure** - 
    Due to misconfigurations, attackers view sensitive source code and secrets.
